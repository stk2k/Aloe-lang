Aloe Build Tool Specification (Draft, English Version)

This document defines the specification of the official command line tool aloe for building and running projects using the Aloe language and AloeVM.
The structures of the package formats themselves (.abc / .meta.json / package-db.json) are defined in a separate document, the “Aloe Package Specification”. This document only refers to them.

0. Overview and Positioning

0.1 Purpose

  • Define a standard CLI for building and running source code written in the Aloe language.
  • Provide stable command names and option semantics that can be used in common by multiple IDEs, editors, and CI systems.
  • Assuming .meta.json and package-db.json defined in the Package Specification, clarify the build tool’s responsibilities for generating and updating them.

0.2 Scope

This specification covers the following:

  • Basic structure and global options of the aloe command
  • Command specifications for compile / build / run / pack
  • Specifications for obtaining dependency packages (restore / install)
  • Specifications for operating on package-db.json via the package-db subcommands
  • The command framework for static checks (check / lint)

The following are out of scope for this document:

  • Syntax, type system, and execution semantics of the Aloe language
    • → See the “Aloe Language Specification”.
  • AloeVM bytecode format (AloeBC) and opcode specification
    • → See the “AloeVM Specification”.
  • JSON structures of .meta.json and package-db.json
    • → See the “Aloe Package Specification”.

0.3 Stability

This specification is a draft and may change based on implementation experience.
However, for command names and options that are already widely used, care should be taken to preserve backward compatibility as much as possible.

1. Basic Structure and Terminology

1.1 Command Line Structure

aloe [global-options] <command> [command-options] [args...]
  • aloe
    • The official build tool and launcher for Aloe.
  • <command>
    • The subcommand to execute, e.g. compile, build, run, etc.
  • Global options
    • Options that can be used with all commands.
    • They may be placed either before or after the command name (an implementation may restrict the order, but that is implementation-defined).

1.2 Typical Usage Scenarios

  • Compile a single file directly: 
    aloe compile main.aloe -o main.abc
  • Build according to a project definition: 
    aloe build
  • Build and then run: 
    aloe run src/Main.aloe --args "foo bar"
  • Restore dependency packages: 
    aloe restore

2. Global Options

Global options are options that can be used with all subcommands. Some subcommands may ignore particular options, but this should not be treated as an error.

2.1 List

  • -h, --help
    • Display help and exit.
    • If specified after a subcommand, it may display help specific to that command.
  • --version
    • Display the version of the Aloe CLI itself and exit.
    • The displayed value should preferably be consistent with ALOE_VERSION.
  • -v, --verbose
    • Output detailed logs to standard error.
    • Whether specifying it multiple times (e.g. -vv) further increases verbosity is implementation-defined.
  • -q, --quiet
    • Reduce output to the minimum.
    • If specified together with --verbose, which one takes precedence is implementation-defined, but the behavior must be consistent.
  • --no-color
    • Disable colored output.
  • --package-db <path>
    • Explicitly specify the path to the package-db.json to use.
    • If omitted, an implementation-defined search path is used (current directory, standard user paths, etc.).
  • --config <path>
    • Path to the project configuration file (for example aloe.config.json).
    • Default values and the file format when omitted will be defined in future specifications.
  • --working-dir <path>
    • Specify the working directory (workspace root).
    • If omitted, the process’s current directory is used.

3. compile Command

3.1 Overview

The compile command compiles one or more Aloe source files directly to .abc and .meta.json.
It is primarily intended for small samples and simple compilation scenarios where no project file is prepared.

3.2 Syntax

aloe compile [options] <input-files...>

3.3 Options

  • -o, --out <path>
    • Path to the output .abc file.
    • If omitted, it is automatically determined from the package name (for example myapp.core.abc).
  • --meta-out <path>
    • Path to the output .meta.json file.
    • If omitted, <package>.meta.json is output in the same directory as the .abc file.
  • --package-name <name>
    • Package name (for example myapp.core).
    • Stored in package.name inside .meta.json.
  • --package-version <version>
    • Package version (for example 0.1.0).
    • Stored in package.version inside .meta.json.
  • --configuration <config>
    • Build configuration (for example Debug, Release).
    • Stored in package.build.configuration inside .meta.json.
  • --output-dir <dir>
    • Specify the output directory for .abc and .meta.json in one go.
    • If -o / --meta-out are specified, they take precedence over --output-dir.
  • -I, --import-dir <dir>
    • Add a directory to search for dependency packages. May be specified multiple times.
    • How .abc, .meta.json, and package-db.json are handled within those directories is implementation-defined, but must comply with the Package Specification.
  • --no-package-db
    • Do not search or reference any package-db.json.
    • Dependency resolution uses only information explicitly specified by --import-dir and so on.
  • --emit-meta-only
    • Generate only .meta.json, and do not output .abc.
  • --emit-abc-only
    • Generate only .abc, and do not output .meta.json.
    • From the perspective of tool integration, outputting .meta.json is generally recommended.

4. build Command

4.1 Overview

The build command builds a set of Aloe source files according to project and workspace definitions.
It is intended to build multiple packages while resolving their dependencies.

4.2 Syntax

aloe build [options] [path/to/project]
  • If path/to/project is omitted, the project root is treated as the directory specified by --working-dir or the current directory.

4.3 Options

  • --configuration <config>
    • Build configuration (Debug / Release, etc.).
    • Overrides the configuration defined in the project configuration file.
  • --no-deps
    • Do not build dependency packages; only build the specified project.
  • --force
    • Ignore incremental build and always perform a full build.
  • --out-dir <dir>
    • Specify the output directory for build artifacts (.abc, .meta.json, etc.).
  • --update-package-db
    • After the build completes, automatically register/update information about built packages in package-db.json.
    • If --package-db is specified, the file at that path is used as the update target.

5. run Command

5.1 Overview

The run command builds the specified source file or project and then runs it on AloeVM.
Implementations that directly execute already built .abc files are also allowed.

5.2 Syntax

aloe run [options] <main-file.or.project>

5.3 Options

  • --args "<arg1 arg2 ...>"
    • Specify arguments to pass to the main function (for example function main(args: string[]): int) separated by spaces.
    • Parsing and escaping of arguments depend on the execution platform (shell) of the implementation.
  • --no-build
    • Do not rebuild; run using existing build artifacts (.abc, etc.).
    • Behavior when no build artifacts exist (e.g. treating this as an error) is defined by implementation.
  • --configuration <config>
    • If building before execution, this specifies the build configuration (Debug / Release, etc.).

6. pack Command

6.1 Overview

The pack command creates a distributable package (archive) from built .abc, .meta.json, and related files.
The package format itself (for example .zip) is outside the scope of this specification; only the minimal interface is defined here.

6.2 Syntax

aloe pack [options] [project-or-meta]

6.3 Options

  • --format <fmt>
    • Package format, for example zip, dir.
    • The default is zip.
  • --output <path>
    • Path to the generated package file.
  • --include-source
    • Include the source code used for the build in the package.
  • --no-meta
    • Do not include .meta.json in the package.
    • From the perspective of tool integration and IDE support, including .meta.json is generally recommended.

7. Dependency Packages (restore / install)

7.1 restore Command

The restore command retrieves dependency packages declared in a project configuration in bulk.
The concrete way of declaring dependencies (configuration file format, etc.) will be defined in future specifications.

7.1.1 Syntax

aloe restore [options] [project]

7.1.2 Options

  • --source <url-or-path>
    • Package repository or local path from which to retrieve packages.
    • Whether multiple sources may be specified is left to implementation.
  • --packages-dir <dir>
    • Directory in which to install packages.
    • If omitted, an environment-specific default directory is used per user.
  • --no-package-db-update
    • Do not automatically update package-db.json after installation.
    • In this case, the user or another tool must update package-db.json separately.

7.2 install Command

The install command installs a specific package individually.
It is mainly intended for scenarios where a developer explicitly adds a package.

7.2.1 Syntax

aloe install [options] <package>[@version]

7.2.2 Options

  • --source <url-or-path>
    • Package repository or local path from which to retrieve the package.
  • --packages-dir <dir>
    • Directory in which to install the package.
  • --global
    • Install into a user-wide global location.
    • The path of the global location is defined by the implementation.

8. package-db Subcommands

8.1 Overview

The package-db subcommand group provides utilities to operate directly on package-db.json, which is defined in the Package Specification.
Only the basic interface is defined here; more detailed operational rules are left to implementation.

8.2 list

aloe package-db list [options]
  • --name <pattern>
    • Filter by package name.
    • The pattern matching semantics (prefix match, wildcard support, etc.) are implementation-defined.

8.3 add

aloe package-db add [options]
  • --name <name>
    • Package name.
  • --version <version>
    • Package version.
  • --abc <path>
    • Path to the corresponding .abc file.
  • --meta <path>
    • Path to the corresponding .meta.json file.

8.4 remove

aloe package-db remove [options]
  • --name <name>
    • Package name.
  • --version <version>
    • Package version.

9. Static Checks (check / lint)

9.1 Overview

This chapter defines the command framework for static analysis and API compatibility checking.
Concrete check contents and warning kinds will be added in future specifications.

9.2 check

aloe check [options] [project]
  • Check for analysis errors based on the project definition and dependency packages.
  • The process’s return value (exit code) indicates whether errors occurred.

9.3 lint

aloe lint [options] [paths...]
  • Check specified files for syntactic and stylistic problems.
  • At present, the specification for warning levels and rule sets is left undefined and may be extended in the future.

9.4 API Difference Check (proposal)

aloe check --api-diff <old-meta> <new-meta>
  • Intended for checking whether there are breaking changes by comparing two .meta.json files.
  • Rules for judging what constitutes a breaking change will be defined and tuned together with the Package Specification.

Appendix: Editing Policy for Generative AI (Specific to This Specification)

This section is an internal guideline for editing the Aloe Build Tool Specification with a generative AI. It is not part of the normative specification itself.

  • Treat this document as the canonical “Aloe Build Tool Specification”.
  • If an English or other language version is created in the future, its chapter structure and semantics must match this version.
  • edit-version is a counter dedicated to this specification, and is managed separately from the Aloe Language Specification, AloeVM Specification, and Aloe Package Specification.
  • Only increment edit-version when making changes that alter the meaning of the specification text (including command descriptions, options, and examples).
  • Always treat the latest HTML provided by the user as the canonical version, and apply only the minimal diff necessary to that version.
  • The JSON structure of the package formats (.meta.json / package-db.json) must always be defined in the “Aloe Package Specification” and must not be redundantly defined here.
  • Do not add options related to target switching (such as --target) to this specification for now. If such options are to be added, do so only after agreeing with the user and revising the specification.
