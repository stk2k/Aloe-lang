1. Introduction

Aloe is an experimental programming language with static typing, a simple syntax, and a stack-based virtual machine (AloeVM). It is especially designed with a focus on Pipe / Filter style dataflow and interoperability with WebAssembly.

  • Statically typed (type checking at compile time)
  • C-like surface syntax, but an intentionally small feature set
  • Designed to make it pleasant to write dataflow-centric code for text processing, REST calls, logging, etc.
  • A stack-based VM (AloeVM) and a future WebAssembly backend are assumed.

This document is a draft of the language specification.
It may differ from the actual compiler/VM implementation.

2. Design Goals

2.1 Goals

  • Readability: the specification itself should be readable as documentation.
  • Ease of implementation: a small compiler and VM should suffice.
  • Consistency of Pipe / Filter: a type-safe dataflow representation.
  • Bridge to Wasm: should be easy to connect to the WebAssembly component model in the future.

2.2 Non-Goals

  • Being a “do-everything” general-purpose language.
  • Competing with highly optimized JIT compilers in performance.
  • Supporting complex metaprogramming (macros, etc.) within this spec.

3. Basic Syntax

3.1 Source Files

  • Source files are assumed to be UTF-8 encoded.
  • Single-line comment: // comment
  • Multi-line comment: /* ... */

3.2 Tokens

  • Identifier: first character is an alphabetic character or _, followed by alphanumeric characters or _.
  • Keywords are reserved and cannot be used as identifiers.

3.3 Representative Keywords

CategoryKeywords
Declarations let, var, type, struct, enum, class, interface, trait, sealed
Control if, else, while, for, break, continue, return, lock
Functions method
Auxiliary with, delete, clone, where

3.4 Statements and Expressions

  • Most statements end with a semicolon ;.
  • Blocks are surrounded by { ... }.
  • Conditions in control statements must have type Bool.
// Variable declarations
let Threshold: Int = 10;
var Count: Int = 0;

// if statement
if (Count > Threshold) {
  Print("over");
} else {
  Print("ok");
}

4. Type System

4.1 Primitive Types

TypeDescription
Booltrue / false
IntSigned integer (bit-width is implementation dependent, 32-bit assumed here)
FloatFloating-point number (32 or 64 bits, implementation dependent)
StringUTF-8 string (internal representation is implementation dependent)

4.2 Arrays and Collections

  • Fixed-length array: T[]
  • For variable-length and lazily evaluated sequences, use Pipe<T> (see section 6).

4.3 struct

struct User {
  Id: Int;
  Name: String;
  IsActive: Bool;
}
  • struct is a value type; assignment copies the value.
  • The copy is shallow; sharing of nested reference-type fields depends on the VM implementation.
  • A struct may inherit from at most one other struct (single inheritance for structs).
  • A struct is not allowed to implement an interface. (How it interacts with traits and generic constraints will be specified later.)

4.4 enum

  • enum is always represented internally as Int.
  • “Generic enums” such as enum<String> or enum<Byte> are forbidden.
  • Bitflag enums are limited to at most 32 flags.
enum LogLevel {
  Trace = 0;
  Debug = 1;
  Info  = 2;
  Warn  = 3;
  Error = 4;
}

4.5 Classes and Inheritance (class / interface / trait / sealed)

4.5.1 Basics of Classes

  • class is always a heap-allocated reference type.
  • Assignment performs a reference copy (shallow copy).
  • The clone keyword creates a shallow copy of an object and returns a new reference.
  • All classes are cloneable; the language provides no way to declare a class as “non-cloneable”. If cloning needs to be discouraged, it should be expressed via API design or coding guidelines.
class User {
  Name: String;
  Age: Int;
}

var A: User = NewUser();
var B: User = A;        // Reference copy (A and B refer to the same instance)

var C: User = clone A;  // Returns a new instance that is a shallow copy

A shallow copy means “copy the field values as-is”.
If a field itself is a reference type, its reference value is copied. Types that require deep copy or custom cloning logic can expose user-defined APIs such as a Clone method, layered on top of clone.

4.5.2 Class Inheritance (Single Inheritance)

Aloe class supports single inheritance only.
A class can specify exactly one base class. 

class Base {
  Id: Int;
}

class Derived : Base {
  Name: String;
}

// The following is a compile error (multiple inheritance of classes is forbidden)
class Bad : Base, OtherBase {
}
  • The class hierarchy forms a tree (or chain); diamond inheritance does not occur.
  • Class inheritance is a mechanism to reuse state and behavior.

4.5.3 interface and trait

Instead of allowing multiple inheritance of classes, Aloe uses interface and trait to compose behavior.

  • interface defines method signatures only and does not hold implementations (whether to allow default implementations is a future design topic).
  • trait is a mixin unit that can contain method implementations and is used to add behavior to existing classes.
  • A class has one base class, can implement multiple interfaces, and can apply multiple traits via with.
interface IDisposable {
  method Dispose();
}

trait Printable {
  method PrintSelf() {
    Print(this.ToString());
  }
}

class FileHandle : BaseFile implements IDisposable with Printable {
  method Dispose() {
    // Resource clean-up
  }

  method ToString(): String {
    return "FileHandle(...)";
  }
}

This draft does not fully define the syntax and rules for interface and trait (inheritance, conflict resolution when multiple traits define the same method, etc.).
Only the direction is shown here.

4.5.4 sealed Classes

Aloe provides the sealed modifier to forbid inheritance from a particular class. 

sealed class Token {
  Kind: TokenKind;
  Text: String;
}

// Compile error (Token is sealed)
class KeywordToken : Token {
}
  • A sealed class cannot be used as a base class.
  • Even if a class is sealed, it can still implement interfaces and apply traits via with.
  • In the future, combining sealed classes with unions/pattern matching to support exhaustiveness checks will be considered.

4.5.5 readonly and Classes

  • Class fields are mutable by default.
  • Adding the readonly modifier to a field makes it immutable:
    • Assignments to that field are only allowed within field initializers or the constructors of the same class.
    • Any assignment outside of those places is a compile error.
  • readonly class makes the entire class logically immutable:
    • All instance fields are implicitly treated as if they were readonly.
    • Assignments to fields are only allowed in field initializers and within the class’s constructors.
    • After a constructor completes, no code is allowed to modify fields of that instance.
  • readonly and clone are orthogonal: even a readonly class can be cloned using clone.
    The result of cloning is another instance with the same immutable state.

In summary, classes are heap-allocated reference types with single inheritance, behavior is composed via interface + trait, sealed expresses “cannot be inherited”, and readonly expresses immutability on field or class level.
Details such as method resolution order and trait conflict resolution will be specified separately.

4.6 Generics

4.6.1 Overview

Aloe provides a minimal generics mechanism for types, functions, methods, and pipe-related declarations. The goal is to allow type-safe reuse of collections and Pipe / Filter components, not to support advanced type-level computation. 

type Box<T> {
  Value: T;
}

struct Pair<TLeft, TRight> {
  Left: TLeft;
  Right: TRight;
}

method Map<T, U>(Source: Pipe<T>, Selector: Func<T:U>): Pipe<U> {
  // Implementation omitted
}

4.6.2 Generic Type Parameters

  • Types, structs, classes, interfaces, traits, methods, and producer/filter/consumer declarations can take type parameters with <T, U, ...>.
  • Type parameter names are recommended to start with an uppercase letter (e.g. T, TItem, TKey).
  • At use sites, type arguments are bound to concrete types.
type List<T> {
  Items: T[];
}

method FirstOrDefault<T>(Source: Pipe<T>): T {
  // Implementation omitted
}

4.6.3 where Type Constraints (Overview)

Aloe generics support arbitrary type constraints via a where clause.
By default there is no constraint, but version 0.1 supports several basic constraints. 

method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  // Sum implementation for Int
}

Here, where exists purely for compile-time type checking.
How code is represented at runtime (monomorphization vs. shared code with type tags, etc.) is left to the VM specification.

4.6.4 Supported Kinds of Constraints

In version 0.1, the following kinds of constraints are supported:

  • Value-type / Reference-type constraints
    • where T: struct – T must be a value type.
    • where T: class – T must be a reference type.
  • Primitive type constraints
    • Primitive type names such as Int, Float, Bool can be used directly in constraints: where T: Int, where T: Float, etc.
  • User-defined type constraints
    • where T: SomeTrait – types that implement trait SomeTrait.
    • where T: MyStruct – type exactly equal to struct MyStruct.
    • where T: MyClass – type exactly equal to class MyClass.
  • Any concrete type name
    • Any existing concrete type name is allowed, e.g. where T: Pipe<Int>.
// Generic restricted to value types
method Zero<T>(): T
  where T: struct
{
  // Implementation omitted
}

// Pipe for Int only
method Average<T>(Source: Pipe<T>): Float
  where T: Int
{
  // Implementation omitted
}

// Generic restricted to a specific struct type
method PrintUser<T>(User: T)
  where T: UserInfo
{
  // Implementation omitted
}

4.6.5 Behavior on Constraint Violation

  • If a type argument given to a generic function/type does not satisfy its where constraints, that is a compile-time error.
  • If multiple constraints are specified, the type must satisfy all of them at once.
  • Cyclic constraints or advanced typeclass-like structures are not supported at this time.
// Example of a compile error:

// Because of where T: Int, T cannot be String
method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  ...
}

var Lines: Pipe<String> = ReadLines("log.txt");
var Result = Sum(Lines);  // <-- compile error

The internal representation of generics (monomorphized vs. shared code with type tags, etc.) is defined by AloeVM.
This specification focuses on type-safe reuse and behavior of constraints based on where.

5. Methods and Functions

5.1 method Declarations

method Add(a: Int, b: Int): Int {
  return a + b;
}
  • method defines top-level functions or methods bound to types.
  • If the return type is omitted, the method is treated as having “no return value” (void).
  • Aloe does not have a Unit type; return-less functions are not expressed as a type.

5.2 Binding Methods with bound

In Aloe, the bound keyword is used to add methods to an existing type. 

type Pipe<T> {
  // Internal representation is VM-dependent
}

bound Pipe<T> {
  method Where(Predicate: Func<T:Bool>): Pipe<T> {
    // Implementation omitted
  }
}

For Pipe<T>, methods declared within a bound block are subject to a restriction on return types (see section 6.2).

5.3 Representation of Function Types (Func<...:...>)

In Aloe, all function types are expressed using the Func<...:...> notation. A function-type notation using a separate arrow syntax is not part of the specification.

  • Func<T:U> represents “a function taking T and returning U”.
  • Func<A, B:R> represents “a function taking A and B and returning R”.
  • Functions with no return value (void) can be written as Func<T:> or Func<T:void>.

6. Pipe / Filter Model

A distinctive feature of Aloe is its ability to express Pipe / Filter style dataflow in a type-safe and consistent way. 

ReadLines("access.log")
  |> Where (.StartsWith("[ERROR]"))
  |> Map   (ParseLogLine)
  |> Where (.Level == LogLevel.Error)
  |> ToArray();

Conceptually, the above is equivalent to the following nested calls: 

ToArray(
  Where(
    Map(
      Where(ReadLines("access.log"), ...),
      ...),
    ...));

The |> operator in Aloe is syntactic sugar for such function composition.

6.1 Pipe<T> Type

Pipe<T> represents the “pipe world” in Aloe, corresponding to a lazy sequence, stream, or iterator.

  • Pipe<T> represents “a pipe through which values of type T flow”.
  • Its internal representation (chained iterators, coroutines, etc.) is VM-dependent.
  • The standard library provides producers, transformers, and consumers based on Pipe<T>.
type Pipe<T> {
  // Implementation-dependent internal fields
}

The term pipe world refers to the area consisting of functions/objects that accept and/or return Pipe<T> (producers, transformers, consumers).
Consumers are the exit points from the pipe world.

6.2 Producer / Transformer / Consumer

In the Pipe / Filter model, functions and objects are classified into three roles:

6.2.1 Producers

  • Definition: Those whose return type is Pipe<*> and appear at the head of a pipeline.
  • Role: Read data from the outside world (files, HTTP, arrays, etc.) and feed it into the pipe world as a Pipe<T>.
  • Type: They conceptually have type Func<X:Pipe<T>>. A producer is always designed to produce a Pipe<*>.

6.2.2 Transformers (Intermediate Filters)

  • Definition: Functions/methods of type Func<Pipe<T>:Pipe<U>>.
  • Role: Transform, filter, or sort data inside the pipe world. They do not leave the pipe world.

In Aloe, methods declared inside bound Pipe<T> are treated as transformers (intermediate filters) by design.

Constraint:
Methods declared inside a bound Pipe<T> block must always return a Pipe<*>.
In other words, their type must be Func<Pipe<T>:Pipe<U>>. 
bound Pipe<T> {
  // OK: transformer
  method Where(Predicate: Func<T:Bool>): Pipe<T> { ... }

  // OK: changes element type
  method Map<U>(Selector: Func<T:U>): Pipe<U> { ... }

  // NG: return type is not Pipe, so this must be a compile error
  // method Count(): Int { ... }
}

6.2.3 Consumers (Terminal Filters)

  • Definition: Functions/objects of type Func<Pipe<T>:R>, where R is not a Pipe<*>.
  • Role: Act as exits from the pipe world, consuming the pipe and converting it into arrays, scalar values, external I/O, etc.
  • Return type: R may be any type as long as it is not Pipe<*>: T[], Int, Unit, Bool, etc.
// Consumer that materializes a pipe into an array
method ToArray<T>(Source: Pipe<T>): T[] { ... }

// Consumer that counts elements
method Count<T>(Source: Pipe<T>): Int { ... }

6.3 Pipe-Related Declarations (producer / pipe / filter / consumer)

To build Pipe / Filter components, Aloe introduces four declaration forms: producer, pipe, filter, and consumer.
All of them are treated as “objects with constructors”, and their bound block is executed when they are connected into a pipeline.

6.3.1 producer Declarations

producer ProducerName(Params...) {
  out: Pipe<T>;

  bound(output) {
    // Executed when the producer is connected into a pipeline
    // Reads data from external resources and writes into output
  }
}
  • ProducerName is the name of the producer.
  • out specifies the output type in the pipe world. It must be a Pipe<*> type.
  • bound(output) defines the behavior upon pipeline connection. The type of output is the type specified in out (e.g. Pipe<T>).

Constructor arguments (Params...) can be referenced inside the bound block. 

producer ReadLines(Path: String) {
  out: Pipe<String>;

  bound(output) {
    // Pseudo-code:
    //   let file = OpenFile(Path);
    //   for each line in file:
    //     output.write(line);
    //   file.close();
  }
}

6.3.2 pipe Declarations

pipe PipeName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Connect pipes and perform transfer/transformation
  }
}
  • PipeName is the name of the pipe declaration.
  • in is the type that flows in from upstream.
  • out is the type that flows out downstream.
  • bound(input, output) describes how data is transformed from input to output.

A pipe declaration is intended for connecting external resources (sockets, HTTP, etc.) and for bridging between different stream types (e.g. text → JSON). 

pipe JsonDecode<T>() {
  in:  String;   // JSON text
  out: T;        // Decoded object

  bound(input, output) {
    // Pseudo-code:
    //   for each line in input:
    //     let obj: T = JsonParse<T>(line);
    //     output.write(obj);
  }
}

6.3.3 filter Declarations

filter FilterName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Define how elements flow from input to output
  }
}
  • FilterName is the name of the filter.
  • in / out specify input and output types.
  • In Pipe / Filter chains, intermediate filters usually use Pipe<*> for both in and out.
  • bound(input, output) describes the conversion from input to output.
filter Where<T>(Predicate: Func<T:Bool>) {
  in:  Pipe<T>;
  out: Pipe<T>;

  bound(input, output) {
    // Pseudo-code:
    //   for each item in input:
    //     if (Predicate(item)):
    //       output.write(item);
  }
}

filter Map<T, U>(Selector: Func<T:U>) {
  in:  Pipe<T>;
  out: Pipe<U>;

  bound(input, output) {
    // Pseudo-code:
    //   for each item in input:
    //     output.write(Selector(item));
  }
}

6.3.4 consumer Declarations

consumer ConsumerName(Params...) {
  in: InputType;

  bound(input) {
    // Executed at the end of the pipeline
    // Consumes input and returns a value
  }
}
  • ConsumerName is the name of the consumer.
  • in specifies the input type. In the Pipe / Filter model this is usually Pipe<T>.
  • bound(input) is executed at the end of the pipeline, consumes input completely, and returns a value of type R.
  • R can be any type such as T[], Int, Bool, Unit, but must not be Pipe<*>.
consumer ToArray<T> {
  in: Pipe<T>;

  bound(input) {
    var buffer: List<T>;
    // Pseudo-code:
    //   for each item in input:
    //     buffer.add(item);
    //   return buffer.toArray();
    return buffer.toArray();
  }
}

consumer Count<T> {
  in: Pipe<T>;

  bound(input) {
    var c: Int = 0;
    // Pseudo-code:
    //   for each item in input:
    //     c = c + 1;
    //   return c;
    return c;
  }
}

To summarize, roles of pipe-related declarations are:

  • producer: external world → Pipe<*> (always outputs a Pipe)
  • pipe: bridge between different stream types and external I/O
  • filter: primarily Pipe<T> → Pipe<U> transformations
  • consumer: Pipe<T> → non-Pipe type (exit from the pipe world)

6.4 Pipe Operator |>

6.4.1 Language-Level Definition

In Aloe, |> is defined as syntactic sugar for simple function composition. 

Expr |> F |> G

is exactly equivalent to:

G(F(Expr))
  • If F has type Func<A:B> and G has type Func<B:C>, then Expr |> F |> G has type C.
  • This operator is applicable to any types and is not limited to Pipe<T>.
Thus, from the type system’s point of view, “changing the type of a value in the middle of a pipeline” is not a problem.
Constraints in the Pipe / Filter model are provided by Pipe<T> and library design based on pipe-related declarations, not by the operator itself.

6.4.2 Using Pipe<T> with |>

ReadLines("users.json")
  |> Where (.Trim().Length > 0)
  |> Map   (ParseUser)
  |> Where (.IsActive)
  |> ToArray();
  • ReadLines: producer (Func<String:Pipe<String>>)
  • Where, Map: transformers (Func<Pipe<T>:Pipe<U>>)
  • ToArray: consumer (Func<Pipe<User>:User[]>)
  • The type of the whole pipeline is User[], the return type of ToArray.

Note:
From the language’s perspective, it is legal to keep using |> after a consumer, but stylistically it is discouraged.
For example, patterns like ToArray() |> ToArray() should be avoided.

6.5 Standard Filters and Producers

6.5.1 Representative Transformers (Filters)

filter Where<T>(Predicate: Func<T:Bool>) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

filter Map<T, U>(Selector: Func<T:U>) {
  in:  Pipe<T>;
  out: Pipe<U>;
  bound(input, output) { ... }
}

filter Take<T>(Count: Int) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

6.5.2 Representative Producers

producer FromArray<T>(Items: T[]) {
  out: Pipe<T>;
  bound(output) { ... }
}

producer ReadLines(Path: String) {
  out: Pipe<String>;
  bound(output) { ... }
}

6.5.3 Representative Consumers

consumer ToArray<T> {
  in: Pipe<T>;
  bound(input) { ... }   // Returns T[]
}

consumer Count<T> {
  in: Pipe<T>;
  bound(input) { ... }   // Returns Int
}

6.5.4 Example Usage

let ActiveUserNames: String[] =
  ReadLines("users.json")
    |> Where (.Trim().Length > 0)
    |> Map   (ParseUser)
    |> Where (.IsActive)
    |> Map   (.Name)
    |> ToArray();

7. with Statement

7.1 Overview

The with statement is syntactic sugar for repeated property assignments and method calls on the same target object.
Additionally, Aloe gives it the role of a “lifetime lock” that keeps the target object alive for the duration of the block. 

struct HttpRequest {
  Url: String;
  Method: String;
  TimeoutSeconds: Int;
}

var Request: HttpRequest;

with (Request) {
  Url = "https://example.com/api";
  Method = "GET";
  TimeoutSeconds = 5;
}

7.2 Semantics

The above with block is semantically equivalent to: 

{
  Request.Url = "https://example.com/api";
  Request.Method = "GET";
  Request.TimeoutSeconds = 5;
}
  • with (Expr) { ... } evaluates Expr exactly once.
  • Simple identifier assignments inside the block are interpreted as member access on the target object.

7.3 Lifetime Lock (Behavior as a GC Root)

AloeVM is assumed to use a memory management model combining reference counting and sliding compaction.
In this model, a with block behaves as a “lifetime lock”:

  • The result of evaluating with (Expr) is stored in a hidden variable (a GC root).
  • This hidden variable keeps the object alive until the block ends,
    guaranteeing that the object is not freed during the block.
  • Therefore, the object is guaranteed to be alive while the block is executing.

7.4 Relationship Between with and delete

  • It is a compile-time error to apply delete to the same variable that is used as the with target inside its with block.
  • The reason is that with expresses the intent “this object must stay alive in this block”, which conflicts with delete.
  • However, it is allowed to delete another variable that refers to the same object.
    Because the with target is still a GC root, the object stays alive.
var Request: HttpRequest = GetRequest();
var Alias: HttpRequest = Request;

with (Request) {
  // Request is protected by with and guaranteed to be alive in this block

  delete Alias;   // OK: only removes the Alias reference
  Url = "https://example.com/api";  // Still usable
}

// After leaving the with block, the GC root is removed.
// If no other references remain, the object will eventually be freed.

To summarize, with is both “syntactic sugar for member access” and “a mechanism to keep the target object as a temporary GC root”.
To protect the same target for concurrency as well, the lock with construct in section 9.3 can be used.

8. delete Statement

8.1 Overview

In Aloe, delete does not directly destroy an object.
Instead, it is a statement used to remove a reference from a variable.

  • Actual object destruction (memory free, finalizers, etc.) is handled by the runtime (AloeVM) using reference counting + GC.
  • delete expresses, in a type-safe way, the intent “this variable no longer refers to this object”, and is used instead of assigning null.

8.2 Grammar

DeleteStatement ::= 'delete' Identifier ';'

8.3 Static Rules (Compile Time)

  • Identifier must be a variable declared in the current scope.
  • After a delete, reading from that variable or accessing members/methods through it is a compile-time error.
  • However, re-assigning a new value to that variable is allowed. Conceptually, delete returns the variable to an “uninitialized” state.
var Buf: Buffer = NewBuffer();

// Use Buf
DoSomething(Buf);

delete Buf;

// After this, reading Buf or Buf.Length is a compile error

Buf = NewBuffer();  // After re-assignment, it is usable again

8.4 Runtime Semantics

When delete Var; is executed, the VM behaves roughly as follows:

  • It decrements the reference count of the object currently referenced by Var (assuming reference counting).
  • Var itself is marked as “uninitialized/deleted”.
  • When the object’s reference count reaches 0, it is freed at the discretion of the GC.
    In other words, delete does not guarantee immediate destruction.

This specification leaves the exact timing of object destruction to the VM/runtime.
delete only removes one reference to that object from a single variable.

8.5 Interaction with with

  • As described in section 7.4, applying delete to the variable that is the target of a with statement inside that with block is a compile-time error.
  • It is allowed to delete another variable referencing the same object, since the with still holds a GC root and keeps the object alive.

If Aloe introduces “ownership-based resource types” and an ownership model in the future, integration of delete with that model will be considered.
This draft assumes no ownership model.

9. lock Statement and Synchronization

9.1 Overview

The lock statement acquires a lock on a specific object and guarantees exclusive access and optimized read barriers for that object while the block executes. 

var SharedCounter: Int = 0;

lock (SharedCounter) {
  SharedCounter = SharedCounter + 1;
}

9.2 Semantics

  • lock (Expr) { Body } acquires a lock on the object represented by Expr and guarantees exclusive access within the Body block.
  • Lock acquisition and release are implemented by the VM.
  • Updates to data shared across threads should generally be performed inside lock blocks.

AloeVM is assumed to use sliding compaction for memory management.
Inside a lock block, the VM may optimize read barriers and reduce per-object checks as much as possible.

9.3 lock with Statement (Combination of lock + with)

To apply both “exclusive lock” and “lifetime lock (with)” to the same target object, Aloe provides the lock with construct. 

lock with (Conn) {
  // In this block:
  //   - Conn is locked exclusively
  //   - Conn is kept as a GC root by with
  //   - Members of Conn can be used without qualification (RequestUrl, etc.)
}

9.3.1 Grammar

LockWithStatement ::= 'lock' 'with' '(' Expression ')' Block

lock with (Expr) { Body } is a combined construct where Expr is evaluated once and the result is used as the target for both lock and with.

9.3.2 Desugaring (Semantically Equivalent Code)

lock with (Expr) { Body } is semantically equivalent to: 

{
  let __tmp = Expr;
  lock (__tmp) {
    with (__tmp) {
      Body
    }
  }
}
  • Expr is evaluated exactly once and bound to a temporary variable __tmp.
  • A nested combination of lock and with is applied to __tmp.
  • Thus, during the execution of Body:
    • Exclusive access to the object is guaranteed by lock.
    • The object is kept alive as a GC root by with.
    • The object’s members can be referenced in the same shorthand as with with.

9.3.3 Style and Restrictions

  • lock with is syntactic sugar for the pattern “operate on a resource exclusively and within a safe lifetime boundary” in a single construct.
  • Alternative notations like with lock (Expr) or lock & with (Expr) are not defined in the language.
    Use lock with for this purpose.
  • Behavior inside a lock with block is interpreted as the combination of the rules of lock (section 9.2) and with (section 7.3).
var Conn: Connection = OpenConnection();

lock with (Conn) {
  Send("PING");
  var Resp: String = Receive();
  Print(Resp);
}

// When this point is reached, the lock is released and the with lifetime lock is also released.

10. Exception Handling

10.1 try / catch / finally

try {
  DoSomething();
} catch (ex: Error) {
  LogError(ex);
} finally {
  Cleanup();
}
  • Exceptions thrown in a try block can be caught by corresponding catch blocks.
  • The finally block is always executed, regardless of whether an exception is thrown.
  • The details of exception types and their hierarchy are defined in the VM specification.

11. Execution Model and Memory Management

11.1 Overview of AloeVM

  • A stack-based bytecode VM.
  • Function calls push frames onto the call stack.
  • GC is assumed to use reference counting plus sliding compaction.

11.2 Memory Management (Overview)

  • Objects are allocated on the heap.
  • When certain conditions are met, GC runs, slides live objects, and compacts memory.
  • with works as a temporary GC root that keeps objects alive, and when combined with lock, allows optimization of read barriers.
  • Normally, there is no need to call delete just for memory management. delete is used only when explicitly dropping references.

A detailed opcode list and details of the GC implementation will be provided in the AloeVM specification.

12. Limitations and Future Work

12.1 Current Limitations

  • The expressiveness of generics is minimal; typeclasses and higher-kinded generics are not defined.
  • Module system and package management are outside the scope of this document.
  • Metaprogramming (macros, reflection, etc.) is not defined.

12.2 Future Directions

  • Parallelization and fusion optimization of Pipe / Filter pipelines.
  • Integration with the Wasm component model.
  • Introduction of error handling constructs such as a Result type in addition to exceptions.
  • Standard library extensions for distributed processing and streaming.
  • Integration of ownership-based resource types with delete.
  • Integration of trait / interface / sealed / readonly with pattern matching and exhaustiveness checking.

This draft primarily aims to clarify:
The relationship between Pipe / Filter and Producer / Transformer / Consumer / Pipe / Consumer, and
the relationship between with / delete / lock / class inheritance / readonly / clone / generics and the AloeVM memory model.

13. event Syntax and Event Model

13.1 Purpose and Positioning

The event syntax in Aloe is used to express, at the language level, “this is a notification point where external code can register handlers”.

  • External code can register/unregister handlers (callbacks) to events.
  • Event firing (executing notifications) is in principle allowed only to the code that declares the event.

At the implementation level, event is treated as syntactic sugar for the standard library type Event<TArgs>.

  • event Clicked<ClickedArgs>; is equivalent to having a field of type Event<ClickedArgs>.
  • However, unlike regular fields, external code cannot reassign that field or call Raise directly (compile-time errors).

13.2 Event<TArgs> and Handler Type

13.2.1 Handler Type

Event handler function types are defined via a standard type alias: 

type EventHandler<TArgs> = Func<TArgs:Task>;
  • Argument: TArgs (event argument type)
  • Return: Task
    To naturally support asynchronous handlers, event handlers are modeled as “functions that return a Task indicating completion”.

In the future, there is room for an extension such as Event<TArgs, TResult> whose handlers have type Func<TArgs:Task<TResult>>, but version 0.1 uses Task as the basic return type for handlers.

13.2.2 Role of Event<TArgs> (Concept)

Event<TArgs> is a type that holds a set of event handlers and the logic to invoke them. 

type Event<TArgs> {
  Handlers: List<EventHandler<TArgs>>;
}

bound Event<TArgs> {
  method Add(Handler: EventHandler<TArgs>): Task {
    Handlers.Add(Handler);
  }

  method Remove(Handler: EventHandler<TArgs>): Task {
    // Details of handler identity comparison are left to VM/runtime specification
    Handlers.Remove(Handler);
  }

  method Raise(Args: TArgs): Task {
    // Await all handlers sequentially and return a Task that completes when all are done
    foreach (H in Handlers) {
      await H(Args);
    }
  }
}

In this language specification, Event<TArgs> is the “actual collection of event handlers”, and event syntax provides field declaration plus access rules as syntactic sugar.

13.3 event Declaration Grammar

13.3.1 Declaration Locations

event can be declared in the following locations:

  • Inside a struct body
  • Inside a class body
  • Inside an interface body

Declarations at the top level (file scope) or inside trait are not defined in version 0.1.

13.3.2 Grammar (Overview)

EventDeclaration ::=
    'event' Identifier '<' Type '>' ';'

Example:

struct Button {
  event Clicked<ClickedArgs>;
}

Semantically, the above is equivalent to the following field declaration: 

struct Button {
  private Clicked: Event<ClickedArgs>;
}

However, additional rules apply that distinguish it from a normal field:

  • External code cannot reassign Clicked (compile-time error).
  • External code cannot directly call Clicked.Raise(...).
  • External code is not allowed to issue delete Clicked; either.

13.4 Subscription / Unsubscription Syntax (+= / -=)

13.4.1 Grammar

EventAddStatement    ::= PrimaryExpression '.' Identifier '+=' EventHandlerExpression ';'
EventRemoveStatement ::= PrimaryExpression '.' Identifier '-=' EventHandlerExpression ';'
  • PrimaryExpression.Identifier must be an event-declared member.
  • EventHandlerExpression must be an expression implicitly convertible to Func<TArgs:Task>.

13.4.2 Type Rules

If event X<TArgs>; is declared:

  • Target.X has type Event<TArgs>.
  • Target.X += H; is allowed only if H has type Func<TArgs:Task> (or is implicitly convertible to it).
  • The same constraint applies to Target.X -= H;.

13.4.3 Desugaring (Semantically Equivalent Code)

Semantically, these are interpreted as follows:

Target.X += H;
// Semantically:
await Target.X.Add(H);
Target.X -= H;
// Semantically:
await Target.X.Remove(H);

By making Add/Remove return Task, subscription and unsubscription using +=/-= can be treated as asynchronous statements.
In the future, a fire-and-forget variant may be considered if needed.

13.5 Rules for Firing Events (Raise)

13.5.1 Firing Allowed Only on the Defining Side

Calls to Raise on an event-declared member are allowed only from:

  • Methods of the struct/class that declares the event
  • bound blocks for that same type

Calls such as Clicked.Raise(...) from other code (external modules, other classes, etc.) are compile-time errors.

This guarantees that only the side that publishes an event has the right to fire it, preserving separation of responsibilities.

13.5.2 Basic Firing Pattern

struct ClickedArgs {
  ButtonId: String;
}

struct Button {
  event Clicked<ClickedArgs>;
}

bound Button {
  method OnNativeClick(Id: String): Task {
    let Args: ClickedArgs = { ButtonId = Id };
    await Clicked.Raise(Args);
  }
}

13.6 Events in interface and Implementations

interface can also declare events.

struct ClickedArgs {
  ButtonId: String;
}

interface IButton {
  event Clicked<ClickedArgs>;
}

A struct/class that implements IButton must declare a corresponding event. 

struct Button implements IButton {
  event Clicked<ClickedArgs>;
}
  • If the event name differs from Clicked, or
  • if the type argument is different from ClickedArgs,

then the implementation is considered incomplete and results in a compile-time error.

13.7 Usage Examples

13.7.1 Button and Clicked Event

struct ClickedArgs {
  ButtonId: String;
}

struct Button {
  event Clicked<ClickedArgs>;
}

bound Button {
  method OnNativeClick(Id: String): Task {
    let Args: ClickedArgs = { ButtonId = Id };
    await Clicked.Raise(Args);
  }
}

method SetupButton(): Task {
  var Btn: Button = CreateButton("OK");

  // Register an async handler
  Btn.Clicked += async (Args: ClickedArgs): Task {
    Print("Clicked: " + Args.ButtonId);
  };

  // Register another handler
  Btn.Clicked += async (Args: ClickedArgs): Task {
    Log("button clicked");
  };
}

13.7.2 Unsubscribing a Handler

method SetupAndTearDown(Btn: Button): Task {
  let Handler: Func<ClickedArgs:Task> =
    async (Args: ClickedArgs): Task {
      Print("temp handler");
    };

  // Subscribe
  Btn.Clicked += Handler;

  // Do something...

  // Unsubscribe
  Btn.Clicked -= Handler;
}

13.8 Combining with with / lock (Reference)

Since Event<TArgs> is just a normal reference type, it can be combined with with or lock with if a thread-safe firing pattern is needed. 

struct Hub {
  event MessageReceived<String>;
}

bound Hub {
  method RaiseMessage(Msg: String): Task {
    lock with (MessageReceived) {
      await MessageReceived.Raise(Msg);
    }
  }
}

13.9 Summary

  • event provides syntactic sugar and access-control rules for Event<TArgs>.
  • External code can only subscribe/unsubscribe via +=/-=; it cannot fire events or replace references.
  • Handlers are unified as Func<TArgs:Task> types, implementing an asynchronous event model.
  • Events can be declared in interfaces, enabling event contracts at the type level.
  • The design provides a C#-like event usage experience without introducing a dedicated delegate type, simply using Aloe function types and Task.
