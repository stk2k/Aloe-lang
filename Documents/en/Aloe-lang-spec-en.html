1. Introduction

Aloe is an experimental programming language with static typing, a simple syntax, and a stack-based virtual machine (AloeVM). It is designed with a particular emphasis on pipe/filter style data flow and interoperability with WebAssembly.

  • Statically typed (type checking is performed at compile time).
  • C-like syntactic appearance, but the feature set is intentionally kept small.
  • Focus on writing “data flow and transformation” pleasantly, such as for text processing, REST calls, and log processing.
  • Designed with a stack-based VM (AloeVM) and a future WebAssembly backend in mind.

This document is a draft language specification.
It may differ from the actual compiler/VM implementation.

2. Design Goals

2.1 Goals

  • Readability: The specification itself should be readable as documentation.
  • Ease of implementation: A small compiler and VM should be able to implement it.
  • Consistency of pipe/filter: Type-safe representation of data flow.
  • Bridge to Wasm: It should be easy to connect to the future Wasm component model.
  • Safety in concurrent processing: Code written using pipes is designed such that, via the type system, there is no shared mutable state that races across threads.

2.2 Non-goals

  • To become a “do-everything” general-purpose language.
  • To compete with highly optimized JIT compilers purely on performance.
  • Complex metaprogramming (macros, etc.) is out of scope for now.

One particularly important point is that you can write Aloe programs without ever using delete/with/lock.
When you want to explicitly control memory management and lifetimes, you can opt in to those constructs. If you stick to the pipe model, the type system ensures that multi-threaded code does not have contended shared mutable state.

3. Basic Syntax

3.1 Source Files

  • Source files are expected to be UTF-8 encoded.
  • Single-line comments: // comment
  • Multi-line comments: /* ... */

3.2 Tokens

  • Identifiers: first character is a letter or _; subsequent characters can be letters, digits, or _.
  • Keywords are reserved and cannot be used as identifiers.

3.3 Representative Keywords

CategoryKeywords
Declarations let, var, type, struct, enum, class, interface, trait, sealed
Control if, else, while, for, break, continue, return, lock
Functions method, function
Auxiliary with, delete, clone, where

3.4 Statements and Expressions

  • Most statements end with a semicolon ;.
  • Blocks are enclosed in { ... }.
  • Conditions in control structures must have type Bool.
// Variable declarations
let Threshold: Int = 10;
var Count: Int = 0;

// if statement
if (Count > Threshold) {
  Print("over");
} else {
  Print("ok");
}

4. Type System

4.1 Primitive Types

TypeDescription
Booltrue / false
IntSigned integer (bit width implementation-defined; assume 32-bit here)
FloatFloating-point number (32 or 64-bit, implementation-defined)
StringUTF-8 string (internal representation is implementation-defined)

4.2 Arrays and Collections

  • Fixed-length array: T[]
  • Variable-length and lazily evaluated sequences use Pipe<T> (see Section 6).

4.3 struct

struct User {
  Id: Int;
  Name: String;
  IsActive: Bool;
}
  • struct is a value type; assignment copies the value.
  • The copy is shallow; sharing of nested reference-type fields depends on the VM implementation.
  • A struct may inherit from at most one other struct (single inheritance for structs).
  • It is forbidden for a struct to implement an interface. (How struct participates in traits and generic constraints will be specified in future revisions.)

4.4 enum

  • enum is always represented internally as Int.
  • “Generic enums” such as enum<String> or enum<Byte> are forbidden.
  • Enum used as bit flags is limited to at most 32 flags.
enum LogLevel {
  Trace = 0;
  Debug = 1;
  Info  = 2;
  Warn  = 3;
  Error = 4;
}

4.5 First-class Objects

Aloe treats the following as “first-class objects.” They can be assigned to variables, stored in arrays or Pipes, and used as function parameters and return values.

  • Primitive values (Bool, Int, Float, String)
  • struct values
  • class instances
  • enum values
  • Pipe<T> (values in the “pipe world”)
  • Functions (values of type Func<...:...>)

4.5 Classes and Inheritance (class / interface / trait / sealed)

4.5.1 Class Basics

  • class is always a heap-allocated reference type.
  • Assignment performs a copy of the reference (shallow copy).
  • clone creates a shallow copy of the object and returns a new reference.
  • All classes are cloneable. There is no type-level way to declare a “non-cloneable” class.
    If you need to express “do not clone” or similar constraints, use API design or coding conventions on top of clone.
class User {
  Name: String;
  Age: Int;
}

var A: User = NewUser();
var B: User = A;        // reference copy (A and B refer to the same instance)

var C: User = clone A;  // new instance created by shallow copy

A shallow copy means “copy the field values as-is.”
If a field is itself a reference type, the reference is copied. If you need deep copy or custom cloning logic, you can implement user-defined APIs such as a Clone method on top of clone.

4.5.2 Root Class Object and ToString / Dump

All reference-type classes implicitly have a root class Object as their base class. Object defines common methods for debugging and logging. 

class Object {
  method ToString(): String {
    // Concise string representation (type name and some fields, etc.)
  }

  method Dump(): String {
    // Detailed string representation for debugging
  }
}
  • All class instances have ToString and Dump.
  • ToString is intended to return a short human-readable description (e.g., the type name).
  • Dump returns detailed information for debugging. The standard implementation traverses the field structure and produces a tree-shaped string representation.
  • The standard implementation of Dump detects cyclic references. When it is about to revisit the same object, it skips dumping that sub-tree and outputs a marker that indicates “cycle detected.”
  • User-defined classes can override ToString and Dump as needed.

The built-in consumer dump in the pipe pipeline calls Dump for each element and outputs it for debugging. See Section 6.5 for details.

4.5.3 Class Inheritance (Single Inheritance)

Aloe class supports single inheritance only.
In other words, a class can specify exactly one base class (excluding the implicit Object). 

class Base {
  Id: Int;
}

class Derived : Base {
  Name: String;
}

// The following is a compile error (multiple inheritance of classes is forbidden)
class Bad : Base, OtherBase {
}
  • Class hierarchies form a tree (or chain); diamond inheritance does not occur.
  • Class inheritance is a mechanism for reusing state and behavior.

4.5.4 interface and trait

Instead of allowing multiple inheritance of classes, Aloe uses interface and trait to compose behavior.

  • interface defines method signatures but no implementations (whether to allow default implementations is a future design point).
  • trait is a mixin unit that can contain method implementations and can be used to add behavior to existing classes.
  • A class may have one base class, implement multiple interfaces, and apply multiple traits using with.
interface IDisposable {
  method Dispose();
}

trait Printable {
  method PrintSelf() {
    Print(this.ToString());
  }
}

class FileHandle : BaseFile implements IDisposable with Printable {
  method Dispose() {
    // Resource cleanup
  }

  method ToString(): String {
    return "FileHandle(...)";
  }
}

The detailed syntax and rules for interface and trait (inheritance, conflict resolution when multiple traits define the same method, etc.) are not fully specified in this draft. Only the general direction is provided.

4.5.5 sealed Classes

To prohibit certain classes from being inherited, Aloe provides the sealed modifier. 

sealed class Token {
  Kind: TokenKind;
  Text: String;
}

// This is a compile error (Token is sealed)
class KeywordToken : Token {
}
  • A sealed class cannot be used as a base class.
  • Even if it is sealed, the class may still implement interfaces and have traits applied via with.
  • In the future, we plan to combine sealed classes with unions / pattern matching to support exhaustiveness checking.

4.5.6 readonly and Classes

  • Class fields are mutable by default.
  • Adding the readonly modifier to a field makes it immutable:
    • Assignments to that field are allowed only in the field initializer or within the constructor of the same class.
    • Assignments from anywhere else are compile errors.
  • readonly class makes the entire class logically immutable:
    • All instance fields are implicitly treated as readonly.
    • Assignments to fields are allowed only in field initializers and in the class’s constructor.
    • After the constructor finishes, no code is allowed to modify the fields of that instance.
  • readonly and clone are orthogonal: even a readonly class can be cloned via clone. The clone result is another instance with the same immutable state.

This section defines the high-level policy that classes are heap reference types with single inheritance, behavior is composed via interfaces + traits, sealed expresses “non-inheritable” classes, and readonly expresses immutability at field or class level. Details such as method resolution order and conflict resolution between traits will be specified separately.

4.6 Generics

4.6.1 Overview

Aloe provides a minimal generics mechanism for types, functions, methods, and pipe-related declarations.
The goal is to make collections and Pipe/Filter reusable in a type-safe way, not to support sophisticated type-level computation. 

type Box<T> {
  Value: T;
}

struct Pair<TLeft, TRight> {
  Left: TLeft;
  Right: TRight;
}

method Map<T, U>(Source: Pipe<T>, Selector: Func<T:U>): Pipe<U> {
  // Implementation omitted
}

4.6.2 Generic Type Parameters

  • Types, structs, classes, interfaces, traits, methods, and producer/filter/consumer declarations can take type parameters via <T, U, ...>.
  • Type parameter names are recommended to start with uppercase letters (e.g., T, TItem, TKey).
  • Type arguments are bound to concrete types at use sites.
type List<T> {
  Items: T[];
}

method FirstOrDefault<T>(Source: Pipe<T>): T {
  // Implementation omitted
}

4.6.3 where Type Constraints (Overview)

Generics in Aloe support arbitrary type constraints using where clauses.
By default, there are no constraints, but in version 0.1 a few basic constraints are supported. 

method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  // Int-only sum
}

Here, where exists purely for compile-time type checking.
The runtime representation (monomorphization vs shared code with type tags, etc.) is left to the VM specification.

4.6.4 Supported Kinds of Constraints

In version 0.1, the following kinds of constraints are supported:

  • Value type / reference type constraints
    • where T: struct – T must be a value type.
    • where T: class – T must be a reference type.
  • Primitive-type constraints
    • where T: Int, where T: Float, where T: Bool, etc. Primitive type names can be specified directly.
  • User-defined type constraints
    • where T: SomeTrait – types implementing trait SomeTrait.
    • where T: MyStruct – exactly the struct type MyStruct.
    • where T: MyClass – exactly the class type MyClass.
  • Any concrete type names
    • Any existing concrete type name can be specified (e.g., where T: Pipe<Int>).
// Generic function for value types only
method Zero<T>(): T
  where T: struct
{
  // Implementation omitted
}

// Int-only pipe average
method Average<T>(Source: Pipe<T>): Float
  where T: Int
{
  // Implementation omitted
}

// Generic function restricted to a particular struct
method PrintUser<T>(User: T)
  where T: UserInfo
{
  // Implementation omitted
}

4.6.5 Behavior on Constraint Violation

  • If a type argument does not satisfy the where constraint of a generic function or type, it is a compile-time error.
  • If there are multiple constraints, they must all be satisfied simultaneously.
  • Cyclic constraints and advanced typeclass-like structures are not supported at this stage.
// Example of a compile-time error:

// where T: Int, so T cannot be String
method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  ...
}

var Lines: Pipe<String> = ReadLines("log.txt");
var Result = Sum(Lines);  // <-- Compile error

The internal representation of generics (monomorphization vs shared code with type tags, etc.) is defined by AloeVM.
The language specification focuses on type-safe reuse and constraint checking via where.

5. Functions and Methods

5.1 function Declarations (Top-level Functions)

Aloe has first-class functions. Functions are values of type Func<...:...> and can be stored in variables, passed as arguments, and returned from functions. 

function Add(a: Int, b: Int): Int {
  let Sum: Int = a + b;
  return Sum;
}
  • function defines a top-level function that is not bound to any particular type.
  • The signature (parameter list and return type) follows the same rules as method.
  • The function body consists of multiple statements and may contain at most one return statement.
  • If a return statement is present, the type of its expression must match the declared return type.
  • If the return type is omitted, the function is treated as having no return value (void-like).
    • In this case, a value-less return; may appear at most once.
    • It is also allowed to reach the end of the body without a return statement.
  • At the VM level, function and method share the same calling convention and representation; they only differ in whether they receive an implicit this.

5.2 method Declarations (Methods Bound to Types)

method Add(a: Int, b: Int): Int {
  return a + b;
}
  • method defines either a top-level function or a method bound to a type.
  • If no return type is specified, the method is treated as having no return value (void).
  • The rules for function bodies, the number of return statements, and return type matching are the same as for function.
  • When bound to a type, this is implicitly available if needed.

5.3 bound for Method Binding

Aloe uses bound to add methods to an existing type. 

type Pipe<T> {
  // Internal representation is VM-dependent
}

bound Pipe<T> {
  method Where(Predicate: Func<T:Bool>): Pipe<T> {
    // Implementation omitted
  }
}

For Pipe<T>, we introduce constraints on the return type of methods declared in a bound block (see Section 6.2 for details).

5.4 Representation of Function Types (Func<...:...>)

Aloe represents all function types with the notation Func<...:...>. Things appearing on the right-hand side of | are also essentially of this function type; the pipe operator is merely a syntactic sugar to connect them in a readable way.

  • Func<T:U> represents “a function that takes an argument of type T and returns U.”
  • Func<A, B:R> represents “a function that takes A and B and returns R.”
  • A function with no return value (void) can be written as Func<T:> or Func<T:void>.

5.5 main Function (Entry Point)

The entry point of an Aloe program is declared as a top-level function named main. 

function main(): Int {
  // Program entry point
  return 0;
}
  • main follows the same rules as a normal function.
  • Its body can consist of multiple statements and at most a single return statement.
  • The type of the value returned from return must match the declared return type.
  • From the language’s perspective, main is just “one of the top-level functions,” with the special role of being the program’s entry point.
  • Whether the return type must be Int or may be omitted (void), and the details of the ABI, are specified by the VM / platform.

6. Pipe / Filter Model

The distinctive feature of Aloe is that it can express pipe/filter-style data flow in a type-safe and consistent style. 

ReadLines("access.log")
  | Where (.StartsWith("[ERROR]"))
  | Map   (ParseLogLine)
  | Where (.Level == LogLevel.Error)
  | ToArray();

Conceptually, the above is equivalent to the following nested function calls: 

ToArray(
  Where(
    Map(
      Where(ReadLines("access.log"), ...),
      ...),
    ...));

The | operator in Aloe is syntactic sugar for such function composition.

6.1 Pipe<T> Type

Pipe<T> represents the “pipe world” in Aloe, analogous to lazy sequences, streams, and iterators.

  • Pipe<T> represents a pipe through which elements of type T flow.
  • Its internal representation (chained iterators, coroutines, etc.) is VM-dependent.
  • The standard library provides producer/transformer/consumer functions using Pipe<T>.
type Pipe<T> {
  // VM-dependent internal fields
}

The pipe world refers to the space of functions (producers/transformers/consumers) that accept and/or return Pipe<T>. Consumers are the exit points from the pipe world.

6.2 Producer / Transformer / Consumer

In the pipe/filter model, we classify functions and objects into three roles:

6.2.1 Producer

  • Definition: a function/object whose return type is Pipe<*> and which appears at the head of a pipeline.
  • Role: read data from the external world (files, HTTP, arrays, etc.) and “pour” it into the pipe world as a Pipe<T>.
  • Type: typically of the form Func<X:Pipe<T>>. Producers are always designed to output Pipe<*>.

6.2.2 Transformer (Intermediate Filter)

  • Definition: functions/methods of type Func<Pipe<T>:Pipe<U>>.
  • Role: transform/filter/sort data inside the pipe world; they do not leave the pipe world.

In Aloe, methods declared in a bound Pipe<T> block are intended to be transformers (intermediate filters) only.

Constraint:
Methods declared inside a bound Pipe<T> block must always return Pipe<*>.
In other words, their types must be Func<Pipe<T>:Pipe<U>>. 
bound Pipe<T> {
  // OK: transformer
  method Where(Predicate: Func<T:Bool>): Pipe<T> { ... }

  // OK: change element type
  method Map<U>(Selector: Func<T:U>): Pipe<U> { ... }

  // NG: not returning a Pipe, should be a compile error
  // method Count(): Int { ... }
}

6.2.3 Consumer (Terminal Filter)

  • Definition: functions/objects of type Func<Pipe<T>:R> where R is not Pipe<*>.
  • Role: act as exit points from the pipe world by fully consuming the pipe and converting it into arrays, scalar values, external I/O, etc.
  • Return type: R can be any type as long as it is not Pipe<*>: T[], Int, Unit, Bool, etc.
// Consumer that materializes a pipe into an array
method ToArray<T>(Source: Pipe<T>): T[] { ... }

// Consumer that counts elements
method Count<T>(Source: Pipe<T>): Int { ... }

6.3 Pipe-related Declarations (producer / pipe / filter / consumer)

Aloe introduces four declaration forms for building pipe/filter components: producer, pipe, filter, and consumer.
All of these are treated as “objects with constructors.” When they are connected into a pipeline, their bound blocks are executed.

6.3.1 producer Declarations

producer ProducerName(Params...) {
  out: Pipe<T>;

  bound(output) {
    // Executed when the producer is connected to a pipeline
    // Reads data from external resources and writes to output
  }
}
  • ProducerName is the producer’s name.
  • out specifies the output type in the pipe world. It must be a Pipe<*> type.
  • bound(output) defines the behavior when the producer is connected into a pipeline. output has the same type as specified in out (e.g., Pipe<T>).

Constructor arguments (Params...) are accessible from inside the bound block. 

producer ReadLines(Path: String) {
  out: Pipe<String>;

  bound(output) {
    // Pseudocode:
    //   let file = OpenFile(Path);
    //   for each line in file:
    //     output.write(line);
    //   file.close();
  }
}

6.3.2 pipe Declarations

pipe PipeName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Connects pipes and performs transfer/transformation
  }
}
  • PipeName is the pipe’s name.
  • in is the type coming from upstream.
  • out is the type flowing to downstream.
  • bound(input, output) describes the transformation from input to output.

A pipe declaration is intended for connecting to external resources (sockets, HTTP, etc.) and bridging different stream types (e.g., text → JSON). 

pipe JsonDecode<T>() {
  in:  String;   // JSON text
  out: T;        // decoded object

  bound(input, output) {
    // Pseudocode:
    //   for each line in input:
    //     let obj: T = JsonParse<T>(line);
    //     output.write(obj);
  }
}

6.3.3 filter Declarations

filter FilterName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Defines how elements flow from input to output
  }
}
  • FilterName is the filter’s name.
  • in / out specify input and output types.
  • In pipe/filter chains, in / out are typically both Pipe<*>.
  • bound(input, output) describes the transformation between them.
filter Where<T>(Predicate: Func<T:Bool>) {
  in:  Pipe<T>;
  out: Pipe<T>;

  bound(input, output) {
    // Pseudocode:
    //   for each item in input:
    //     if (Predicate(item)):
    //       output.write(item);
  }
}

filter Map<T, U>(Selector: Func<T:U>) {
  in:  Pipe<T>;
  out: Pipe<U>;

  bound(input, output) {
    // Pseudocode:
    //   for each item in input:
    //     output.write(Selector(item));
  }
}

6.3.4 consumer Declarations

consumer ConsumerName(Params...) {
  in: InputType;

  bound(input) {
    // Executed at the end of the pipeline
    // Consumes input and returns some value
  }
}
  • ConsumerName is the consumer’s name.
  • in specifies the input type. In the pipe/filter model this is usually Pipe<T>.
  • bound(input) is executed at the pipeline tail; it fully consumes input and returns a value of some type R.
  • R can be T[], Int, Unit, Bool, etc., but must not be Pipe<*>.
consumer ToArray<T> {
  in: Pipe<T>;

  bound(input) {
    var Buffer: List<T>;
    // Pseudocode:
    //   for each item in input:
    //     Buffer.Add(item);
    //   return Buffer.ToArray();
    return Buffer.ToArray();
  }
}

consumer Count<T> {
  in: Pipe<T>;

  bound(input) {
    var C: Int = 0;
    // Pseudocode:
    //   for each item in input:
    //     C = C + 1;
    //   return C;
    return C;
  }
}

In summary, the roles of pipe-related declarations are:

  • producer: external world → Pipe<*> (always outputs a Pipe)
  • pipe: bridge between heterogeneous streams and external I/O
  • filter: mainly Pipe<T> → Pipe<U> transformations
  • consumer: Pipe<T> → non-Pipe type (exit from the pipe world)

6.4 Pipe Operators | and |>

6.4.1 Language-level Definition (Pipe Operator |)

The fundamental pipe operator in Aloe is |, defined as syntactic sugar for simple function composition. 

Expr | F | G

is exactly equivalent in meaning to:

G(F(Expr))
  • If F has type Func<A:B> and G has type Func<B:C>, then Expr | F | G has type C.
  • This operator can be applied to any types; it is not limited to Pipe<T>.
Thus, from the type system’s perspective, “the return type changes in the middle of the pipeline” is not a problem in itself.
Constraints specific to the pipe/filter model are provided by Pipe<T> and the design of the pipe-related declarations in the library, not by the operator itself.

6.4.2 Relationship Between |> and | with

To make frequent use of with in pipelines easier to write, Aloe defines |> as syntactic sugar for | with.

Intuitively, the following transformation occurs: 

Source
  |> FilterX(Args)
  |> FilterY()

is conceptually equivalent to:

Source
  | with (FilterX(Args)) {
      // Result of applying FilterX to the pipe flows to the next step
    }
  | with (FilterY()) {
      // FilterY is applied similarly
    }
  • |> is purely syntactic sugar and does not introduce new type rules.
  • The evaluation order and type inference for pipelines are intended to match those of the equivalent code with explicit | with.
  • Pipe expressions themselves are always interpreted in terms of the fundamental operator |. |> is just a shorthand for readability.

6.4.3 Using Pipe<T> with |

ReadLines("users.json")
  | Where (.Trim().Length > 0)
  | Map   (ParseUser)
  | Where (.IsActive)
  | ToArray();
  • ReadLines: producer (Func<String:Pipe<String>>)
  • Where, Map: transformers (Func<Pipe<T>:Pipe<U>>)
  • ToArray: consumer (Func<Pipe<User>:User[]>)
  • The entire pipe expression has type User[], the return type of ToArray.

Note:
At the language level, you may syntactically chain | or |> after a consumer, but stylistically this is not recommended.
For example, patterns like ToArray() | ToArray() should be avoided.

6.5 Standard Filters / Producers / Consumers

6.5.1 Representative Transformers (filters)

filter Where<T>(Predicate: Func<T:Bool>) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

filter Map<T, U>(Selector: Func<T:U>) {
  in:  Pipe<T>;
  out: Pipe<U>;
  bound(input, output) { ... }
}

filter Take<T>(Count: Int) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

6.5.2 Representative Producers

producer FromArray<T>(Items: T[]) {
  out: Pipe<T>;
  bound(output) { ... }
}

producer ReadLines(Path: String) {
  out: Pipe<String>;
  bound(output) { ... }
}

6.5.3 Representative Consumers

consumer ToArray<T> {
  in: Pipe<T>;
  bound(input) { ... }   // Returns T[]
}

consumer Count<T> {
  in: Pipe<T>;
  bound(input) { ... }   // Returns Int
}

6.5.4 Built-in Consumer: dump

Aloe provides dump as a built-in consumer for debugging. dump calls Dump on each element and outputs the result to logs or standard output. 

consumer dump<T> {
  in: Pipe<T>;

  bound(input) {
    // Pseudocode:
    //   for each item in input:
    //     DumpOne(item);
    //   return;  // No return value
  }
}

The standard implementation of DumpOne is expected to have the following behavior:

  • For ordinary objects:
    • Calls Dump() on the object and outputs its string representation.
  • For collections (arrays, Pipe<T>, etc.):
    • Iterates over each element and calls DumpOne on them.
  • For recursive structures such as trees:
    • Recursively calls DumpOne on child elements.
    • When a cyclic reference is detected, it stops dumping that sub-tree, and outputs only a marker indicating that a cycle exists.
    • However, the overall processing of the collection continues, and dumping proceeds for subsequent elements.
ReadLines("users.json")
  | Map(ParseUser)
  | dump;  // Dump is called for each User object

6.5.5 Example Usage

let ActiveUserNames: String[] =
  ReadLines("users.json")
    | Where (.Trim().Length > 0)
    | Map   (ParseUser)
    | Where (.IsActive)
    | Map   (.Name)
    | ToArray();

7. with Statement

7.1 Overview

The with statement is syntactic sugar to repeatedly write property assignments or method calls on the same target object.
In Aloe, it also acts as a “lifetime lock” that keeps the target object alive during the block. 

struct HttpRequest {
  Url: String;
  Method: String;
  TimeoutSeconds: Int;
}

var Request: HttpRequest;

with (Request) {
  Url = "https://example.com/api";
  Method = "GET";
  TimeoutSeconds = 5;
}

7.2 Semantics

The above with block is semantically equivalent to: 

{
  Request.Url = "https://example.com/api";
  Request.Method = "GET";
  Request.TimeoutSeconds = 5;
}
  • with (Expr) { ... } evaluates Expr exactly once.
  • Simple identifier assignments in the block are interpreted as member accesses on the target object.

7.3 Lifetime Lock (Behavior as a GC Root)

AloeVM is assumed to use a combination of reference counting and sliding compaction for memory management.
In this model, a with block acts as a “lifetime lock”:

  • The result of evaluating with (Expr) is stored in a hidden variable (a GC root).
  • This hidden variable holds the object until the block is exited, guaranteeing that the object is not collected during that time.
  • Therefore, the object is guaranteed to be alive while the block is executing.

7.4 Relationship Between with and delete

  • Within a with block targeting a particular variable, using delete on that same variable is a compile-time error.
  • The reason is that with expresses the intention “this object should stay alive here,” which conflicts with delete.
  • However, performing delete on a different variable that refers to the same object is allowed. Because the with block still holds a GC root, the object itself remains alive.
var Request: HttpRequest = GetRequest();
var Alias: HttpRequest = Request;

with (Request) {
  // Request is protected by with and is guaranteed to be alive in this block

  delete Alias;   // OK: only removes the reference from Alias
  Url = "https://example.com/api";  // Still usable
}

// After leaving the with block, the GC root is removed.
// If no other references remain, the object is eventually collected.

In summary, with is both a “syntactic sugar for member access” and a “mechanism to keep the target object as a temporary GC root.”
To protect the same target for concurrent access, with can be combined with lock via the lock with syntax (Section 9.3).

8. delete Statement

8.1 Overview

In Aloe, delete does not directly destroy objects.
Instead, it is a statement for removing a reference from a variable.

  • Actual destruction of objects (memory deallocation, finalizers, etc.) is carried out by the runtime (AloeVM) using reference counting + GC.
  • delete expresses in a type-safe way that “this variable no longer refers to this object” and is used instead of assigning null.

8.2 Syntax

DeleteStatement ::= 'delete' Identifier ';'

8.3 Static Rules (Compile Time)

  • Identifier must be a variable declared in the current scope.
  • After delete, any reads from that variable or member access / method calls on it are compile errors.
  • However, assigning a new value to that variable is allowed. Conceptually, delete makes the variable “uninitialized” again.
var Buf: Buffer = NewBuffer();

// Use Buf
DoSomething(Buf);

delete Buf;

// From here on, reading Buf or Buf.Length is a compile error

Buf = NewBuffer();  // After re-assignment, Buf can be used again

8.4 Runtime Semantics

When delete Var; executes, the VM behaves roughly as follows:

  • It decrements the reference count of the object currently referenced by Var (assuming reference counting is used).
  • Var itself is marked as “uninitialized / deleted.”
  • When the object’s reference count reaches 0, the object is freed at the GC’s discretion.
    In other words, the object is not guaranteed to be destroyed immediately when delete is executed.

In this language specification, the timing of object destruction is a responsibility of the VM/runtime.
delete merely removes one variable’s reference to the object.

8.5 Interaction with with

  • As described in Section 7.4, it is a compile-time error to use delete on the variable that is currently the target of a with block.
  • Using delete on another variable that refers to the same object is allowed, because the with block still holds a GC root, so the object remains alive.

In the future, if Aloe introduces “owned resource types” and an ownership model, delete may be integrated into that model.
In this draft, we assume no ownership model exists.

9. lock Statement and Synchronization

9.1 Overview

The lock statement acquires a lock on a specific object and guarantees exclusive access and optimized read barriers for that object while the block is executing. 

var SharedCounter: Int = 0;

lock (SharedCounter) {
  SharedCounter = SharedCounter + 1;
}
  • You do not need to use lock to write typical Aloe programs.
  • By designing with pipe-based data flow and minimizing shared mutable state, you can write concurrent code that rarely needs lock.

9.2 Semantics

  • lock (Expr) { Body } acquires a lock on the object denoted by Expr and guarantees exclusive access within Body.
  • Lock acquisition and release are implemented by the VM.
  • Updates to data shared across threads should essentially be confined within lock blocks.

AloeVM is assumed to use sliding compaction for memory management.
Within a lock block, the VM may optimize read barriers, minimizing per-object checks as much as possible.

9.3 lock with Syntax (lock + with Composition)

For cases where you want to apply both “exclusive lock” and “lifetime lock (with)” to the same target object, Aloe provides the lock with syntax. 

lock with (Conn) {
  // Here:
  //   - Conn is exclusively locked
  //   - Conn is kept as a GC root via with
  //   - Conn's members can be accessed in a shortened form (e.g., RequestUrl)
}

9.3.1 Syntax

LockWithStatement ::= 'lock' 'with' '(' Expression ')' Block

lock with (Expr) { Body } is a composite syntax that evaluates Expr once and treats the result as the target of both lock and with.

9.3.2 Desugaring (Semantically Equivalent Code)

Semantically, lock with (Expr) { Body } is equivalent to: 

{
  let __tmp = Expr;
  lock (__tmp) {
    with (__tmp) {
      Body
    }
  }
}
  • Expr is evaluated exactly once and bound to a temporary variable __tmp.
  • lock and with are then applied in nested form to __tmp.
  • Thus, while Body executes:
    • Exclusive access is guaranteed by lock.
    • Lifetime is guaranteed by treating __tmp as a GC root via with.
    • Members of __tmp can be referenced in the short form provided by with.

9.3.3 Style and Constraints

  • lock with is syntactic sugar to concisely express “operating on a particular resource exclusively and safely within a lifetime scope.”
  • Alternative spellings like with lock (Expr) or lock & with (Expr) are not defined by the language.
    For this purpose, always use lock with.
  • The behavior inside a lock with block is interpreted as the combination of lock (Section 9.2) and with (Section 7.3).
var Conn: Connection = OpenConnection();

lock with (Conn) {
  Send("PING");
  var Resp: String = Receive();
  Print(Resp);
}

// When we reach here, the lock is released and the with-based lifetime lock is also cleared.

10. Exception Handling

10.1 try / catch / finally

try {
  DoSomething();
} catch (Ex: Error) {
  LogError(Ex);
} finally {
  Cleanup();
}
  • Exceptions thrown inside a try block can be caught by the corresponding catch block.
  • The finally block is always executed regardless of whether an exception was thrown.
  • The details of exception types and hierarchy are defined in the VM specification.

11. Runtime Model and Memory Management

11.1 Overview of AloeVM

  • Stack-based bytecode VM.
  • Function calls push frames onto the call stack.
  • GC is assumed to be a combination of reference counting and sliding compaction.

11.2 Memory Management (Overview)

  • Objects are allocated on the heap.
  • GC runs under certain conditions and compacts memory by sliding live objects.
  • with acts as a temporary GC root that keeps objects alive, and when combined with lock, allows optimization of read barriers.
  • Normally, you do not need to call delete for memory management. Use delete only when you explicitly want to drop a reference.

A detailed opcode list and GC internals will be provided in the AloeVM specification.

12. Limitations and Future Work

12.1 Current Limitations

  • The expressiveness of generics is minimal; typeclasses and higher-kinded generics are not defined.
  • Module system and package management are out of scope for this document.
  • Metaprogramming (macros, reflection, etc.) is not defined.

12.2 Future Work

  • Parallelization and fusion optimizations of pipe/filter pipelines.
  • Integration with the Wasm component model.
  • Introducing Result-like types for error handling in addition to exceptions.
  • Extending the standard library for distributed processing and streaming.
  • Integrating ownership-based resource types with delete where appropriate.
  • Integrating traits / interfaces / sealed / readonly with pattern matching and exhaustiveness checking.

In this draft, we focus especially on:
the relationships between the Pipe / Filter model and Producer / Transformer / Consumer / Pipe / Consumer declarations,
and the relationships between with / delete / lock / class inheritance / readonly / clone / generics and the AloeVM memory model.

13. event Syntax and Event Model

13.1 Purpose and Positioning

The event syntax in Aloe expresses at the language level that “this is a notification point where external code can register handlers.”

  • External code can register and unregister event handlers (callbacks).
  • Event firing (executing notifications) is, in principle, only allowed for the code that declared the event.

At the implementation level, event is treated as syntactic sugar for the standard library type Event<TArgs>.

  • event Clicked<ClickedArgs>; is equivalent to having a field of type Event<ClickedArgs>.
  • However, unlike ordinary fields, reassignment of that field or direct Raise calls from external code are forbidden (compile-time error).

13.2 Event<TArgs> and Handler Types

13.2.1 Handler Type

The function type for event handlers is defined as a standard type alias: 

type EventHandler<TArgs> = Func<TArgs:Task>;
  • Argument: TArgs (type of the event arguments)
  • Return type: Task
    To naturally support asynchronous handlers, event handlers are modeled as “functions that return a Task representing completion.”

13.2.2 Role of Event<TArgs> (Conceptual)

Event<TArgs> is a type that holds the collection of event handlers and logic for invoking them. 

type Event<TArgs> {
  Handlers: List<EventHandler<TArgs>>;
}

bound Event<TArgs> {
  method Add(Handler: EventHandler<TArgs>): Task {
    Handlers.Add(Handler);
  }

  method Remove(Handler: EventHandler<TArgs>): Task {
    // The details of handler identity comparison are left to the VM / runtime spec
    Handlers.Remove(Handler);
  }

  method Raise(Args: TArgs): Task {
    // Sequentially await all handlers and return a Task
    // that completes when they all finish.
    foreach (H in Handlers) {
      await H(Args);
    }
  }
}

From the language specification’s perspective, Event<TArgs> is the concrete representation of “an event collection,” and event syntax is just sugar plus access control on top of that.

13.3 event Declaration Syntax

13.3.1 Declaration Locations

event may be declared in:

  • Inside a struct body
  • Inside a class body
  • Inside an interface body

Declarations at the top level (file scope) or inside a trait are not defined in version 0.1.

13.3.2 Syntax (Overview)

EventDeclaration ::=
    'event' Identifier '<' Type '>' ';'

Example:

struct Button {
  event Clicked<ClickedArgs>;
}

This is semantically equivalent to the following field declaration: 

struct Button {
  private Clicked: Event<ClickedArgs>;
}

However, unlike a normal field, the following additional constraints apply:

  • External code cannot reassign Clicked (compile-time error).
  • External code cannot directly call Clicked.Raise(...).
  • External code cannot perform delete Clicked;.

13.4 Event Subscription / Unsubscription Syntax (+= / -=)

13.4.1 Syntax

EventAddStatement    ::= PrimaryExpression '.' Identifier '+=' EventHandlerExpression ';'
EventRemoveStatement ::= PrimaryExpression '.' Identifier '-=' EventHandlerExpression ';'
  • PrimaryExpression.Identifier must denote a member declared as event.
  • EventHandlerExpression must be an expression that can be implicitly converted to Func<TArgs:Task>.

13.4.2 Type Rules

When event X<TArgs>; is declared:

  • Target.X has type Event<TArgs>.
  • Target.X += H; is allowed only if H has type Func<TArgs:Task> (or can be implicitly converted to it).
  • The same constraints apply to Target.X -= H;.

13.4.3 Desugaring (Semantically Equivalent Code)

The following meaning is assumed:

Target.X += H;
// is semantically:
await Target.X.Add(H);
Target.X -= H;
// is semantically:
await Target.X.Remove(H);

Having Add / Remove return Task allows subscription/unsubscription via += / -= to be treated as asynchronous statements.

13.5 Rules for Event Firing (Raise)

13.5.1 Firing Allowed Only on the Defining Side

Calls to Raise on a member declared as event are allowed only from the following scopes:

  • Methods of the struct/class that declares that event
  • Methods within a bound block for the same type

Any other code (external modules or other classes) calling Clicked.Raise(...) is a compile-time error.

This ensures that “only the side that exposes the event has the authority to fire it,” maintaining separation of responsibilities.

13.5.2 Basic Firing Form

struct ClickedArgs {
  ButtonId: String;
}

struct Button {
  event Clicked<ClickedArgs>;
}

bound Button {
  method OnNativeClick(Id: String): Task {
    let Args: ClickedArgs = { ButtonId = Id };
    await Clicked.Raise(Args);
  }
}

13.6 event in interface and Implementations

interface can also declare events.

struct ClickedArgs {
  ButtonId: String;
}

interface IButton {
  event Clicked<ClickedArgs>;
}

In this case, any struct / class that implements IButton must declare the corresponding event. 

struct Button implements IButton {
  event Clicked<ClickedArgs>;
}
  • If the event name Clicked does not match, or
  • If the type argument ClickedArgs differs,

then it is a compile-time error for incomplete interface implementation.

13.7 Usage Examples

13.7.1 Button and Clicked Event

struct ClickedArgs {
  ButtonId: String;
}

struct Button {
  event Clicked<ClickedArgs>;
}

bound Button {
  method OnNativeClick(Id: String): Task {
    let Args: ClickedArgs = { ButtonId = Id };
    await Clicked.Raise(Args);
  }
}

method SetupButton(): Task {
  var Btn: Button = CreateButton("OK");

  // Register an async handler
  Btn.Clicked += async (Args: ClickedArgs): Task {
    Print("Clicked: " + Args.ButtonId);
  };

  // Register another handler
  Btn.Clicked += async (Args: ClickedArgs): Task {
    Log("button clicked");
  };
}

13.7.2 Event Unsubscription

method SetupAndTearDown(Btn: Button): Task {
  let Handler: Func<ClickedArgs:Task> =
    async (Args: ClickedArgs): Task {
      Print("temp handler");
    };

  // Subscribe
  Btn.Clicked += Handler;

  // ... Some processing ...

  // Unsubscribe
  Btn.Clicked -= Handler;
}

13.8 Combining with with / lock (Reference)

Since Event<TArgs> is a normal reference type, it can be combined with with or lock with to perform thread-safe firing of events when necessary. 

struct Hub {
  event MessageReceived<String>;
}

bound Hub {
  method RaiseMessage(Msg: String): Task {
    lock with (MessageReceived) {
      await MessageReceived.Raise(Msg);
    }
  }
}

13.9 Summary

  • event provides syntactic sugar and access control rules on top of Event<TArgs>.
  • External code can only subscribe/unsubscribe via +=/-=; it cannot fire events or reassign the event field.
  • Handlers are unified as Func<TArgs:Task>, making the event model inherently asynchronous.
  • Interfaces can declare events, expressing event contracts at the type level.
  • It offers a usage experience similar to C# event, but without introducing a delegate type; instead it uses Aloe’s function types and Task directly.
