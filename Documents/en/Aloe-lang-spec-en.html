1. Introduction

Aloe is an experimental programming language with static typing, a simple syntax, and a stack-based virtual machine (AloeVM). It is particularly designed to emphasize pipe / filter style data flows and interoperability with WebAssembly.

  • Static typing (type checking is done at compile time).
  • C-like surface syntax, but an intentionally small feature set.
  • Optimized for text processing, REST calls, logging, and other tasks where “data flow and transformation” should feel good to write.
  • Assumes a stack-based VM (AloeVM) and future WebAssembly backends.

This document is a draft language specification.
It may differ from an actual compiler/VM implementation.

2. Design goals

2.1 Goals

  • Readability: the specification itself should be readable as documentation.
  • Ease of implementation: should be implementable with a small compiler and VM.
  • Consistency of Pipe / Filter: type-safe representation of data flows.
  • Bridge to Wasm: should integrate smoothly with future Wasm component models.

2.2 Non-goals

  • Becoming a “do-everything” general-purpose language.
  • Competing with highly optimized JIT compilers in raw performance.
  • Complex metaprogramming (macros, etc.) is out of scope for now.

3. Basic syntax

3.1 Source files

  • Source files are assumed to be UTF-8 encoded.
  • Single-line comments: // comment
  • Multi-line comments: /* ... */

3.2 Tokens

  • Identifiers: start with a letter or _, followed by letters, digits, or _.
  • Keywords are reserved and cannot be used as identifiers.

3.3 Representative keywords

CategoryKeywords
Declarations let, var, type, struct, enum, class, interface, trait, sealed
Control if, else, while, for, break, continue, return, lock
Functions method
Auxiliary with, delete, clone, where

3.4 Statements and expressions

  • Most statements end with a semicolon ;.
  • Blocks are enclosed in { ... }.
  • Conditions in control statements must have type Bool.
// Variable declarations
let Threshold: Int = 10;
var Count: Int = 0;

// if statement
if (Count > Threshold) {
  Print("over");
} else {
  Print("ok");
}

4. Type system

4.1 Primitive types

TypeDescription
Booltrue / false
IntSigned integer (bit width is implementation-defined, assume 32-bit here).
FloatFloating-point number (32 or 64 bit, implementation-defined).
StringUTF-8 string (internal representation is implementation-defined).

4.2 Arrays and collections

  • Fixed-length array: T[]
  • Variable-length and lazily evaluated sequences use Pipe<T> (see chapter 6).

4.3 struct

struct User {
  Id: Int;
  Name: String;
  IsActive: Bool;
}
  • struct is a value type; assignments copy the value.
  • The copy is shallow; sharing of nested reference-type fields depends on the VM implementation.
  • A struct may inherit from at most one other struct (single inheritance of structs).
  • Structs are not allowed to implement interface. (How they interact with traits and generic constraints will be clarified in later revisions.)

4.4 enum

  • enum is always represented internally as Int.
  • “Generic enums” such as enum<String> or enum<Byte> are forbidden.
  • Bitfield enums are limited to at most 32 flags.
enum LogLevel {
  Trace = 0;
  Debug = 1;
  Info  = 2;
  Warn  = 3;
  Error = 4;
}

4.5 Classes and inheritance (class / interface / trait / sealed)

4.5.1 Class basics

  • class is always a heap-allocated reference type.
  • Assignments copy the reference (shallow copy).
  • clone creates a shallow copy of an object and returns a new reference.
  • All classes are cloneable; there is no type-level way to declare “non-cloneable” classes.
    If cloning should be discouraged, this is left to API design and coding conventions. 
class User {
  Name: String;
  Age: Int;
}

var A: User = NewUser();
var B: User = A;        // reference copy (A and B point to the same instance)

var C: User = clone A;  // shallow copy; C points to a new instance

A shallow copy means that field values are copied as-is.
If a field itself is a reference type, its reference value is copied. Types requiring deep copies or custom cloning logic should provide user-defined APIs, such as a Clone method layered on top of the language-level clone.

4.5.2 Class inheritance (single inheritance)

Aloe class supports single inheritance only.
A class can specify at most one base class. 

class Base {
  Id: Int;
}

class Derived : Base {
  Name: String;
}

// Compile-time error: multiple inheritance of classes is not allowed
class Bad : Base, OtherBase {
}
  • Class hierarchies form trees (or chains); diamond inheritance does not occur.
  • Class inheritance is the mechanism for reusing state and behavior.

4.5.3 interface and trait

Instead of allowing multiple inheritance of classes, Aloe uses interface and trait to compose behavior.

  • interface defines method signatures only and has no implementation (whether default implementations are allowed is undecided in this draft).
  • trait is a mixin unit that can provide method implementations to be applied to existing classes using with.
  • A class has one base class, can implement multiple interface types, and can apply multiple traits via with. 
interface IDisposable {
  method Dispose(): Unit;
}

trait Printable {
  method PrintSelf(): Unit {
    Print(this.ToString());
  }
}

class FileHandle : BaseFile implements IDisposable with Printable {
  method Dispose(): Unit {
    // resource cleanup
  }

  method ToString(): String {
    return "FileHandle(...)";
  }
}

Detailed rules for interface and trait— such as inheritance between them and conflict resolution when multiple traits define the same method—are not fully defined in this draft. Only the high-level direction is given.

4.5.4 sealed classes

Aloe provides the sealed modifier to forbid inheritance from specific classes. 

sealed class Token {
  Kind: TokenKind;
  Text: String;
}

// Compile-time error: Token is sealed
class KeywordToken : Token {
}
  • A sealed class cannot be used as a base class.
  • Even if a class is sealed, it may still implement interfaces and apply traits via with.
  • In the future, we plan to combine sealed classes with union types and pattern matching to support exhaustiveness checking.

4.5.5 readonly and classes

  • Fields of classes are mutable by default.
  • A field with the readonly modifier becomes immutable:
    • Assignments to that field are only allowed in a field initializer or inside a constructor of the same class.
    • Any other assignment is a compile-time error.
  • readonly class makes the entire class logically immutable:
    • All instance fields are implicitly treated as if they were declared readonly.
    • Field assignments are allowed only in field initializers and constructors of that class.
    • After a constructor finishes, no code can modify fields of that instance.
  • readonly and clone are orthogonal: a readonly class is still cloneable. The cloned instance has the same immutable state.

This section defines a high-level policy: classes are heap-based reference types with single inheritance; multiple behaviors are combined via interface + trait; sealed forbids inheritance; readonly expresses immutability at field or class level.
Details like method resolution order and trait conflict resolution will be specified separately.

4.6 Generics

4.6.1 Overview

Aloe provides minimal generics for types, structs, classes, interfaces, traits, methods, and pipe-related declarations.
The goal is to enable type-safe reuse of collections and Pipe / Filter components, not to support advanced type-level computation. 

type Box<T> {
  Value: T;
}

struct Pair<TLeft, TRight> {
  Left: TLeft;
  Right: TRight;
}

method Map<T, U>(Source: Pipe<T>, Selector: T -> U): Pipe<U> {
  // Implementation omitted
}

4.6.2 Generic type parameters

  • Types, structs, classes, interfaces, traits, methods, and producer/filter/consumer declarations can all have type parameters <T, U, ...>.
  • Type parameter names are generally recommended to start with an uppercase letter (T, TItem, TKey, etc.).
  • Type arguments are bound to concrete types at use sites.
type List<T> {
  Items: T[];
}

method FirstOrDefault<T>(Source: Pipe<T>): T {
  // Implementation omitted
}

4.6.3 where constraints (overview)

Aloe generics support arbitrary type constraints using where clauses.
By default there is no constraint, but version 0.1 supports several basic constraint forms. 

method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  // Int-only sum
}

Here, where exists purely for compile-time type checking.
Runtime representation (monomorphization vs. shared generic code with type tags, etc.) is left to the VM specification.

4.6.4 Supported constraint kinds

Version 0.1 supports the following constraint kinds:

  • Value-type / reference-type constraints
    • where T: struct – T must be a value type.
    • where T: class – T must be a reference type.
  • Primitive-type constraints
    • where T: Int, where T: Float, where T: Bool, etc. Primitive type names can be used directly.
  • User-defined type constraints
    • where T: SomeTrait – types implementing trait SomeTrait.
    • where T: MyStruct – T must be exactly struct MyStruct.
    • where T: MyClass – T must be exactly class MyClass.
  • Arbitrary concrete type names
    • Any concrete type name can be specified (e.g. where T: Pipe<Int>).
// Value-type-only generic
method Zero<T>(): T
  where T: struct
{
  // Implementation omitted
}

// Pipe for Int only
method Average<T>(Source: Pipe<T>): Float
  where T: Int
{
  // Implementation omitted
}

// Generic constrained to a specific struct type
method PrintUser<T>(User: T)
  where T: UserInfo
{
  // Implementation omitted
}

4.6.5 Behavior on constraint violation

  • If a type argument violates its where constraint, this is a compile-time error.
  • All constraints in a where clause must be satisfied simultaneously.
  • Cyclic constraints and advanced typeclass-style structures are not supported at this time.
// Example of compile-time error:

// where T: Int, so T cannot be String
method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  ...
}

var Lines: Pipe<String> = ReadLines("log.txt");
var Result = Sum(Lines);  // <-- compile-time error

The internal representation of generics (monomorphization vs. shared code with type tags, etc.) is defined by AloeVM.
This language specification focuses on type-safe reuse and constraint checking via where.

5. Methods and functions

5.1 method declarations

method Add(a: Int, b: Int): Int {
  return a + b;
}
  • method declares either a top-level function or a method bound to a type.
  • If no return type is specified, the method is assumed to return Unit.

5.2 bound for method binding

Aloe uses bound blocks to add methods to existing types. 

type Pipe<T> {
  // Internal representation is VM-dependent
}

bound Pipe<T> {
  method Where(Predicate: T -> Bool): Pipe<T> {
    // Implementation omitted
  }
}

For Pipe<T>, the bound block introduces a restriction on return types of methods declared inside it (see section 6.2 for details).

6. Pipe / Filter model

A distinctive feature of Aloe is that it can express pipe / filter style data flows in a type-safe and consistent way. 

ReadLines("access.log")
  |> Where (.StartsWith("[ERROR]"))
  |> Map   (ParseLogLine)
  |> Where (.Level == LogLevel.Error)
  |> ToArray();

Conceptually, the above is equivalent to the following nested call: 

ToArray(
  Where(
    Map(
      Where(ReadLines("access.log"), ...),
      ...),
    ...));

Aloe’s |> operator is syntactic sugar for such function composition.

6.1 Pipe<T> type

Pipe<T> represents the “pipe world” in Aloe, analogous to lazy sequences, streams, or iterators.

  • Pipe<T> is “a pipe through which elements of type T flow”.
  • Its internal representation (iterator chains, coroutines, etc.) is VM-dependent.
  • The standard library provides producers / transformers / consumers based on Pipe<T>.
type Pipe<T> {
  // Internal fields are implementation-dependent
}

The pipe world is the universe of functions and objects that accept and/or return Pipe<T>: producers, transformers, and consumers.
Consumers are the exit points from the pipe world.

6.2 Producer / Transformer / Consumer

In the Pipe / Filter model, we classify functions and objects into three roles:

6.2.1 Producer

  • Definition: A function/object whose return type is Pipe<*> and which appears at the head of a pipeline.
  • Role: Reads from the external world (files, HTTP, arrays, etc.) and pushes data into the pipe world (Pipe<T>).
  • Type: Conceptually, a producer has type X -> Pipe<T>. It is always designed to output a Pipe<*>.

6.2.2 Transformer (intermediate filter)

  • Definition: A function/method with type Pipe<T> -> Pipe<U>.
  • Role: Transforms, filters, or sorts data inside the pipe world, without exiting it.

In Aloe, methods declared inside bound Pipe<T> are intended to be transformers only.

Constraint:
Methods declared inside a bound Pipe<T> block must always return Pipe<*>.
That is, their type must be Pipe<T> -> Pipe<U>. 
bound Pipe<T> {
  // OK: transformer
  method Where(Predicate: T -> Bool): Pipe<T> { ... }

  // OK: changes element type
  method Map<U>(Selector: T -> U): Pipe<U> { ... }

  // NG: return type is not a Pipe; should be a compile-time error
  // method Count(): Int { ... }
}

6.2.3 Consumer (terminal filter)

  • Definition: A function/object with type Pipe<T> -> R, where R is not Pipe<*>.
  • Role: Acts as an exit from the pipe world by completely consuming a pipe and converting it into arrays, scalars, or external I/O, etc.
  • Return type: R is unconstrained as long as it is not Pipe<*>. It may be T[], Int, Unit, etc. 
// Consumer that materializes a pipe into an array
method ToArray<T>(Source: Pipe<T>): T[] { ... }

// Consumer that counts elements
method Count<T>(Source: Pipe<T>): Int { ... }

6.3 Pipe-related declarations (producer / pipe / filter / consumer)

Aloe introduces four declaration forms to construct Pipe / Filter components: producer, pipe, filter, and consumer.
All of them are treated as objects with constructors; when attached to a pipeline, their bound block is executed.

6.3.1 producer declaration

producer ProducerName(Params...) {
  out: Pipe<T>;

  bound(output) {
    // Executed when the producer is attached to a pipeline
    // Reads from external resources and writes into 'output'
  }
}
  • ProducerName is the name of the producer.
  • out declares the type of the pipe output and must be Pipe<*>.
  • bound(output) defines behavior when the producer is connected. The type of output is the type given in out (e.g. Pipe<T>).

Constructor parameters (Params...) are visible inside the bound block. 

producer ReadLines(Path: String) {
  out: Pipe<String>;

  bound(output) {
    // Pseudo-code:
    //   let file = OpenFile(Path);
    //   for each line in file:
    //     output.write(line);
    //   file.close();
  }
}

6.3.2 pipe declaration

pipe PipeName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Connects upstream to downstream and performs transfer/transformation
  }
}
  • PipeName is the name of the pipe.
  • in is the type of incoming data from upstream.
  • out is the type of outgoing data to downstream.
  • bound(input, output) defines the transformation between them.

Pipe declarations are intended for bridging heterogeneous streams and external I/O (e.g. text → JSON). 

pipe JsonDecode<T>() {
  in:  String;   // JSON text
  out: T;        // decoded object

  bound(input, output) {
    // Pseudo-code:
    //   for each line in input:
    //     let obj: T = JsonParse<T>(line);
    //     output.write(obj);
  }
}

6.3.3 filter declaration

filter FilterName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Defines how elements flow from input to output
  }
}
  • FilterName is the name of the filter.
  • in / out specify input and output types.
  • In typical Pipe / Filter chains, both in and out are Pipe<*>.
  • bound(input, output) defines the transformation logic between them.
filter Where<T>(Predicate: T -> Bool) {
  in:  Pipe<T>;
  out: Pipe<T>;

  bound(input, output) {
    // Pseudo-code:
    //   for each item in input:
    //     if (Predicate(item)):
    //       output.write(item);
  }
}

filter Map<T, U>(Selector: T -> U) {
  in:  Pipe<T>;
  out: Pipe<U>;

  bound(input, output) {
    // Pseudo-code:
    //   for each item in input:
    //     output.write(Selector(item));
  }
}

6.3.4 consumer declaration

consumer ConsumerName(Params...) {
  in: InputType;

  bound(input) {
    // Executed at the end of the pipeline
    // Consumes 'input' and returns some value
  }
}
  • ConsumerName is the name of the consumer.
  • in specifies the input type; in this model it is usually Pipe<T>.
  • bound(input) is executed at the end of the pipeline, completely consumes input and returns a value of some type R.
  • R may be any type such as T[], Int, Bool, Unit, etc., but must not be Pipe<*>. 
consumer ToArray<T> {
  in: Pipe<T>;

  bound(input) {
    var buffer: List<T>;
    // Pseudo-code:
    //   for each item in input:
    //     buffer.add(item);
    //   return buffer.toArray();
    return buffer.toArray();
  }
}

consumer Count<T> {
  in: Pipe<T>;

  bound(input) {
    var c: Int = 0;
    // Pseudo-code:
    //   for each item in input:
    //     c = c + 1;
    //   return c;
    return c;
  }
}

In summary, the roles of pipe-related declarations are:

  • producer: external world → Pipe<*> (always outputs a Pipe).
  • pipe: bridge between different stream types and external I/O.
  • filter: typically Pipe<T> -> Pipe<U> transformations.
  • consumer: Pipe<T> → non-Pipe type (exit from the pipe world).

6.4 Pipe operator |>

6.4.1 Language-level definition

Aloe’s |> operator is defined as syntactic sugar for simple function composition. 

Expr |> F |> G

is exactly equivalent to:

G(F(Expr))
  • If F has type A -> B and G has type B -> C, then Expr |> F |> G has type C.
  • The operator can be applied to any types; it is not restricted to Pipe<T>.
Thus, from the type system’s point of view, “changing the return type in the middle of a pipeline” is not a problem.
Constraints for the Pipe / Filter model come from Pipe<T> and the design of pipe-related libraries, not from the operator itself.

6.4.2 Using Pipe<T> with |>

ReadLines("users.json")
  |> Where (.Trim().Length > 0)
  |> Map   (ParseUser)
  |> Where (.IsActive)
  |> ToArray();
  • ReadLines: producer (String -> Pipe<String>).
  • Where, Map: transformers (Pipe<T> -> Pipe<U>).
  • ToArray: consumer (Pipe<User> -> User[]).
  • The overall type of the pipeline expression is User[], the return type of ToArray.

Note:
At the language level, you may chain |> even after a consumer, but stylistically it is not recommended.
For example, patterns like ToArray() |> ToArray() should be avoided.

6.5 Standard filters and producers

6.5.1 Typical transformers (filters)

filter Where<T>(Predicate: T -> Bool) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

filter Map<T, U>(Selector: T -> U) {
  in:  Pipe<T>;
  out: Pipe<U>;
  bound(input, output) { ... }
}

filter Take<T>(Count: Int) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

6.5.2 Typical producers

producer FromArray<T>(Items: T[]) {
  out: Pipe<T>;
  bound(output) { ... }
}

producer ReadLines(Path: String) {
  out: Pipe<String>;
  bound(output) { ... }
}

6.5.3 Typical consumers

consumer ToArray<T> {
  in: Pipe<T>;
  bound(input) { ... }   // returns T[]
}

consumer Count<T> {
  in: Pipe<T>;
  bound(input) { ... }   // returns Int
}

6.5.4 Example usage

let ActiveUserNames: String[] =
  ReadLines("users.json")
    |> Where (.Trim().Length > 0)
    |> Map   (ParseUser)
    |> Where (.IsActive)
    |> Map   (.Name)
    |> ToArray();

7. with syntax

7.1 Overview

The with syntax is syntactic sugar for repeated property assignments and method calls on the same target object.
In Aloe, it also serves as a “lifetime lock” that keeps the target object alive for the duration of the block. 

struct HttpRequest {
  Url: String;
  Method: String;
  TimeoutSeconds: Int;
}

var Request: HttpRequest;

with (Request) {
  Url = "https://example.com/api";
  Method = "GET";
  TimeoutSeconds = 5;
}

7.2 Semantics

The with block above is semantically equivalent to: 

{
  Request.Url = "https://example.com/api";
  Request.Method = "GET";
  Request.TimeoutSeconds = 5;
}
  • with (Expr) { ... } evaluates Expr exactly once.
  • Simple identifier assignments inside the block are interpreted as member access on the target object.

7.3 Lifetime lock (GC root behavior)

AloeVM assumes a memory model based on reference counting plus sliding compaction.
In this model, a with block acts as a “lifetime lock”:

  • The result of with (Expr) is stored in a hidden variable (a GC root).
  • This hidden variable keeps the object alive until the end of the block; the object cannot be collected while inside the block.
  • Thus, the object is guaranteed to be alive during execution of the block.

7.4 Interaction with delete

  • It is a compile-time error to apply delete to the same variable that is the target of with inside that with block.
  • The reason is that with expresses the intention “this object should stay alive here”, which conflicts with delete.
  • However, it is allowed to delete a different variable that happens to refer to the same object.
    The GC root created by with still keeps the object alive. 
var Request: HttpRequest = GetRequest();
var Alias: HttpRequest = Request;

with (Request) {
  // Request is protected by with; it is guaranteed to stay alive in this block.

  delete Alias;   // OK: removes the Alias reference only
  Url = "https://example.com/api";  // still usable
}

// After leaving the with block, the GC root is released.
// If no other references remain, the object may eventually be reclaimed.

In summary, with is both syntactic sugar for member access and a temporary GC root that keeps the target object alive.

8. delete syntax

8.1 Overview

In Aloe, delete does not directly destroy objects.
Instead, it is a construct for removing the reference from a variable.

  • Actual object destruction (memory reclamation, finalizers, etc.) is done by the runtime (AloeVM), using reference counting + GC.
  • delete is intended as a type-safe way to express “this variable no longer refers to this object”, as an alternative to assigning null.

8.2 Grammar

DeleteStatement ::= 'delete' Identifier ';'

8.3 Static rules (compile time)

  • Identifier must be a variable declared in the current scope.
  • After delete, reading from that variable or performing member access / method calls on it is a compile-time error.
  • However, you may assign a new value to the variable. Conceptually, delete resets the variable to an “uninitialized” state. 
var Buf: Buffer = NewBuffer();

// Use Buf
DoSomething(Buf);

delete Buf;

// After this, reading Buf or Buf.Length is a compile-time error

Buf = NewBuffer();  // After re-assignment, it is usable again

8.4 Runtime semantics

When delete Var; is executed, the VM behaves roughly as follows:

  • Decrement the reference count of the object currently referenced by Var (assuming a reference-counting scheme).
  • Mark Var as “uninitialized / deleted”.
  • When the object’s reference count reaches zero, the object is reclaimed at the GC’s discretion.
    Thus, the object is not necessarily destroyed immediately when delete is executed.

This specification leaves the exact timing of object destruction up to the VM/runtime.
delete simply removes one reference from a variable to an object.

8.5 Interaction with with

  • As stated in section 7.4, it is a compile-time error to delete the same variable that is the target of a with within that block.
  • It is allowed to delete another variable that refers to the same object. The GC root created by with keeps the object alive.

If Aloe later gains “owned resource types” or an ownership model, delete may be integrated with that model.
This draft assumes no ownership model.

9. lock syntax and synchronization

9.1 Overview

The lock syntax acquires a lock on a particular object and guarantees exclusive access and optimized read barriers to that object while the block is executing. 

var SharedCounter: Int = 0;

lock (SharedCounter) {
  SharedCounter = SharedCounter + 1;
}

9.2 Semantics

  • lock (Expr) { Body } acquires a lock on the object represented by Expr, guaranteeing exclusive access inside Body.
  • Lock acquisition and release are implemented by the VM.
  • Updates to data shared between threads should generally be confined to lock blocks.

AloeVM assumes memory management with sliding compaction.
Inside a lock block, the VM can optimize read barriers and reduce per-object checks as much as possible.

9.3 lock with syntax (combining lock + with)

Often, you want both:

  • Exclusive access to an object (via lock), and
  • Convenient repeated access to its members plus a lifetime lock (via with).

Aloe provides a combined syntax for this: 

lock with (SharedCounter) {
  SharedCounter = SharedCounter + 1;
}

9.3.1 Syntax

LockWithStatement ::= 'lock' 'with' '(' Expression ')' Block

That is, the only combined form is lock with (Expr) { ... }.

9.3.2 Desugaring

Conceptually, the following transformation holds: 

lock with (Expr) {
  Body
}

is equivalent to:

{
  let __tmp = Expr;
  lock (__tmp) {
    with (__tmp) {
      Body
    }
  }
}
  • Expr is evaluated exactly once and bound to a hidden local __tmp.
  • lock (__tmp) acquires a lock on that object.
  • with (__tmp) creates a lifetime lock and provides syntactic sugar for member access inside the block.

9.3.3 Style and restrictions

  • lock with exists purely as convenient syntax; it does not introduce a new underlying primitive beyond lock and with.
  • The language does not define alternative spellings such as with lock (Expr) or lock & with (Expr).
    For this purpose you must always use lock with (Expr).
  • If you need more complex ordering—for example, nested locks on different objects— you should write explicit nested lock and with constructs rather than relying on additional syntactic sugar.

10. Exception handling

10.1 try / catch / finally

try {
  DoSomething();
} catch (ex: Error) {
  LogError(ex);
} finally {
  Cleanup();
}
  • Exceptions thrown inside a try block can be caught by compatible catch clauses.
  • The finally block is always executed, regardless of whether an exception is thrown.
  • Exception types and hierarchies are defined on the VM side.

11. Execution model and memory management

11.1 Overview of AloeVM

  • Stack-based bytecode VM.
  • Function calls push frames onto a call stack.
  • GC is assumed to be reference counting plus sliding compaction.

11.2 Memory management (overview)

  • Objects are allocated on the heap.
  • When certain conditions are met, GC runs and compacts memory by sliding live objects.
  • with acts as a temporary GC root that keeps objects alive, and combined with lock it allows optimization of read barriers.
  • Typically, you do not need to call delete for memory management; only use it if you want to explicitly remove a reference from a variable.

Detailed opcode listings and GC implementation details are provided in the AloeVM specification.

12. Limitations and future work

12.1 Current limitations

  • Expressiveness of generics is minimal; typeclasses and higher-kinded generics are not defined.
  • The module system and package management are out of scope for this document.
  • Metaprogramming (macros, reflection, etc.) is not defined.

12.2 Future directions

  • Parallelization and fusion optimizations for Pipe / Filter pipelines.
  • Integration with the Wasm component model.
  • Introduction of Result-like types as an alternative to exceptions.
  • Extended standard libraries for distributed processing and streaming.
  • Possible integration of an ownership-based resource model with delete.
  • Integration of trait / interface / sealed / readonly with pattern matching and exhaustiveness checks.

In this draft, we especially focus on:
The relationship between Pipe / Filter and Producer / Transformer / Consumer / Pipe / Consumer declarations,
and
The relationship between with / delete / lock / class inheritance / readonly / clone / generics and the AloeVM memory model.
