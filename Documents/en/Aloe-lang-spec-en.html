1. Overview

Aloe is an experimental programming language with static typing, simple syntax, and a stack-based VM (AloeVM). It is designed with a particular focus on dataflow description via Pipe / Filter and on being friendly to WebAssembly.

  • Statically typed (type checking at compile time)
  • Syntax is similar to C-like languages, but the feature set is intentionally small
  • Aims to make it pleasant to write code that “flows and transforms data”, such as text processing, REST calls, and log processing
  • Implementation assumes a stack-based VM (AloeVM) and future WebAssembly output

This document is a draft of the language specification. It may diverge from the actual compiler / VM implementation.

2. Design Goals

2.1 Goals

  • Readability: can be read as a specification document
  • Ease of implementation: can be implemented by a small compiler and VM
  • Consistency of Pipe / Filter: type-safe expression of dataflow
  • Bridge to Wasm: designed to connect well with Wasm’s component model in the future

2.2 Non-goals

  • Being a “do everything” general-purpose language
  • Competing in performance with JIT-optimized languages
  • Complex metaprogramming (macros, etc.) is out of scope for now

3. Basic Syntax

3.1 Source Code and Files

  • Source files are assumed to be encoded in UTF-8.
  • Single-line comments: // comment
  • Multi-line comments: /* ... */

3.2 Tokens

  • Identifiers: start with an alphabetic character or _, subsequent characters may include digits.
  • Keywords are reserved and cannot be used as identifiers.

3.3 Representative Keywords

CategoryKeywords
Declarationslet, var, type, struct, enum
Controlif, else, while, for, break, continue, return
Functionsmethod
Auxiliarywith, lock, delete

3.4 Statements and Expressions

  • In general, a statement must end with a semicolon ;.
  • Blocks are surrounded by { ... }.
  • Conditions in control constructs must be of type Bool.
// Variable declarations
let Threshold: Int = 10;
var Count: Int = 0;

// if statement
if (Count > Threshold) {
  Print("over");
} else {
  Print("ok");
}

4. Type System

4.1 Primitive Types

TypeDescription
Booltrue / false
IntSigned integer (bit width is implementation-defined; 32-bit is assumed)
FloatFloating point (32 or 64 bit, implementation-defined)
StringUTF-8 string (internal representation is implementation-defined)

4.2 Arrays and Collections

  • Fixed-length array: T[]
  • Variable-length collections and lazy sequences are represented by Pipe<T> (see Chapter 6)

4.3 Structs

struct User {
  Id: Int;
  Name: String;
  IsActive: Bool;
}
  • struct is a value type and is copied on assignment.
  • It has shallow-copy semantics; sharing of nested reference types depends on the VM spec.

4.4 Enums

  • enum always uses Int as its underlying representation.
  • Generic enums such as enum<String> or enum<Byte> are forbidden.
  • For bit-flag enums, the maximum number of flags is 32.
enum LogLevel {
  Trace = 0;
  Debug = 1;
  Info  = 2;
  Warn  = 3;
  Error = 4;
}

5. Functions and Methods

5.1 method Declaration

method Add(a: Int, b: Int): Int {
  return a + b;
}
  • method declares either a top-level function or a method bound to a type.
  • If there is no return value, it is treated as type Unit.

5.2 Binding Methods with bound

Aloe introduces bound to attach methods “after the fact” to an existing type. 

type Pipe<T> {
  // Internal representation is VM-dependent
}

bound Pipe<T> {
  method Where(Predicate: T -> Bool): Pipe<T> {
    // Implementation omitted
  }
}

In this specification, especially for Pipe<T>, we impose a restriction on return types for methods defined in bound (see Section 6.2).

6. Pipe / Filter Model

A key feature of Aloe is that it lets you write dataflow via Pipe / Filter in a type-safe and consistent style. 

ReadLines("access.log")
  |> Where (.StartsWith("[ERROR]"))
  |> Map   (ParseLogLine)
  |> Where (.Level == LogLevel.Error)
  |> ToArray();

Conceptually, this is equivalent to: 

ToArray(
  Where(
    Map(
      Where(ReadLines("access.log"), ...),
      ...),
    ...));

The |> operator in Aloe is simply syntactic sugar for such function composition.

6.1 Pipe<T> Type

Pipe<T> represents the “pipe world” in Aloe. It corresponds to lazy sequences, streams, or iterators.

  • Pipe<T> represents “a pipe that flows elements of type T”.
  • Its internal representation (iterator chains, coroutines, etc.) is VM-dependent.
  • The standard library is built around Pipe<T>, providing producers, transformers, and consumers. 
type Pipe<T> {
  // Implementation-specific internal fields
}

We refer to the pipe world as the realm where functions that take and return Pipe<T> (producers / transformers / consumers) are composed, and exiting this world is done only through consumers.

6.2 Producer / Transformer / Consumer

In the Pipe / Filter model, functions (or objects) play one of three roles:

6.2.1 Producer

  • Definition: a function or object whose return type is Pipe<*> and which appears at the beginning of a pipeline.
  • Role: read data from the external world (files, HTTP, arrays, etc.) and push it into the pipe world (Pipe<T>).
  • Type-level view: it has the form X -> Pipe<T>. In this specification, anything designed as a producer must always output Pipe<*>.

6.2.2 Transformer (Intermediate Filter)

  • Definition: a function or method of type Pipe<T> -> Pipe<U>.
  • Role: transform, filter, or sort data within the pipe world, without leaving it yet.

In Aloe, methods defined in bound Pipe<T> are conceptually reserved for transformers.

Restriction:
Any method declared inside a bound Pipe<T> block must return a type of the form Pipe<*>.
That is, its shape is limited to Pipe<T> -> Pipe<U>. 
bound Pipe<T> {
  // OK: transformer
  method Where(Predicate: T -> Bool): Pipe<T> { ... }

  // OK: transformer that changes element type
  method Map<U>(Selector: T -> U): Pipe<U> { ... }

  // NG: return type is not a Pipe, so this violates the spec
  // method Count(): Int { ... }  // <-- should be a compile-time error
}

6.2.3 Consumer (Terminal Filter)

  • Definition: a function or object of type Pipe<T> -> R, where the return type R is not Pipe<*>.
  • Role: a gate that exits the pipe world. It consumes the pipe and materializes it as an array, a single value, external I/O, etc.
  • Return type restriction: there is no restriction on R for consumers; it may be T[], Int, Unit, etc., but it must not be Pipe<*>. 
// Pure consumer that materializes to an array
method ToArray<T>(Source: Pipe<T>): T[] { ... }

// Consumer that counts elements
method Count<T>(Source: Pipe<T>): Int { ... }

6.3 Pipe Declarations (producer / pipe / filter / consumer)

Aloe provides four declaration forms as building blocks for the Pipe / Filter model: producer / pipe / filter / consumer. They are all defined as “objects with constructors”, and their bound blocks are executed when they are connected in a pipeline.

6.3.1 producer Declaration

producer ProducerName(Params...) {
  out: Pipe<T>;

  bound(output) {
    // Code executed when connected at the head of a pipeline
    // Read from external resources and write to output
  }
}
  • ProducerName is the name of the producer.
  • out specifies the pipe-world output type this producer generates. By specification it must be a Pipe<*> type.
  • bound(output) describes what happens when the producer is actually connected at the beginning of a pipeline. The type of output is the type declared in out (i.e. Pipe<T>).

Constructor parameters (Params...) can be referenced directly from within the bound block. 

producer ReadLines(Path: String) {
  out: Pipe<String>;

  bound(output) {
    // Pseudo:
    //   let file = OpenFile(Path);
    //   for each line in file:
    //     output.write(line);
    //   file.close();
  }
}

6.3.2 pipe Declaration

pipe PipeName(Params...) {
  in:  InputDataType;
  out: OutputDataType;

  bound(input, output) {
    // Connect pipes, intermediate routing / transformation
  }
}
  • PipeName is the name of the pipe.
  • in is the type of data received from “upstream”.
  • out is the type of data sent “downstream”.
  • bound(input, output) describes how to move data from input to output.

Pipe declarations are intended to represent connections to the external world (sockets, HTTP) or bridging between streams of different types (e.g., text → JSON). 

pipe JsonDecode<T>() {
  in:  String;   // JSON text
  out: T;        // decoded object

  bound(input, output) {
    // Pseudo:
    //   for each line in input:
    //     let obj: T = JsonParse<T>(line);
    //     output.write(obj);
  }
}

6.3.3 filter Declaration

filter FilterName(Params...) {
  in:  InputDataType;
  out: OutputDataType;

  bound(input, output) {
    // Code executed when this filter is connected
    // Describe how to move data from input to output
  }
}
  • FilterName is the name of the filter.
  • in / out specify the input/output types handled by the filter.
  • For intermediate filters in the Pipe / Filter model, in / out will typically both be of type Pipe<*>.
  • bound(input, output) describes how to transform data from input to output.
filter Where<T>(Predicate: T -> Bool) {
  in:  Pipe<T>;
  out: Pipe<T>;

  bound(input, output) {
    // Pseudo:
    //   for each item in input:
    //     if (Predicate(item)):
    //       output.write(item);
  }
}

filter Map<T, U>(Selector: T -> U) {
  in:  Pipe<T>;
  out: Pipe<U>;

  bound(input, output) {
    // Pseudo:
    //   for each item in input:
    //     output.write(Selector(item));
  }
}

6.3.4 consumer Declaration

consumer ConsumerName(Params...) {
  in: InputDataType;

  bound(input) {
    // Code executed at the end of a pipeline
    // Consume input and return some value
  }
}
  • ConsumerName is the name of the consumer.
  • in specifies the input data type for this consumer. In the Pipe / Filter model, this is typically Pipe<T>.
  • bound(input) is executed when the consumer is attached to the end of a pipeline; it consumes input and returns a value of some type R.
  • R may be T[], Int, Bool, Unit, etc., but it must not be Pipe<*>. 
consumer ToArray<T> {
  in: Pipe<T>;

  bound(input) {
    var buffer: List<T>;
    // Pseudo:
    //   for each item in input:
    //     buffer.add(item);
    //   return buffer.toArray();
    return buffer.toArray();
  }
}

consumer Count<T> {
  in: Pipe<T>;

  bound(input) {
    var c: Int = 0;
    // Pseudo:
    //   for each item in input:
    //     c = c + 1;
    //   return c;
    return c;
  }
}

In summary, the roles of pipe declarations are:

  • producer: external world → Pipe<*> (must always output Pipe)
  • pipe: bridging different kinds of streams or external I/O
  • filter: primarily Pipe<T> -> Pipe<U> transformation logic
  • consumer: Pipe<T> → any non-Pipe type (exit from the pipe world)

6.4 Pipe Operator |>

6.4.1 Language-level Definition

Aloe’s |> is defined as syntactic sugar for function composition. 

Expr |> F |> G

is exactly equivalent to:

G(F(Expr))
  • If F has type A -> B and G has type B -> C, then Expr |> F |> G has type C.
  • It is applicable to any types, not just Pipe<T>.
Therefore, from the type system’s perspective
there is no inherent issue with “changing the return type in the middle of a pipeline”.
Constraints for the Pipe / Filter model are provided by library design (centered around Pipe<T> and pipe declarations).

6.4.2 Using Pipe<T> with the Operator

ReadLines("users.json")
  |> Where (.Trim().Length > 0)
  |> Map   (ParseUser)
  |> Where (.IsActive)
  |> ToArray();
  • ReadLines: Producer (String -> Pipe<String>)
  • Where, Map: Transformers (Pipe<T> -> Pipe<U>)
  • ToArray: Consumer (Pipe<User> -> User[])
  • The entire pipeline’s type is the return type of the final ToArray, i.e. User[].

Note:
While it is not forbidden by the language spec to chain further with |> after a consumer, it is almost always discouraged as a style.
For example, code like ToArray() |> ToArray() should be avoided.

6.5.1 Representative Transformers (Filters)

filter Where<T>(Predicate: T -> Bool) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

filter Map<T, U>(Selector: T -> U) {
  in:  Pipe<T>;
  out: Pipe<U>;

}

filter Take<T>(Count: Int) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

6.5.2 Representative Producers

producer FromArray<T>(Items: T[]) {
  out: Pipe<T>;
  bound(output) { ... }
}

producer ReadLines(Path: String) {
  out: Pipe<String>;
  bound(output) { ... }
}

6.5.3 Representative Consumers

consumer ToArray<T> {
  in: Pipe<T>;
  bound(input) { ... }   // returns T[]
}

  in: Pipe<T>;
  bound(input) { ... }   // returns Int
}

6.5.4 Example Usage

let ActiveUserNames: String[] =
  ReadLines("users.json")
    |> Where (.Trim().Length > 0)
    |> Map   (ParseUser)
    |> Where (.IsActive)
    |> Map   (.Name)
    |> ToArray();

7. with Statement

7.1 Overview

The with statement is syntactic sugar for writing a sequence of property assignments and method calls on a particular object more concisely. In Aloe, it additionally functions as a “release lock” that guarantees the lifetime of the target object within the block. 

struct HttpRequest {
  Url: String;
  Method: String;
  TimeoutSeconds: Int;
}

var Request: HttpRequest;

with (Request) {
  Url = "https://example.com/api";
  Method = "GET";
  TimeoutSeconds = 5;
}

7.2 Semantics

The above with block is, superficially, equivalent to: 

{
  Request.Url = "https://example.com/api";
  Request.Method = "GET";
  Request.TimeoutSeconds = 5;
}
  • with (Expr) { ... } evaluates Expr exactly once.
  • Simple identifier assignments inside the block are interpreted as member accesses on the target object.

7.3 Release Lock (GC Root Behavior)

AloeVM is assumed to use reference counting plus sliding compaction for memory management. A with block behaves as a kind of “release lock” as follows:

  • The result of evaluating with (Expr) is stored in a hidden variable that acts as a GC root.
  • That hidden variable is kept out of GC’s collection targets until the block is exited.
  • Thus, during execution of the with block, the target object is guaranteed to remain alive.

7.4 Relation between with and delete

  • It is a compile-time error to apply delete to the same variable that is used as the with target (e.g., Request in the example) within the same with block.
  • Reason: with expresses the intent “this object should stay alive here”, and deleting that same variable inside the block contradicts that intent.
  • However, it is allowed to apply delete to another variable that happens to reference the same object, because the hidden variable from with still keeps one GC root. 
var Request: HttpRequest = GetRequest();
var Alias: HttpRequest = Request;

with (Request) {
  // Request is protected by with, so it is guaranteed to be alive in this block

  delete Alias;   // OK: only cuts Alias's reference
  Url = "https://example.com/api";  // still usable
}

// When the with block ends, its GC root is removed.
// If there are no other references, the object may eventually be freed by GC.

In summary, with acts as “syntactic sugar for property initialization” and “a temporary GC lock that protects the target object”.

8. delete Statement

8.1 Overview

In Aloe, delete is not object destruction itself, but a construct for cutting the reference from a variable.

  • Actual object destruction (memory deallocation, finalizers, etc.) is handled by the runtime (AloeVM) with reference counting and GC.
  • delete is intended to express, in a type-safe manner, the intent of “this variable no longer uses that object”, and serves as a replacement for assigning null.

8.2 Grammar

DeleteStatement ::= 'delete' Identifier ';'

8.3 Static Rules (Compile-time)

  • Identifier must be a variable declared in the current scope.
  • After a delete, using that variable for reading, member access, or method calls is a compile-time error.
  • However, assigning a new value to that variable is allowed. That is, delete is treated as returning the variable to an “uninitialized / deleted” state. 
var Buf: Buffer = NewBuffer();

// Use Buf
DoSomething(Buf);

delete Buf;

// From here on, any read of Buf is a compile-time error.
// Buf.Length is also a compile-time error.

Buf = NewBuffer();  // Reassigning makes it usable again from this point on.

8.4 Runtime Semantics

When delete Var; is executed, the VM behaves roughly as follows:

  • Decrement the reference count of the object currently referred to by Var (assuming reference counting).
  • Mark the variable Var itself as “uninitialized/deleted”.
  • If the reference count reaches 0, the object will be released at a GC timing. The object is not necessarily destroyed immediately when delete is executed.

From the language spec’s perspective, the timing of object destruction is the responsibility of the VM/runtime. delete only “cuts the reference from one variable”.

8.5 Relation to with

  • As described in Section 7.4, it is a compile-time error to apply delete within a with block to the very same variable that is used as the with target.
  • It is allowed to apply delete to another variable that references the same object, because the with block still holds a GC root to that object.

In the future, if an “ownership-based resource type” model is introduced, delete might be integrated with that ownership model. In this draft, however, we define delete under the assumption that no ownership model is introduced.

9. lock Statement and Synchronization

9.1 Overview

The lock statement is used to execute a block of code exclusively with respect to a particular reference-type object. At the same time, it provides AloeVM with a hint that “while this block is being executed, the locked object must not be moved by GC”, i.e., it acts as a pin. 

class Counter {
  Value: Int;
}

var Shared: Counter = NewCounter();

lock (Shared) {
  Shared.Value = Shared.Value + 1;
}

9.2 Syntax and Type Constraints

LockStatement ::= 'lock' '(' Expression ')' Block
  • Expression must be an expression of a reference type (such as a class instance). Using a value type (Int, struct, etc.) as the lock target is a compile-time error.
  • Block follows the usual block syntax and scoping rules.

9.3 Evaluation Order and Re-entrant Locks

  1. The Expression part of lock (Expression) is evaluated exactly once.
  2. The VM obtains a lock and GC pin for the resulting reference value.
  3. After acquiring the lock, the statements in Block are executed in order.
  4. Whether the block exits normally or via an exception, the VM must always release both the lock and the pin (equivalent to a finally guarantee).

Re-entrant Locks

  • The same thread is allowed to lock the same object again (nested locks).
  • In such a case, the implementation may use an internal counter, etc., to keep the lock and pin until all corresponding lock / unlock pairs are balanced.
  • If a different thread attempts to lock the same object, only one thread succeeds at a time; other threads must wait until the lock is released. The waiting strategy (blocking, timeouts, etc.) is left to the VM specification.

9.4 Relation to GC and Read Barriers (Informative)

This section is informative guidance for implementers. It does not directly affect the language-level semantics of programs. An implementation may choose not to perform the optimizations described here.

  • When entering a lock block, the VM may apply a single read barrier to the lock target object to resolve it to its latest location, and then obtain the lock and pin.
  • The locked object is expected to not be moved by GC during the execution of the block (i.e., it is excluded from sliding compaction).
  • As a result, within the lock block, accesses to the locked object itself (field reads, method calls) may omit additional read barriers.
  • On the other hand, objects reachable from the lock target (e.g., child objects) are still subject to normal read barriers. lock pins only one object, not the entire object graph.

9.5 Interaction with with / delete and Usage Guidelines

  • with acts as a “release lock” via GC roots, while lock acts as a “synchronization lock” providing exclusion and movement prohibition. Using them together allows you to safely write “exclusive modification of an object whose lifetime is guaranteed”.
  • Although it is not forbidden by the spec to delete the lock target itself inside the lock block, this is strongly discouraged from an implementation and style perspective (since it contradicts the usual intent of protecting the object while it is in use).
  • Holding a lock for a long time or locking a large object that is a root of a large object graph may degrade compaction efficiency. Critical sections should therefore be kept as small as possible.

In summary, lock is a construct that gives the VM “exclusive access for a reference-type object” plus “a hint to temporarily exclude that object from sliding compaction”. The language-level semantics only require simple exclusion; detailed behavior regarding GC and read barriers is left to the VM specification and implementation.

10. Exception Handling

10.1 try / catch / finally

try {
  DoSomething();
} catch (ex: Error) {
  LogError(ex);
} finally {
  Cleanup();
}
  • Exceptions thrown within the try block can be caught by a corresponding catch.
  • The finally block is executed regardless of whether an exception occurred or not.
  • The details of exception types and hierarchies are defined in the VM specification.

11. Execution Model and Memory Management

11.1 Overview of AloeVM

  • Stack-based bytecode VM.
  • Function calls are implemented by pushing frames on a call stack.
  • GC is assumed to be reference counting plus sliding compaction.

11.2 Memory Management (Overview)

  • Objects are allocated on a heap.
  • GC is triggered under certain conditions and compacts memory by sliding live objects.
  • The with statement acts as a temporary GC root to protect an object from collection, and the lock statement provides a hint to temporarily exclude a particular object from compaction while also enabling fewer read barrier checks.
  • In typical use, programmers do not need to use delete for memory reclamation. It is only needed when explicitly expressing the intent to cut references.

A detailed opcode list and GC implementation details are covered by the AloeVM specification.

12. Limitations and Future Work

12.1 Current Limitations

  • Expressiveness of generics is minimal (advanced type inference and type classes are unspecified).
  • The module system and package management are not addressed in this document.
  • Metaprogramming features (macros, reflection, etc.) are unspecified.

12.2 Future Topics

  • Parallelization and fusion optimization of Pipe / Filter chains.
  • Integration with the Wasm component model.
  • Error handling using Result-like types (non-exception-based error handling).
  • Standard library extensions for distributed and streaming processing.
  • Unified design of delete and ownership-based resource types (if needed).

This draft focuses in particular on clarifying: the relationships among Pipe / Filter and Producer / Transformer / Consumer / Pipe / Consumer, along with their declaration forms, as well as the behavior of with / delete / lock and their relationship with AloeVM’s memory management model.
