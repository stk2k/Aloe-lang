1. Introduction

Aloe is an experimental programming language with static typing, a simple syntax, and a stack-based virtual machine (AloeVM). It is designed with a special focus on pipe / filter style dataflow and interoperability with WebAssembly.

  • Statically typed (type checking is performed at compile time)
  • C-like surface syntax with an intentionally small feature set
  • Optimized for making it pleasant to write “data flow and transformation” code such as text processing, REST calls, and logging
  • Assumes a stack-based VM (AloeVM) and future WebAssembly backends from the start

This document is a draft of the language specification.
The actual compiler / VM implementation may differ from what is described here.

2. Design Goals

2.1 Goals

  • Readability: The specification itself should be readable as documentation.
  • Implementation Simplicity: A small compiler and VM should be able to implement the language.
  • Consistency of Pipe / Filter: A type-safe dataflow expression.
  • Bridge to Wasm: Easy to connect to the future WebAssembly component model.
  • delete / with / lock are “escape hatches” for low-level resource control and special concurrency. Typical Aloe programs are assumed to be writeable without using these constructs, as long as they are written in terms of pipes.

2.2 Non-goals

  • Becoming a “do everything” general-purpose language.
  • Competing with highly optimized JIT compilers on performance.
  • Providing complex metaprogramming (macros etc.) at this stage.

3. Basic Syntax

3.1 Source Files

  • Source files are assumed to be UTF-8 encoded.
  • Single-line comments: // comment
  • Multi-line comments: /* ... */

3.2 Tokens

  • Identifiers: start with a letter or _, followed by letters, digits, or _.
  • Keywords are reserved words and cannot be used as identifiers.

3.3 Representative Keywords

CategoryKeywords
Declaration let, var, type, struct, enum, class, interface, trait, sealed
Control if, else, while, for, break, continue, return, lock
Functions method
Auxiliary with, delete, clone, where

3.4 Statements and Expressions

  • Most statements end with a semicolon ;.
  • Blocks are enclosed in { ... }.
  • Conditions in control structures must have type Bool.
// Variable declarations
let Threshold: Int = 10;
var Count: Int = 0;

// if statement
if (Count > Threshold) {
  Print("over");
} else {
  Print("ok");
}

4. Type System

4.1 Primitive Types

TypeDescription
Booltrue / false
IntSigned integer (bit width is implementation dependent; assume 32-bit here)
FloatFloating-point number (32 or 64 bits, implementation dependent)
StringUTF-8 string (internal representation is implementation dependent)

4.2 Arrays and Collections

  • Fixed-length arrays: T[]
  • Variable-length and lazily evaluated sequences are represented with Pipe<T> (see Chapter 6).

4.3 struct

struct User {
  Id: Int;
  Name: String;
  IsActive: Bool;
}
  • struct is a value type; assignment copies the value.
  • The copy is shallow; sharing of nested reference-type fields depends on the VM implementation.
  • A struct may inherit from at most one other struct (single inheritance for structs).
  • It is forbidden for a struct to implement an interface. (How structs participate in traits and generic constraints will be specified later.)

4.4 enum

  • enum is always represented internally as Int.
  • “Generic enums” such as enum<String> or enum<Byte> are forbidden.
  • Bitfield enums used for flags are limited to at most 32 flags.
enum LogLevel {
  Trace = 0;
  Debug = 1;
  Info  = 2;
  Warn  = 3;
  Error = 4;
}

4.5 Classes and Inheritance (class / interface / trait / sealed)

4.5.1 Class Basics

  • class is always a heap-allocated reference type.
  • Assignment copies the reference (shallow copy).
  • The clone keyword creates a shallow copy of the object and returns a new reference.
  • All classes are cloneable; the language does not provide a way to declare “non-cloneable” at the type level.
    If you want to express that cloning is prohibited or discouraged, do so via API design or coding conventions on top of clone. 
class User {
  Name: String;
  Age: Int;
}

var A: User = NewUser();
var B: User = A;        // reference copy (A and B refer to the same instance)

var C: User = clone A;  // shallow copy, returns a reference to a new instance

A shallow copy means “copy field values as-is”.
If a field itself is a reference type, the reference value is copied. Types that need deep copy or custom clone logic can provide user-defined APIs such as a Clone method on top of clone.

4.5.2 Class Inheritance (Single Inheritance)

Aloe class supports single inheritance only.
That is, a class may have exactly one base class. 

class Base {
  Id: Int;
}

class Derived : Base {
  Name: String;
}

// The following is a compile error (multiple inheritance of classes is forbidden)
class Bad : Base, OtherBase {
}
  • The class hierarchy forms a tree (or chain); diamond inheritance does not occur.
  • Class inheritance is a mechanism for reusing state and behavior.

4.5.3 interface and trait

Instead of allowing multiple inheritance of classes, Aloe uses interfaces and traits to compose behavior.

  • interface defines method signatures only, with no implementation (whether default implementations are allowed is future work).
  • trait is a mixin unit that can contain method implementations and is used to add behavior to existing classes.
  • A class may have one base class, implement multiple interfaces, and apply multiple traits using with. 
interface IDisposable {
  method Dispose();
}

trait Printable {
  method PrintSelf() {
    Print(this.ToString());
  }
}

class FileHandle : BaseFile implements IDisposable with Printable {
  method Dispose() {
    // resource cleanup
  }

  method ToString(): String {
    return "FileHandle(...)";
  }
}

The detailed syntax and rules for interface and trait (inheritance, conflict resolution when multiple traits define the same method, etc.) are not fully specified in this draft. Here we only state the direction.

4.5.4 Sealed Classes

To forbid inheritance from a specific class, Aloe provides the sealed modifier. 

sealed class Token {
  Kind: TokenKind;
  Text: String;
}

// This is a compile error (Token is sealed)
class KeywordToken : Token {
}
  • A sealed class cannot be used as a base class.
  • Even if a class is sealed, it may still implement interfaces and have traits applied via with.
  • In the future, we are considering combining sealed classes with unions / pattern matching to support exhaustiveness checking.

4.5.5 readonly and Classes

  • Class fields are mutable by default.
  • Attaching a readonly modifier to a field makes it immutable:
    • Assignment to that field is allowed only in field initializers and constructors of the same class.
    • Assignment from anywhere else is a compile error.
  • readonly class makes the whole class logically immutable:
    • All instance fields are implicitly treated as readonly.
    • Field assignment is allowed only in field initializers and constructors of that class.
    • After the constructor finishes, no code is allowed to mutate any field of that instance.
  • readonly and clone are orthogonal: even a readonly class can be cloned with clone. The clone result is another instance with the same immutable state.

4.5.6 Common Base Type Object and ToString / Dump

In Aloe, all reference types and most value types are conceptually considered to have a common base type Object (primitive types such as Int / Bool / Float / String may be treated specially in the implementation, but they are considered to have equivalent capabilities).

  • Object defines at least the following two methods:
    • method ToString(): String – a short human-readable string representation.
    • method Dump(): String – a detailed structured string representation for debugging.
  • User-defined struct / class types may override these methods as needed.
  • Dump is the standard hook used by the built-in consumer dump, allowing each type to customize how its structure is expanded in debug output and logs.

This section establishes a high-level policy: classes are heap reference types with single inheritance; multiple behavior is composed via interfaces and traits; sealed expresses “no further inheritance”; and readonly expresses immutability on fields or entire classes; all of this is in relation to the AloeVM’s memory model and debugging hooks such as Dump().
Details of method resolution order and trait conflict resolution will be specified separately.

4.6 Generics

4.6.1 Overview

Aloe provides a minimal generics mechanism for types, functions, methods, and pipe-related declarations.
The goal is to enable type-safe reuse for collections and Pipe / Filter constructs, not to support advanced type-level computation. 

type Box<T> {
  Value: T;
}

struct Pair<TLeft, TRight> {
  Left: TLeft;
  Right: TRight;
}

method Map<T, U>(Source: Pipe<T>, Selector: Func<T:U>): Pipe<U> {
  // Implementation omitted
}

4.6.2 Generic Type Parameters

  • Types, structs, classes, interfaces, traits, methods, and producer/filter/consumer declarations may take type parameters using <T, U, ...>.
  • Type parameter names are recommended to start with uppercase (e.g., T, TItem, TKey).
  • Type arguments are bound to concrete types at use sites.
type List<T> {
  Items: T[];
}

method FirstOrDefault<T>(Source: Pipe<T>): T {
  // Implementation omitted
}

4.6.3 where Type Constraints (Overview)

Aloe generics support arbitrary type constraints using a where clause.
The default is no constraints, but version 0.1 supports a few basic forms of constraints. 

method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  // Int-specific sum
}

Here, where exists purely for compile-time type checking.
The runtime representation (monomorphization vs. shared code with type tags, etc.) is delegated to the VM specification.

4.6.4 Supported Constraint Kinds

In version 0.1, the following constraint kinds are supported:

  • Value type / Reference type constraints
    • where T: struct – T must be a value type.
    • where T: class – T must be a reference type.
  • Primitive type constraints
    • where T: Int, where T: Float, where T: Bool, etc., i.e., primitive type names can be used directly.
  • User-defined type constraints
    • where T: SomeTrait – types that implement trait SomeTrait.
    • where T: MyStruct – type must be exactly struct MyStruct.
    • where T: MyClass – type must be exactly class MyClass.
  • Any concrete type name
    • Any existing concrete type name can be used (e.g., where T: Pipe<Int>).
// Generic restricted to value types
method Zero<T>(): T
  where T: struct
{
  // Implementation omitted
}

// Pipe specialized for Int
method Average<T>(Source: Pipe<T>): Float
  where T: Int
{
  // Implementation omitted
}

// Generic restricted to a specific struct type
method PrintUser<T>(User: T)
  where T: UserInfo
{
  // Implementation omitted
}

4.6.5 Behavior When Constraints Are Violated

  • If a type argument to a generic function/type does not satisfy the where constraints, it is a compile error.
  • If multiple constraints are specified, all of them must be satisfied simultaneously.
  • Cyclic constraints and advanced typeclass-like patterns are not supported at this time.
// Example of a compile error:

// where T: Int, so T cannot be String
method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  ...
}

var Lines: Pipe<String> = ReadLines("log.txt");
var Result = Sum(Lines);  // <-- compile error

How generics are represented internally (monomorphization vs. shared code with type tags, etc.) is defined by AloeVM.
This language specification focuses on type-safe reuse and where-based constraint checking behavior.

5. Methods and Functions

5.1 method Declarations

method Add(a: Int, b: Int): Int {
  return a + b;
}
  • method defines either a top-level function or a method bound to a type.
  • If the return type is omitted, the method is treated as having “no return value” (void).
  • Aloe does not have a Unit type; “no return value” is not represented as a type.

5.2 bound for Method Binding

In Aloe, bound is used to add methods to existing types. 

type Pipe<T> {
  // Internal representation is VM dependent
}

bound Pipe<T> {
  method Where(Predicate: Func<T:Bool>): Pipe<T> {
    // Implementation omitted
  }
}

For Pipe<T>, we introduce a restriction on methods declared in bound blocks regarding their return type (see Section 6.2 for details).

5.3 Function Types (Func<...:...>)

In Aloe, all function types are expressed using Func<...:...> notation. Function type notation using -> does not exist in the language specification.

  • Func<T:U> means “a function that takes a T and returns a U”.
  • Func<A, B:R> means “a function that takes A and B and returns R”.
  • A function with no return value (void) can be written as Func<T:> or Func<T:void>.

6. Pipe / Filter Model

The distinctive point of Aloe is that it can express pipe / filter style dataflow in a type-safe and consistent style. 

ReadLines("access.log")
  | Where (.StartsWith("[ERROR]"))
  | Map   (ParseLogLine)
  | Where (.Level == LogLevel.Error)
  | ToArray();

Conceptually, the above is equivalent to the following nested calls: 

ToArray(
  Where(
    Map(
      Where(ReadLines("access.log"), ...),
      ...),
    ...));

Aloe’s | operator is syntactic sugar for this kind of function composition.

6.1 Pipe<T> Type

Pipe<T> represents the “pipe world” in Aloe and corresponds to lazy sequences, streams, or iterators.

  • Pipe<T> represents “a pipe through which elements of type T flow”.
  • The internal representation (chained iterators, coroutines, etc.) is VM-dependent.
  • The standard library provides producers / transformers / consumers that operate on Pipe<T>.
type Pipe<T> {
  // Implementation-dependent internal fields
}

The pipe world is the domain formed by functions (producers / transformers / consumers) that take and/or return Pipe<T>.
Consumers are the exit points from the pipe world.

6.2 Producer / Transformer / Consumer

In the Pipe / Filter model, functions and objects are classified into three roles:

6.2.1 Producer

  • Definition: A function/object whose return type is Pipe<*> and which appears at the beginning of a pipeline.
  • Role: Read data from the external world (files, HTTP, arrays, etc.) and “pour” it into the pipe world as Pipe<T>.
  • Type: Typically has the shape Func<X:Pipe<T>>. Producers are always designed to output a Pipe<*>.

6.2.2 Transformer (Intermediate Filter)

  • Definition: A function/method of type Func<Pipe<T>:Pipe<U>>.
  • Role: Transform, filter, or sort data within the pipe world; does not leave the pipe world.

In Aloe, methods declared inside bound Pipe<T> are in principle reserved for transformers (intermediate filters).

Restriction:
Methods declared inside bound Pipe<T> blocks must
return some Pipe<*>.
That is, their type must be Func<Pipe<T>:Pipe<U>>. 
bound Pipe<T> {
  // OK: transformer
  method Where(Predicate: Func<T:Bool>): Pipe<T> { ... }

  // OK: change element type
  method Map<U>(Selector: Func<T:U>): Pipe<U> { ... }

  // NG: return type is not Pipe, so this must be a compile error
  // method Count(): Int { ... }
}

6.2.3 Consumer (Terminal Filter)

  • Definition: A function/object of type Func<Pipe<T>:R>, where R is not Pipe<*>.
  • Role: Serves as the exit from the pipe world by consuming the pipe completely and converting it into arrays, scalar values, or external I/O.
  • Return type: As long as R is not Pipe<*>, it is unconstrained: it may be T[], Int, Unit, Bool, etc. 
// Consumer that materializes a pipe as an array
method ToArray<T>(Source: Pipe<T>): T[] { ... }

// Consumer that counts elements
method Count<T>(Source: Pipe<T>): Int { ... }

6.3 Pipe-Related Declarations (producer / pipe / filter / consumer)

To build Pipe / Filter components, Aloe introduces four declaration forms: producer, pipe, filter, and consumer.
All of these are treated as “objects with constructors”, and their bound blocks are executed when they are connected into a pipeline.

6.3.1 producer Declaration

producer ProducerName(Params...) {
  out: Pipe<T>;

  bound(output) {
    // Executed when the producer is connected into a pipeline
    // Read from external resources and write into output
  }
}
  • ProducerName is the name of the producer.
  • out specifies the output type in the pipe world. It must be a Pipe<*> type.
  • bound(output) defines behavior when the producer is connected into a pipeline. The type of output is the type specified in out (e.g., Pipe<T>).

Constructor parameters (Params...) are accessible from inside the bound block. 

producer ReadLines(Path: String) {
  out: Pipe<String>;

  bound(output) {
    // Pseudo code:
    //   let file = OpenFile(Path);
    //   for each line in file:
    //     output.write(line);
    //   file.close();
  }
}

6.3.2 pipe Declaration

pipe PipeName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Connect pipes and perform transfer/transformation
  }
}
  • PipeName is the name of the pipe.
  • in is the type that flows in “from upstream”.
  • out is the type that flows “to downstream”.
  • bound(input, output) describes how to transform input into output.

A pipe declaration is intended for connecting to external resources (sockets, HTTP, etc.) and bridging heterogeneous streams such as text → JSON. 

pipe JsonDecode<T>() {
  in:  String;   // JSON text
  out: T;        // decoded object

  bound(input, output) {
    // Pseudo code:
    //   for each line in input:
    //     let obj: T = JsonParse<T>(line);
    //     output.write(obj);
  }
}

6.3.3 filter Declaration

filter FilterName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Define how elements flow from input to output
  }
}
  • FilterName is the name of the filter.
  • in / out specify input and output types.
  • In Pipe / Filter chains, both in and out typically take Pipe<*> types.
  • bound(input, output) describes the transformation between them.
filter Where<T>(Predicate: Func<T:Bool>) {
  in:  Pipe<T>;
  out: Pipe<T>;

  bound(input, output) {
    // Pseudo code:
    //   for each item in input:
    //     if (Predicate(item)):
    //       output.write(item);
  }
}

filter Map<T, U>(Selector: Func<T:U>) {
  in:  Pipe<T>;
  out: Pipe<U>;

  bound(input, output) {
    // Pseudo code:
    //   for each item in input:
    //     output.write(Selector(item));
  }
}

6.3.4 consumer Declaration

consumer ConsumerName(Params...) {
  in: InputType;

  bound(input) {
    // Executed at the end of the pipeline
    // Consume input and return some value
  }
}
  • ConsumerName is the name of the consumer.
  • in specifies the input type. In the Pipe / Filter model, this is usually Pipe<T>.
  • bound(input) is executed at the end of the pipeline, consumes input completely, and returns a value of some type R.
  • R may be T[], Int, Bool, Unit, etc., but must not be Pipe<*>. 
consumer ToArray<T> {
  in: Pipe<T>;

  bound(input) {
    var buffer: List<T>;
    // Pseudo code:
    //   for each item in input:
    //     buffer.add(item);
    //   return buffer.toArray();
    return buffer.toArray();
  }
}

consumer Count<T> {
  in: Pipe<T>;

  bound(input) {
    var c: Int = 0;
    // Pseudo code:
    //   for each item in input:
    //     c = c + 1;
    //   return c;
    return c;
  }
}

In summary, the roles of pipe-related declarations are:

  • producer: external world → Pipe<*> (always outputs a Pipe)
  • pipe: bridge between heterogeneous streams and external I/O
  • filter: mainly Pipe<T> → Pipe<U> transformations
  • consumer: Pipe<T> → non-Pipe type (exit from the pipe world)

6.4 Pipe Operator | and |>

6.4.1 Language-Level Definition of |

Aloe’s fundamental pipe operator is |, defined as syntactic sugar for simple function composition. 

Expr | F | G

is exactly equivalent to:

G(F(Expr))
  • If F has type Func<A:B> and G has type Func<B:C>, then the type of Expr | F | G is C.
  • This operator can be applied to any types; it is not limited to Pipe<T>.
Therefore, from the viewpoint of the type system, “having the return type change in the middle of the pipeline” is not a problem in itself.
The constraints of the Pipe / Filter model are provided by Pipe<T> and the design of pipe-related declarations in the library, not by the operator itself.

6.4.2 Relationship Between |> and | with

Aloe provides | with syntax for inserting filter objects or configured consumers into a pipeline, and defines |> as syntactic sugar for it. 

Expr |> Filter

is semantically interpreted as:

Expr | with (Filter)

Here, Filter is an already-created filter/consumer object, or a temporary object created on the spot (such as Where(.IsActive)).

In summary:

  • The pipe expression itself is always written using |.
  • |> is syntactic sugar for inserting existing objects using | with; it does not introduce a new, independent pipe operator.

6.4.3 Pipe<T> and |

ReadLines("users.json")
  | Where (.Trim().Length > 0)
  | Map   (ParseUser)
  | Where (.IsActive)
  | ToArray();
  • ReadLines: producer (Func<String:Pipe<String>>)
  • Where, Map: transformers (Func<Pipe<T>:Pipe<U>>)
  • ToArray: consumer (Func<Pipe<User>:User[]>)
  • The type of the entire pipe expression is User[], the return type of ToArray.

Note:
At the language level, it is allowed to chain | even after a consumer, but stylistically this is not recommended.
For example, patterns like ToArray() | ToArray() should be avoided.

6.5 Standard Filters and Producers

6.5.1 Representative Transformers (Filters)

filter Where<T>(Predicate: Func<T:Bool>) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

filter Map<T, U>(Selector: Func<T:U>) {
  in:  Pipe<T>;
  out: Pipe<U>;
  bound(input, output) { ... }
}

filter Take<T>(Count: Int) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

6.5.2 Representative Producers

producer FromArray<T>(Items: T[]) {
  out: Pipe<T>;
  bound(output) { ... }
}

producer ReadLines(Path: String) {
  out: Pipe<String>;
  bound(output) { ... }
}

6.5.3 Representative Consumers

consumer ToArray<T> {
  in: Pipe<T>;
  bound(input) { ... }   // Returns T[]
}

consumer Count<T> {
  in: Pipe<T>;
  bound(input) { ... }   // Returns Int
}

6.5.4 Built-In Consumer dump

Aloe provides a built-in consumer dump for debugging. dump calls the target object’s Dump() method and outputs the result to standard output or a debug console.

  • For X | dump, X’s Dump() is called once.
  • For Pipe<T> | dump, Dump() is called sequentially on each element in the pipe.
  • Default Dump() implementations for collections and tree-like objects recurse through their structure, but if a cycle is detected, traversal below that point is skipped. For collections, only the problematic element is skipped and dumping continues for subsequent elements.

dump is a debug-only tool and its return value has no semantic meaning (it may be treated as a Unit-like value in implementations). In production code, it is recommended to control the amount of output from Dump().

6.5.5 Example Usage

let ActiveUserNames: String[] =
  ReadLines("users.json")
    | Where (.Trim().Length > 0)
    | Map   (ParseUser)
    | Where (.IsActive)
    | Map   (.Name)
    | ToArray();

6.6 Thread Safety and the Pipe World

In Aloe, as long as code is written using Pipe<T> and producer/filter/consumer according to the pipeline model, user code does not need to be aware of multithreading.

  • Pipe<T> is logically treated as a single-consumer stream.
  • The standard API does not provide a way to read from the same Pipe<T> concurrently from multiple threads.
  • Public APIs of producers/filters/consumers are designed such that shared mutable state is not directly exposed to other threads.

AloeVM may parallelize internal processing of individual producers/filters/consumers at its discretion, but from the outside their behavior must be equivalent to “processed sequentially on a single thread”.

Therefore, if you write code purely in terms of Pipe<T>-based pipelines, data races due to multithreading are eliminated by the type system and library design.
delete / with / lock are provided as low-level control mechanisms for dealing with raw shared state or native resources outside the pipe world, and are not required for ordinary Pipe / Filter code.

7. with Construct

7.1 Overview

The with construct is syntactic sugar for repeatedly writing property assignments and method calls on the same target object.
In Aloe it additionally serves as a “lifetime lock” that keeps the target object alive for the duration of the block. 

struct HttpRequest {
  Url: String;
  Method: String;
  TimeoutSeconds: Int;
}

var Request: HttpRequest;

with (Request) {
  Url = "https://example.com/api";
  Method = "GET";
  TimeoutSeconds = 5;
}

In typical Aloe programs you do not need to use with explicitly; Pipe / Filter alone is enough to write safe code. with is used when you want to explicitly state that a particular object’s lifetime is guaranteed and to simplify repeated member access.

7.2 Semantics

The with block above is semantically equivalent to: 

{
  Request.Url = "https://example.com/api";
  Request.Method = "GET";
  Request.TimeoutSeconds = 5;
}
  • In with (Expr) { ... }, Expr is evaluated exactly once.
  • Simple identifier assignments inside the block are interpreted as member accesses on the target object.

7.3 Lifetime Lock (Behavior as GC Root)

AloeVM is assumed to use a combination of reference counting and sliding compaction for memory management.
Under this model, a with block acts as a “lifetime lock”:

  • The evaluation result of with (Expr) is stored in a hidden variable (GC root).
  • This hidden variable keeps the object alive until the block exits, guaranteeing that the object will not be reclaimed during the block.
  • Therefore, the object is guaranteed to be alive for the duration of the block’s execution.

7.4 Relationship Between with and delete

  • Within a with block that targets a variable, it is a compile error to call delete on that same variable.
  • The reason is that with expresses the intent “this object must be alive here”, which contradicts delete.
  • However, calling delete on a different variable that happens to reference the same object is allowed. Because the with GC root remains, the object itself stays alive. 
var Request: HttpRequest = GetRequest();
var Alias: HttpRequest = Request;

with (Request) {
  // Request is protected by with and is guaranteed to live during this block

  delete Alias;   // OK: just removes the reference from Alias
  Url = "https://example.com/api";  // still usable
}

// After leaving the with block, the GC root is removed.
// If no other references remain, the object will eventually be reclaimed.

In summary, with is both “syntactic sugar for member access” and “a mechanism that keeps the target object as a temporary GC root”.
When you want to protect the same target with a lock, you can use the lock with construct in Section 9.3.

8. delete Construct

8.1 Overview

In Aloe, delete does not directly destroy an object.
Instead, it is treated as a construct that removes a reference from a variable.

  • Actual object destruction (memory reclamation, finalizers, etc.) is performed by the runtime (AloeVM) using reference counting + GC.
  • delete is for expressing “this variable no longer refers to that object” in a type-safe way, and is used as a replacement for assigning null.

In normal code you do not need to call delete explicitly; reference destruction and memory reclamation are handled by AloeVM’s GC. delete is a construct for special cases where you want to explicitly sever a reference.

8.2 Syntax

DeleteStatement ::= 'delete' Identifier ';'

8.3 Static Rules (Compile-Time)

  • Identifier must be a variable declared in the current scope.
  • After delete, reading the variable, accessing members, or calling methods on it is a compile error.
  • However, assigning a new value to that variable is allowed. Conceptually, delete can be seen as returning the variable to an “uninitialized” state. 
var Buf: Buffer = NewBuffer();

// Use Buf
DoSomething(Buf);

delete Buf;

// After this point, reading Buf or Buf.Length is a compile error

Buf = NewBuffer();  // After re-assignment, it can be used again

8.4 Runtime Semantics

When delete Var; is executed, the VM behaves roughly as follows:

  • Decrements the reference count for the object currently referenced by Var (assuming a reference counting scheme).
  • Marks Var itself as “uninitialized/deleted”.
  • When the object’s reference count becomes 0, the object is eventually reclaimed by the GC.
    In other words, delete does not guarantee that the object is destroyed immediately.

In this language specification, the timing of object destruction is entirely the responsibility of the VM/runtime.
delete only removes a reference from one variable to that object.

8.5 Interaction with with

  • As stated in Section 7.4, in a with block targeting a given variable, calling delete on that same variable is a compile error.
  • Calling delete on a different variable that refers to the same object is allowed, because the with GC root still keeps the object alive.

In the future, if Aloe introduces “owned resource types” or an ownership model, delete may be integrated with that model.
In this draft, we describe the language under the assumption that no ownership model is present.

9. lock Construct and Synchronization

9.1 Overview

The lock construct obtains a lock on a specific object and guarantees mutual exclusion and optimized read barriers for that object during the execution of the block. 

var SharedCounter: Int = 0;

lock (SharedCounter) {
  SharedCounter = SharedCounter + 1;
}

Typical Aloe programs are self-contained within Pipe<T> and the Pipe / Filter model, and do not need to use lock or lock with. These constructs are “escape hatches” for low-level control in special cases where you must deal with shared mutable state or OS/native resources outside the pipe world.

9.2 Semantics

  • lock (Expr) { Body } acquires a lock on the object represented by Expr and guarantees exclusive access within the Body block.
  • Lock acquisition and release are implemented by the VM.
  • Updates to data shared between threads should generally be confined within lock blocks.

AloeVM is expected to use sliding compaction for memory management.
Inside a lock block, the VM may optimize read barriers and reduce per-object checks to at most once per object.

9.3 lock with Construct (lock + with Combined)

For cases where you want to apply both “mutual exclusion” and “lifetime lock (with)” to the same target object, Aloe provides the lock with construct. 

lock with (Conn) {
  // Here:
  //   - Conn is locked exclusively
  //   - Conn is kept alive as a GC root by with
  //   - Conn's members can be accessed with abbreviated syntax (RequestUrl, etc.)
}

9.3.1 Syntax

LockWithStatement ::= 'lock' 'with' '(' Expression ')' Block

lock with (Expr) { Body } is a compound construct which evaluates Expr once and treats the result as the target of both lock and with.

9.3.2 Desugaring (Semantically Equivalent Code)

lock with (Expr) { Body } is semantically equivalent to: 

{
  let __tmp = Expr;
  lock (__tmp) {
    with (__tmp) {
      Body
    }
  }
}
  • Expr is evaluated exactly once and bound to a temporary variable __tmp.
  • Then lock and with are nested on __tmp.
  • Thus, during the execution of Body:
    • Exclusive access to the target object is guaranteed (semantics of lock).
    • The target object is kept alive as a GC root (semantics of with).
    • Members of the target object can be accessed with shorthand syntax, same as with.

9.3.3 Style and Restrictions

  • lock with is syntactic sugar to express the intent “operate on a specific resource exclusively and with a guaranteed lifetime” in a single construct.
  • Other notations such as with lock (Expr) or lock & with (Expr) are not defined by the language.
    For the same purpose you must always use lock with.
  • The behavior inside a lock with block is interpreted as the combination of lock (Section 9.2) and with (Section 7.3).
var Conn: Connection = OpenConnection();

lock with (Conn) {
  Send("PING");
  var Resp: String = Receive();
  Print(Resp);
}

// At this point the lock is released and the with lifetime lock is also removed.

10. Exception Handling

10.1 try / catch / finally

try {
  DoSomething();
} catch (ex: Error) {
  LogError(ex);
} finally {
  Cleanup();
}
  • Exceptions thrown inside the try block can be caught by the corresponding catch block.
  • The finally block is always executed whether an exception is thrown or not.
  • The set of exception types and their hierarchy is defined on the VM side.

11. Execution Model and Memory Management

11.1 Overview of AloeVM

  • Stack-based bytecode VM.
  • Function calls push frames onto a call stack.
  • GC is assumed to use reference counting plus sliding compaction.

11.2 Memory Management (Overview)

  • Objects are allocated on the heap.
  • When certain conditions are met, the GC runs and slides live objects to compact memory.
  • with acts as a temporary GC root that keeps objects alive, and combining it with lock enables read-barrier optimization.
  • In general, you do not need to call delete for memory management. Use delete only when you explicitly want to sever references.

Detailed opcode lists and specific GC implementation details are provided in the AloeVM specification.

12. Limitations and Future Work

12.1 Current Limitations

  • Expressiveness of generics is minimal; typeclasses and higher-kinded generics are not defined.
  • The module system and package management are outside the scope of this document.
  • Metaprogramming (macros, reflection, etc.) is not defined.

12.2 Future Work

  • Parallelization and fusion optimizations of Pipe / Filter pipelines.
  • Integration with the Wasm component model.
  • Introduction of error handling mechanisms such as a Result type in addition to exceptions.
  • Standard library extensions for distributed and streaming workloads.
  • Integration of delete with owned resource types and ownership models, if introduced.
  • Integration of traits / interfaces / sealed / readonly with pattern matching and exhaustiveness checking.

In this draft, we particularly focus on clarifying:
the relationship between Pipe / Filter and the Producer / Transformer / Consumer / Pipe / Consumer declarations, and
the relationship between with / delete / lock / class inheritance / readonly / clone / generics and the AloeVM memory model.

13. event Construct and Event Model

13.1 Purpose and Positioning

Aloe’s event construct expresses, at the language level, the contract “this is a notification point where external code can register handlers.”

  • External code can register/unregister handlers (callbacks) to the event.
  • Raising (firing) the event is, in principle, only allowed by the side that declares the event.

At the implementation level, event is treated as syntactic sugar for the standard library type Event<TArgs>.

  • event Clicked<ClickedArgs>; is equivalent to having a field of type Event<ClickedArgs>.
  • However, unlike a normal field, it forbids external code from reassigning it or calling Raise directly (compile error).

13.2 Event<TArgs> and Handler Types

13.2.1 Handler Type

The function type for event handlers is defined as a standard type alias: 

type EventHandler<TArgs> = Func<TArgs:Task>;
  • Argument: TArgs (type of event arguments)
  • Return type: Task
    To support async handlers naturally, event handlers are modeled as “functions that return a Task representing completion.”

In the future, we may consider an extension such as Event<TArgs, TResult> for handlers of type Func<TArgs:Task<TResult>>, but in version 0.1 we standardize on Task return as the basis.

13.2.2 Role of Event<TArgs> (Concept)

Event<TArgs> is a type that holds a collection of event handlers and the logic to invoke them as a group. 

type Event<TArgs> {
  Handlers: List<EventHandler<TArgs>>;
}

bound Event<TArgs> {
  method Add(Handler: EventHandler<TArgs>): Task {
    Handlers.Add(Handler);
  }

  method Remove(Handler: EventHandler<TArgs>): Task {
    // Details of handler identity comparison are delegated to the VM / runtime specification
    Handlers.Remove(Handler);
  }

  method Raise(Args: TArgs): Task {
    // Await each handler in order and return a Task that completes after all are done
    foreach (H in Handlers) {
      await H(Args);
    }
  }
}

In the language specification, we only define that Event<TArgs> is the “event collection” entity and event is syntactic sugar plus access-control rules on top of it.

13.3 Syntax of event Declarations

13.3.1 Declaration Locations

event can be declared in:

  • struct bodies
  • class bodies
  • interface bodies

Declaration at the top level (file scope) or inside trait is not defined in version 0.1.

13.3.2 Syntax (Overview)

EventDeclaration ::=
    'event' Identifier '<' Type '>' ';'

Example:

struct Button {
  event Clicked<ClickedArgs>;
}

The above declaration is semantically equivalent to the following field declaration: 

struct Button {
  private Clicked: Event<ClickedArgs>;
}

However, unlike a normal field, the following additional restrictions apply:

  • External code cannot reassign Clicked (compile error).
  • External code cannot directly call Clicked.Raise(...).
  • External code cannot delete Clicked;.

13.4 Subscription / Unsubscription Syntax (+= / -=)

13.4.1 Syntax

EventAddStatement    ::= PrimaryExpression '.' Identifier '+=' EventHandlerExpression ';'
EventRemoveStatement ::= PrimaryExpression '.' Identifier '-=' EventHandlerExpression ';'
  • PrimaryExpression.Identifier must be an event-declared member.
  • EventHandlerExpression must be an expression implicitly convertible to Func<TArgs:Task>.

13.4.2 Type Rules

When event X<TArgs>; is declared:

  • The type of Target.X is Event<TArgs>.
  • Target.X += H; is allowed only if H has type Func<TArgs:Task> (or is implicitly convertible to it).
  • The same constraints apply to Target.X -= H;.

13.4.3 Desugaring (Semantically Equivalent Code)

Semantically, these are interpreted as:

Target.X += H;
// is equivalent to:
await Target.X.Add(H);
Target.X -= H;
// is equivalent to:
await Target.X.Remove(H);

By making Add / Remove return Task, subscriptions/unsubscriptions via += / -= can themselves be treated as async statements. In future versions we may consider fire-and-forget variations if needed.

13.5 Rules for Raising Events (Raise)

13.5.1 Only the Defining Side May Raise

Raise calls on event-declared members are only allowed from:

  • Methods of the struct / class that declares the event
  • bound blocks for that same type

Calls such as Clicked.Raise(...) from other code (external modules or other classes) are compile errors.

This guarantees a separation of responsibilities: “the side that exposes an event alone has the authority to fire it.”

13.5.2 Basic Form of Raising

struct ClickedArgs {
  ButtonId: String;
}

struct Button {
  event Clicked<ClickedArgs>;
}

bound Button {
  method OnNativeClick(Id: String): Task {
    let Args: ClickedArgs = { ButtonId = Id };
    await Clicked.Raise(Args);
  }
}

13.6 event in interface and Implementation

interface may also declare event.

struct ClickedArgs {
  ButtonId: String;
}

interface IButton {
  event Clicked<ClickedArgs>;
}

In this case, any struct / class that implements IButton must declare the corresponding event. 

struct Button implements IButton {
  event Clicked<ClickedArgs>;
}
  • If the event name Clicked does not match, or
  • If the type argument ClickedArgs differs,

then it is a compile error as an incomplete interface implementation.

13.7 Examples

13.7.1 Button and Clicked Event

struct ClickedArgs {
  ButtonId: String;
}

struct Button {
  event Clicked<ClickedArgs>;
}

bound Button {
  method OnNativeClick(Id: String): Task {
    let Args: ClickedArgs = { ButtonId = Id };
    await Clicked.Raise(Args);
  }
}

method SetupButton(): Task {
  var Btn: Button = CreateButton("OK");

  // Register async handler
  Btn.Clicked += async (Args: ClickedArgs): Task {
    Print("Clicked: " + Args.ButtonId);
  };

  // Register another handler
  Btn.Clicked += async (Args: ClickedArgs): Task {
    Log("button clicked");
  };
}

13.7.2 Unsubscribing from Events

method SetupAndTearDown(Btn: Button): Task {
  let Handler: Func<ClickedArgs:Task> =
    async (Args: ClickedArgs): Task {
      Print("temp handler");
    };

  // Subscribe
  Btn.Clicked += Handler;

  // Some work ...

  // Unsubscribe
  Btn.Clicked -= Handler;
}

13.8 Combining with with / lock (Reference)

Since Event<TArgs> is itself a normal reference type, you can combine it with with or lock with to perform thread-safe raising if necessary. 

struct Hub {
  event MessageReceived<String>;
}

bound Hub {
  method RaiseMessage(Msg: String): Task {
    lock with (MessageReceived) {
      await MessageReceived.Raise(Msg);
    }
  }
}

13.9 Summary

  • event provides syntactic sugar and access-control rules on top of Event<TArgs>.
  • External code can only subscribe/unsubscribe handlers via += / -=; it cannot fire the event, reassign it, or delete it.
  • Handlers are standardized to type Func<TArgs:Task>, forming an event model that assumes async by default.
  • interface may also declare event, expressing event contracts at the type level.
  • This design provides a C#-like event experience while avoiding the need to add a dedicated delegate type to the language, instead leveraging Aloe’s function types and Task directly.
