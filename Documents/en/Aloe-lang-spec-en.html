1. Introduction

Aloe is an experimental programming language with static typing, simple syntax, and a stack-based virtual machine (AloeVM). It is designed with a strong focus on Pipe / Filter style dataflow and good interoperability with WebAssembly.

  • Statically typed (type checking at compile-time)
  • Syntax is C-like, but the feature set is deliberately small
  • Designed to write “data flows and transformations” comfortably: text processing, REST calls, log processing, etc.
  • Implementation assumes a stack-based VM (AloeVM) and a future WebAssembly backend

This document is a draft language specification. There may be discrepancies with the actual compiler / VM implementation.

2. Design Goals

2.1 Goals

  • Readability: The spec should itself be readable as documentation.
  • Implementability: A small compiler and VM should be able to implement it.
  • Consistency for Pipe / Filter: Type-safe expression of dataflow.
  • Bridge to Wasm: Easy to connect with the Wasm component model in the future.

2.2 Non-goals

  • Being a “do everything” general-purpose language.
  • Competing on performance with highly optimized JIT compilers.
  • Complex metaprogramming (macros, etc.) is out of scope for now.

3. Basic Syntax

3.1 Source Files

  • Source files are assumed to be UTF-8 encoded.
  • Single-line comment: // comment
  • Multi-line comment: /* ... */

3.2 Tokens

  • Identifiers: Start with a letter or _, followed by letters, digits, or _.
  • Keywords are reserved words and cannot be used as identifiers.

3.3 Representative Keywords

CategoryKeywords
Declarations let, var, type, struct, enum, class, interface, trait, sealed
Control if, else, while, for, break, continue, return, lock
Functions method
Auxiliary with, delete, clone, where

3.4 Statements and Expressions

  • Statements generally end with a semicolon ;.
  • Blocks are enclosed with { ... }.
  • Conditions in control structures must be of type Bool.
// variable declarations
let Threshold: Int = 10;
var Count: Int = 0;

// if-statement
if (Count > Threshold) {
  Print("over");
} else {
  Print("ok");
}

4. Type System

4.1 Primitive Types

TypeDescription
Booltrue / false
IntSigned integer (bit-width implementation-dependent; 32 bits assumed)
FloatFloating-point number (32 or 64 bits, implementation-dependent)
StringUTF-8 string (internal representation implementation-dependent)

4.2 Arrays and Collections

  • Fixed-size array: T[]
  • Variable-size and lazy sequences use Pipe<T> (details in section 6).

4.3 Structs

struct User {
  Id: Int;
  Name: String;
  IsActive: Bool;
}
  • struct is a value type; assignment copies the value.
  • Copy semantics are shallow; sharing of nested reference types is VM-defined.

4.4 Enums

  • enum is always represented internally as an Int.
  • “Generic enums” such as enum<String> or enum<Byte> are prohibited.
  • Bitflag enums are limited to at most 32 flags.
enum LogLevel {
  Trace = 0;
  Debug = 1;
  Info  = 2;
  Warn  = 3;
  Error = 4;
}

4.5 Classes and Inheritance (class / interface / trait / sealed)

4.5.1 Basics of Classes

  • class is always a heap-allocated reference type.
  • Assignment performs a reference copy (shallow copy).
  • The clone keyword creates a shallow copy of an object and returns a new reference.
class User {
  Name: String;
  Age: Int;
}

var A: User = NewUser();
var B: User = A;        // reference copy (A and B point to the same instance)

var C: User = clone A;  // creates a new instance (shallow copy) and returns its reference

A shallow copy means that field values are copied as-is. If a field itself is a reference type, the reference value is copied. Types that require deep copy or special cloning semantics may later use user-defined Clone methods or similar mechanisms.

4.5.2 Class Inheritance (Single Inheritance)

Aloe class supports single inheritance only. That is, a class may specify exactly one base class. 

class Base {
  Id: Int;
}

class Derived : Base {
  Name: String;
}

// The following is a compile-time error (class multiple inheritance is forbidden)
class Bad : Base, OtherBase {
}
  • Class hierarchies form tree (or chain) structures; diamond inheritance does not occur.
  • Class inheritance is intended for reuse of state and behavior.

4.5.3 interface and trait

Instead of supporting multiple inheritance for classes, Aloe uses interface and trait to compose behavior.

  • interface defines method signatures as contracts and has no implementation (whether default implementations are allowed is future work).
  • trait is a mixin unit that can contain method implementations; it is used to add behavior to existing classes.
  • A class may have one base class, and may implement multiple interfaces and apply multiple traits with with. 
interface IDisposable {
  method Dispose(): Unit;
}

trait Printable {
  method PrintSelf(): Unit {
    Print(this.ToString());
  }
}

class FileHandle : BaseFile implements IDisposable with Printable {
  method Dispose(): Unit {
    // resource cleanup
  }

  method ToString(): String {
    return "FileHandle(...)";
  }
}

Detailed syntax and rules for interface and trait (inheritance, conflict resolution when multiple traits define the same method, etc.) are not fully specified in this draft; here we only define the general direction.

4.5.4 sealed Classes

To forbid inheritance from a specific class, Aloe provides the sealed modifier. 

sealed class Token {
  Kind: TokenKind;
  Text: String;
}

// This is a compile-time error (Token is sealed)
class KeywordToken : Token {
}
  • A sealed class must not be used as a base class.
  • Even if sealed, a class may still implement interfaces and use traits via with.
  • In the future, we may connect sealed classes with union/pattern matching to support exhaustiveness checking.

4.5.5 readonly and Classes

  • Class fields are mutable by default.
  • A readonly modifier on a field makes that field immutable (assignments are restricted to constructor or initialization context; details TBD).
  • Introducing readonly class as a way to make entire classes immutable is a candidate for future design.

This section sets high-level policies: classes are heap-allocated reference types, support single inheritance only, use interface + trait for multiple behavior composition, and sealed forbids inheritance. Detailed method resolution order and trait conflict handling will be specified separately.

4.6 Generics

4.6.1 Overview

Aloe provides a minimal generics mechanism for types, functions, methods, and Pipe-related declarations. The goal is to make collections and Pipe / Filter reusable in a type-safe way, not to support advanced type-level computation. 

type Box<T> {
  Value: T;
}

struct Pair<TLeft, TRight> {
  Left: TLeft;
  Right: TRight;
}

method Map<T, U>(Source: Pipe<T>, Selector: T -> U): Pipe<U> {
  // implementation omitted
}

4.6.2 Generic Type Parameters

  • Types, structs, classes, interfaces, traits, methods, and producer/filter/consumer declarations may take type parameters using <T, U, ...>.
  • Type parameter names should generally start with an uppercase letter (T, TItem, TKey, etc.).
  • Generic arguments are bound to concrete types at usage sites.
type List<T> {
  Items: T[];
}

method FirstOrDefault<T>(Source: Pipe<T>): T {
  // implementation omitted
}

4.6.3 where Type Constraints (Overview)

Aloe generics support optional type constraints using the where clause. Unconstrained generics are the default, but version 0.1 already supports some basic constraints. 

method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  // Summation specialized to Int
}

Here, where is purely a compile-time type check. The runtime representation (monomorphization vs shared code with type tags, etc.) is left to the VM specification.

4.6.4 Supported Constraint Kinds

For version 0.1, the following constraints are supported:

  • Value vs Reference type constraints
    • where T: struct – T must be a value type.
    • where T: class – T must be a reference type.
  • Primitive type constraints
    • where T: Int, where T: Float, where T: Bool, etc., naming primitive types directly.
  • User-defined type constraints
    • where T: SomeTrait – types that implement trait SomeTrait.
    • where T: MyStruct – types that are exactly struct MyStruct.
    • where T: MyClass – types that are exactly class MyClass.
  • Arbitrary concrete type names
    • You may use any existing concrete type name directly, e.g. where T: Pipe<Int>.
// Value-type-only generic
method Zero<T>(): T
  where T: struct
{
  // implementation omitted
}

// Int-only Pipe
method Average<T>(Source: Pipe<T>): Float
  where T: Int
{
  // implementation omitted
}

// Specialization for a specific struct type
method PrintUser<T>(User: T)
  where T: UserInfo
{
  // implementation omitted
}

4.6.5 Behavior on Constraint Violation

  • If a generic function or type is used with a type argument that does not satisfy its where constraints, this is a compile-time error.
  • If multiple constraints are specified, all of them must be satisfied.
  • Cyclic constraints or advanced typeclass-like constructs are not supported at this time.
// Example of compile-time error:

// where T: Int, so T cannot be String
method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  ...
}

var Lines: Pipe<String> = ReadLines("log.txt");
var Result = Sum(Lines);  // <-- compile-time error

The internal representation of generics (monomorphization vs code sharing with type tags, etc.) is defined by AloeVM; this language spec focuses on type-safe reuse and the behavior of where-based constraint checking.

5. Functions and Methods

5.1 method Declarations

method Add(a: Int, b: Int): Int {
  return a + b;
}
  • method defines either a top-level function or a method bound to a type.
  • If there is no return value, the method is considered to return Unit.

5.2 bound for Method Binding

Aloe uses bound to attach methods to existing types. 

type Pipe<T> {
  // internal representation is VM-dependent
}

bound Pipe<T> {
  method Where(Predicate: T -> Bool): Pipe<T> {
    // implementation omitted
  }
}

For Pipe<T>, this spec introduces a restriction on return types of methods declared in bound blocks (see section 6.2).

6. Pipe / Filter Model

Aloe’s distinguishing feature is its ability to express Pipe / Filter dataflow in a type-safe and uniform style. 

ReadLines("access.log")
  |> Where (.StartsWith("[ERROR]"))
  |> Map   (ParseLogLine)
  |> Where (.Level == LogLevel.Error)
  |> ToArray();

Conceptually, the above is equivalent to: 

ToArray(
  Where(
    Map(
      Where(ReadLines("access.log"), ...),
      ...),
    ...));

Aloe’s |> operator is syntactic sugar for such function composition.

6.1 Pipe<T> Type

Pipe<T> represents the “pipe world” in Aloe: it is analogous to lazy sequences, streams, or iterators.

  • Pipe<T> represents a “pipe through which elements of type T flow”.
  • Internal representation (iterator chains, coroutines, etc.) is VM-dependent.
  • The standard library provides producers / transformers / consumers based on Pipe<T>.
type Pipe<T> {
  // Implementation-dependent internal fields
}

The pipe world is the realm of functions that take and return Pipe<T> (producers / transformers / consumers). Consumers are the points where we leave this world.

6.2 Producer / Transformer / Consumer

In the Pipe / Filter model, we classify functions or objects into three roles:

6.2.1 Producer

  • Definition: Functions/objects whose return type is Pipe<*> and which appear at the head of a pipeline.
  • Role: Read data from the external world (files, HTTP, arrays, etc.) and “push” it into the pipe world (Pipe<T>).
  • Type: They have the form X -> Pipe<T>. Producers are designed so they always output Pipe<*>.

6.2.2 Transformer (Intermediate Filter)

  • Definition: Functions/methods of type Pipe<T> -> Pipe<U>.
  • Role: Transform, filter, or sort data within the pipe world, without leaving it.

In Aloe, methods defined in bound Pipe<T> are, in principle, reserved for transformers (intermediate filters).

Restriction:
Methods declared in a bound Pipe<T> block must always return a Pipe<*>.
In other words, they must have the form Pipe<T> -> Pipe<U>. 
bound Pipe<T> {
  // OK: transformer
  method Where(Predicate: T -> Bool): Pipe<T> { ... }

  // OK: changes element type
  method Map<U>(Selector: T -> U): Pipe<U> { ... }

  // NG: return type is not Pipe; should be a compile-time error
  // method Count(): Int { ... }
}

6.2.3 Consumer (Terminal Filter)

  • Definition: Functions/objects of type Pipe<T> -> R where R is not Pipe<*>.
  • Role: Gateways out of the pipe world: they consume the pipe entirely and turn it into arrays, scalar values, or external I/O effects.
  • Return Type: There is no restriction on R other than it must not be Pipe<*>. It may be T[], Int, Unit, etc. 
// pure consumer that materializes a pipe as an array
method ToArray<T>(Source: Pipe<T>): T[] { ... }

// consumer that counts the number of elements
method Count<T>(Source: Pipe<T>): Int { ... }

6.3 Pipe-related Declarations (producer / pipe / filter / consumer)

Aloe introduces four declaration forms for building Pipe / Filter components: producer, pipe, filter, and consumer. All of them are “objects with constructors”, whose bound blocks are executed when they are attached to a pipeline.

6.3.1 producer Definition

producer ProducerName(Params...) {
  out: Pipe<T>;

  bound(output) {
    // Executed when the producer is attached to a pipeline
    // Reads data from external resources and writes to output
  }
}
  • ProducerName is the name of the producer.
  • out specifies the output type of the pipe world. It must be a Pipe<*> type.
  • bound(output) describes what happens when attached to a pipeline. The type of output is the type given in out (e.g. Pipe<T>).

Constructor parameters (Params...) are visible within the bound block. 

producer ReadLines(Path: String) {
  out: Pipe<String>;

  bound(output) {
    // Pseudo:
    //   let file = OpenFile(Path);
    //   for each line in file:
    //     output.write(line);
    //   file.close();
  }
}

6.3.2 pipe Definition

pipe PipeName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Connects pipes and performs forwarding / transformation
  }
}
  • PipeName is the name of the pipe.
  • in is the type coming from “upstream”.
  • out is the type flowing “downstream”.
  • bound(input, output) describes how data is passed from input to output.

Pipe definitions are intended for connecting external resources (sockets, HTTP, etc.) or bridging between different stream types (e.g. text → JSON). 

pipe JsonDecode<T>() {
  in:  String;   // JSON text
  out: T;        // decoded object

  bound(input, output) {
    // Pseudo:
    //   for each line in input:
    //     let obj: T = JsonParse<T>(line);
    //     output.write(obj);
  }
}

6.3.3 filter Definition

filter FilterName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Defines how elements are moved from input to output
  }
}
  • FilterName is the filter name.
  • in / out specify input and output types.
  • For intermediate filters in a Pipe / Filter chain, both in and out are usually Pipe<*>.
  • bound(input, output) describes the transformation between the two.
filter Where<T>(Predicate: T -> Bool) {
  in:  Pipe<T>;
  out: Pipe<T>;

  bound(input, output) {
    // Pseudo:
    //   for each item in input:
    //     if (Predicate(item)):
    //       output.write(item);
  }
}

filter Map<T, U>(Selector: T -> U) {
  in:  Pipe<T>;
  out: Pipe<U>;

  bound(input, output) {
    // Pseudo:
    //   for each item in input:
    //     output.write(Selector(item));
  }
}

6.3.4 consumer Definition

consumer ConsumerName(Params...) {
  in: InputType;

  bound(input) {
    // Executed at the end of the pipeline
    // Consumes input and returns some value
  }
}
  • ConsumerName is the consumer name.
  • in specifies the input type. In the Pipe / Filter model, this is usually Pipe<T>.
  • bound(input) is executed at the pipeline end, consumes input completely and returns a value R of any type.
  • R may be T[], Int, Bool, Unit, etc., but must not be Pipe<*>. 
consumer ToArray<T> {
  in: Pipe<T>;

  bound(input) {
    var buffer: List<T>;
    // Pseudo:
    //   for each item in input:
    //     buffer.add(item);
    //   return buffer.toArray();
    return buffer.toArray();
  }
}

consumer Count<T> {
  in: Pipe<T>;

  bound(input) {
    var c: Int = 0;
    // Pseudo:
    //   for each item in input:
    //     c = c + 1;
    //   return c;
    return c;
  }
}

In summary, the roles of Pipe-related declarations are:

  • producer: external world → Pipe<*> (always outputs a Pipe).
  • pipe: bridges between heterogeneous streams and/or external I/O.
  • filter: primarily Pipe<T> -> Pipe<U> transformations.
  • consumer: Pipe<T> → arbitrary non-Pipe type (exit from the pipe world).

6.4 Pipe Operator |>

6.4.1 Language-Level Definition

Aloe’s |> is defined as syntactic sugar for plain function composition. 

Expr |> F |> G

is exactly equivalent to:

G(F(Expr))
  • If F has type A -> B and G has type B -> C, then Expr |> F |> G has type C.
  • The operator applies to any types; it is not limited to Pipe<T>.
Thus, from the perspective of the type system, “changing the return type mid-chain” is not a problem at all.
Constraints for the Pipe / Filter model are provided by library design (Pipe<T> and Pipe-related declarations), not by the operator itself.

6.4.2 Using Pipe<T> with |>

ReadLines("users.json")
  |> Where (.Trim().Length > 0)
  |> Map   (ParseUser)
  |> Where (.IsActive)
  |> ToArray();
  • ReadLines: producer (String -> Pipe<String>)
  • Where, Map: transformer (Pipe<T> -> Pipe<U>)
  • ToArray: consumer (Pipe<User> -> User[])
  • The overall type of the pipe expression is the return type of ToArray, i.e. User[].

Note:
From a language perspective, it is legal to keep chaining after a consumer, but stylistically this is rarely recommended.
For example, ToArray() |> ToArray() is a pattern best avoided.

6.5 Standard Filters and Producers

6.5.1 Typical transformers (filters)

filter Where<T>(Predicate: T -> Bool) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

filter Map<T, U>(Selector: T -> U) {
  in:  Pipe<T>;
  out: Pipe<U>;
  bound(input, output) { ... }
}

filter Take<T>(Count: Int) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

6.5.2 Typical producers

producer FromArray<T>(Items: T[]) {
  out: Pipe<T>;
  bound(output) { ... }
}

producer ReadLines(Path: String) {
  out: Pipe<String>;
  bound(output) { ... }
}

6.5.3 Typical consumers

consumer ToArray<T> {
  in: Pipe<T>;
  bound(input) { ... }   // returns T[]
}

consumer Count<T> {
  in: Pipe<T>;
  bound(input) { ... }   // returns Int
}

6.5.4 Usage Example

let ActiveUserNames: String[] =
  ReadLines("users.json")
    |> Where (.Trim().Length > 0)
    |> Map   (ParseUser)
    |> Where (.IsActive)
    |> Map   (.Name)
    |> ToArray();

7. with Syntax

7.1 Overview

The with syntax is sugar for a block that repeatedly accesses the same target object for property assignments and method calls. In Aloe, it also serves as a “lifetime lock” that keeps the target object alive for the duration of the block. 

struct HttpRequest {
  Url: String;
  Method: String;
  TimeoutSeconds: Int;
}

var Request: HttpRequest;

with (Request) {
  Url = "https://example.com/api";
  Method = "GET";
  TimeoutSeconds = 5;
}

7.2 Semantics

The with block above is semantically equivalent to: 

{
  Request.Url = "https://example.com/api";
  Request.Method = "GET";
  Request.TimeoutSeconds = 5;
}
  • Expr in with (Expr) { ... } is evaluated exactly once.
  • Simple identifier assignments within the block are treated as member accesses on the target object.

7.3 Lifetime Lock (GC Root Behavior)

AloeVM assumes reference counting plus sliding compaction for memory management. Under this model, a with block acts as a “lifetime lock”:

  • The result of with (Expr) is stored in a hidden variable (a GC root).
  • This hidden variable prevents the object from being collected until the block is exited.
  • Thus, the object is guaranteed to be alive throughout the block.

7.4 with and delete

  • It is a compile-time error to use delete on the same variable that is the target of with inside that with block.
  • The reason: with expresses the intent “this object should stay alive here”, which conflicts with deleting it.
  • However, deleting another variable that refers to the same object is allowed (the with GC root still keeps the object alive).
var Request: HttpRequest = GetRequest();
var Alias: HttpRequest = Request;

with (Request) {
  // Request is protected by with; it is guaranteed to be alive in this block

  delete Alias;   // OK: only removes Alias's reference
  Url = "https://example.com/api";  // still usable
}

// Leaving the with block removes the GC root; if no other references remain,
// the object will eventually be collected.

In short, with acts as both “syntactic sugar for member access” and “a temporary GC root that keeps the object alive”.

8. delete Syntax

8.1 Overview

In Aloe, delete does not directly destroy the object. Instead, it is syntax for removing a reference from a variable.

  • Actual object destruction (memory reclamation, finalizers, etc.) is handled by the runtime (AloeVM) using reference counting + GC.
  • delete expresses the intent “this variable will no longer refer to this object” in a type-safe way, and serves as a replacement for null assignment.

8.2 Grammar

DeleteStatement ::= 'delete' Identifier ';'

8.3 Static Rules (Compile-time)

  • Identifier must be a variable declared in the current scope.
  • After a delete, it is a compile-time error to read from that variable or to access its members or call methods on it.
  • However, you may reassign a new value to the variable. Conceptually, delete returns the variable to an “uninitialized” state. 
var Buf: Buffer = NewBuffer();

// use Buf
DoSomething(Buf);

delete Buf;

// After this point, reading Buf or Buf.Length is a compile-time error.

Buf = NewBuffer();  // After reassignment, Buf can be used again.

8.4 Runtime Semantics

When delete Var; executes, the VM behaves roughly as follows:

  • Decrement the reference count for the object that Var currently points to (assuming reference counting).
  • Mark Var itself as “uninitialized / deleted”.
  • When the reference count of an object reaches zero, it is collected at the GC’s discretion. The object is not necessarily destroyed immediately at the delete site.

In this language spec, the timing of object destruction is the responsibility of the VM/runtime. delete only removes one variable’s reference to the object.

8.5 Interaction with with

  • As stated in section 7.4, it is a compile-time error to delete the variable that is the target of a with within that with block.
  • Deleting another variable that refers to the same object is allowed, because with still holds a GC root for that object.

In the future, if Aloe introduces “owned resource types” or an ownership model, delete may integrate with that model. This draft, however, is written under the assumption that there is no ownership model.

9. lock Syntax and Synchronization

9.1 Overview

The lock syntax locks a specific object and provides mutual exclusion and read-barrier optimization for the duration of the lock block. 

var SharedCounter: Int = 0;

lock (SharedCounter) {
  SharedCounter = SharedCounter + 1;
}

9.2 Semantics

  • lock (Expr) { Body } acquires a lock on the object denoted by Expr and guarantees exclusive access within the block.
  • Lock acquisition and release are implemented by the VM.
  • Updates to shared data between threads should generally be confined to lock blocks.

AloeVM assumes sliding-compaction memory management. Inside a lock block, the VM may optimize read barriers so that each object is checked at most once where feasible.

10. Exception Handling

10.1 try / catch / finally

try {
  DoSomething();
} catch (ex: Error) {
  LogError(ex);
} finally {
  Cleanup();
}
  • Exceptions thrown in the try block can be caught in the corresponding catch.
  • The finally block executes regardless of whether an exception is thrown.
  • The details of exception types and hierarchies are defined in the VM specification.

11. Execution Model and Memory Management

11.1 Overview of AloeVM

  • Stack-based bytecode VM.
  • Function calls push frames onto a call stack.
  • GC is assumed to use reference counting plus sliding compaction.

11.2 Memory Management (Overview)

  • Objects are allocated on the heap.
  • When certain conditions are met, GC runs, slides live objects, and compacts memory.
  • with acts as a temporary GC root to keep objects alive, and in combination with lock it allows optimizing read barrier checks.
  • Programmers usually do not need to call delete for memory management. It is only needed when explicitly expressing the intent to remove a reference.

A detailed opcode list and GC implementation details are provided in the AloeVM specification.

12. Limitations and Future Work

12.1 Current Limitations

  • Expressiveness of generics is minimal (typeclasses and higher-order generics are not defined).
  • Module system and package management are out of scope for this document.
  • Metaprogramming (macros, reflection, etc.) is not defined.

12.2 Future Work

  • Parallelization and fusion optimizations for Pipe / Filter pipelines.
  • Integration with the Wasm component model.
  • Error handling using Result-like types (in addition to exceptions).
  • Standard library extensions for distributed and streaming workloads.
  • Integration of delete with owned-resource types if needed.
  • Integration of trait / interface / sealed with pattern matching and exhaustiveness checking.

This draft especially clarifies the relationship between Pipe / Filter and Producer / Transformer / Consumer / Pipe / Consumer, as well as their declaration forms, and the behavior of with / delete / lock / class inheritance / generics and their connection to AloeVM’s memory model.
