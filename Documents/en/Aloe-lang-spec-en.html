1. Introduction

Aloe is an experimental programming language with static typing, simple syntax, and a stack-based virtual machine (AloeVM). It is designed with a strong focus on Pipe / Filter style dataflow and interoperability with WebAssembly.

  • Statically typed (type checking is done at compile time).
  • C-like surface syntax, with an intentionally small set of features.
  • Optimized for writing data flows and transformations comfortably: text processing, REST calls, logging, etc.
  • Assumes a stack-based VM (AloeVM) and future WebAssembly backend.

This document is a draft language specification.
It may differ from any actual compiler/VM implementation.

2. Design Goals

2.1 Goals

  • Readability: the specification itself should read like documentation.
  • Ease of implementation: implementable with a small compiler and VM.
  • Consistency of Pipe / Filter: type-safe dataflow representation.
  • Bridge to Wasm: easy integration with future Wasm component model.

2.2 Non-goals

  • Becoming a “do everything” general-purpose language.
  • Competing on performance with highly optimized JIT compilers.
  • Supporting complex metaprogramming (macros, etc.) in this version.

3. Basic Syntax

3.1 Source Files

  • Source files are assumed to be encoded in UTF-8.
  • Single-line comment: // comment
  • Multi-line comment: /* ... */

3.2 Tokens

  • Identifiers: first character is a letter or _, followed by letters, digits, or _.
  • Keywords are reserved words and cannot be used as identifiers.

3.3 Representative Keywords

CategoryKeywords
Declarations let, var, type, struct, enum, class, interface, trait, sealed
Control if, else, while, for, break, continue, return, lock
Functions method
Auxiliary with, delete, clone, where

3.4 Statements and Expressions

  • Most statements end with a semicolon ;.
  • Blocks are delimited by { ... }.
  • Conditions in control statements must have type Bool.
// Variable declarations
let Threshold: Int = 10;
var Count: Int = 0;

// if statement
if (Count > Threshold) {
  Print("over");
} else {
  Print("ok");
}

4. Type System

4.1 Primitive Types

TypeDescription
Booltrue / false
IntSigned integer (bit width is implementation-defined, assume 32-bit here).
FloatFloating point (32 or 64 bit, implementation-defined).
StringUTF-8 string (internal representation is implementation-defined).

4.2 Arrays and Collections

  • Fixed-length array: T[]
  • For variable-length and lazy sequences, use Pipe<T> (see section 6).

4.3 struct

struct User {
  Id: Int;
  Name: String;
  IsActive: Bool;
}
  • struct is a value type; assignment copies the value.
  • The copy is shallow; sharing of nested reference-type fields depends on the VM implementation.
  • A struct may inherit from at most one other struct (single inheritance for structs).
  • A struct is not allowed to implement interface. (How structs may participate in traits or generic constraints will be specified in later revisions.)

4.4 enum

  • enum is always represented internally as Int.
  • “Generic enums” such as enum<String> or enum<Byte> are forbidden.
  • Bit-flag enums are limited to at most 32 flags.
enum LogLevel {
  Trace = 0;
  Debug = 1;
  Info  = 2;
  Warn  = 3;
  Error = 4;
}

4.5 Classes and Inheritance (class / interface / trait / sealed)

4.5.1 Basics of Classes

  • class is always a heap-allocated reference type.
  • Assignment copies the reference (shallow copy).
  • clone creates a shallow copy of an object and returns a new reference.
  • All classes are cloneable; there is no way to declare “non-cloneable” at the type level.
    If cloning needs to be discouraged or controlled, that should be expressed via API design or coding conventions. 
class User {
  Name: String;
  Age: Int;
}

var A: User = NewUser();
var B: User = A;        // reference copy (A and B refer to the same instance)

var C: User = clone A;  // shallow copy: new instance with copied fields

A shallow copy means “copy field values as-is”.
If a field itself is a reference type, that reference value is copied. Types that require deep copy or custom cloning logic can add user-defined APIs (e.g. a Clone method) on top of clone.

4.5.2 Class Inheritance (Single Inheritance)

Aloe class supports single inheritance only.
That is, a class may specify exactly one base class. 

class Base {
  Id: Int;
}

class Derived : Base {
  Name: String;
}

// The following is a compile-time error (multiple inheritance is forbidden)
class Bad : Base, OtherBase {
}
  • Class hierarchies form trees (or chains); diamond inheritance does not occur.
  • Class inheritance is a mechanism for reuse of state and behavior.

4.5.3 interface and trait

Instead of allowing multiple inheritance of classes, Aloe uses interface and trait to compose behavior.

  • interface defines method signatures only, with no implementation (whether to allow default implementations is left for later revisions).
  • trait is a mixin unit that can provide method implementations and be applied to existing classes.
  • A class may have one base class, implement multiple interfaces, and apply multiple traits using with. 
interface IDisposable {
  method Dispose();
}

trait Printable {
  method PrintSelf() {
    Print(this.ToString());
  }
}

class FileHandle : BaseFile implements IDisposable with Printable {
  method Dispose() {
    // resource cleanup
  }

  method ToString(): String {
    return "FileHandle(...)";
  }
}

Detailed syntax and rules for interface and trait (inheritance, conflict resolution when multiple traits define the same method, etc.) are not fully defined in this draft. Only the general direction is specified.

4.5.4 sealed Classes

Aloe provides the sealed modifier to forbid deriving from a specific class. 

sealed class Token {
  Kind: TokenKind;
  Text: String;
}

// This is a compile-time error (Token is sealed)
class KeywordToken : Token {
}
  • A sealed class cannot be used as a base class.
  • A sealed class may still implement interfaces and have traits applied via with.
  • In the future, combining sealed classes with union/pattern matching may enable exhaustiveness checking.

4.5.5 readonly and Classes

  • Class fields are mutable by default.
  • Adding the readonly modifier to a field makes that field immutable:
    • Assignments to that field are only allowed in the field initializer or in the constructor of the same class.
    • Any other assignment is a compile-time error.
  • readonly class makes the entire class logically immutable:
    • All instance fields are implicitly treated as if they were readonly.
    • Field assignments are allowed only in field initializers and constructors of that class.
    • After the constructor completes, no code may change the fields of that instance.
  • readonly and clone are orthogonal: even a readonly class can be cloned. The result of clone is a new instance with the same immutable state.

In summary, classes are heap-based reference types with single inheritance; multiple behaviors are composed via interface + trait, sealed expresses “no further inheritance”, and readonly expresses immutability at field or class level. The details of method resolution and trait conflict resolution are specified elsewhere.

4.6 Generics

4.6.1 Overview

Aloe provides minimal generics for types, structs, classes, interfaces, traits, methods, and pipe-related declarations. The purpose is to make collections and Pipe / Filter reusable in a type-safe manner, rather than to support advanced type-level computation. 

type Box<T> {
  Value: T;
}

struct Pair<TLeft, TRight> {
  Left: TLeft;
  Right: TRight;
}

method Map<T, U>(Source: Pipe<T>, Selector: Func<T:U>): Pipe<U> {
  // implementation omitted
}

4.6.2 Generic Type Parameters

  • Types, structs, classes, interfaces, traits, methods, and producer/filter/consumer declarations may take type parameters written as <T, U, ...>.
  • Type parameter names are recommended to start with an uppercase letter (e.g. T, TItem, TKey).
  • Type arguments are bound to concrete types at use sites.
type List<T> {
  Items: T[];
}

method FirstOrDefault<T>(Source: Pipe<T>): T {
  // implementation omitted
}

4.6.3 where Type Constraints (Overview)

Aloe generics support arbitrary type constraints using a where clause.
By default there are no constraints; version 0.1 supports a few basic kinds. 

method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  // Int-specific sum
}

Here, where exists solely for compile-time type checking.
The runtime representation (monomorphization vs tagged shared code, etc.) is defined by the VM spec.

4.6.4 Supported Kinds of Constraints

Version 0.1 supports the following kinds of constraints:

  • Value-type / reference-type constraints
    • where T: struct – T must be a value type.
    • where T: class – T must be a reference type.
  • Primitive-type constraints
    • where T: Int, where T: Float, where T: Bool, etc. Primitive type names may be specified directly.
  • User-defined type constraints
    • where T: SomeTrait – types that implement trait SomeTrait.
    • where T: MyStruct – exactly the struct type MyStruct.
    • where T: MyClass – exactly the class type MyClass.
  • Any concrete type name
    • Any existing concrete type name may be used (e.g. where T: Pipe<Int>).
// Generic constrained to value types
method Zero<T>(): T
  where T: struct
{
  // implementation omitted
}

// Int-only Pipe
method Average<T>(Source: Pipe<T>): Float
  where T: Int
{
  // implementation omitted
}

// Specific struct only
method PrintUser<T>(User: T)
  where T: UserInfo
{
  // implementation omitted
}

4.6.5 Behavior on Constraint Violations

  • If a type argument does not satisfy a where constraint, this is a compile-time error.
  • When multiple constraints are specified, all of them must be satisfied simultaneously.
  • Cyclic constraints and typeclass-like advanced structures are not supported at this time.
// Example of compile-time error:

// where T: Int, so T cannot be String
method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  ...
}

var Lines: Pipe<String> = ReadLines("log.txt");
var Result = Sum(Lines);  // <-- compile-time error

The internal representation of generics (monomorphization vs tagged shared code, etc.) is defined by AloeVM.
This language specification focuses on type-safe reuse and where-based constraint checking.

5. Functions and Methods

5.1 method Declarations

method Add(a: Int, b: Int): Int {
  return a + b;
}
  • method defines top-level functions or methods bound to types.
  • If a return type is omitted, the method is treated as having “no return value” (void).
  • Aloe does not have a Unit type; “no return value” is not represented as a type.

5.2 Method Binding via bound

Aloe uses bound to add methods to existing types. 

type Pipe<T> {
  // internal representation is VM-dependent
}

bound Pipe<T> {
  method Where(Predicate: Func<T:Bool>): Pipe<T> {
    // implementation omitted
  }
}

For Pipe<T>, methods declared inside bound blocks are subject to a return-type constraint (see section 6.2).

5.3 Function Type Representation (Func<...:...>)

Aloe represents all function types using Func<...:...> notation. Function type notation using -> does not exist in the language specification.

  • Func<T:U> means “a function that takes T and returns U”.
  • Func<A, B:R> means “a function that takes A and B and returns R”.
  • A function with no return value (void) may be written as Func<T:> or Func<T:void>.

6. Pipe / Filter Model

A defining feature of Aloe is the ability to express Pipe / Filter style dataflows in a type-safe and consistent way. 

ReadLines("access.log")
  | Where (.StartsWith("[ERROR]"))
  | Map   (ParseLogLine)
  | Where (.Level == LogLevel.Error)
  | ToArray();

Conceptually, this is equivalent to the following nested calls: 

ToArray(
  Where(
    Map(
      Where(ReadLines("access.log"), ...),
      ...),
    ...));

Aloe has only one pipe operator: |.
In this specification, when we say “pipe expression” or “pipe operator”, we mean expressions that contain | directly in the syntax.
The |> that appears later is merely syntactic sugar for | with, and is not considered a pipe operator itself.

Section 6.4 defines the relationship between | (the pipe operator) and |> (sugar), while sections 6.2–6.3 describe how this relates to Pipe / Filter.

6.1 Pipe<T> Type

Pipe<T> represents the “pipe world” in Aloe: lazy sequences, streams, iterators, etc.

  • Pipe<T> represents “a pipe through which elements of type T flow”.
  • Internal representation (concatenated iterators, coroutines, etc.) is VM-dependent.
  • The standard library provides producers/transformers/consumers built around Pipe<T>.
type Pipe<T> {
  // internal fields are implementation-dependent
}

The pipe world is the space of functions that take and/or return Pipe<T> (producer / transformer / consumer).
Consumers are exits from the pipe world.

6.2 Producer / Transformer / Consumer

In the Pipe / Filter model, functions or objects are classified into three roles:

6.2.1 Producer

  • Definition: A function/object whose return type is Pipe<*> and appears at the head of a pipeline.
  • Role: Reads data from the external world (files, HTTP, arrays, etc.) and feeds it into the pipe world (Pipe<T>).
  • Type: Generally of the form Func<X:Pipe<T>>. Producers are always designed to output a Pipe<*>.

6.2.2 Transformer (Intermediate Filter)

  • Definition: Functions/methods of type Func<Pipe<T>:Pipe<U>>.
  • Role: Transform/filter/sort data within the pipe world, without leaving it.

In Aloe, methods declared inside bound Pipe<T> are reserved for transformers.

Constraint:
Methods declared inside bound Pipe<T> must always return a Pipe<*>.
That is, they must have a type of the form Func<Pipe<T>:Pipe<U>>. 
bound Pipe<T> {
  // OK: transformer
  method Where(Predicate: Func<T:Bool>): Pipe<T> { ... }

  // OK: changes element type
  method Map<U>(Selector: Func<T:U>): Pipe<U> { ... }

  // NG: return type is not Pipe, so should be a compile-time error
  // method Count(): Int { ... }
}

6.2.3 Consumer (Terminal Filter)

  • Definition: Functions/objects with type Func<Pipe<T>:R>, where R is not Pipe<*>.
  • Role: Exit from the pipe world, consuming the pipe and turning it into arrays, scalar values, external I/O, etc.
  • Return type: R may be any type so long as it is not Pipe<*>: T[], Int, Unit, Bool, etc. 
// Consumer that materializes a pipe into an array
method ToArray<T>(Source: Pipe<T>): T[] { ... }

// Consumer that counts elements
method Count<T>(Source: Pipe<T>): Int { ... }

6.3 Pipe-related Declarations (producer / pipe / filter / consumer)

Aloe introduces four declaration forms for constructing Pipe / Filter components: producer, pipe, filter, and consumer.
All of these are treated as “objects with constructors”. When connected in a pipeline, their bound block is executed.

6.3.1 producer Declaration

producer ProducerName(Params...) {
  out: Pipe<T>;

  bound(output) {
    // Executed when the producer is connected into a pipeline
    // Reads from external resources and writes to output
  }
}
  • ProducerName is the name of the producer.
  • out specifies the output type of the pipe world. It must be a Pipe<*> type.
  • bound(output) defines the behavior when the producer is connected. The type of output is the type given by out (e.g. Pipe<T>).

Constructor parameters (Params...) may be referenced inside the bound block. 

producer ReadLines(Path: String) {
  out: Pipe<String>;

  bound(output) {
    // Pseudo code:
    //   let file = OpenFile(Path);
    //   for each line in file:
    //     output.write(line);
    //   file.close();
  }
}

6.3.2 pipe Declaration

pipe PipeName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Connect pipes and perform transfer/transformation
  }
}
  • PipeName is the name of the pipe.
  • in is the type coming from “upstream”.
  • out is the type flowing to “downstream”.
  • bound(input, output) describes the transformation from input to output.

A pipe declaration is intended for bridging external resources (sockets, HTTP, etc.) or heterogeneous streams (e.g. text → JSON). 

pipe JsonDecode<T>() {
  in:  String;   // JSON text
  out: T;        // decoded object

  bound(input, output) {
    // Pseudo code:
    //   for each line in input:
    //     let obj: T = JsonParse<T>(line);
    //     output.write(obj);
  }
}

6.3.3 filter Declaration

filter FilterName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Define how elements flow from input to output
  }
}
  • FilterName is the name of the filter.
  • in / out specify the input/output types.
  • In a Pipe / Filter chain, in and out are typically both Pipe<*>.
  • bound(input, output) describes the transformation between them.
filter Where<T>(Predicate: Func<T:Bool>) {
  in:  Pipe<T>;
  out: Pipe<T>;

  bound(input, output) {
    // Pseudo code:
    //   for each item in input:
    //     if (Predicate(item)):
    //       output.write(item);
  }
}

filter Map<T, U>(Selector: Func<T:U>) {
  in:  Pipe<T>;
  out: Pipe<U>;

  bound(input, output) {
    // Pseudo code:
    //   for each item in input:
    //     output.write(Selector(item));
  }
}

6.3.4 consumer Declaration

consumer ConsumerName(Params...) {
  in: InputType;

  bound(input) {
    // Executed at the end of the pipeline
    // Consumes input and returns some value
  }
}
  • ConsumerName is the name of the consumer.
  • in specifies the input type. In Pipe / Filter usage, this is usually Pipe<T>.
  • bound(input) is executed at the pipeline end, fully consumes input, and returns some value of type R.
  • R may be any type (T[], Int, Bool, Unit, etc.), but must not be Pipe<*>. 
consumer ToArray<T> {
  in: Pipe<T>;

  bound(input) {
    var buffer: List<T>;
    // Pseudo code:
    //   for each item in input:
    //     buffer.add(item);
    //   return buffer.toArray();
    return buffer.toArray();
  }
}

consumer Count<T> {
  in: Pipe<T>;

  bound(input) {
    var c: Int = 0;
    // Pseudo code:
    //   for each item in input:
    //     c = c + 1;
    //   return c;
    return c;
  }
}

In summary, the roles of pipe-related declarations are:

  • producer: external world → Pipe<*> (always outputs a Pipe).
  • pipe: bridge between heterogeneous streams and external I/O.
  • filter: typically Pipe<T> → Pipe<U>.
  • consumer: Pipe<T> → non-Pipe type (exit from pipe world).

6.4 Pipe Operator | and Sugar |>

6.4.1 Language-level Definition (Base Operator |)

Aloe’s pipe operator is | only.
First we define the base operator |. 

Expr | F | G

This is exactly equivalent to:

G(F(Expr))
  • If F has type Func<A:B> and G has type Func<B:C>, then Expr | F | G has type C.
  • This operator can be applied to any types and is not limited to Pipe<T>.
Therefore, from the type system’s perspective, it is not inherently problematic for the return type to change in the middle of a pipeline.
Constraints of the Pipe / Filter model come from Pipe<T> and pipe-related declarations in the library design, not from the operator itself.

6.4.2 Relationship Between |> and | with (Sugar)

In this specification, |> is not a pipe operator.
|> is just syntactic sugar for | with. The language processor (parser) expands |> in source code into | with before normal parsing and type checking. 

Expr |> Something

When the parser sees this code, it first rewrites it to:

Expr | with Something

Then | with itself is interpreted using the base operator | defined in 6.4.1 as a regular pipe composition.
In other words, in the end it is treated as a pipe expression using only |.

In summary:
  • “Pipe operator” and “pipe expression” refer only to syntax that directly contains |.
  • |> is a visual convenience for source code; the parser internally expands it to | with, then further to forms using only |.

6.4.3 Using Pipe<T> with | / |>

ReadLines("users.json")
  | Where (.Trim().Length > 0)
  | Map   (ParseUser)
  | Where (.IsActive)
  | ToArray();
  • ReadLines: producer (Func<String:Pipe<String>>).
  • Where, Map: transformers (Func<Pipe<T>:Pipe<U>>).
  • ToArray: consumer (Func<Pipe<User>:User[]>).
  • The overall pipe expression type is User[], the return type of ToArray.

The same logic can be written using |> sugar. Even then, the specification treats it as being expanded to | with, and then further into a pipe expression using only the base operator |. 

ReadLines("users.json")
  |> Where (.Trim().Length > 0)
  |> Map   (ParseUser)
  |> Where (.IsActive)
  |> ToArray();

Syntactically, this is interpreted first (at the “sugar” level) as the following (at this stage we do not yet call it a “pipe expression”): 

ReadLines("users.json")
  | with Where (.Trim().Length > 0)
  | with Map   (ParseUser)
  | with Where (.IsActive)
  | with ToArray();

After that, each | with is converted, according to 6.4.1, into a composition using only the base operator |. Only at that point is it treated as a “pipe expression”.

Note:
From the language’s perspective, it is allowed to chain | or |> even after a consumer. However, stylistically this is discouraged.
For example, patterns such as ToArray() | ToArray() or ToArray() |> ToArray() should be avoided.

6.5 Standard Filters and Producers

6.5.1 Representative Transformers (Filters)

filter Where<T>(Predicate: Func<T:Bool>) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

filter Map<T, U>(Selector: Func<T:U>) {
  in:  Pipe<T>;
  out: Pipe<U>;
  bound(input, output) { ... }
}

filter Take<T>(Count: Int) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

6.5.2 Representative Producers

producer FromArray<T>(Items: T[]) {
  out: Pipe<T>;
  bound(output) { ... }
}

producer ReadLines(Path: String) {
  out: Pipe<String>;
  bound(output) { ... }
}

6.5.3 Representative Consumers

consumer ToArray<T> {
  in: Pipe<T>;
  bound(input) { ... }   // returns T[]
}

consumer Count<T> {
  in: Pipe<T>;
  bound(input) { ... }   // returns Int
}

6.5.4 Example Usage

let ActiveUserNames: String[] =
  ReadLines("users.json")
    | Where (.Trim().Length > 0)
    | Map   (ParseUser)
    | Where (.IsActive)
    | Map   (.Name)
    | ToArray();

7. with Syntax

7.1 Overview

with is syntactic sugar for repeatedly writing property assignments and method calls on the same target object.
In Aloe it additionally has the role of a “lifetime lock” that keeps the target object alive for the duration of the block. 

struct HttpRequest {
  Url: String;
  Method: String;
  TimeoutSeconds: Int;
}

var Request: HttpRequest;

with (Request) {
  Url = "https://example.com/api";
  Method = "GET";
  TimeoutSeconds = 5;
}

7.2 Semantics

The above with block is semantically equivalent to: 

{
  Request.Url = "https://example.com/api";
  Request.Method = "GET";
  Request.TimeoutSeconds = 5;
}
  • with (Expr) { ... } evaluates Expr exactly once.
  • Within the block, simple identifier assignments are interpreted as member access on the target object.

7.3 Lifetime Lock (GC Root Behavior)

AloeVM is assumed to use memory management combining reference counting and sliding compaction.
Within this model, a with block acts as a “lifetime lock”:

  • The result of evaluating with (Expr) is stored in a hidden variable (a GC root).
  • This hidden variable keeps the object alive until the block is exited.
  • Thus, the object is guaranteed to be live throughout the block execution.

7.4 Interaction Between with and delete

  • Within a with block that targets variable V, issuing delete V; is a compile-time error.
  • The reason is that with expresses the intent “keep this object alive in this scope”, which conflicts with delete.
  • However, delete is allowed on other variables that refer to the same object. Because the with block keeps a GC root, the object itself remains alive.
var Request: HttpRequest = GetRequest();
var Alias: HttpRequest = Request;

with (Request) {
  // Request is protected by with; it is guaranteed to be alive in this block

  delete Alias;   // OK: only removes the reference from Alias
  Url = "https://example.com/api";  // still usable
}

// After leaving the with block, the GC root is removed.
// If no other references remain, the object may eventually be collected.

In summary, with is both syntactic sugar for member access and a mechanism to hold the target object as a temporary GC root.
To protect the same target with both locking and lifetime semantics, use lock with as described in section 9.3.

8. delete Syntax

8.1 Overview

In Aloe, delete does not directly destroy objects.
Instead, it is a syntax that removes the reference from a variable.

  • Actual object destruction (memory free, finalizers, etc.) is performed by the runtime (AloeVM) using reference counting + GC.
  • delete is used to express “this variable no longer refers to this object” in a type-safe way, as an alternative to assigning null.

8.2 Syntax

DeleteStatement ::= 'delete' Identifier ';'

8.3 Static Rules (Compile-time)

  • Identifier must be a variable declared in the current scope.
  • After delete, reading from that variable, or performing member access / method calls on it is a compile-time error.
  • However, assigning a new value to that variable is allowed. Conceptually, delete returns the variable to an “uninitialized” state. 
var Buf: Buffer = NewBuffer();

// Use Buf
DoSomething(Buf);

delete Buf;

// From here on, reading Buf or Buf.Length is a compile-time error

Buf = NewBuffer();  // After re-assignment, it can be used again

8.4 Runtime Semantics

When delete Var; executes, the VM behaves roughly as follows:

  • Decrements the reference count for the object that Var currently references (assuming reference counting).
  • Marks Var as “uninitialized/deleted”.
  • When the object’s reference count reaches zero, it will be freed at the GC’s discretion.
    In other words, delete does not guarantee immediate destruction.

This specification leaves the timing of object destruction to the VM/runtime.
delete only removes the reference from one variable.

8.5 Interaction with with

  • As stated in 7.4, issuing delete on the variable used as the with target inside that with block is a compile-time error.
  • Deleting another variable that refers to the same object is allowed, because the with root still keeps the object alive.

If Aloe introduces an ownership/resource type model in the future, integrating delete with that model may be considered.
This draft assumes no ownership model.

9. lock Syntax and Synchronization

9.1 Overview

The lock syntax acquires a lock on a specific object and guarantees exclusive access and optimized read barriers on that object during the execution of the block. 

var SharedCounter: Int = 0;

lock (SharedCounter) {
  SharedCounter = SharedCounter + 1;
}

9.2 Semantics

  • lock (Expr) { Body } acquires a lock on the object represented by Expr and guarantees exclusive access within the Body block.
  • Lock acquisition and release are implemented by the VM.
  • Updates to data shared between threads should generally be confined to lock blocks.

AloeVM is assumed to use sliding compaction for memory management.
Within a lock block, the VM may optimize read barriers and reduce repeated checks on the same object.

9.3 lock with Syntax (lock + with Combination)

To apply both “exclusive lock” and “lifetime lock (with)” to the same target object, Aloe provides the lock with syntax. 

lock with (Conn) {
  // Here:
  //   - Conn is protected by an exclusive lock
  //   - Conn is kept as a GC root via with
  //   - Members of Conn can be written without qualification (RequestUrl, etc.)
}

9.3.1 Syntax

LockWithStatement ::= 'lock' 'with' '(' Expression ')' Block

lock with (Expr) { Body } is a compound syntax that evaluates Expr exactly once and treats the result as the target of both lock and with.

9.3.2 Desugaring (Semantically Equivalent Code)

lock with (Expr) { Body } is semantically equivalent to: 

{
  let __tmp = Expr;
  lock (__tmp) {
    with (__tmp) {
      Body
    }
  }
}
  • Expr is evaluated exactly once and bound to a temporary variable __tmp.
  • lock and with are nested and applied to __tmp.
  • Thus, during execution of Body:
    • The object is protected by an exclusive lock (lock semantics).
    • The object is held as a GC root via with, guaranteeing liveness.
    • Members can be accessed with the same shorthand as with.

9.3.3 Style and Constraints

  • lock with is sugar for expressing “operate on this resource with both exclusive lock and safe lifetime” in a single construct.
  • Alternative spellings such as with lock (Expr) or lock & with (Expr) are not defined in the language.
    For this purpose, always use lock with.
  • The behavior inside a lock with block is understood as the combination of the rules in section 9.2 (lock) and section 7.3 (with).
var Conn: Connection = OpenConnection();

lock with (Conn) {
  Send("PING");
  var Resp: String = Receive();
  Print(Resp);
}

// At this point, the lock is released and the with lifetime lock is also released.

10. Exception Handling

10.1 try / catch / finally

try {
  DoSomething();
} catch (ex: Error) {
  LogError(ex);
} finally {
  Cleanup();
}
  • Exceptions thrown inside the try block can be caught by catch.
  • The finally block is always executed, regardless of whether an exception was thrown.
  • The details of the exception types and hierarchy are defined in the VM spec.

11. Execution Model and Memory Management

11.1 AloeVM Overview

  • Stack-based bytecode VM.
  • Function calls push frames on a call stack.
  • GC is assumed to combine reference counting with sliding compaction.

11.2 Memory Management (Overview)

  • Objects are allocated on the heap.
  • When certain conditions are met, GC runs, slides live objects, and compacts memory.
  • with acts as a temporary GC root to keep objects alive and, combined with lock, may allow optimization of read barriers.
  • Normally there is no need to call delete for memory management. Use delete only when you explicitly want to drop a reference.

Detailed opcode lists and GC behavior will be specified in a separate AloeVM specification.

13. event Syntax and Event Model

13.1 Purpose and Positioning

Aloe’s event syntax is used to express, at the language level, that “this is a notification point where external code can register handlers”.

  • External code can register/unregister handlers (callbacks) on events.
  • Raising (firing) an event is, in principle, allowed only on the declaring side.

At implementation level, event is treated as syntactic sugar for the standard library type Event<TArgs>.

  • event Clicked<ClickedArgs>; is equivalent to having a field of type Event<ClickedArgs>.
  • However, unlike normal fields, reassigning the event or calling Raise directly from external code is forbidden (compile-time error).

13.2 Event<TArgs> and Handler Type

13.2.1 Handler Type

The function type of event handlers is defined via a standard type alias: 

type EventHandler<TArgs> = Func<TArgs:Task>;
  • Argument: TArgs (type of event arguments).
  • Return type: Task.
    Handlers are modeled as “functions that return a Task representing completion”, so that asynchronous handlers are naturally supported.

Future versions may consider an Event<TArgs, TResult> that handles handlers of type Func<TArgs:Task<TResult>>, but version 0.1 uses Task-returning handlers as the base.

13.2.2 Role of Event<TArgs> (Conceptual)

Event<TArgs> is a type that holds a set of event handlers and the logic to invoke them. 

type Event<TArgs> {
  Handlers: List<EventHandler<TArgs>>;
}

bound Event<TArgs> {
  method Add(Handler: EventHandler<TArgs>): Task {
    Handlers.Add(Handler);
  }

  method Remove(Handler: EventHandler<TArgs>): Task {
    // Details of handler identity comparison are delegated to the VM/runtime spec
    Handlers.Remove(Handler);
  }

  method Raise(Args: TArgs): Task {
    // Await all handlers in order and return a Task that completes when they all finish
    foreach (H in Handlers) {
      await H(Args);
    }
  }
}

From a language specification perspective, Event<TArgs> is the concrete “event collection” type, and event syntax is sugar plus access-control rules on top of it.

13.3 event Declaration Grammar

13.3.1 Declaration Sites

event may be declared in:

  • struct bodies.
  • class bodies.
  • interface bodies.

Declaring event at top level (file scope) or inside trait is not defined in version 0.1.

13.3.2 Grammar (Overview)

EventDeclaration ::=
    'event' Identifier '<' Type '>' ';'

Example:

struct Button {
  event Clicked<ClickedArgs>;
}

Semantically, the above is equivalent to the following field declaration: 

struct Button {
  private Clicked: Event<ClickedArgs>;
}

With additional constraints that differ from normal fields:

  • External code may not reassign Clicked (compile-time error).
  • External code may not call Clicked.Raise(...) directly.
  • External code may not delete Clicked; either.

13.4 Subscription/Unsubscription Syntax (+= / -=)

13.4.1 Grammar

EventAddStatement    ::= PrimaryExpression '.' Identifier '+=' EventHandlerExpression ';'
EventRemoveStatement ::= PrimaryExpression '.' Identifier '-=' EventHandlerExpression ';'
  • PrimaryExpression.Identifier must be an event-declared member.
  • EventHandlerExpression must be implicitly convertible to Func<TArgs:Task>.

13.4.2 Type Rules

When event X<TArgs>; is declared:

  • Target.X has type Event<TArgs>.
  • Target.X += H; is allowed only if H has type Func<TArgs:Task> (or is implicitly convertible to it).
  • Likewise for Target.X -= H;.

13.4.3 Desugaring (Semantically Equivalent Code)

Semantically, they are interpreted as:

Target.X += H;
// is equivalent to:
await Target.X.Add(H);
Target.X -= H;
// is equivalent to:
await Target.X.Remove(H);

By giving Add / Remove return type Task, subscription/unsubscription via += / -= also behave as asynchronous statements. Future versions may consider fire-and-forget variants if needed.

13.5 Rules for Raising Events

13.5.1 Raising Allowed Only from the Defining Side

Calling Raise on an event-declared member is allowed only from:

  • Methods of the struct / class that declares the event.
  • bound blocks for the same type.

Any other code (external modules or other classes) calling Clicked.Raise(...) is a compile-time error.

This guarantees that only the declaring side of an event has the authority to fire it.

13.5.2 Basic Pattern for Raising

struct ClickedArgs {
  ButtonId: String;
}

struct Button {
  event Clicked<ClickedArgs>;
}

bound Button {
  method OnNativeClick(Id: String): Task {
    let Args: ClickedArgs = { ButtonId = Id };
    await Clicked.Raise(Args);
  }
}

13.6 interface Events and Implementations

interface may also declare event.

struct ClickedArgs {
  ButtonId: String;
}

interface IButton {
  event Clicked<ClickedArgs>;
}

In this case, any struct / class that implements IButton must declare a corresponding event. 

struct Button implements IButton {
  event Clicked<ClickedArgs>;
}
  • If the event name Clicked does not match,
  • or the type argument ClickedArgs differs,

then it is a compile-time error due to improper interface implementation.

13.7 Examples

13.7.1 Button and Clicked Event

struct ClickedArgs {
  ButtonId: String;
}

struct Button {
  event Clicked<ClickedArgs>;
}

bound Button {
  method OnNativeClick(Id: String): Task {
    let Args: ClickedArgs = { ButtonId = Id };
    await Clicked.Raise(Args);
  }
}

method SetupButton(): Task {
  var Btn: Button = CreateButton("OK");

  // Register async handler
  Btn.Clicked += async (Args: ClickedArgs): Task {
    Print("Clicked: " + Args.ButtonId);
  };

  // Register another handler
  Btn.Clicked += async (Args: ClickedArgs): Task {
    Log("button clicked");
  };
}

13.7.2 Unsubscribing Handlers

method SetupAndTearDown(Btn: Button): Task {
  let Handler: Func<ClickedArgs:Task> =
    async (Args: ClickedArgs): Task {
      Print("temp handler");
    };

  // Subscribe
  Btn.Clicked += Handler;

  // Do something...

  // Unsubscribe
  Btn.Clicked -= Handler;
}

13.8 Combining with with / lock (Reference)

Since Event<TArgs> is also a normal reference type, it may be combined with with or lock with to fire events in a thread-safe manner if desired. 

struct Hub {
  event MessageReceived<String>;
}

bound Hub {
  method RaiseMessage(Msg: String): Task {
    lock with (MessageReceived) {
      await MessageReceived.Raise(Msg);
    }
  }
}

13.9 Summary

  • event provides sugar and access-control rules on top of Event<TArgs>.
  • External code can only subscribe/unsubscribe using += / -=; it cannot fire the event or replace the underlying Event instance.
  • Handlers use the function type Func<TArgs:Task>, forming an async-first event model.
  • event can be declared in interfaces, expressing event contracts at the type level.
  • Similar usability to C#’s event is provided without introducing delegate as a language feature, instead reusing Aloe’s function types and Task.
