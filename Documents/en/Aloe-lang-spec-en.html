1. Introduction

Aloe is an experimental programming language with static typing, simple syntax, and a stack-based virtual machine (AloeVM). It is designed with a strong emphasis on Pipe / Filter style data flow and interoperability with WebAssembly.

  • Statically typed (type-checked at compile time).
  • C-like surface syntax, but with an intentionally small feature set.
  • Focus on making it pleasant to express
    “data flow and transformation” such as text processing, REST calls, and logging.
  • Assumes a stack-based VM (AloeVM) and future WebAssembly backends.

This document is a draft of the language specification.
It may differ from actual compiler / VM implementations.

2. Design Goals

2.1 Goals

  • Readability: the specification itself should be readable as documentation.
  • Ease of implementation: a small compiler and VM should be able to implement it.
  • Consistency of Pipe / Filter: type-safe data flow representation.
  • Bridge to Wasm: easy integration with future Wasm component models.

2.2 Non-Goals

  • Becoming an “everything can be done here” general-purpose language.
  • Competing with highly optimized JIT compilers purely on performance.
  • Supporting complex metaprogramming (macros, etc.) in this version.

3. Basic Syntax

3.1 Source Files

  • Source files are assumed to be UTF-8 encoded.
  • Single-line comments: // comment
  • Multi-line comments: /* ... */

3.2 Tokens

  • Identifiers: start with a letter or _, followed by letters, digits, or _.
  • Keywords are reserved and cannot be used as identifiers.

3.3 Representative Keywords

CategoryKeywords
Declarations let, var, type, struct, enum, class, interface, trait, sealed
Control if, else, while, for, break, continue, return, lock, using
Functions method
Auxiliary with, delete, clone, where

3.4 Statements and Expressions

  • Most statements end with a semicolon ;.
  • Blocks are enclosed in { ... }.
  • Conditional expressions in control constructs must be of type Bool.
// Variable declarations
let Threshold: Int = 10;
var Count: Int = 0;

// if statement
if (Count > Threshold) {
  Print("over");
} else {
  Print("ok");
}

4. Type System

4.1 Primitive Types

TypeDescription
Booltrue / false
IntSigned integer (bit width is implementation-defined; assume 32-bit here).
FloatFloating point number (32 or 64 bit, implementation-defined).
StringUTF-8 string (internal representation is implementation-defined).

4.2 Arrays and Collections

  • Fixed-length array: T[]
  • For variable-length and lazy sequences, use Pipe<T> (details in section 6).

4.3 struct

struct User {
  Id: Int;
  Name: String;
  IsActive: Bool;
}
  • struct is a value type. Assignment copies the value.
  • Copying is shallow; sharing of nested reference-type fields depends on the VM implementation.
  • A struct may inherit from at most one other struct (single inheritance of structs).
  • Structs are not allowed to implement interface. (How exactly they interact with traits and generic constraints will be detailed later.)

4.4 enum

  • enum is always represented internally as Int.
  • “Generic enums” like enum<String> or enum<Byte> are forbidden.
  • Bit-flag enums are limited to at most 32 flags.
enum LogLevel {
  Trace = 0;
  Debug = 1;
  Info  = 2;
  Warn  = 3;
  Error = 4;
}

4.5 Classes and Inheritance (class / interface / trait / sealed)

4.5.1 Basics of Classes

  • class is always a heap-allocated reference type.
  • Assignment copies the reference (shallow copy).
  • clone creates a shallow copy of the object and returns a new reference.
  • All classes are cloneable; the language does not provide a way to mark a type as “not clonable”.
    If you want to express that an object should not be cloned, use API design or coding guidelines on top of clone. 
class User {
  Name: String;
  Age: Int;
}

var A: User = NewUser();
var B: User = A;        // Reference copy (A and B point to the same instance)

var C: User = clone A;  // Shallow copy: a new instance with the same field values

A shallow copy means “copy field values as-is”.
If a field is itself a reference type, the reference is copied. For deeper copies or custom cloning logic, you can provide user-defined APIs like a Clone method built on top of clone.

4.5.2 Class Inheritance (Single Inheritance)

Aloe class supports single inheritance only.
A class can specify exactly one base class. 

class Base {
  Id: Int;
}

class Derived : Base {
  Name: String;
}

// Compile error: multiple inheritance of classes is forbidden
class Bad : Base, OtherBase {
}
  • The class hierarchy is a tree (or chain); diamond inheritance does not occur.
  • Class inheritance is a mechanism for reusing state and behavior.

4.5.3 interface and trait

Instead of allowing multiple inheritance, Aloe uses interface and trait to compose behaviors for classes.

  • interface defines method signatures only and carries no implementation (whether default implementations are allowed is TBD).
  • trait is a mixin unit that can contain method implementations, used to add behavior to existing classes.
  • A class may have one base class, implement multiple interfaces, and apply multiple traits via with. 
interface IDisposable {
  method Dispose(): Unit;
}

trait Printable {
  method PrintSelf(): Unit {
    Print(this.ToString());
  }
}

class FileHandle : BaseFile implements IDisposable with Printable {
  method Dispose(): Unit {
    // Release resources
  }

  method ToString(): String {
    return "FileHandle(...)";
  }
}

The detailed syntax and rules for interface and trait (inheritance between them, conflict resolution when multiple traits define the same method, etc.) are not fully specified in this draft. Only the general direction is described here.

4.5.4 sealed Classes

To forbid inheritance from a particular class, Aloe provides the sealed modifier. 

sealed class Token {
  Kind: TokenKind;
  Text: String;
}

// Compile error: Token is sealed
class KeywordToken : Token {
}
  • A sealed class cannot be used as a base class.
  • Even if sealed, the class may still implement interfaces and have traits applied via with.
  • In the future, we plan to consider combining sealed classes with unions and pattern matching to support exhaustiveness checks.

4.5.5 readonly and Classes

  • Class fields are mutable by default.
  • Adding a readonly modifier to a field makes it immutable:
    • Assignments to that field are allowed only in field initializers or constructors of the same class.
    • Assignments from any other place are compile-time errors.
  • readonly class makes the whole class logically immutable:
    • All instance fields are treated as if they were declared readonly.
    • Assignments to fields are only allowed inside field initializers and constructors of that class.
    • After the constructor completes, no code may mutate the fields of that instance.
  • readonly and clone are orthogonal: even a readonly class can be cloned with clone. The clone is a separate instance with the same immutable state.

In summary, classes are heap-based reference types with single inheritance. Multiple behaviors are composed via interface + trait. sealed is used to forbid inheritance, and readonly expresses immutability at the field or class level.
Details such as method resolution order and trait conflict resolution will be specified elsewhere.

4.6 Generics

4.6.1 Overview

Aloe provides a minimal generics mechanism for types, functions, methods, and pipe-related declarations.
The goal is to make collections and Pipe / Filter components reusable and type-safe, not to support advanced type-level computation. 

type Box<T> {
  Value: T;
}

struct Pair<TLeft, TRight> {
  Left: TLeft;
  Right: TRight;
}

method Map<T, U>(Source: Pipe<T>, Selector: T -> U): Pipe<U> {
  // Implementation omitted
}

4.6.2 Generic Type Parameters

  • Types, structs, classes, interfaces, traits, methods, and producer/filter/consumer declarations can take type parameters via <T, U, ...>.
  • Type parameter names are recommended to start with uppercase (T, TItem, TKey, etc.).
  • Type arguments are bound to concrete types at usage sites.
type List<T> {
  Items: T[];
}

method FirstOrDefault<T>(Source: Pipe<T>): T {
  // Implementation omitted
}

4.6.3 where Type Constraints (Overview)

Aloe generics support arbitrary type constraints via a where clause.
By default, there are no constraints, but version 0.1 supports a few basic ones. 

method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  // Sum specialized for Int
}

Here, where exists purely for compile-time type checking.
Runtime representation (monomorphization, type-tagged shared code, etc.) is left to the VM specification.

4.6.4 Supported Constraint Kinds

Version 0.1 supports the following kinds of constraints:

  • Value type / reference type constraints
    • where T: struct – T must be a value type.
    • where T: class – T must be a reference type.
  • Primitive type constraints
    • where T: Int, where T: Float, where T: Bool, etc. Primitive type names may be used directly.
  • User-defined type constraints
    • where T: SomeTrait – types that implement trait SomeTrait.
    • where T: MyStruct – exactly the struct type MyStruct.
    • where T: MyClass – exactly the class type MyClass.
  • Any concrete type name
    • Any existing concrete type may be specified (for example, where T: Pipe<Int>). 
// Generic for value types only
method Zero<T>(): T
  where T: struct
{
  // Implementation omitted
}

// Pipe specialized for Int
method Average<T>(Source: Pipe<T>): Float
  where T: Int
{
  // Implementation omitted
}

// Generic restricted to a specific struct type
method PrintUser<T>(User: T)
  where T: UserInfo
{
  // Implementation omitted
}

4.6.5 Behavior on Constraint Violations

  • If a type argument does not satisfy the where constraints of a generic function or type, it is a compile-time error.
  • If multiple constraints are specified, they all must be satisfied simultaneously.
  • Cyclic constraints or advanced typeclass-like structures are not supported at this time.
// Example of a compile error:

// where T: Int, so T cannot be String
method Sum<T>(Source: Pipe<T>): T
  where T: Int
{
  ...
}

var Lines: Pipe<String> = ReadLines("log.txt");
var Result = Sum(Lines);  // <-- Compile error

The internal representation of generics (monomorphization, type-tagged shared code, etc.) is defined by AloeVM.
The language specification focuses on type-safe reuse and where-based constraint checking.

5. Functions and Methods

5.1 method Declarations

method Add(a: Int, b: Int): Int {
  return a + b;
}
  • method defines top-level functions or methods bound to a type.
  • If no return type is specified, the method is assumed to return Unit.

5.2 Method Binding via bound

Aloe uses bound to add methods to existing types. 

type Pipe<T> {
  // Internal representation is VM-dependent
}

bound Pipe<T> {
  method Where(Predicate: T -> Bool): Pipe<T> {
    // Implementation omitted
  }
}

For Pipe<T>, we introduce a restriction on methods declared inside bound blocks regarding their return types (see section 6.2 for details).

6. Pipe / Filter Model

A key feature of Aloe is that it can express Pipe / Filter style data flow in a type-safe and stylistically consistent way. 

ReadLines("access.log")
  |> Where (.StartsWith("[ERROR]"))
  |> Map   (ParseLogLine)
  |> Where (.Level == LogLevel.Error)
  |> ToArray();

Conceptually, this is equivalent to the following nested calls: 

ToArray(
  Where(
    Map(
      Where(ReadLines("access.log"), ...),
      ...),
    ...));

The |> operator in Aloe is syntactic sugar for such function composition.

6.1 Pipe<T> Type

Pipe<T> represents the “pipe world” in Aloe: lazy sequences / streams / iterators.

  • Pipe<T> represents a pipe through which elements of type T flow.
  • Its internal representation (iterator chains, coroutines, etc.) is VM-dependent.
  • The standard library provides producers / transformers / consumers using Pipe<T>.
type Pipe<T> {
  // VM-dependent internal fields
}

The pipe world is the realm of functions that take and/or return Pipe<T>
(producers / transformers / consumers). A consumer is an exit from the pipe world.

6.2 Producer / Transformer / Consumer

In the Pipe / Filter model, functions and objects are classified into three roles:

6.2.1 Producer

  • Definition: a function/object whose return type is Pipe<*> and which appears at the head of a pipeline.
  • Role: read from the external world (files, HTTP, arrays, etc.) and push data into the pipe world (Pipe<T>).
  • Type: has the shape X -> Pipe<T>. Producers are always designed to output Pipe<*>.

6.2.2 Transformer (Intermediate Filter)

  • Definition: a function/method of type Pipe<T> -> Pipe<U>.
  • Role: transforms / filters / sorts data within the pipe world; it does not exit the pipe world.

In Aloe, methods defined inside bound Pipe<T> are intended to be transformers (intermediate filters) only.

Restriction:
Methods declared inside a bound Pipe<T> block must always
return a Pipe<*>.
In other words, their type must be Pipe<T> -> Pipe<U>. 
bound Pipe<T> {
  // OK: transformer
  method Where(Predicate: T -> Bool): Pipe<T> { ... }

  // OK: changes element type
  method Map<U>(Selector: T -> U): Pipe<U> { ... }

  // NG: return type is not Pipe, so this must be a compile error
  // method Count(): Int { ... }
}

6.2.3 Consumer (Terminal Filter)

  • Definition: a function/object of type Pipe<T> -> R where R is not Pipe<*>.
  • Role: exits from the pipe world by consuming the pipe and converting it into arrays, scalar values, external I/O, etc.
  • Return Type: R may be any type as long as it is not Pipe<*> (e.g. T[], Int, Unit). 
// Consumer that materializes a pipe into an array
method ToArray<T>(Source: Pipe<T>): T[] { ... }

// Consumer that counts the number of elements
method Count<T>(Source: Pipe<T>): Int { ... }

6.3 Pipe-related Declarations (producer / pipe / filter / consumer)

Aloe introduces four declaration forms to build Pipe / Filter components: producer, pipe, filter, and consumer.
All of them are treated as “objects with constructors”. Their bound blocks are executed when they are connected to a pipeline.

6.3.1 producer Declarations

producer ProducerName(Params...) {
  out: Pipe<T>;

  bound(output) {
    // Executed when the producer is connected to a pipeline
    // Reads data from external resources and writes to output
  }
}
  • ProducerName is the name of the producer.
  • out denotes the output type in the pipe world. It must be a Pipe<*> type.
  • bound(output) specifies behavior when the producer is connected to a pipeline. The type of output is the type specified by out (e.g. Pipe<T>).

Constructor parameters (Params...) are accessible inside the bound block. 

producer ReadLines(Path: String) {
  out: Pipe<String>;

  bound(output) {
    // Pseudo-code:
    //   let file = OpenFile(Path);
    //   for each line in file:
    //     output.write(line);
    //   file.close();
  }
}

6.3.2 pipe Declarations

pipe PipeName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Connect pipes and perform forwarding / transformation
  }
}
  • PipeName is the name of the pipe.
  • in is the type flowing in from “upstream”.
  • out is the type flowing to “downstream”.
  • bound(input, output) describes the transformation from input to output.

The pipe declaration is intended for bridging external resources or converting between different stream types (e.g. text → JSON). 

pipe JsonDecode<T>() {
  in:  String;   // JSON text
  out: T;        // Decoded object

  bound(input, output) {
    // Pseudo-code:
    //   for each line in input:
    //     let obj: T = JsonParse<T>(line);
    //     output.write(obj);
  }
}

6.3.3 filter Declarations

filter FilterName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Define how to pass elements from input to output
  }
}
  • FilterName is the name of the filter.
  • in / out specify the input and output types.
  • In the Pipe / Filter chain, typical filters have both in and out as Pipe<*>.
  • bound(input, output) describes the transformation between the two.
filter Where<T>(Predicate: T -> Bool) {
  in:  Pipe<T>;
  out: Pipe<T>;

  bound(input, output) {
    // Pseudo-code:
    //   for each item in input:
    //     if (Predicate(item)):
    //       output.write(item);
  }
}

filter Map<T, U>(Selector: T -> U) {
  in:  Pipe<T>;
  out: Pipe<U>;

  bound(input, output) {
    // Pseudo-code:
    //   for each item in input:
    //     output.write(Selector(item));
  }
}

6.3.4 consumer Declarations

consumer ConsumerName(Params...) {
  in: InputType;

  bound(input) {
    // Executed at the end of the pipeline
    // Consumes input and returns some value
  }
}
  • ConsumerName is the name of the consumer.
  • in specifies the input type. In the Pipe / Filter model, this is usually Pipe<T>.
  • bound(input) is executed at the end of a pipeline, consumes input, and returns a value of some type R.
  • R can be any type (T[], Int, Bool, Unit, etc.) as long as it is not Pipe<*>. 
consumer ToArray<T> {
  in: Pipe<T>;

  bound(input) {
    var buffer: List<T>;
    // Pseudo-code:
    //   for each item in input:
    //     buffer.add(item);
    //   return buffer.toArray();
    return buffer.toArray();
  }
}

consumer Count<T> {
  in: Pipe<T>;

  bound(input) {
    var c: Int = 0;
    // Pseudo-code:
    //   for each item in input:
    //     c = c + 1;
    //   return c;
    return c;
  }
}

To summarize, the roles of pipe-related declarations are:

  • producer: external world → Pipe<*> (always outputs Pipe)
  • pipe: bridging heterogeneous streams and external I/O
  • filter: mainly transforms Pipe<T> -> Pipe<U>
  • consumer: Pipe<T> → non-Pipe type (exit from the pipe world)

6.4 Pipe Operator |>

6.4.1 Language-level Definition

In Aloe, |> is defined as syntactic sugar for simple function composition. 

Expr |> F |> G

is exactly equivalent to:

G(F(Expr))
  • If F has type A -> B and G has type B -> C, then Expr |> F |> G has type C.
  • This operator is applicable to any types; it is not limited to Pipe<T>.
Therefore, from the type system’s point of view, there is no problem with “changing the return type inside a pipeline”.
Restriction in the Pipe / Filter model is provided by the Pipe<T> type and library design, not by the operator itself.

6.4.2 Usage with Pipe<T> and |>

ReadLines("users.json")
  |> Where (.Trim().Length > 0)
  |> Map   (ParseUser)
  |> Where (.IsActive)
  |> ToArray();
  • ReadLines: producer (String -> Pipe<String>).
  • Where, Map: transformers (Pipe<T> -> Pipe<U>).
  • ToArray: consumer (Pipe<User> -> User[]).
  • The type of the entire pipe expression is User[], the return type of ToArray.

Note:
At the language level, it is allowed to put more |> after a consumer, but stylistically this is discouraged.
For example, ToArray() |> ToArray() should be avoided.

6.5 Standard Filters and Producers

6.5.1 Typical Transformers (Filters)

filter Where<T>(Predicate: T -> Bool) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

filter Map<T, U>(Selector: T -> U) {
  in:  Pipe<T>;
  out: Pipe<U>;
  bound(input, output) { ... }
}

filter Take<T>(Count: Int) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

6.5.2 Typical Producers

producer FromArray<T>(Items: T[]) {
  out: Pipe<T>;
  bound(output) { ... }
}

producer ReadLines(Path: String) {
  out: Pipe<String>;
  bound(output) { ... }
}

6.5.3 Typical Consumers

consumer ToArray<T> {
  in: Pipe<T>;
  bound(input) { ... }   // Returns T[]
}

consumer Count<T> {
  in: Pipe<T>;
  bound(input) { ... }   // Returns Int
}

6.5.4 Usage Example

let ActiveUserNames: String[] =
  ReadLines("users.json")
    |> Where (.Trim().Length > 0)
    |> Map   (ParseUser)
    |> Where (.IsActive)
    |> Map   (.Name)
    |> ToArray();

7. with Statement

7.1 Overview

The with statement is syntactic sugar for repeatedly writing property assignments and method calls on the same target object.
In Aloe, it also has a “lifetime lock” role to keep the target object alive during the block. 

struct HttpRequest {
  Url: String;
  Method: String;
  TimeoutSeconds: Int;
}

var Request: HttpRequest;

with (Request) {
  Url = "https://example.com/api";
  Method = "GET";
  TimeoutSeconds = 5;
}

7.2 Semantics

The above with block is semantically equivalent to: 

{
  Request.Url = "https://example.com/api";
  Request.Method = "GET";
  Request.TimeoutSeconds = 5;
}
  • with (Expr) { ... } evaluates Expr exactly once.
  • Within the block, simple identifier assignments are interpreted as member access on the target object.

7.3 Lifetime Lock (Behavior as a GC Root)

AloeVM assumes a memory management model that combines reference counting and sliding compaction.
In this model, the with block acts as a “lifetime lock”:

  • The result of evaluating with (Expr) is stored in a hidden variable (a GC root).
  • This hidden variable keeps the object alive until the block exits, guaranteeing that the object is not freed during the block.
  • Therefore, the object is guaranteed to be alive while the block is executing.

7.4 Relationship Between with and delete

  • It is a compile-time error to delete the same variable that is the target of a with block inside that very with block.
  • The reason is that with expresses the intention “this object should be alive here”, which conflicts with the delete intention.
  • However, it is allowed to delete another variable that refers to the same object. The GC root from with will still keep the object alive. 
var Request: HttpRequest = GetRequest();
var Alias: HttpRequest = Request;

with (Request) {
  // Request is protected by with and is guaranteed to be alive inside this block

  delete Alias;   // OK: only drops the reference held by Alias
  Url = "https://example.com/api";  // Still usable
}

// After the with block exits, the GC root is removed.
// If no other references exist, the object may eventually be freed.

In summary, with is both “syntactic sugar for member access” and “a mechanism to hold the target object as a temporary GC root”.
When you want to protect the same target together with synchronization, use the lock with syntax described in section 9.3.

8. delete Statement

8.1 Overview

In Aloe, delete does not directly destroy objects.
Instead, it is a construct for removing a reference from a variable.

  • Actual object destruction (memory freeing, finalizers, etc.) is handled by the runtime (AloeVM) using reference counting + GC.
  • delete is used as a type-safe way to express “this variable no longer refers to this object”, in place of explicitly assigning null.

8.2 Syntax

DeleteStatement ::= 'delete' Identifier ';'

8.3 Static Rules (Compile Time)

  • Identifier must be a variable declared in the current scope.
  • After delete, it is a compile-time error to read from that variable or to use it for member access / method calls.
  • However, reassigning a new value to that variable is allowed. Conceptually, delete returns the variable to an “uninitialized” state. 
var Buf: Buffer = NewBuffer();

// Use Buf
DoSomething(Buf);

delete Buf;

// From here on, reading Buf or Buf.Length is a compile-time error

Buf = NewBuffer();  // After re-assignment, Buf is usable again

8.4 Runtime Semantics

When delete Var; is executed, the VM behaves roughly as follows:

  • Decrement the reference count of the object currently referred to by Var (assuming reference counting).
  • Mark Var itself as “uninitialized / deleted”.
  • When the object’s reference count reaches 0, it is freed at the discretion of the GC.
    Thus, it is not guaranteed that the object is destroyed immediately when delete executes.

This language specification delegates the exact timing of object destruction to the VM / runtime.
delete merely removes a reference from a single variable to that object.

8.5 Interaction with with

  • As stated in section 7.4, it is a compile-time error to delete a variable that is the target of a with block within that block.
  • It is allowed to delete another variable that refers to the same object. The object itself remains alive due to the GC root created by with.

If Aloe introduces an “ownership-aware resource type” or ownership model in the future, delete may be integrated into that model.
In this draft, it is written under the assumption that no ownership model exists.

9. lock Statement and Synchronization

9.1 Overview

The lock statement acquires a lock for a given object and, during the execution of the block, guarantees exclusive access to that object and enables optimization of read barriers. 

var SharedCounter: Int = 0;

lock (SharedCounter) {
  SharedCounter = SharedCounter + 1;
}

9.2 Semantics

  • lock (Expr) { Body } acquires a lock on the object represented by Expr and guarantees exclusive access within Body.
  • Lock acquisition and release are implemented by the VM.
  • Updates to data shared between threads should generally be enclosed within lock blocks.

AloeVM assumes memory management using sliding compaction.
Within a lock block, the VM can optimize read barriers and reduce checks on each object to at most once.

9.3 lock with Statement (Combination of lock + with)

For cases where you want to apply both an “exclusive lock” and a “lifetime lock (with)” to the same target object at once, Aloe provides lock with. 

lock with (Conn) {
  // Here:
  //   - Conn is protected by an exclusive lock
  //   - Conn is kept alive as a GC root via with
  //   - Members of Conn can be accessed in a shortened form (RequestUrl, etc.)
}

9.3.1 Syntax

LockWithStatement ::= 'lock' 'with' '(' Expression ')' Block

lock with (Expr) { Body } is a composite construct meaning “evaluate Expr once and treat the result as the target of both lock and with”.

9.3.2 Desugaring (Semantically Equivalent Code)

lock with (Expr) { Body } is semantically equivalent to: 

{
  let __tmp = Expr;
  lock (__tmp) {
    with (__tmp) {
      Body
    }
  }
}
  • Expr is evaluated exactly once and bound to the temporary variable __tmp.
  • lock and with are then nested on __tmp.
  • Thus, during Body:
    • Exclusive access to the target object is guaranteed by lock.
    • The target object is kept alive as a GC root by with.
    • Members of the target object can be accessed in the shorthand form provided by with.

9.3.3 Style and Restrictions

  • lock with is a sugar for expressing the intent “operate on this resource in an exclusive and lifetime-safe block” in a single construct.
  • Alternative notations such as with lock (Expr) or lock & with (Expr) are not defined in the language.
    Use lock with for this purpose.
  • The behavior inside a lock with block should be interpreted as the combination of the rules in section 9.2 (lock) and section 7.3 (with).
var Conn: Connection = OpenConnection();

lock with (Conn) {
  Send("PING");
  var Resp: String = Receive();
  Print(Resp);
}

// Upon exiting the block, the lock is released and the with-based GC root is also removed.

10. using Statement (RAII Resource Management)

10.1 Overview

The using statement binds the lifetime of a resource (file handles, sockets, DB connections, etc.) to a block,
and guarantees that Dispose() is called when the block exits.
This is an RAII-like resource management mechanism similar to C#’s using. 

class FileHandle implements IDisposable {
  method Dispose(): Unit {
    // Release OS handle and buffers
  }
}

var File: FileHandle = OpenFile("data.txt");

using (File) {
  // Use File for I/O
  // File.Dispose() is guaranteed to be called when leaving the block
}

using is about external resources (OS resources), not memory.
Memory itself is still managed by AloeVM’s GC.

10.2 Syntax

UsingStatement ::= 'using' '(' Expression ')' Block
  • Expression must be an expression evaluating to an object that implements the IDisposable interface (section 4.5.3).
  • Typical examples are variables (like File) or constructor calls.
using (OpenFile("data.txt")) {
  // Use the handle returned from OpenFile
}

10.3 Static Rules (Compile Time)

  • In using (Expr), the static type of Expr must implement the IDisposable interface (see section 4.5.3).
  • The compiler checks for the presence of a Dispose(): Unit method.
  • It is a compile-time error to call delete on the same variable that is the target of using within that using block:
    • Example: using (File) { delete File; } is an error.
    • However, calling delete on another variable that refers to the same object is allowed.

Just like with with, there is a conflict between the intent “keep this object alive in this block” and “remove this reference” (delete).
Therefore, using using and delete on the same variable in the same block is forbidden.

10.4 Runtime Semantics and Desugaring

using (Expr) { Body } is semantically equivalent to: 

{
  let __res = Expr;        // Evaluate Expr exactly once
  try {
    Body                   // Code using the resource
  } finally {
    __res.Dispose();       // Called exactly once, regardless of normal exit or exception
  }
}
  • Expr is evaluated once and bound to the hidden variable __res.
  • Even if Body throws an exception, Dispose() is guaranteed to be called once by the finally clause.
  • Within Body, reading and writing via the original variable (e.g. File) is allowed, but calling delete on that variable is forbidden by the rules in 10.3.
  • __res is a hidden-scoped temporary and its lifetime ends when the block ends (decrementing the reference count by one).

10.5 Combining using with with / lock

using handles resource lifetime, with handles GC-root lifetime guarantees, and lock handles mutual exclusion.
These can be nested to combine their respective responsibilities. 

var Conn: Connection = OpenConnection();

using (Conn) {
  // Conn.Dispose() is guaranteed at the end of this block

  lock with (Conn) {
    // Here:
    //   - Conn has an exclusive lock
    //   - Conn is kept alive by with
    //   - The resource is released by using at block end
    Send("PING");
    var Resp: String = Receive();
    Print(Resp);
  }

  // Even after leaving the lock with block, Conn is still valid
  // as long as we remain inside the using block.
}

// When exiting the using block, Conn.Dispose() is called.
// If there are no other references, the memory may later be reclaimed by GC.

In summary, with handles “memory lifetime guarantees (GC root)”, using handles “resource lifetime (Dispose call)”, and lock handles thread-safe access.
Each has a distinct role, and they can be combined as needed.

11. Exception Handling

11.1 try / catch / finally

try {
  DoSomething();
} catch (ex: Error) {
  LogError(ex);
} finally {
  Cleanup();
}
  • Exceptions thrown inside the try block can be caught by matching catch clauses.
  • The finally block is always executed, regardless of whether an exception is thrown.
  • The detailed hierarchy and semantics of exception types are defined in the VM specification.

12. Execution Model and Memory Management

12.1 Overview of AloeVM

  • Stack-based bytecode VM.
  • Function calls push frames on a call stack.
  • GC is assumed to use reference counting combined with sliding compaction.

12.2 Memory Management (Overview)

  • Objects are allocated on a heap.
  • GC runs when certain conditions are met, sliding live objects and compacting memory.
  • with acts as a temporary GC root to keep objects alive, and can be combined with lock to optimize read barriers.
  • Normally, you do not need to call delete for memory management. Use delete only when you explicitly want to drop a reference.
  • using manages resource lifetime, not memory: it guarantees a call to Dispose() at the end of the block, nothing more.

Detailed opcode lists and GC implementation details are provided in the AloeVM specification.

13. Limitations and Future Work

13.1 Current Limitations

  • Generics have minimal expressive power; typeclasses and higher-kinded generics are not defined.
  • The module system and package management are out of scope for this document.
  • Metaprogramming (macros, reflection, etc.) is not defined.

13.2 Future Directions

  • Parallelization and fusion optimization of Pipe / Filter pipelines.
  • Integration with the Wasm component model.
  • Introduction of error handling mechanisms like Result types alongside exceptions.
  • Standard library extensions for distributed processing and streaming.
  • Potential integration of ownership-aware resource types with delete.
  • Integration of trait / interface / sealed / readonly with pattern matching and exhaustiveness checks.
  • Stronger RAII patterns that combine using with future ownership models.

In this draft, we focus especially on:
the relationship between Pipe / Filter and Producer / Transformer / Consumer / Pipe / Consumer declarations,
and the relationship between with / delete / lock / using / class inheritance / readonly / clone / generics and the AloeVM memory model.
