1. Introduction

Aloe is an experimental programming language with static typing, a simple syntax, and a stack-based virtual machine (AloeVM). It is especially designed around Pipe / Filter style dataflow, and good affinity with WebAssembly.

  • Statically typed (types are checked at compile time)
  • C-like surface syntax with a minimal feature set
  • Designed to write “flow & transform data” code comfortably: text processing, REST calls, log processing, and so on
  • Implementation assumes a stack-based VM (AloeVM), with a future WebAssembly backend in mind

This document is a draft language specification. Actual compiler and VM implementations may diverge from this text.

2. Design Goals

2.1 Goals

  • Readable: can be read as a language spec / documentation
  • Implementable: a small compiler and VM should be enough
  • Consistent Pipe / Filter: type-safe expression of dataflow
  • Wasm bridge: easy to connect to the Wasm component model in the future

2.2 Non-goals

  • Being a “do everything” general-purpose language
  • Competing with JIT-optimized languages in raw performance
  • Complex metaprogramming (macros, etc.) is out of scope for now

3. Basic Syntax

3.1 Source Files

  • Source encoding is assumed to be UTF-8.
  • Single-line comments: // comment
  • Multi-line comments: /* ... */

3.2 Tokens

  • Identifiers: start with a letter or _; following characters may include digits.
  • Keywords are reserved and cannot be used as identifiers.

3.3 Representative Keywords

CategoryKeywords
Declarationslet, var, type, struct, enum
Controlif, else, while, for, break, continue, return
Functionsmethod
Helperswith, lock, delete

3.4 Statements and Expressions

  • Statements usually end with a semicolon ;.
  • Blocks are delimited by { ... }.
  • Conditions in control constructs must have type Bool.
// variable declarations
let Threshold: Int = 10;
var Count: Int = 0;

// if statement
if (Count > Threshold) {
  Print("over");
} else {
  Print("ok");
}

4. Type System

4.1 Primitive Types

TypeDescription
Booltrue / false
IntSigned integer (bit width is implementation-defined, 32-bit assumed)
FloatFloating point (32 or 64-bit, implementation-defined)
StringUTF-8 string (internal representation is implementation-defined)

4.2 Arrays & Collections

  • Fixed-length arrays: T[]
  • Variable or lazy collections are represented by Pipe<T> (see section 6)

4.3 Structs

struct User {
  Id: Int;
  Name: String;
  IsActive: Bool;
}
  • struct is a value type and is copied on assignment.
  • Copy is shallow; sharing of nested reference types depends on the VM design.

4.4 Enums

  • enum is internally represented as Int in all cases.
  • Generic enums such as enum<String> or enum<Byte> are forbidden.
  • For bitflag enums, the maximum number of flags is 32.
enum LogLevel {
  Trace = 0;
  Debug = 1;
  Info  = 2;
  Warn  = 3;
  Error = 4;
}

5. Functions and Methods

5.1 method Declaration

method Add(a: Int, b: Int): Int {
  return a + b;
}
  • method declares a top-level function or a method bound to a type.
  • If no return type is specified, it is treated as Unit.

5.2 bound for Method Binding

Aloe introduces bound to attach methods to existing types. 

type Pipe<T> {
  // internal representation is VM-dependent
}

bound Pipe<T> {
  method Where(Predicate: T -> Bool): Pipe<T> {
    // omitted
  }
}

In this spec, bound Pipe<T> is given a return type restriction (see section 6.2).

6. Pipe / Filter Model

One major feature of Aloe is that it allows you to write Pipe / Filter style dataflow in a typing-consistent way. 

ReadLines("access.log")
  |> Where (.StartsWith("[ERROR]"))
  |> Map   (ParseLogLine)
  |> Where (.Level == LogLevel.Error)
  |> ToArray();

Conceptually, this is: 

ToArray(
  Where(
    Map(
      Where(ReadLines("access.log"), ...),
      ...),
    ...));

The |> operator is sugar for such function composition.

6.1 Pipe<T> Type

Pipe<T> represents the “pipe world” in Aloe. It corresponds to lazy sequences, streams, or iterators.

  • Pipe<T> is “a pipe that flows elements of type T”.
  • Internal representation (iterator chains, coroutines, etc.) is VM-dependent.
  • The standard library is designed around Pipe<T> with producers / transformers / consumers.
type Pipe<T> {
  // VM-dependent internal fields
}

The pipe world is the space of functions and objects that take and return Pipe<T>. Consumers are the exits from this world.

6.2 Producer / Transformer / Consumer

In the Pipe / Filter model, functions (or objects) are categorized into three roles:

6.2.1 Producer

  • Definition: Functions or objects whose return type is Pipe<*> and are used at the beginning of pipelines.
  • Role: Read data from the outside world (files, HTTP, arrays, etc.) and inject it into the pipe world (Pipe<T>).
  • Type-level: Have the shape X -> Pipe<T>. In this spec, anything designed as a Producer must always output some Pipe<*>.

6.2.2 Transformer (Intermediate Filter)

  • Definition: Functions or methods with type Pipe<T> -> Pipe<U>.
  • Role: Transform, filter, sort, etc. inside the pipe world, without leaving it.

In Aloe, methods defined in bound Pipe<T> are considered transformers by default.

Restriction:
Methods declared inside bound Pipe<T> must always return some Pipe<*>.
In other words, they must have the shape Pipe<T> -> Pipe<U>. 
bound Pipe<T> {
  // OK: transformer
  method Where(Predicate: T -> Bool): Pipe<T> { ... }

  // OK: transforms element type
  method Map<U>(Selector: T -> U): Pipe<U> { ... }

  // NG: not returning Pipe -- should be a compile-time error
  // method Count(): Int { ... }
}

6.2.3 Consumer (Terminal Filter)

  • Definition: Functions or objects with type Pipe<T> -> R, where R is not Pipe<*>.
  • Role: Exit points from the pipe world. Consume a pipe and materialize it into arrays, single values, external I/O, etc.
  • Return type restriction: There is no restriction on the return type R, as long as it is not Pipe<*>. It may be T[], Int, Unit, etc.
// pure consumer: materialize to array
method ToArray<T>(Source: Pipe<T>): T[] { ... }

// consumer counting elements
method Count<T>(Source: Pipe<T>): Int { ... }

6.3 Pipe Declarations (producer / pipe / filter / consumer)

Aloe introduces four kinds of declarations for structuring the Pipe / Filter model: producer, pipe, filter, and consumer. They are all “constructors with a bound block”. The bound block is executed when the component is actually attached into a pipeline.

6.3.1 producer

producer ProducerName(Params...) {
  out: Pipe<T>;

  bound(output) {
    // Executed when attached at the head of a pipeline
    // Read from external resource and write to output
  }
}
  • ProducerName is the producer name.
  • out is the output type in the pipe world. It must always be a Pipe<*> type.
  • bound(output) describes what happens when this producer is placed at the head of a pipeline. The type of output is the same as out (i.e. Pipe<T>).

Constructor parameters (Params...) are visible inside the bound block. 

producer ReadLines(Path: String) {
  out: Pipe<String>;

  bound(output) {
    // Pseudo:
    //   let file = OpenFile(Path);
    //   for each line in file:
    //     output.write(line);
    //   file.close();
  }
}

6.3.2 pipe

pipe PipeName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Glue logic between upstream and downstream
  }
}
  • PipeName is the pipe name.
  • in is the type received from upstream.
  • out is the type passed to downstream.
  • bound(input, output) describes how to move data from input to output.

Pipe declarations are intended to bridge external I/O (sockets, HTTP, etc.) and type translations (e.g. text → JSON). 

pipe JsonDecode<T>() {
  in:  String;   // JSON text
  out: T;        // decoded object

  bound(input, output) {
    // Pseudo:
    //   for each line in input:
    //     let obj: T = JsonParse<T>(line);
    //     output.write(obj);
  }
}

6.3.3 filter

filter FilterName(Params...) {
  in:  InputType;
  out: OutputType;

  bound(input, output) {
    // Filter logic when attached to a pipeline
    // Describe how to flow elements from input to output
  }
}
  • FilterName is the filter name.
  • in / out specify input and output types.
  • In the Pipe / Filter model, intermediate filters usually have both in and out of type Pipe<*>.
  • bound(input, output) implements the transformation logic from input to output.
filter Where<T>(Predicate: T -> Bool) {
  in:  Pipe<T>;
  out: Pipe<T>;

  bound(input, output) {
    // Pseudo:
    //   for each item in input:
    //     if (Predicate(item)):
    //       output.write(item);
  }
}

filter Map<T, U>(Selector: T -> U) {
  in:  Pipe<T>;
  out: Pipe<U>;

  bound(input, output) {
    // Pseudo:
    //   for each item in input:
    //     output.write(Selector(item));
  }
}

6.3.4 consumer

consumer ConsumerName(Params...) {
  in: InputType;

  bound(input) {
    // Logic executed at the end of the pipeline
    // Consume input and return some value
  }
}
  • ConsumerName is the consumer name.
  • in is the input type. In the Pipe / Filter model, this is usually Pipe<T>.
  • bound(input) is executed when this consumer is attached as the terminal of a pipeline. It consumes input and returns a value of arbitrary type R.
  • R can be any type (T[], Int, Bool, Unit, ...), but it must not be Pipe<*>.
consumer ToArray<T> {
  in: Pipe<T>;

  bound(input) {
    var buffer: List<T>;
    // Pseudo:
    //   for each item in input:
    //     buffer.add(item);
    //   return buffer.toArray();
    return buffer.toArray();
  }
}

consumer Count<T> {
  in: Pipe<T>;

  bound(input) {
    var c: Int = 0;
    // Pseudo:
    //   for each item in input:
    //     c = c + 1;
    //   return c;
    return c;
  }
}

Summary of roles:

  • producer: Outside world → Pipe<*> (must always output a Pipe)
  • pipe: Bridges between different streams or external I/O
  • filter: Mainly Pipe<T> -> Pipe<U> transformations
  • consumer: Pipe<T> → any non-Pipe type (exit from the pipe world)

6.4 Pipe Operator |>

6.4.1 Language-level Definition

Aloe’s |> is defined as syntactic sugar for function composition. 

Expr |> F |> G

is exactly equivalent to:

G(F(Expr))
  • If F has type A -> B and G has type B -> C, then Expr |> F |> G has type C.
  • It is applicable to any type, not only Pipe<T>.
Therefore, from the language type system point of view,
there is no problem in “changing return types along the pipeline”.
The constraints of the Pipe / Filter model are imposed by the library design around Pipe<T> and the declarations described above.

6.4.2 Pipe<T> Usage Example

ReadLines("users.json")
  |> Where (.Trim().Length > 0)
  |> Map   (ParseUser)
  |> Where (.IsActive)
  |> ToArray();
  • ReadLines: Producer (String -> Pipe<String>)
  • Where, Map: Transformers (Pipe<T> -> Pipe<U>)
  • ToArray: Consumer (Pipe<User> -> User[])
  • The whole pipeline has type User[], the return type of ToArray.

Note:
While it is valid in the language to chain more |> after a consumer, this is usually considered bad style. For example, ToArray() |> ToArray() is discouraged.

6.5 Standard Filters & Producers

6.5.1 Representative Transformers (filters)

filter Where<T>(Predicate: T -> Bool) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

filter Map<T, U>(Selector: T -> U) {
  in:  Pipe<T>;
  out: Pipe<U>;
  bound(input, output) { ... }
}

filter Take<T>(Count: Int) {
  in:  Pipe<T>;
  out: Pipe<T>;
  bound(input, output) { ... }
}

6.5.2 Representative Producers

producer FromArray<T>(Items: T[]) {
  out: Pipe<T>;
  bound(output) { ... }
}

producer ReadLines(Path: String) {
  out: Pipe<String>;
  bound(output) { ... }
}

6.5.3 Representative Consumers

consumer ToArray<T> {
  in: Pipe<T>;
  bound(input) { ... }   // returns T[]
}

consumer Count<T> {
  in: Pipe<T>;
  bound(input) { ... }   // returns Int
}

6.5.4 Example

let ActiveUserNames: String[] =
  ReadLines("users.json")
    |> Where (.Trim().Length > 0)
    |> Map   (ParseUser)
    |> Where (.IsActive)
    |> Map   (.Name)
    |> ToArray();

7. with Statement

7.1 Overview

The with statement is syntactic sugar to simplify repeated property assignments and method calls on a single object. Additionally, in Aloe it also works as a “liveness lock” that protects the target object from being freed by GC during the block. 

struct HttpRequest {
  Url: String;
  Method: String;
  TimeoutSeconds: Int;
}

var Request: HttpRequest;

with (Request) {
  Url = "https://example.com/api";
  Method = "GET";
  TimeoutSeconds = 5;
}

7.2 Semantics (sugar)

From a surface syntax standpoint, the with block above is equivalent to: 

{
  Request.Url = "https://example.com/api";
  Request.Method = "GET";
  Request.TimeoutSeconds = 5;
}
  • with (Expr) { ... } evaluates Expr exactly once.
  • Simple assignments to bare identifiers inside the block are treated as member assignments on the target object.

7.3 Liveness Lock (as a GC Root)

AloeVM assumes reference counting + sliding compaction for memory management. The with block acts as a “liveness lock” as follows:

  • The result of with (Expr) is bound to a hidden GC root variable.
  • This hidden variable is kept as a GC root until the block is exited.
  • Therefore, while executing inside the with block, the target object is guaranteed to remain alive.

7.4 Relationship with delete

  • It is a compile-time error to apply delete to the variable used as the with target (e.g. Request in the example) inside that block.
  • Reason: with expresses the intention “this object should stay alive in this block”, so deleting it is semantically contradictory.
  • However, it is allowed to delete another variable that refers to the same underlying object. The GC root created by with still protects it.
var Request: HttpRequest = GetRequest();
var Alias: HttpRequest = Request;

with (Request) {
  // Request is protected by with; it remains alive during this block.

  delete Alias;   // OK: only the Alias reference is removed
  Url = "https://example.com/api";  // still usable
}

// Exiting the block removes the GC root.
// If there are no other references, the object will eventually be collected.

In summary, with functions as “sugar for property setup” and also “a temporary GC root that protects the target object”.

8. delete Statement

8.1 Overview

Aloe’s delete is not an object destruction primitive. It is a construct to remove the reference stored in a variable.

  • Actual object destruction (memory reclamation, finalization, etc.) is handled by the runtime (AloeVM) via reference counting + GC.
  • delete is meant as a type-safe way to express “this variable will no longer use this object”, and can be seen as an alternative to null assignment.

8.2 Grammar

DeleteStatement ::= 'delete' Identifier ';'

8.3 Static Rules (Compile Time)

  • Identifier must be a variable declared in the current scope.
  • From the point where delete is executed onward, using that variable for reading, member access, or method calls is a compile-time error.
  • However, reassigning a new value to that variable is allowed. Conceptually, delete puts the variable back into an “uninitialized” state.
var Buf: Buffer = NewBuffer();

// Use Buf
DoSomething(Buf);

delete Buf;

// From here, using Buf on the right-hand side is a compile error.
// Buf.Length is also a compile error.

Buf = NewBuffer();  // After re-assignment, Buf is valid again.

8.4 Runtime Semantics

When delete Var; is executed, AloeVM behaves roughly as follows:

  • Decrement the reference count for the object currently pointed to by Var (assuming a reference-counting scheme).
  • Mark the variable Var itself as “uninitialized/deleted”.
  • Objects whose reference count reaches 0 are reclaimed by GC at some later time. delete does not guarantee immediate destruction.

In this language specification, the timing of object destruction is purely the responsibility of the VM/runtime. delete only removes a reference from a single variable.

8.5 Relationship with with

  • As stated in section 7.4, it is a compile-time error to apply delete to the variable used as the with target inside that block.
  • It is allowed to delete other variables that refer to the same object, because the with block provides a GC root that keeps the object alive.

In the future, if an “ownership-aware resource type” is introduced, delete may also interact with that ownership model. In this draft, however, the spec is written under the assumption that no ownership type system is present.

9. lock Statement & Synchronization

9.1 Overview

The lock statement locks a particular object and provides mutual exclusion and GC read barrier optimization within the block. 

var SharedCounter: Int = 0;

lock (SharedCounter) {
  SharedCounter = SharedCounter + 1;
}

9.2 Semantics

  • lock (Expr) { Body } guarantees exclusive access to the object denoted by Expr during the execution of Body.
  • Lock acquisition and release are handled by the VM.
  • Updates to shared data should generally be limited to lock blocks.

AloeVM is assumed to use a sliding compaction GC. Inside a lock block, read barrier checks for the same object may be optimized to run at most once.

10. Exceptions

10.1 try / catch / finally

try {
  DoSomething();
} catch (ex: Error) {
  LogError(ex);
} finally {
  Cleanup();
}
  • Exceptions thrown inside the try block are caught by matching catch blocks.
  • The finally block is always executed, regardless of exceptions.
  • The detailed hierarchy and properties of exception types are defined by the VM specification.

11. Runtime Model & Memory Management

11.1 AloeVM Overview

  • Stack-based bytecode virtual machine.
  • Function calls are implemented via stack frames.
  • GC is assumed to be reference counting + sliding compaction.

11.2 Memory Management (Summary)

  • Objects are allocated on a heap.
  • GC runs under certain conditions and compacts live objects while sliding them in memory.
  • The with statement acts as a temporary GC root to protect an object, and combined with lock it enables optimization of read barrier checks.
  • In normal programming, there is no need to use delete for memory reclamation. It is only needed when you want to explicitly express that a particular variable will no longer reference a given object.

A detailed opcode list and GC implementation specifics are described in the AloeVM spec.

12. Limitations & Future Work

12.1 Current Limitations

  • Generic features are minimal (no advanced type inference or typeclass-like features yet).
  • Module system and package management are out of scope in this document.
  • Metaprogramming (macros, reflection, etc.) is not defined.

12.2 Future Topics

  • Parallelization and fusion optimizations for Pipe / Filter.
  • Integration with the Wasm component model.
  • Error handling via Result-like types (in addition to exceptions).
  • Standard libraries for distributed and streaming workloads.
  • Integration of delete with any future ownership-aware resource types, if introduced.

This draft mainly clarifies: the relationships between Pipe / Filter and Producer / Transformer / Consumer / Pipe / Consumer, and their declaration forms, and the behavior of with / delete / lock and how they interact with AloeVM’s memory model.
