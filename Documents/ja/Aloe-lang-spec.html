Aloe 言語仕様書（ドラフト、日本語版）

edit-version: 30
edit-date: 2025-12-18T00:19:20+09:00

この文書は、英語版仕様 https://stk2k.github.io/Aloe-lang/Documents/en/Aloe-lang-spec-en.html をベースにした日本語ドラフトです。
その上で 4.2 with-blocks に class / struct / enum を対象にした型ベース with（static メンバー／列挙子への短縮アクセス） の拡張を追加しています。
また、本版では null を仕様書から排除し、 valid / invalid を比較・判定用の特別なリテラルとして導入しています。
それ以外の仕様は原文にできるだけ忠実に訳しています。

0. Overview（概要）

Aloe は静的型付けのスクリプト言語であり、シンプルで直感的な構文を目指している。
スタックベースの仮想マシン（AloeVM）上で動作し、主に次のような目標を持つ。

  • 教育用途や小規模アプリケーションで読み書きしやすいこと
  • C# ランタイムおよび WebAssembly (WASM) 環境で動作すること
  • pipe と filter によって、I/O と並行処理を静的型付けのまま制御できること

特徴：

  • 静的型付け
    • var による型推論
    • let による明示的な型指定
  • 値型 / 参照型
    • struct ... 値型
    • class ... 参照型
  • GC ベースのメモリ管理（具体的なアルゴリズムは VM 実装依存）
  • valid / invalid による参照の有効性概念
    • 参照型は「有効」または「無効（invalid）」の状態を取り得る。
    • 無効参照へのアクセスは実行時例外となる。
    • ソースコード上で invalid を値として代入することはできない（8章参照）。
  • pipe / filter によるデータフロースタイルの設計

1. Basic Syntax（基本文法）

1.1 Comments（コメント）

// single-line comment

/* multi-line
   comment */
  • // から行末までが行コメント。
  • /* ～ */ が複数行コメント。
  • 複数行コメントは、開始 /* から最初に現れる */ までを まるごとコメントとして扱う。 コメントの内部に追加の /* や */ が現れても、 それらはすべてコメントの一部として無視される（ネストしたコメントの構文解析は行わない）。
    したがって、コメント内の「コメント構文」が破綻していても、 いったん /* が始まってから最初の */ が来るまでは常にコメント扱いとなる。

1.2 Identifiers（識別子）

  • 使用可能文字：英字 (A-Z, a-z)、数字 (0-9)、アンダースコア _
  • 先頭文字は英字または _ でなければならない。
  • 大文字小文字は区別される。
  • キーワードと同じ綴りは使用できない。
  • _ で始まる識別子は「プライベート」扱い（外部からは参照しない慣習）。
  • ユニコード文字も使用できる（空白や区切り記号などは除外）。

1.4 Literals（リテラル）

  • 整数リテラル：10, 0, -5, 0x1A, 0b1010 など
  • 浮動小数点リテラル：10.5, 0.0, -3.14
  • 10進リテラル：10.12345678901234567890:d など
  • 真偽値：true, false
  • 文字列："Hello, World!"
  • 文字：'A', 'z' など
  • valid, invalid：参照の有効性を判定するための特別なリテラル（代入用途には使用できない）
  • 配列リテラル：[1, 2, 3]
  • リストリテラル：(1, 2, 3)
  • セットリテラル：{1, 2, 3}
  • マップリテラル：["key1": 10, "key2": 20]
  • enum リテラル：EnumType.Value1
  • struct リテラル：Point { x: 0, y: 1 }
  • オブジェクト生成：new ClassName()

1.5 Statement terminator（文の終端）

var x = 10;
x = x + 1;
print(x);
  • すべての文はセミコロン ; で終わる。
  • 改行による自動セミコロン挿入は行わない。

1.6 Whitespace（空白）

  • スペース、タブ、改行はトークン区切りとして扱われる。
  • 連続する空白は 1 つの区切りとみなされる。
  • インデントに意味はない（Python とは異なる）。

1.7 Case sensitivity（大文字小文字）

  • 識別子とキーワードは大文字小文字を区別する。
  • キーワードはすべて小文字（Result を除く）。

2. Type System（型システム）

2.1 Primitive types（プリミティブ型）

Aloe における「プリミティブ型」は、次の 8 種類とする。

  • bool / boolean - 真偽値。
  • int / integer - 符号付き整数。
  • float - 浮動小数点数。
  • decimal - 高精度 10 進数。
  • string - 文字列（内部的には Unicode）。
  • byte - 0-255 の符号なし 8 ビット整数。
  • char - 1 文字（Unicode スカラー値を想定）。
  • void - 戻り値なし（メソッド戻り値専用）。

valid / invalid は型ではなく、参照の有効性を表す比較・判定用の特別リテラルである（8章・9章参照）。
any 型を導入し、必要な箇所で任意の型の値を扱える。

2.2 Composite / user-defined types（合成型 / ユーザー定義型）

  • 配列：T[]
  • リスト：T()
  • セット：T{}
  • マップ：map<T> または map<K/V>
  • struct
  • class
  • enum, bitfield enum
  • 非同期タスク：task
  • パイプ：pipe<T>（18章参照）
  • 結果型：Result<T>（18章参照）

any は任意の型の値を扱うための特別な型であり、必要な箇所で利用できる。

2.3 Value types / reference types（値型 / 参照型）

  • 値型：
    • int, float, decimal, bool, byte, char
    • struct
    • enum, bitfield enum（内部表現は int32）
  • 参照型：
    • class
    • string（実装上は参照型）
    • task, pipe<T>, Result<T> など

any type（any 型）

  • any は、void を除く任意の値を保持できる特別な型である。
  • 任意の型 T の値は any へ暗黙に変換できる（代入・引数渡し）。
  • any から具体型 T への暗黙変換はない。
  • if (x is T) { ... } のブロック内では、x を T として扱ってよい（フロー依存の型絞り込み）。

Struct restrictions and copying（struct の制約とコピー）

  • struct のフィールドにはプリミティブ型と他の struct のみを持てる。
  • class などの参照型フィールドは持てない。
  • struct の代入は常に値コピー。
  • 最大サイズなどは仕様で制限しない。

Class vs struct（class と struct）

  • class は struct をフィールドとして持てる。
  • class インスタンスを別変数に代入すると参照がコピーされる（オブジェクト本体は共有）。

3. Numbers & Conversions（数値と暗黙変換）

3.1 int and float

  • int と float を混在させた式の結果は float になる。
    • int + float -> float
    • int - float -> float
    • int * float -> float
    • int / float -> float

3.2 decimal

  • decimal は高精度な 10 進数。
  • int / float と混在する式は、必要に応じて decimal に昇格しうる。
  • オーバーフロー / アンダーフロー / 精度問題は OverflowException として扱う。

3.3 enum and int

  • Aloe にはキャスト構文がない。
  • enum と int の暗黙 / 明示変換はいずれも禁止。
  • enum と数値型の混在式はコンパイルエラー（bitfield enum のビット演算を除く）。
  • 内部表現としては enum / bitfield enum はすべて int32。

3.4 byte and char

  • byte：
    • 範囲は 0-255。
    • byte 同士の演算は許されるが、範囲外はエラーになり得る。
    • byte と int の混在式は int に昇格。
  • char：
    • char の算術演算は不可。
    • 比較演算（==, <, > 等）は char 同士のみ。
    • 数値型との暗黙変換はない。

3.5 NaN / Infinity

  • 数値オーバーフロー/アンダーフローは NaN/Infinity ではなく例外で扱う。
  • ZeroDivisionException は 0 / 0.0 で常に投げる。

4. Blocks & Scope（ブロックとスコープ）

ブロックは { と } で区切られ、各ブロックが新しいスコープを導入する。

  • main ブロック
  • クラス定義ブロック
  • メソッド定義ブロック
  • 任意の { ... } ブロック

4.1 Static (lexical) scope and shadowing（静的スコープとシャドウイング）

class Sample {
    field value: int = 1;
    method foo(): void {
        var value = 2;        // フィールドをシャドウ
        print(value);         // 2
        print(this.value);    // 1
    }
}
  • Aloe は静的（レキシカル）スコープ。
  • 内側ブロックで同名変数を宣言すると、外側の変数をシャドウする。
  • フィールド名とローカル変数名がぶつかった場合、ローカル変数が優先される。 フィールドは this. で参照する。

4.2 with-blocks（メンバー省略記法）

Aloe には、同一オブジェクトのメンバーアクセスを簡潔に書くための with ブロックがある。 

with (user) {
    .name = "Alice";
    .age = .age + 1;
    print(.name);
}
  • with (expr) { ... } ブロック内では：
    • .Member は (expr).Member の省略形。
    • expr はブロックの先頭で 1 度だけ評価され、内部一時変数に保持される。
    • 特別な一時変数 _ は「with 対象オブジェクト」を指す。
      • log(_); のように渡すことができる。
  • ネストした with ブロック：
    • 内側の with が . の解決および _ をシャドウする。
    • 外側オブジェクトへは通常の変数名でアクセスできる。
  • C# の using スタイルのように、同じブロックに複数の with を並べられる：
with (user), (profile) {
    .name = "Alice";
}
  • 複数ターゲットの with における .Member の解決は、左から順に探索し、最初に見つかった候補に解決される。
  • ただし、同一ブロック内で複数ターゲットが同名メンバー Member を持つ場合は、曖昧さを避けるため 必ずコンパイルエラーとする。
  • 曖昧解決は実装依存にしない（常にコンパイルエラー）。

with はあくまで構文糖衣であり、意味的にはすべて 明示的な obj.member 呼び出しに書き換えた上で 型チェック・コード生成を行うことができる。

4.2.1 Type-based with（class / struct / enum の static / 列挙子ショートカット）

通常の with (expr) に加えて、型名 を対象にした with を利用できる。 

with (TypeName) {
    .SomeStaticMethod();
    .SomeStaticField = 1;
}

ここで TypeName は次のいずれか：

  • class 型
  • struct 型
  • enum / bitfield enum

この場合、with は「型コンテキスト with」として解釈される：

  • () 内の式は実行時に値として評価されない。
    • コンパイラは TypeName を型として解決し、インスタンス生成などは行わない。
  • ブロック内では：
    • .Member は TypeName.Member の省略形。
      • 対象となる Member は
        • static フィールド
        • static メソッド
        • enum / bitfield enum の列挙子
    • 一時変数 _ は、TypeName の static コンテキストを表す擬似オブジェクトとして振る舞う。
      • _.Member も TypeName.Member と等価。
    • 型ベース with 内でインスタンスメンバーにアクセスすることはコンパイルエラー。

型ベース with でもローカル変数や外側スコープの変数が同名で存在すればそちらが優先される。 列挙子や static メンバーを明示したいときは TypeName.Member または .Member / _.Member を用いる。

例：class / struct の static アクセス
struct AngleUtil {
    static field Pi: float = 3.14159;

    static method degToRad(deg: float): float {
        return deg * Pi / 180.0;
    }
}

class MathUtil {
    static method max(a: int, b: int): int {
        if (a > b) return a;
        return b;
    }
}

function main(args: string[]): int {
    with (AngleUtil) {
        var rad = degToRad(90.0);
        print(.Pi);
    }

    with (MathUtil) {
        var m = max(10, 20);
        print(m);
    }
    return 0;
}
例：enum の列挙子アクセス
enum Color {
    Red,
    Green,
    Blue
}

function main(args: string[]): int {
    with (Color) {
        var c1: Color = Red;
        var c2: Color = Blue;
        print(c1);
        print(c2);
    }
    return 0;
}

型ベース with も複数ターゲット構文に混在させられる： 

with (user), (Color) {
    .name = "Alice";
    var c: Color = Red;
}

この場合の .Member 解決も、左から順に探索して最初に見つかった候補に解決される。 ただし同名候補が複数存在する場合は必ずコンパイルエラーとする。

5. Variables & Constants（変数と定数）

5.1 var（型推論）

var i = 10;
var f = 10.5;
var s = "Hello";
var c = 'A';
var b = 255;
var arr = [1, 2, 3];
var lst = (1, 2, 3);
var st  = {1, 2, 3};
var mp  = ["key": 10];
var fb  = new FizzBuzz();
  • 変数の型は右辺の式から推論される。
  • var x = 1; は常に int。
  • var x = 1.0; は float として扱われる。
  • var x = invalid; はコンパイルエラー（invalid は代入・初期化に使用できない）。
  • any 型の式を右辺に置いた場合、推論結果の型も any になる。
function SomeFunc(): any {
    return 123;
}

function main(args: string[]): int {
    var x = 1;            // int
    var a = 123;          // int
    var v = SomeFunc();   // any
    return 0;
}

5.2 let（明示的な型）

let i: int = 10;
let f: float = 10.5;
let d: decimal = 10.12345678901234567890:d;
let s: string = "Hello, Aloe!";
let c: char = 'A';
let b: byte = 255;

let arr: int[] = [1, 2, 3];
let lst: int() = (1, 2, 3);
let st : int{} = {1, 2, 3};
let mp : map<string/int> = ["key": 10];
  • let x: Foo = invalid; のような invalid 代入はコンパイルエラー（invalid は代入・初期化に使用できない）。

5.3 Constants（const）

const MaxCount: int = 100;
const Pi: float = 3.14159;
const AppName: string = "Aloe VM";
  • const 変数は再代入不可。
  • const には型注釈が必須。

6. Fields（フィールド宣言）

class / struct のフィールドは field キーワードで宣言する。

class Sample {
    field count: int = 0;
    field name: string = "default";

    readonly field id: int;

    construct(id: int) {
        this.id = id;
    }
    method change(): void {
        this.count = this.count + 1;
        // this.id = 2; // NG
    }
}
  • field ... 通常のフィールド。
  • readonly field ... コンストラクタ内のみ代入可、それ以降は読み取り専用。
  • struct / class の両方で同じ意味。

7. Class / Struct / Interface / Trait

7.1 class

class FizzBuzz {
    field _count: int = 0;
    field name: string = "FizzBuzz";

    construct() {
    }

    method run(): void {
        var i = 1;
        while (i <= 100) {
            // ...
            i = i + 1;
        }
    }
}
  • class は参照型。
  • sealed class は extends による継承不可。
  • 単一継承のみサポート。

7.2 struct

struct Point {
    field x: int;
    field y: int;

    construct(x: int, y: int) {
        this.x = x;
        this.y = y;
    }
}
  • struct は値型。
  • フィールドはプリミティブ型か他の struct に限定（参照型フィールドは禁止）。
  • struct のコピーは常に値コピー。
  • struct の継承はサポートしない（sealed struct は「拡張禁止」マーカー）。

Struct layout annotations: @StructLayout / @FieldOffset

ネイティブコードとの相互運用などで、struct のメモリレイアウトを制御したい場合のために、 struct には限定的なアノテーション @StructLayout / @FieldOffset を付与できる。 これらは Aloe に組み込みで定義されているアノテーションであり、後述の汎用アノテーション機構の一例でもある。 

@StructLayout("Explicit")
struct NativePoint {
    @FieldOffset(0) field X: int;
    @FieldOffset(4) field Y: int;
}
  • @StructLayout("...")
    • struct 宣言の直前にのみ付けられるアノテーション。
    • 引数にはレイアウト種別を表す文字列を指定する。
      • "Auto"（または省略時）: レイアウトは VM 実装依存。C 構造体との ABI 互換は保証しない。
      • "Sequential": フィールド宣言順に並べることを VM に要求する（アラインメントなどの詳細は実装依存）。
      • "Explicit": 各フィールドに @FieldOffset を指定して、バイトオフセットを明示する。
    • 現時点で仕様として定義するのは第 1 引数のレイアウト種別文字列のみとする。
  • @FieldOffset(n)
    • struct 内の field 宣言の直前にのみ付けられるアノテーション。
    • 引数 n は 0 以上の整数リテラルで、struct 先頭からのバイトオフセットを表す。
    • @StructLayout("Explicit") が付いた struct 内でのみ有効であり、それ以外のレイアウト種別で使用した場合はコンパイルエラー。
    • 同じ struct 内でメモリ領域が重なった場合の扱いは実装依存だが、多くの実装ではコンパイルエラーまたは警告とすることが推奨される。
  • これらのアノテーションは VM / ランタイムのネイティブインターフェイスに対するヒントであり、 通常の Aloe コードにおける型システムの意味論を変えるものではない。

7.2.1 汎用アノテーション機構（annotation 定義）

Aloe では、@StructLayout や @FieldOffset のような組み込みアノテーションに加えて、 ユーザが独自のアノテーションを定義・利用できる。 アノテーションは annotation 宣言で定義する。 

annotation MyAnnotation {
    text:   "アノテーションテキスト";
    target: TargetKind;
    params: map<string/any>;

    build:  function (ctx: AnnotationBuildContext): void {
        // コンパイラから呼び出されるフック関数（擬似シグネチャ例）
    }
}
  • annotation 宣言は、アノテーションのメタ定義を行うための専用構文とする。
  • text は説明用文字列（任意）。
  • target は付与できる対象種別を表す。
  • params は引数の論理表現。
  • build はコンパイラから呼び出されるフック関数。

7.3 Inheritance and sealed

class Base {
    method foo(): void { }
}

class Derived extends Base {
    method bar(): void { }
}

sealed class FinalDerived extends Base {
}

7.4 interface

interface ILogger {
    method log(msg: string): void;
}

class ConsoleLogger implements ILogger {
    method log(msg: string): void {
        print(msg);
    }
}
  • implements キーワードでインタフェースを実装。
  • is 演算子でインタフェース実装有無を判定。

7.5 trait

trait Trait_A {
    method hello(): void {
        print("Hello from Trait_A");
    }
}

with による trait の適用

var obj = new SomeClass() with Trait_A;
obj.hello();

非破壊的な追加 / 削除（+ / -）

var obj2 = obj + Trait_A;
var obj3 = obj2 - Trait_A;

as によるエイリアス

var obj2 = obj + Trait_A as a;
obj2.a.hello();

Trait conflicts（競合）

  • 複数 trait が同名フィールド / メソッドを定義する場合、右側に記述した trait が左側の定義を上書きする。
  • クラス本体のメンバーと trait メンバーが同名の場合も同様に、最後に適用された trait のメンバーで上書きされるとみなす。
  • 上書きを避けたい場合は as a のようなエイリアスを用いる。
  • エイリアス付き適用は衝突判定の対象外とする。

trait の expects 句

interface IHasId {
    method GetId(): int;
}

trait HasIdFormatting expects IHasId {
    method PrintId(): void {
        print("ID = " + this.GetId());
    }
}
  • trait X expects IA, IB, ... { ... } のように、expects 句で 1 個以上のインタフェース型を指定できる。
  • expects で指定された各インタフェースは、trait を適用されるオブジェクトがすべて実装していなければならない。満たさない場合はコンパイルエラー。
  • is 演算子の左辺は必ず式（値）でなければならない（9章参照）。

8. delete & invalid

var obj = new FizzBuzz();
delete obj;  // obj becomes invalid (at VM level)
  • delete obj; は「参照変数 obj を無効（invalid）状態にする」コマンドである。
  • 同一オブジェクトを参照している別の参照変数には干渉しない。
  • オブジェクトの実際の回収は実行系が決定する（回収時刻は未規定）。
  • ソースコード中で obj = invalid; のように invalid を値として代入することはできない（コンパイルエラー）。
  • invalid は比較・判定のためにのみ使用できる。（9章参照）
  • invalid 状態でのメンバーアクセスは InvalidReferenceException を引き起こす。
  • obj がすでに invalid の場合、delete obj; は無効果であり例外を送出してはならない。

9. is Operator（is 演算子）

if (obj is FizzBuzz) {
}

if (obj is ILogger) {
}

if (obj is Trait_A) {
}
  • is は実行時の型チェックを行う。
  • 継承関係を考慮する。
  • インタフェースや trait に対しても使用できる。
  • is 演算子の左辺は必ず式（値）でなければならない。
  • 参照の有効性判定として、次をサポートする：
if (obj is valid) {
}

if (obj is invalid) {
}
  • obj is valid は obj が有効参照であるとき true。
  • obj is invalid は obj が無効参照であるとき true。
  • valid / invalid は代入値としては使用できない（8章参照）。

10. Collections（配列 / リスト / セット / マップ）

10.1 Arrays (T[])

var arr: int[] = [1, 2, 3];
var matrix: float[][] = [[1.0, 2.0], [3.0, 4.0]];

10.2 Lists (T())

var lst: int() = (1, 2, 3);
var objLst: FizzBuzz() = (new FizzBuzz(), new FizzBuzz());

10.3 Sets (T{})

var st: int{} = {1, 2, 3};
var structSt: Point{} = {Point { x:0, y:0 }, Point { x:1, y:1 }};

10.4 Maps (map<K/V>)

var mp: map<int> = ["key1": 10, "key2": 20];
var mp2: map<string/float> = ["key1": 10.5, "key2": 20.5];
var objMp: map<FizzBuzz> = ["obj1": new FizzBuzz(), "obj2": new FizzBuzz()];

11. enum and bitfield enum

11.1 Normal enum

enum Color {
    Red,
    Green,
    Blue
}
  • 内部表現は常に int32。
  • 基礎型の変更は許可されない。
  • enum 型も型ベース with の対象となる。

11.2 bitfield enum

bitfield enum LogFlags {
    None = 1 << 0,
    Info = 1 << 3,
    Warn,
}

b(n) 記法

bitfield enum LogFlags {
    None : b(0),
    Info : b(3),
    Warn,
}
  • b(n) はビット n（1 << n）を表す。
  • bitfield enumで使用可能なビットは最大 32（n は 0-31）。
  • 同一ビットを共有する列挙子があっても、コンパイラは警告にとどめ、エラーにはしない。
  • 基礎型は常に int32。

12. main Block（エントリポイント）

function main(args: string[]): int {
    var fb = new FizzBuzz();
    fb.run();
    return 0;
}
  • プログラムのエントリポイント。
  • args はコマンドライン引数。
  • main ブロックは必ず function main(args: string[]): int の形で宣言する。
  • main ブロック内の特別な一時変数 _ の仕様は存在しない（通常の一時変数のみ）。

13. Temp Variable _（一時変数）

function main(args: string[]): int {
    var x = 10;
    var _ = x + 5;
    return 0;
}

method foo(): void {
    var x = 10;
    _ = x + 5;
}
  • _ は任意のブロックで宣言なしに使用できる特別な一時変数として扱われる。
  • main では通常のローカル変数として扱われる。
  • main 以外のブロックでは、そのブロックで最初に代入された式の型が _ の型になる。
  • with ブロック内では「現在の with 対象」を指す意味を持つ（4.2参照）。

14. swap Keyword（swap キーワード）

var a = 10;
var b = 20;
swap (a, b);
print(a);
print(b);
  • swap (x, y); は x と y の値（または参照）を入れ替える。
  • swap が妥当かどうかはコンパイル時にチェックされる。

15. Operators（演算子）

15.1 優先順位（高い順）

  1. メンバー / 呼び出し：x.y, x[y], x(args...)
  2. 単項：+x, -x, !x, delete x
  3. 乗除：*, /, %
  4. 加減：+, -
  5. シフト：<<, >>
  6. 関係：<, <=, >, >=
  7. 等価：==, !=
  8. ビット AND：&
  9. ビット XOR：^
  10. ビット OR：|（pipeline 文では別語義）
  11. 論理 AND：&&（エイリアス：and）
  12. 論理 OR：||（エイリアス：or）
  13. 代入：=

複合代入（+= 等）はまだ仕様化されていない。

15.2 結合規則

  • 単項演算子：右結合。
  • 代入：右結合。
  • その他の 2 項演算子：左結合。

16. Exceptions（例外）

try {
    // code that might throw
}
catch (e) {
    // handle exception
}
finally {
    // cleanup
}

16.1 標準例外の概要

Exception
+- SystemException
|  +- InvalidReferenceException
|  +- IndexOutOfBoundsException
|  +- OverflowException
|  +- ZeroDivisionException
|  +- InvalidOperationException
|  +- NotImplementedException
|  +- TimeoutException
|  +- ArgumentException
|  |  +- ArgumentNullException
|  |  +- ArgumentOutOfRangeException
|  +- FormatException
+- IOException
   +- FileNotFoundException
   +- EndOfStreamException
   +- IOException (general)
  • InvalidReferenceException は invalid 参照へのアクセスで投げる。
  • OverflowException はオーバーフロー / アンダーフロー両方に用いる。
  • ZeroDivisionException は 0 / 0.0 による除算で投げる。

17. Modules / namespace / import

17.1 Module（モジュール / コンパイル単位）

Aloe における モジュール は、コンパイル対象となる 1 つのソースファイルを指す。 言語仕様のスコープ外として、モジュール名やファイルパス・ディレクトリ構成・パッケージ管理は ビルドツールやパッケージマネージャ側の責務とする。

この章では、1 ファイル内部での import / namespace の書式と それによる名前解決のルールのみを定義する。

17.2 ファイル構造の制約

import Foo.Bar;
import Aloe.Core.Logging;
import Aloe.Json.*;
import MyCompany.Project.Module;

namespace My.App {
    // declarations only
}
  • import はファイルの先頭にのみ書ける。
  • namespace は、すべての import の後に 1 回だけ現れなければならない。
  • トップレベルに実行文を書くことは禁止。

17.3 namespace のルール

  • 1 モジュールにつき namespace 文はちょうど 1 回だけ許可される。
  • namespace のネスト構文は禁止とする。
  • 同じ namespace 名を持つ宣言が複数モジュールに存在する場合、論理的にマージされる。

17.4 import のルール

17.4.1 文法と制限

  • import はファイル先頭にのみ書ける。
  • 相対 import はサポートしない。
  • import はすべて絶対指定でなければならない。

17.4.2 ワイルドカード import の挙動

  • import Foo.*; は Foo namespace 直下の公開シンボルを導入する。
  • 同名シンボルが複数導入され解決が曖昧になる場合はコンパイルエラー。

17.5 名前解決の優先順位（概要）

ある識別子 X を解決するときの大まかな優先順位は次の通りとする。 これは仕様として固定され、実装依存にしない。

  1. ローカルスコープ
  2. 現在のクラス / struct / trait / function 自身のメンバー
  3. 現在の namespace 内の宣言
  4. import された namespace 達の公開メンバー

この優先順位で複数の候補が見つかり、どれを指しているかが曖昧になる場合は 必ずコンパイルエラーとする。

17.6 可視性と公開範囲

  • public は他の namespace / モジュールから参照できる。
  • private は同一モジュール内でのみ参照できる。
  • protected は継承関係にある型から参照できる。

17.7 循環 import の扱い

  • 循環 import 自体は禁止しない。
  • 型の矛盾（例：struct の再帰値包含など）は通常の型エラーとしてコンパイルエラー。

18. Pipes & Filters（パイプとフィルタ）

Aloe は pipe<T> と filter(...) を用いて、 サーバ / CLI / Web アプリなどのストリーミング処理を構成する。

18.1 pipe<T> の基本

  • pipe<T> は T 型要素のストリーム。
  • pipe は共有メモリではなく値コピーを前提とする。

18.1.1 組み込みパイプ

  • stdin : pipe<byte>
  • stdout : pipe<byte>
  • stderr : pipe<byte>（任意）

18.2 Result<T> と ROP（Railway Oriented Programming）の概要

Aloe では、パイプラインやフィルタにおけるエラー処理や分岐を Result<T> 型と Railway Oriented Programming（ROP） の考え方で扱う。

  • Result<T> は成功（Success = true）または失敗（Success = false）を表す標準ライブラリ型。
  • Result に丸める（Result を強制する）のは、パイプ（filter / producer / consumer）の中のみとする。
  • 通常の関数・メソッドは戻り値型を自由に選べる（Result 強制なし）。失敗時の扱い（例外を投げる等）はその関数仕様による。

18.2.1 Result<T> の型仕様

class Result<T> {
    field Success: bool;
    field Value: T;
    field ErrorMessage: string;

    static method Ok(value: T): Result<T>;
    static method Error(message: string): Result<T>;
}
  • Success が true の場合：Value が有効。
  • Success が false の場合：ErrorMessage が有効であり、Value の読み出しは不正。
  • 原則（A）：Result の不正アクセスは「静的に確定できる場合はコンパイルエラー、確定できない場合は実行時例外」とする。
    • 例：if (r.Success == false) { r.Value; } のように不正が静的に確定する場合はコンパイルエラーとしてよい。
    • 不正かどうかが静的に確定できない場合は、実行時に InvalidOperationException を投げる。
  • Result<void> は値を持たない成功を表現するために Result<void>.Ok()（引数なし）を特別に許可する。
  • Result<void> に対して Value を読み出そうとした場合は常に不正であり、InvalidOperationException を投げる（静的に確定できる場合はコンパイルエラーとしてよい）。

18.2.2 ROP（Railway Oriented Programming）のイメージ

入力
  +-> [ Filter A ] -> [ Filter B ] -> [ Filter C ] -> 出力
  |        |              |              |
  |        +-> Error -----+--------------+----> エラー経路

18.3 filter(...) の基本と定義

filter(F) はパイプライン中の中間変換ステージ。

18.3.1 フィルタ定義（filter 定義文）

filter lineSplit {
    in:  pipe<string>;
    out: pipe<string>;

    options: LineSplitOptions;

    field _buffer: string = "";

    constructor(opts: LineSplitOptions) {
    }

    bound(input, output): Result<void> {
        foreach (chunk in input) {
            let lines = chunk.split("\n");
            foreach (line in lines) {
                output.write(line);
            }
        }
        output.close();
        return Result<void>.Ok();
    }
}
  • bound(input, output) はフィルタの実行本体であり、戻り値の型は Result<T> もしくはその派生クラス でなければならない。
  • これに違反した場合（戻り値が Result でない、または void の場合）は コンパイルエラーとする。

18.3.2 フィルタオプション（filter オプション指定）

filter(json<User>: JsonOptions.PrettyPrint | JsonOptions.IncludeInvalids)
  • オプションは : の後に | でつなげて指定。
  • 各フィルタは自前のオプション型（bitfield enum 等）を持つ。
  • options に指定された型に属する列挙子であることがコンパイラによって検証される。

18.4 パイプライン文と | / パイプ with 演算子 |>

stdin
    | filter(utf8)
    | filter(json<DateTime>)
    | dates;
  • A | B | C; で A の出力が B へ、B の出力が C へ接続される。
  • 型チェックにより各ステージの入出力型整合性を確認。
  • パイプライン文そのものは値を返さない。

18.4.1 パイプ with 演算子 |>（フィルタ条件ショートカット）

users
    | filter(if : .Age >= 20 and .Gender == Gender.Male);

users |> .Age >= 20 and .Gender == Gender.Male;
  • |> は内部的には filter(if : ...) に展開される。

18.5 foreach と pipe

foreach (line in lines) {
    print("Input: " + line);
}
  • pipe<T> は hasNext() と next() を提供する。
  • 次の要素が存在しない状態で next() を呼び出した場合、EndOfStreamException を投げる。
  • foreach (x in p) は内部的に次の形に展開される：
while (p.hasNext()) {
    let x = p.next();
    // body
}

18.6 map による要素変換（pipe 用）

function toName(u: User): string {
    return u.Name;
}

names = users | map(toName);

18.7 コレクションに対するテンプレート map

customers = users | map Customer {
    Name    = "FixedName";
    Created = now();
    BaseUrl = base.ApiEndpoint;
};
  • テンプレート内では「固定値」または「固定関数呼び出し」のみでフィールドを構成できる。
  • テンプレート内からコレクションの個々の要素には直接アクセスできない。
  • テンプレ map の禁止事項に違反した場合は基本的にコンパイルエラーとする。

18.8 close と EOF 伝播

  • p.close() 後の write() は無効（エラーになり得る）。
  • フィルタは入力 EOF を検知したら出力側も close() すべき。

18.9 pipe を使った並行モデル

  • タスク同士の主な通信手段は pipe<T>。
  • shared-nothing モデルを前提とする。

18.10 match によるパイプ分岐（Result<T> の分岐）

stdin
    | filter(parseUser)
    | match {
        case Ok(u):
            | filter(saveUser)
            | filter(logSuccess);
        case Error(e):
            | filter(logError);
        default:
            break;
    };

18.11 例：簡単な REST 風パイプライン

class User {
    field id: int;
    field name: string;
}

function main(args: string[]): int {
    let users: pipe<User> = pipe<User>.create();

    stdin
        | filter(utf8)
        | filter(json<User>)
        | users;

    if (users.hasNext()) {
        let u: User = users.next();
        print("User: " + u.name);
    }

    return 0;
}
