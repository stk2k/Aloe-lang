0. Overview（概要）

Aloe は静的型付けのスクリプト言語であり、シンプルで直感的な構文を目指している。
スタックベースの仮想マシン（AloeVM）上で動作し、主に次のような目標を持つ。

  • 教育用途や小規模アプリケーションで読み書きしやすいこと
  • C# ランタイムおよび WebAssembly (WASM) 環境で動作すること
  • pipe と filter によって、I/O と並行処理を静的型付けのまま制御できること

特徴：

  • 静的型付け
    • var による型推論
    • let による明示的な型指定
  • 値型 / 参照型
    • struct ... 値型
    • class / object ... 参照型
    • string ... 値型
  • GC ベースのメモリ管理（具体的なアルゴリズムは VM 実装依存）
  • 参照の無効状態（invalid）を言語レベルのモデルとして導入する
    • delete は「参照変数を無効化する命令」であり、無効化後の参照アクセスは例外となる。
    • invalid は代入や初期化に使えない（有効性チェック用途のみ）。
  • object と box object による boxing / unboxing
    • プリミティブ値を object に保持する場合、対応する box object に boxing される。
    • object からプリミティブを取り出す場合、as / 暗黙 unbox キャスト / unbox を用いる。
  • 数値 box object による文字列パース導線
    • Parse は例外を許可する形でパースする。
    • TryParse は例外を出さずにパースする。
  • pipe / filter によるデータフロースタイルの設計

1. Basic Syntax（基本文法）

1.1 Comments（コメント）

// single-line comment

/* multi-line
   comment */
  • // から行末までが行コメント。
  • /* ～ */ が複数行コメント。
  • 複数行コメントは、開始 /* から最初に現れる */ までを まるごとコメントとして扱う。 コメントの内部に追加の /* や */ が現れても、 それらはすべてコメントの一部として無視される（ネストしたコメントの構文解析は行わない）。
    したがって、コメント内の「コメント構文」が破綻していても、 いったん /* が始まってから最初の */ が来るまでは常にコメント扱いとなる。

1.2 Identifiers（識別子）

  • 使用可能文字：英字 (A-Z, a-z)、数字 (0-9)、アンダースコア _
  • 先頭文字は英字または _ でなければならない。
  • 大文字小文字は区別される。
  • キーワードと同じ綴りは使用できない。
  • _ は通常の識別子として使用できる（特別扱いしない）。
  • ユニコード文字も使用できる（空白や区切り記号などは除外）。

1.4 Literals（リテラル）

  • 整数リテラル：10, 0, -5, 0x1A, 0b1010 など
  • 浮動小数点リテラル：10.5, 0.0, -3.14
  • 10進リテラル：10.12345678901234567890:d など
  • 真偽値：true, false
  • 文字列："Hello, World!"
  • 文字：'A', 'z' など
  • 参照有効性：valid, invalid（有効性チェック専用。代入・初期化には使用できない）
  • 配列リテラル：[1, 2, 3]
  • リストリテラル：(1, 2, 3)
  • セットリテラル：{1, 2, 3}
  • マップリテラル：["key1": 10, "key2": 20]
  • enum リテラル：EnumType.Value1
  • struct リテラル：Point { x: 0, y: 1 }
  • オブジェクト生成：new ClassName()

1.5 Statement terminator（文の終端）

var x = 10;
x = x + 1;
print(x);
  • すべての文はセミコロン ; で終わる。
  • 改行による自動セミコロン挿入は行わない。

1.6 Whitespace（空白）

  • スペース、タブ、改行はトークン区切りとして扱われる。
  • 連続する空白は 1 つの区切りとみなされる。
  • インデントに意味はない（Python とは異なる）。

1.7 Case sensitivity（大文字小文字）

  • 識別子とキーワードは大文字小文字を区別する。
  • キーワードはすべて小文字（Result を除く）。

2. Type System（型システム）

2.1 Primitive types（プリミティブ型）

Aloe における「プリミティブ型」は、次の 8 種類とする。

  • bool / boolean - 真偽値。
  • int / integer - 符号付き整数。
  • float - 浮動小数点数。
  • decimal - 高精度 10 進数。
  • string - 文字列（内部表現は実装依存。観測可能な意味は 2.3.1 に固定する）。
  • byte - 0-255 の符号なし整数。
  • char - 1 文字。
  • void - 戻り値なし（戻り値専用）。

valid / invalid は型ではなく、 参照の有効性を判定するための特殊定数（リテラル）である。
仕様書から null は排除し、参照の無効状態は invalid で表現する。

2.2 Composite / user-defined types（合成型 / ユーザー定義型）

  • 配列：T[]
  • リスト：T()
  • セット：T{}
  • マップ：map<T> または map<K/V>
  • struct
  • class
  • enum, bitfield enum
  • 非同期タスク：task
  • 万能参照型：object
  • パイプ：pipe<T>（18章参照）
  • 結果型：Result<T>（18章参照）

2.3 Value types / reference types（値型 / 参照型）

  • 値型：
    • int, float, decimal, bool, byte, char, string
    • struct
    • enum, bitfield enum（内部表現は int32）
  • 参照型：
    • class
    • object
    • task, pipe<T>, Result<T> など
    • 配列 / リスト / セット / マップ などのコレクション型

2.3.1 string 値型の観測可能な意味（固定）

string は値型として扱う。実装が内部で参照共有等を行ってもよいが、 次の意味は常に守られ、利用者が観測できる挙動はこれに固定される。

  • string 変数は invalid 状態にならない。未初期化宣言時も invalid にはならない（5.2.1 参照）。
  • delete の対象は参照型変数のみであり、delete s;（s が string）は コンパイルエラー。
  • string の代入・引数渡し・戻り値は 値として扱う（値セマンティクス）。
  • string を object に代入する場合、必ず String box object に boxing される（2.4.0）。
  • is valid / is invalid は参照の有効性チェックであるため、string に適用することは コンパイルエラー。

object 型（万能参照型）

  • object は、void を除く任意の値を保持できる特別な参照型である。
  • 参照型の値は object へ暗黙に代入できる（参照コピー）。
  • プリミティブ値（string を含む）を object に代入する場合、対応する box object に boxing される（2.4参照）。
  • object から具体型 T への暗黙変換はない。
  • if (x is T) { ... } のブロック内では、x を T として扱ってよい（フロー依存の型絞り込み）。

Struct restrictions and copying（struct の制約とコピー）

  • struct のフィールドにはプリミティブ型と他の struct のみを持てる。
  • class などの参照型フィールドは持てない。
  • struct の代入は常に値コピー。
  • 最大サイズなどは仕様で制限しない。

Class vs struct（class と struct）

  • class は struct をフィールドとして持てる。
  • class インスタンスを別変数に代入すると参照がコピーされる（オブジェクト本体は共有）。

2.4 boxing / unboxing（box object と ToXxx の正準名）

Aloe は、プリミティブ値を object に格納するために、各プリミティブ型に対応する box object を定義する。
box object は object から派生し、名前は大文字で始まる英単語とする。

  • bool / boolean - Boolean
  • int / integer - Integer
  • float - Float
  • decimal - Decimal
  • string - String
  • byte - Byte
  • char - Char

2.4.0 boxing の正準規則（固定）

  • プリミティブ値（string を含む）を object に代入する場合、必ず対応する box object に boxing する。
  • box object を object に代入する場合は、その参照を保持する（再 boxing はしない）。
  • box object は対応する値を内部に保持するが、value プロパティは公開しない（value は非公開）。
  • unboxing は対応する ToXxx() メソッドで行う。
  • ToXxx の正準名は次の通り：
    • ToBool(), ToInt(), ToFloat(), ToDecimal(), ToString(), ToByte(), ToChar()
  • unboxing：object からプリミティブ値を得るには as（15.1）/ 暗黙 unbox キャスト（5.2.2）/ unbox（15.3）を用いる。

2.4.1 角（invalid 時・変換の禁止）

  • invalid な参照に対して as / 暗黙 unbox キャスト / unbox を評価する場合の挙動は、それぞれの節の規則に従う。
  • 型変換は行わない。 例えば数値型同士の変換や文字列のパースなどは、as / 暗黙 unbox キャスト / unbox の範囲では行わない（失敗として扱う）。

2.4.2 Parse / TryParse（数値 box object の文字列パース導線）

as / 暗黙 unbox キャスト / unbox は型変換を行わない。 そのため、文字列から数値を得る導線は数値 box object の Parse / TryParse により提供する。

  • 対象：Integer, Byte, Float, Decimal
  • Parse は例外を許可する形でパースする。
  • TryParse は例外を出さずにパースする。
  • 共通ルール（受理する文字列）：
    • 前後の空白は許可し、パース前に除去する（trim）。
    • 途中の空白は不許可（書式不正）。
    • 区切り記号（例：_, ,）は不許可（書式不正）。
    • 数字は ASCII の 0-9 のみを受理する（全角数字等は不許可）。
  • エラー型（正準）：
    • 書式不正は FormatException。
    • 範囲外 / 表現不能は OverflowException。
2.4.2.1 Parse の規則
  • シグネチャ（概念）：static method Parse(text: string): P
  • 失敗時の例外：
    • 書式不正は FormatException。
    • 範囲外 / 表現不能は OverflowException。
2.4.2.2 TryParse の規則
  • シグネチャ（概念）：static method TryParse(text: string): Result<P>
  • 成功時：値を含む成功結果を返す。
  • 失敗時：例外は投げず、失敗情報を含む失敗結果を返す（失敗理由は FormatException / OverflowException に正規化する）。
2.4.2.3 各型の受理フォーマット（固定）
  • Integer（int）：
    • 10進：[+|-]? + 1 文字以上の数字
    • 16進：[+|-]? + 0x/0X + 1 文字以上の 16 進数字
    • 2進：[+|-]? + 0b/0B + 1 文字以上の 0/1
    • 小数点・指数表記は不許可（書式不正）。
  • Byte（byte）：
    • 10進：1 文字以上の数字（符号は不許可）
    • 16進：0x/0X + 1 文字以上の 16 進数字
    • 2進：0b/0B + 1 文字以上の 0/1
    • 範囲外（0..255）は OverflowException / TryParse 失敗。
  • Float（float）：
    • 10進のみ。符号は任意（+/-）。
    • 小数点 . を許可する。
    • 指数表記 e/E を許可する（指数部の符号 +/- を許可）。
    • 16進 / 2進プレフィックスは不許可（書式不正）。
    • Aloe の数値体系には NaN / Infinity の概念は存在しない。 NaN, Infinity, -Infinity 等の文字列は受理してはならない（書式不正）。
    • 表現不能（オーバーフロー / アンダーフロー等）は OverflowException / TryParse 失敗。
  • Decimal（decimal）：
    • 10進のみ。符号は任意（+/-）。
    • 小数点 . を許可する。
    • 指数表記は不許可（書式不正）。
    • 表現不能（範囲外 / 精度上の表現不能等）は OverflowException / TryParse 失敗。

3. Numbers & Conversions（数値と暗黙変換）

3.1 int and float

  • int と float を混在させた式の結果は float になる。
    • int + float -> float
    • int - float -> float
    • int * float -> float
    • int / float -> float

3.2 decimal

  • decimal は高精度な 10 進数。
  • int / float と混在する式は、必要に応じて decimal に昇格しうる。
  • オーバーフロー / アンダーフロー / 精度問題は OverflowException として扱う。

3.3 enum and int

  • Aloe にはキャスト構文がない。
  • enum と int の暗黙 / 明示変換はいずれも禁止。
  • enum と数値型の混在式はコンパイルエラー（bitfield enum のビット演算を除く）。
  • 内部表現としては enum / bitfield enum はすべて int32。

3.4 byte and char

  • byte：
    • 範囲は 0-255。
    • byte 同士の演算は許されるが、範囲外はエラーになり得る。
    • byte と int の混在式は int に昇格。
  • char：
    • char の算術演算は不可。
    • 比較演算（==, <, > 等）は char 同士のみ。
    • 数値型との暗黙変換はない。

3.5 NaN / Infinity（概念として存在しない）

  • Aloe の数値体系には NaN / Infinity の概念は存在しない。
  • 実行時に NaN / Infinity を生成・保持・観測できてはならない。
  • 数値オーバーフロー / アンダーフロー等の表現不能は常に OverflowException として扱う。
  • ZeroDivisionException は 0 / 0 および 0.0 / 0.0 で常に投げる。
  • Float.Parse / Float.TryParse は NaN, Infinity 等の文字列を受理してはならない（2.4.2.3）。

4. Blocks & Scope（ブロックとスコープ）

ブロックは { と } で区切られ、各ブロックが新しいスコープを導入する。

  • main ブロック
  • クラス定義ブロック
  • メソッド定義ブロック
  • 任意の { ... } ブロック

4.1 Static (lexical) scope and shadowing（静的スコープとシャドウイング）

class Sample {
    field value: int = 1;
    method foo(): void {
        var value = 2;        // フィールドをシャドウ
        print(value);         // 2
        print(this.value);    // 1
    }
}
  • Aloe は静的（レキシカル）スコープ。
  • 内側ブロックで同名変数を宣言すると、外側の変数をシャドウする。
  • フィールド名とローカル変数名がぶつかった場合、ローカル変数が優先される。フィールドは this. で参照する。

4.2 with-blocks（メンバー省略記法）

Aloe には、同一オブジェクトのメンバーアクセスを簡潔に書くための with ブロックがある。 

with (user) {
    .name = "Alice";
    .age = .age + 1;
    print(.name);
}
  • with (expr) { ... } ブロック内では：
    • .Member は (expr).Member の省略形。
    • expr はブロックの先頭で 1 度だけ評価され、内部で保持される（再評価しない）。
  • _ は特別扱いしない。 with 対象を表す暗黙変数は存在しない。
  • ネストした with ブロック：
    • 内側の with が . の解決対象となる。
    • 外側オブジェクトへは通常の変数名でアクセスできる。
  • C# の using スタイルのように、同じブロックに複数の with を並べられる：
with (user), (profile) {
    .name = "Alice";  // resolved against user, then profile
}
  • 解決順序は 左から右 とする。
  • 左から探索して最初に見つかったメンバーに解決する。
  • ただし同名メンバーが複数ターゲットに存在し、解決が曖昧になる場合は コンパイルエラー。

with は構文糖衣であり、意味的にはすべて明示的な obj.member 呼び出しに書き換えた上で 型チェック・コード生成を行うことができる。

4.2.1 Type-based with（class / struct / enum の static / 列挙子ショートカット）

通常の with (expr) に加えて、型名 を対象にした with を利用できる。 

with (TypeName) {
    .SomeStaticMethod();
    .SomeStaticField = 1;
}

ここで TypeName は次のいずれか：

  • class 型
  • struct 型
  • enum / bitfield enum

この場合、with は「型コンテキスト with」として解釈される：

  • () 内の式は実行時に値として評価されない。
    • コンパイラは TypeName を型として解決し、インスタンス生成などは行わない。
  • ブロック内では：
    • .Member は TypeName.Member の省略形。
    • 型ベース with 内でインスタンスメンバーにアクセスすることはコンパイルエラー。

型ベース with も複数ターゲット構文に混在させられる。解決順序は左から右であり、曖昧ならコンパイルエラーとする。

4.3 名前解決の正式ルール（曖昧ならコンパイルエラー）

Aloe は「曖昧ならコンパイルエラー」を原則とし、名前解決の探索順を次のように固定する。 本節は、識別子（非修飾名）および型名の解決に適用される。

4.3.1 非修飾識別子（x）の探索順

  1. 現在のブロックのローカル変数 / 引数（最内側スコープから外側へ）
  2. （メソッド内の場合）同名のフィールド（暗黙 this スコープ）
  3. 同一 namespace 内のトップレベル宣言（関数 / 型 / 定数など）
  4. import により導入された名前（明示 import されたもの）
  5. 外側 namespace（親 namespace へ遡る）
  • 同一段階で複数候補が存在する場合は コンパイルエラー。
  • 型名と値名（変数名など）が同名で衝突し、構文から区別できない場合は コンパイルエラー。

4.3.2 メンバー名（obj.Member）の探索

  • obj.Member は、obj の静的型に基づき、該当するメンバーを解決する。
  • 継承がある場合、派生型から基底型方向へ探索する。
  • 同名候補が複数（オーバーロード等）で、引数型から一意に確定できない場合は コンパイルエラー。

4.3.3 with ブロック内の .Member

  • .Member は、最内側の with から外側へ、かつ同一 with 内では左から右の順で探索する（4.2参照）。
  • 複数ターゲットで解決が曖昧になる場合は コンパイルエラー。

5.2.1 未初期化宣言（デフォルト値 / invalid）

let による宣言では、初期化を省略した宣言を許可する。 この場合の初期状態は型カテゴリで決定する。 

let i: int;        // デフォルト値（0）
let s: string;     // デフォルト値（""）
let obj: Foo;      // invalid（参照型の未初期化宣言）
  • 値型（プリミティブ / struct / enum）はデフォルト値で初期化する。
  • 参照型（class / object / コレクション等）は invalid 状態で初期化する。
  • string は値型であり、デフォルト値は 空文字列 "" とする。
  • 未初期化参照（invalid）へのメンバーアクセスは InvalidReferenceException を投げる。
let obj: Foo;        // invalid
obj.Value;           // 実行時例外（InvalidReferenceException）

5.2.2 暗黙 unbox キャスト（非消費）

期待される型がプリミティブ型 P に固定できるコンテキストでは、 式 E に対して 暗黙 unbox キャスト を許可する。
暗黙 unbox キャストは 参照を消費しない（変数を invalid にしない）。

暗黙 unbox キャストが許可される代表的なコンテキスト：

  • let x:P = E;（初期化）
  • x = E;（代入。x の型が P）
  • return E;（関数の戻り値型が P）
  • f(E)（引数型が P）
  • 暗黙 unbox キャストは、演算子型付け（例：E + 1 のため）には適用しない。 必要な場合は E as P を明示するか、一旦 let t:P = E; のように受ける。

暗黙 unbox キャストは意味的には E as P と等価であり、失敗時は InvalidCastException を投げる。 ifnot による代替値指定は、明示 as（15.1）でのみ行える。 

let o: object = 123;   // boxing: Integer
let i: int = o;        // OK（暗黙 unbox キャスト） == (o as int)
  • E が invalid 参照の場合：InvalidReferenceException（InvalidCastException ではない）。
  • 成功条件：
    • E がプリミティブ P の値である。
    • または E が対応する box object（例：Integer）であり、ToXxx() で取り出せる。
  • 型変換は行わない。 数値型同士の変換や文字列パース等は行わず、該当しなければ失敗とする。

5.2.3 推奨（as / 暗黙 unbox / unbox）

  • 失敗を分岐で扱いたい場合は as P ifnot ... を推奨する。
  • 短く書きたい場合は暗黙 unbox キャストを許可するが、失敗時は例外となる。
  • 値の取り出しと同時に「参照を消費したい」場合のみ unbox を用いる。

5.3 Constants（const）

const MaxCount: int = 100;
const Pi: float = 3.14159;
const AppName: string = "Aloe VM";

  • const には型注釈が必須。

8. delete & invalid

8.1 参照の有効性モデル（valid / invalid）

Aloe では、参照が「有効（valid）」か「無効（invalid）」かという状態をモデル化する。
invalid は 代入や初期化に使用できないが、参照の有効性チェックには使用できる。 

var obj = new FizzBuzz();
delete obj;           // obj becomes invalid
if (obj is invalid) {
    print("invalid");
}
  • delete x; は参照変数 x の状態を invalid にする操作である。
  • delete は解放のためのステートメントではなく、あくまで「無効化にする処理」である。
  • delete は例外を投げてはならない。 既に invalid の場合も無効果である。
  • 無効化後、x を通じた参照先のメンバーアクセスは InvalidReferenceException を引き起こす。
  • 同一オブジェクトを参照している別の参照変数には干渉しない（参照の無効化は変数単位）。
  • オブジェクトの実際の解放（回収）は実行系が決定する（GC）。
  • invalid は比較・判定用であり、x = invalid や let x: Foo = invalid はコンパイルエラー。

8.2 invalid 参照アクセス

let obj: Foo;     // invalid
obj.Value;        // throws InvalidReferenceException
  • invalid 参照へのメンバーアクセスは必ず InvalidReferenceException。
  • （実装が静的に invalid を確定できても）原則として挙動は同じとし、最適化は実装依存とする。

15. Operators（演算子）

15.1 as 演算子と ifnot（キャストと代替）

Aloe には括弧キャスト構文がなく、型変換は as 演算子で行う。
as は失敗時に InvalidCastException を投げる。ifnot により代替値、または throw を指定できる。 

let o: object = 123;                   // boxing: Integer

let i1: int = o as int;                // OK（unboxing）。失敗時は InvalidCastException
let i2: int = o as int ifnot -1;       // 失敗時は -1 を使う
let i3: int = o as int ifnot throw;    // 失敗時は InvalidCastException（省略時と同じ）
  • E as T の右辺 T が参照型の場合、実行時の型チェックを行う。
  • E as P の右辺 P がプリミティブ型の場合：
    • E がプリミティブ P なら成功。
    • E が対応する box object（例：Integer）なら ToXxx() により成功。
    • それ以外は失敗（InvalidCastException）。
  • ifnot を省略した場合は ifnot throw と同義とする。
  • 型変換は行わない。 数値型同士の変換、文字列パース等は行わない。
  • ifnot 節は、キャストが失敗した場合にのみ評価する。
  • E が invalid 参照の場合は InvalidReferenceException。 これはキャスト失敗ではなく参照エラーであり、ifnot では捕捉しない。

15.3 unbox 演算子（消費 unboxing）

unbox は、参照変数からプリミティブ値を取り出し、成功時にその変数を invalid にする「消費 unboxing」である。 

let o: object = 123;         // boxing: Integer
let i: int = unbox o;        // i = 123, o becomes invalid
  • unbox のオペランドは代入可能な参照変数（lvalue）でなければならない。
  • unbox は、期待される型がプリミティブ型 P に固定できるコンテキストでのみ使用できる。
  • 評価規則（x は 1 回だけ評価）：
    • x が invalid の場合：InvalidReferenceException（x は変化しない）。
    • unboxing に失敗する場合：InvalidCastException（x は変化しない）。
    • unboxing に成功する場合：値を返し、その後 delete x; と同様に x を invalid にする。
  • 暗黙 unbox キャスト（5.2.2）は非消費であり、unbox は消費である。
