この文書は、英語版仕様 https://stk2k.github.io/Aloe-lang/Documents/en/Aloe-lang-spec-en.html をベースにした日本語ドラフトです。

0. Overview（概要）

Aloe は静的型付けのスクリプト言語であり、シンプルで直感的な構文を目指している。
スタックベースの仮想マシン（AloeVM）上で動作し、主に次のような目標を持つ。

  • 教育用途や小規模アプリケーションで読み書きしやすいこと
  • C# ランタイムおよび WebAssembly (WASM) 環境で動作すること
  • pipe と filter によって、I/O と並行処理を静的型付けのまま制御できること

特徴：

  • 静的型付け
    • var による型推論
    • let による明示的な型指定
  • 値型 / 参照型
    • struct … 値型
    • class … 参照型
  • GC ベースのメモリ管理（具体的なアルゴリズムは VM 実装依存）
  • = null をソースコード中に直接書かない
    • 参照型は内部的には null を取り得るが、delete による糖衣構文で扱う。
  • pipe / filter によるデータフロースタイルの設計

1. Basic Syntax（基本文法）

1.1 Comments（コメント）

// single-line comment

/* multi-line
   comment */
  • // から行末までが行コメント。
  • /* ～ */ が複数行コメント。
  • ネストした /* */ は未定義（サポートしない想定）。

1.2 Identifiers（識別子）

  • 使用可能文字：英字 (A–Z, a–z)、数字 (0–9)、アンダースコア _
  • 先頭文字は英字または _ でなければならない。
  • 大文字小文字は区別される。
  • キーワードと同じ綴りは使用できない。
  • _ で始まる識別子は「プライベート」扱い（外部からは参照しない慣習）。
  • ユニコード文字も使用できる（空白や区切り記号などは除外）。

  • package は使わず、namespace を使用する。

1.4 Literals（リテラル）

  • 整数リテラル：10, 0, -5, 0x1A, 0b1010 など
  • 浮動小数点リテラル：10.5, 0.0, -3.14
  • 10進リテラル：10.12345678901234567890:d など
  • 真偽値：true, false
  • 文字列："Hello, World!"
  • 文字：'A', 'z' など
  • null リテラル：null
  • 配列リテラル：[1, 2, 3]
  • リストリテラル：(1, 2, 3)
  • セットリテラル：{1, 2, 3}
  • マップリテラル：["key1": 10, "key2": 20]
  • enum リテラル：EnumType.Value1
  • struct リテラル：Point { x: 0, y: 1 }
  • オブジェクト生成：new ClassName()

1.5 Statement terminator（文の終端）

var x = 10;
x = x + 1;
print(x);
  • すべての文はセミコロン ;で終わる。
  • 改行による自動セミコロン挿入は行わない。

1.6 Whitespace（空白）

  • スペース、タブ、改行はトークン区切りとして扱われる。
  • 連続する空白は 1 つの区切りとみなされる。
  • インデントに意味はない（Python とは異なる）。

1.7 Case sensitivity（大文字小文字）

  • 識別子とキーワードは大文字小文字を区別する。
  • キーワードはすべて小文字。

2. Type System（型システム）

2.1 Primitive types（プリミティブ型）

Aloe における「プリミティブ型」は、次の 7 種類とする。

  • bool / boolean – 真偽値。
  • int / integer – 符号付き整数。
  • float – 浮動小数点数。
  • decimal – 高精度 10 進数。
  • string – 文字列（内部的には Unicode）。
  • null – 「値が存在しない」ことを表す特別な型兼値。
  • void – 戻り値なし（メソッド戻り値専用）。

any 型は存在せず、動的型付けはサポートしない。
char や byte は言語としてはサポートされるが、プリミティブ型のカテゴリからは外し、 数値と変換の章（3 章）で別途扱う値型とする。

2.2 Composite / user-defined types（合成型 / ユーザー定義型）

  • 配列：T[]
  • リスト：T()
  • セット：T{}
  • マップ：map<T> または map<K/V>
  • struct
  • class
  • enum, bitfield enum
  • 非同期タスク：task
  • パイプ：pipe<T>（18章参照）
  • 汎用結果型：Result<T>（18.0.1 節参照）

any 型は存在せず、動的型付けはサポートしない。

2.3 Value types / reference types（値型 / 参照型）

  • 値型：
    • int, float, decimal, bool, byte, char
    • struct
    • enum, bitfield enum（内部表現は int32）
  • 参照型：
    • class
    • string（実装上は参照型）
    • task, pipe<T> など

Struct restrictions and copying（struct の制約とコピー）

  • struct のフィールドにはプリミティブ型と他の struct のみを持てる。
  • class などの参照型フィールドは持てない。
  • struct の代入は常に値コピー。
  • 最大サイズなどは仕様で制限しない。

Class vs struct（class と struct）

  • class は struct をフィールドとして持てる。
  • class インスタンスを別変数に代入すると参照がコピーされる（オブジェクト本体は共有）。

3. Numbers & Conversions（数値と暗黙変換）

3.1 int and float

  • int と float を混在させた式の結果は float になる。
    • int + float → float
    • int - float → float
    • int * float → float
    • int / float → float

3.2 decimal

  • decimal は高精度な 10 進数。
  • int / float と混在する式は、必要に応じて decimal に昇格しうる。
  • オーバーフロー / アンダーフロー / 精度問題は OverflowException として扱う。

3.3 enum and int

  • Aloe にはキャスト構文がない。
  • enum と int の暗黙 / 明示変換はいずれも禁止。
  • enum と数値型の混在式はコンパイルエラー（bitfield enum のビット演算を除く）。
  • 内部表現としては enum / bitfield enum はすべて int32。

3.4 byte and char

  • byte：
    • 範囲は 0–255。
    • byte 同士の演算は許されるが、範囲外はエラーになり得る。
    • byte と int の混在式は int に昇格。
  • char：
    • char の算術演算は不可。
    • 比較演算（==, <, > 等）は char 同士のみ。
    • 数値型との暗黙変換はない。

3.5 NaN / Infinity

  • 数値オーバーフロー/アンダーフローは NaN/Infinity ではなく例外で扱う。
  • ZeroDivisionException は 0 / 0.0 で常に投げる。

4. Blocks & Scope（ブロックとスコープ）

ブロックは { と } で区切られ、各ブロックが新しいスコープを導入する。

  • main ブロック
  • クラス定義ブロック
  • メソッド定義ブロック
  • 任意の { ... } ブロック

4.1 Static (lexical) scope and shadowing（静的スコープとシャドウイング）

class Sample {
    field value: int = 1;
    method foo(): void {
        var value = 2;        // shadows the field
        print(value);         // 2
        print(this.value);    // 1
    }
}
  • Aloe は静的（レキシカル）スコープ。
  • 内側ブロックで同名変数を宣言すると、外側の変数をシャドウする。
  • フィールド名とローカル変数名がぶつかった場合、ローカル変数が優先される。 フィールドは this. で参照する。

4.2 with-blocks（メンバー省略記法）

Aloe には、同一オブジェクトのメンバーアクセスを簡潔に書くための with ブロックがある。 

with (user) {
    .name = "Alice";
    .age = .age + 1;
    print(.name);
}
  • with (expr) { ... } ブロック内では：
    • .Member は (expr).Member の省略形。
    • expr はブロックの先頭で 1 度だけ評価され、内部一時変数に保持される。
    • 特別な一時変数 _ は「with 対象オブジェクト」を指す。
      • log(_); のように渡すことができる。
  • ネストした with ブロック：
    • 内側の with が . の解決および _ をシャドウする。
    • 外側オブジェクトへは通常の変数名でアクセスできる。
  • C# の using スタイルのように、同じブロックに複数の with を並べられる：
with (user), (profile) {
    .name = "Alice";  // resolved against both user and profile
}
  • どのオブジェクトのメンバーに解決されるかの順序は実装依存。
  • ただし曖昧な場合（複数オブジェクトが同じメンバーを持つ）はコンパイルエラー。

with はあくまで構文糖衣であり、意味的にはすべて 明示的な obj.member 呼び出しに書き換えた上で 型チェック・コード生成を行うことができる。

4.2.1 Type-based with（class / struct / enum の static / 列挙子ショートカット）

通常の with (expr) に加えて、型名 を対象にした with を利用できる。 

with (TypeName) {
    .SomeStaticMethod();
    .SomeStaticField = 1;
}

ここで TypeName は次のいずれか：

  • class 型
  • struct 型
  • enum / bitfield enum

この場合、with は「型コンテキスト with」として解釈される：

  • () 内の式は実行時に値として評価されない。
    • コンパイラは TypeName を型として解決し、インスタンス生成などは行わない。
  • ブロック内では：
    • .Member は TypeName.Member の省略形。
      • 対象となる Member は
        • static フィールド
        • static メソッド
        • enum / bitfield enum の列挙子
    • 一時変数 _ は、TypeName の static コンテキストを表す擬似オブジェクトとして振る舞う。
      • _.Member も TypeName.Member と等価。
    • 型ベース with 内でインスタンスメンバーにアクセスすることはコンパイルエラー。

なお、型ベース with でもローカル変数や外側スコープの変数が同名で存在すればそちらが優先される。 列挙子や static メンバーを明示したいときは TypeName.Member または .Member / _.Member を用いる。

例：class / struct の static アクセス
struct AngleUtil {
    static field Pi: float = 3.14159;

    static method degToRad(deg: float): float {
        return deg * Pi / 180.0;
    }
}

class MathUtil {
    static method max(a: int, b: int): int {
        if (a > b) return a;
        return b;
    }
}

function main(args: string[]): int {
    with (AngleUtil) {
        var rad = degToRad(90.0);  // AngleUtil.degToRad(90.0)
        print(.Pi);                // AngleUtil.Pi
    }

    with (MathUtil) {
        var m = max(10, 20);       // MathUtil.max(10, 20)
        print(m);
    }

    return 0;
}
例：enum の列挙子アクセス
enum Color {
    Red,
    Green,
    Blue
}

function main(args: string[]): int {
    with (Color) {
        var c1: Color = Red;   // Color.Red
        var c2: Color = Blue;  // Color.Blue
        print(c1);
        print(c2);
    }
    return 0;
}

型ベース with も複数ターゲット構文に混在させられる： 

with (user), (Color) {
    .name = "Alice"; // user.name
    var c: Color = Red; // Color.Red
}

この場合も解決順は実装依存だが、曖昧であればコンパイルエラーとする。

5. Variables & Constants（変数と定数）

5.1 var（型推論）

var i = 10;
var f = 10.5;
var s = "Hello";
var c = 'A';
var b = 255;
var arr = [1, 2, 3];
var lst = (1, 2, 3);
var st  = {1, 2, 3};
var mp  = ["key": 10];
var fb  = new FizzBuzz();
  • 変数の型は右辺の式から推論される。
  • var x = 1; は常に int。
  • var x = 1.0; は float として扱われる。
  • var x = null; はコンパイルエラー（型が推論できない）。

5.2 let（明示的な型）

let i: int = 10;
let f: float = 10.5;
let d: decimal = 10.12345678901234567890:d;
let s: string = "Hello, Aloe!";
let c: char = 'A';
let b: byte = 255;

let arr: int[] = [1, 2, 3];
let lst: int() = (1, 2, 3);
let st : int{} = {1, 2, 3};
let mp : map<string/int> = ["key": 10];

5.3 Constants（const）

const MaxCount: int = 100;
const Pi: float = 3.14159;
const AppName: string = "Aloe VM";
  • const 変数は再代入不可。
  • const には型注釈が必須。

6. Fields（フィールド宣言）

class / struct のフィールドは field キーワードで宣言する。

class Sample {
    field count: int = 0;
    field name: string = "default";

    readonly field id: int;

    construct(id: int) {
        this.id = id; // OK: inside the constructor
    }
    method change(): void {
        this.count = this.count + 1; // OK
        // this.id = 2; // NG: readonly field cannot be changed outside constructor
    }
}
  • field … 通常のフィールド。
  • readonly field … コンストラクタ内のみ代入可、それ以降は読み取り専用。
  • struct / class の両方で同じ意味。

7. Class / Struct / Interface / Trait

7.1 class

class FizzBuzz {
    field _count: int = 0;
    field name: string = "FizzBuzz";

    construct() {
    }

    method run(): void {
        var i = 1;
        while (i <= 100) {
            // ...
            i = i + 1;
        }
    }
}
  • class は参照型。
  • sealed class は extends による継承不可。
  • 単一継承のみサポート。

7.2 struct

struct Point {
    field x: int;
    field y: int;

    construct(x: int, y: int) {
        this.x = x;
        this.y = y;
    }
}
  • struct は値型。
  • フィールドはプリミティブ型か他の struct に限定（参照型フィールドは禁止）。
  • struct のコピーは常に値コピー。
  • struct の継承はサポートしない（sealed struct は「拡張禁止」マーカー）。

7.3 Inheritance and sealed

class Base {
    method foo(): void { }
}

class Derived extends Base {
    method bar(): void { }
}

sealed class FinalDerived extends Base {
    // cannot be extended further
}

7.4 interface

interface ILogger {
    method log(msg: string): void;
}

class ConsoleLogger implements ILogger {
    method log(msg: string): void {
        print(msg);
    }
}
  • implements キーワードでインタフェースを実装。
  • is 演算子でインタフェース実装有無を判定。

7.5 トレイト（trait）

trait Trait_A {
    method hello(): void {
        print("Hello from Trait_A");
    }
}

with によるトレイト適用

var obj = new SomeClass() with Trait_A;
obj.hello(); // method from Trait_A (if no conflicts)

非破壊な追加 / 削除（+ / -）

var obj2 = obj + Trait_A;      // obj unchanged; obj2 has Trait_A
var obj3 = obj2 - Trait_A;     // obj3 is obj2 without Trait_A

as によるエイリアス

var obj2 = obj + Trait_A as a;
obj2.a.hello(); // OK

// obj2.hello(); // NG: alias only; no direct member flattening

トレイトの衝突（Trait conflicts）

  • 複数 trait が同名フィールド / メソッドを定義するとコンパイルエラー。
  • as a などのエイリアス付き適用は衝突判定の対象外。
  • クラス本体と trait メンバとの衝突の扱いは将来バージョンで定義予定。

8. delete & null

var obj = new FizzBuzz();
delete obj;  // equivalent to: obj = null (at VM level)
  • delete obj; は VM レベルでは obj = null と等価。
  • ソースコード中で obj = null; と書くことはできない。
  • 参照型は内部的には null を取り得るが、型システムは null を追跡しない。
  • null 状態でのアクセスは NullReferenceException を引き起こす可能性がある。

9. is Operator（is 演算子）

if (obj is FizzBuzz) {
    // true if runtime type is FizzBuzz or a subclass
}

if (obj is ILogger) {
    // true if obj implements ILogger
}

if (obj is Trait_A) {
    // true if obj has Trait_A applied (with / +)
}
  • is は実行時の型チェックを行う。
  • 継承関係（Derived は Base としても扱う）を考慮する。
  • インタフェースや trait に対しても使用できる。

10. Collections（配列 / リスト / セット / マップ）

10.1 Arrays (T[])

var arr: int[] = [1, 2, 3];
var matrix: float[][] = [[1.0, 2.0], [3.0, 4.0]];

10.2 Lists (T())

var lst: int() = (1, 2, 3);
var objLst: FizzBuzz() = (new FizzBuzz(), new FizzBuzz());

10.3 Sets (T{})

var st: int{} = {1, 2, 3};
var structSt: Point{} = {Point { x:0, y:0 }, Point { x:1, y:1 }};

10.4 Maps (map<K/V>)

var mp: map<int> = ["key1": 10, "key2": 20];          // default key type is string
var mp2: map<string/float> = ["key1": 10.5, "key2": 20.5];
var objMp: map<FizzBuzz> = ["obj1": new FizzBuzz(), "obj2": new FizzBuzz()];

11. enum and bitfield enum

11.1 Normal enum

enum Color {
    Red,
    Green,
    Blue
}
  • 内部表現は常に int32。
  • enum<byte>, enum<string> のような基礎型の変更は許可されない。
  • enum 型も型ベース with の対象となり、with (Color) { Red; } などで列挙子に短縮アクセスできる。

11.2 bitfield enum

bitfield enum LogFlags {
    None = 1 << 0,
    Info = 1 << 3,
    Warn,
}

b(n) 記法

bitfield enum LogFlags {
    None : b(0),
    Info : b(3),
    Warn,            // highest b(n) so far is 3; Warn becomes b(4) = 1 << 4
}
  • b(n) はビット n（1 << n）を表す。
  • Name : b(n) でその列挙子にビット n を割り当てる。
  • bitfield enum で使用可能なビットは最大 32（n は 0–31）。
  • 同一ビットを共有する列挙子があっても、コンパイラは警告にとどめ、エラーにはしない。
  • 基礎型は常に int32 であり、bitfield enum<byte> などはコンパイルエラー。

12. main Block（エントリポイント）

function main(args: string[]): int {
    var fb = new FizzBuzz();
    fb.run();
    return 0; // process exit code (int)
}
  • プログラムのエントリポイントとなる 関数。
  • args はコマンドライン引数。
  • main 関数は int 型の値を return で返し、その値がプロセスの終了コードとなる。

13. Temp Variable _（一時変数）

method foo(): void {
    var x = 10;
    _ = x + 5;   // type of _ is inferred in this block (here: int)
}

method bar(): void {
    _ = "temp";  // here, _ is string in this block
}
  • _ は任意のブロックで宣言なしに使用できる特別な一時変数。
  • 各ブロックごとに、そのブロックで最初に代入された式の型が _ の型になる。
  • 同じブロック内で、_ に異なる型の値を代入することはコンパイルエラー。
  • with ブロック内では、追加で「現在の with 対象」を指す意味を持つ（4.2 参照）。

14. swap Keyword（swap キーワード）

var a = 10;
var b = 20;
swap (a, b);
print(a); // 20
print(b); // 10
  • swap (x, y); は x と y の値（または参照）を入れ替える。
  • swap が妥当かどうかはコンパイル時にチェックされる。

15. Operators（演算子）

15.1 優先順位（高い順）

  1. メンバー / 呼び出し： x.y, x[y], x(args...)
  2. 単項： +x, -x, !x, delete x
  3. 乗除： *, /, %
  4. 加減： +, -
  5. シフト： <<, >>
  6. 関係： <, <=, >, >=
  7. 等価： ==, !=
  8. ビット AND： &
  9. ビット XOR： ^
  10. ビット OR（式コンテキスト）： |
  11. 論理 AND： &&
  12. 論理 OR： ||
  13. パイプ with 演算子： |>
  14. 代入： =
  15. パイプライン接続（文コンテキスト）： |

パイプ with 演算子 |> は左辺の式を先に評価し、その結果を 右辺のパイプ / フィルタに流し込む演算子である。 優先順位は代入より高く、その他の 2 項演算子より低いため、 1 + 2 |> f は (1 + 2) |> f と解釈される。

パイプライン接続演算子 | は stdin | filter(utf8) | filter(json<T>) のような パイプライン文でのみ使用される構文であり、 式の中でビット OR と混在することはない。

複合代入（+= 等）はまだ仕様化されていない。

15.2 結合規則

  • 単項演算子：右結合。
  • 代入：右結合。
  • その他の 2 項演算子：左結合。

18. Pipes & Filters（パイプとフィルタ）

Aloe は pipe<T> と filter(...) を用いて、 サーバ / CLI / Web アプリなどのストリーミング処理を構成する。

18.0 ROP（Railway Oriented Programming）の考え方

ROP（Railway Oriented Programming）は、処理の流れを 「二本の線路（成功レーンと失敗レーン）」として捉える設計スタイルである。 各ステージは「入力を受け取り、成功なら次のステージへ、失敗ならエラーを別レーンに流す」 という形で構成される。

Aloe の pipe / filter は、まさにこのステージの連鎖を表現する仕組みであり、 各フィルタを「同じ入力なら常に同じ出力を返す」関数として設計することで、 ROP 的な読みやすさとテストのしやすさを得られる。 エラーは 18.0.1 節で定義する Result<T> 型や専用のエラーパイプなどに 明示的に流すことを推奨する。

18.0.1 Result<T> 型（成功 / 失敗の 2 値表現）

Aloe の標準ライブラリは、ROP での利用を想定した汎用的な結果型 Result<T> を提供する。 これは「成功した値」または「失敗時のエラー情報」のいずれかを保持する値型である。 以下の定義は擬似コードとしてのイメージであり、実装レベルでは最適化や補助メソッドが追加されうる。 

struct Result<T> {
    field IsOk: bool;
    field Value: T;
    field Error: Exception;

    static method Ok(value: T): Result<T> {
        let r: Result<T>;
        r.IsOk = true;
        r.Value = value;
        // Error は未使用（VM レベルでは未定義または null 相当）
        return r;
    }

    static method Error(error: Exception): Result<T> {
        let r: Result<T>;
        r.IsOk = false;
        // Value は未定義（読み出してはならない）
        r.Error = error;
        return r;
    }

    // 補助的なヘルパー（T = void を想定した成功値）
    // 実装ではジェネリック特殊化や別名型 ResultVoid などで提供してもよい。
    static method Ok(): Result<void> {
        let r: Result<void>;
        r.IsOk = true;
        // Value は使用しない
        // Error は未使用
        return r;
    }
}
  • IsOk … 結果が成功かどうかを表すフラグ（true なら成功）。
  • Value … 成功時の値。IsOk == true の場合のみ有効。
  • Error … 失敗時の例外情報。IsOk == false の場合のみ有効。
  • Ok(value) / Error(error) は、それぞれ成功 / 失敗の Result<T> を生成するヘルパーメソッドである。
  • 値を返さず成功 / 失敗だけを表したい場合には Result<void> を用い、 フィルタの bound などから Result<void>.Ok() を返すことを想定する。

フィルタやメソッドの戻り値として Result<T> を用いることで、 「正常系の値」と「エラー情報」を同一の型で扱うことができ、 ROP 的な「二本の線路」をコード上で明示しやすくなる。 ドメイン固有のエラー型を用いたい場合は、Exception の代わりに 独自の struct / class を用いた 別の Result<TError, TValue> などを定義してもよい。

18.1 pipe<T> の基本

  • pipe<T> は T 型要素のストリーム。
  • pipe<byte> … 標準入力やソケットなどのバイト列。
  • pipe<string> … 文字列ストリーム。
  • pipe は共有メモリではなく値コピーを前提とする。

18.1.1 組み込みパイプ

  • stdin : pipe<byte>
  • stdout : pipe<byte>
  • stderr : pipe<byte>（任意）

18.1.2 パイプ変数の宣言と生成

Aloe には、特別な「パイプ定義文（pipe Name { ... }）」は存在しない。 パイプはすべて、通常のジェネリック型 pipe<T> として宣言・生成する。 

function main(args: string[]): int {
    // ローカルのパイプ変数を宣言し、生成
    let bytes: pipe<byte>   = pipe<byte>.create();
    let lines: pipe<string> = pipe<string>.create();

    stdin
        | filter(utf8)
        | bytes;

    bytes
        | filter(lineSplit)
        | lines;

    foreach (line in lines) {
        print("Line: " + line);
    }

    return 0;
}
  • let p: pipe<T> = pipe<T>.create(); のようにして、新しいパイプインスタンス（チャネル）を生成する。
  • 宣言された pipe<T> 変数は、パイプライン文や関数呼び出しの中で自由に接続・受け渡しができる。
  • 「どのように変換するか（ロジック）」は、フィルタ定義（18.2 節）や通常のメソッドで表現し、 pipe<T> 自体はあくまで「値を運ぶストリーム」である。

18.1.3 パイプインスタンスの共有とスコープ

パイプインスタンス（実際に値を流すチャネル）は、関数ローカルだけでなく、 モジュールレベルやクラスフィールドとしても宣言できる。 

// モジュールレベルでの共有パイプ定義
namespace My.App {
    // リクエストを流すための共有パイプ
    let Requests: pipe<Request> = pipe<Request>.create();

    // 応答を流すための共有パイプ
    let Responses: pipe<Response> = pipe<Response>.create();
}
  • Requests や Responses のような共有パイプは、 複数のタスク・関数・フィルタから同時に使用されることを想定してよい。
  • どのフィルタや関数ロジックからそのインスタンスを利用するかは、 パイプライン文（18.3 節）や明示的な接続コードで決まる。
  • パイプは first-class な値であり、関数の引数や戻り値、 クラス / struct のフィールドとして自由に受け渡しできる。
  • より高度なバッファリング戦略や back-pressure の扱いは、 VM / ランタイムおよび標準ライブラリの設計に委ねられる。

まとめると、 フィルタ定義や通常のメソッドが「ステージのロジック」 を表現し、 pipe<T> 変数と pipe<T>.create() が「実際のチャネルインスタンス」 を それぞれ表現している。両者を組み合わせて、パイプライン全体を構築する。

18.2 filter(...) の基本

filter(F) はパイプライン中の中間変換ステージ。

18.2.1 フィルタ定義

filter lineSplit {
    in:  pipe<string>;
    out: pipe<string>;

    // フィルタ内部状態（各フィルタインスタンスごとに保持される）
    field Separator: string = "\n";

    // このフィルタが受け取るオプションクラス（通常は bitfield enum）
    options: LineSplitOptions;

    // インスタンス生成時に 1 度だけ呼ばれるコンストラクタ
    constructor(opts: LineSplitOptions) {
        // options から内部状態を初期化してもよい
        // 例: オプションに応じて区切り文字を変えるなど
        Separator = "\n";
    }

    // パイプライン実行時に呼ばれる本体。
    // 処理が正常に完了したかどうかを Result<void> で返す。
    bound(input, output): Result<void> {
        try {
            foreach (chunk in input) {
                let parts = chunk.split(Separator);
                foreach (part in parts) {
                    output.write(part);
                }
            }
            output.close();
            return Result<void>.Ok();
        }
        catch (e: Exception) {
            // 必要に応じてログ出力などを行った上でエラーとして返す
            return Result<void>.Error(e);
        }
    }
}

filter 定義は、次の要素から構成される：

  • in: pipe<TIn>; … 入力パイプの型（必須）。
  • out: pipe<TOut>; … 出力パイプの型（必須）。
  • field … class / struct と同様に、フィルタ内部で保持される状態を表すフィールドを宣言できる。
    • 各パイプライン接続ごとにフィルタインスタンスが生成され、そのインスタンスがフィールドを持つ。
    • なお、Railway Oriented Programming（ROP）的な設計思想では、 フィルタは冪等性（同じ入力に対して常に同じ出力を返すこと）を維持することが重要であるため、 副作用を持つ field による内部状態は最小限にとどめることが推奨される （集計・ログ出力・キャッシュなど、実装上やむを得ない用途を除く）。
  • options: SomeOptions; … このフィルタが受け取るオプションクラス（通常は bitfield enum）を宣言する。
    • 18.2.2 で説明する filter(json<T>: JsonOptions.Foo | ...) の JsonOptions に対応する。
    • 省略した場合、そのフィルタはオプションを受け取らない。
  • constructor(...) … フィルタインスタンス生成時に 1 度だけ呼ばれるコンストラクタ。
    • ここでオプション値（bitfield など）や追加の設定値を受け取り、フィールドを初期化できる。
    • constructor の有無は任意であり、省略してもよい。
  • bound(input, output): Result<void> … 実際の変換処理を記述する本体。
    • input と output の型は、それぞれ in / out に対応する。
    • EOF を検知したら output.close() を呼び、下流へ EOF を伝播させる。
    • 正常に完了した場合は Result<void>.Ok() を返し、 何らかのエラーがあれば Result<void>.Error(e) のようにエラー情報付きで返す。
    • 例外を直接スローする実装も可能だが、パイプライン全体の制御性・テスト容易性の観点から、 ROP 的には「例外を捕捉して Result<void> に畳み込む」スタイルを推奨する。

18.2.2 フィルタオプション

filter(json<User>: JsonOptions.PrettyPrint | JsonOptions.IncludeNulls)

// options で JsonOptions を指定している場合、クラス名は省略できる：
filter(json<User>: PrettyPrint | IncludeNulls)
  • オプションは : の後に | でつなげて指定する。
  • 各フィルタは、自身の filter 定義内の options 項目で options: JsonOptions; のように「そのフィルタが受け取るオプションクラス」を指定できる。
  • options で指定したクラスの列挙子をパイプライン文の : の後で使用する場合、 JsonOptions.PrettyPrint のようにクラス名を明示してもよいし、 上記サンプルのように PrettyPrint, IncludeNulls のように クラス名を省略して記述してもよい。
    • クラス名省略時、コンパイラはまず options で指定されたクラス （この例では JsonOptions）の列挙子として解決を試みる。
    • 同名のローカル変数や他のシンボルと衝突する場合、その識別子は曖昧としてコンパイルエラー。
  • パイプライン文でクラス名付きで指定する場合（JsonOptions.PrettyPrint など）は、 その識別子は常にそのクラスのメンバーとして解釈されるため、曖昧さが解消される。
  • 実装側では、これらのオプション値を 1 つの bitfield などに畳み込み、 フィルタの constructor へ渡して内部状態の初期化に利用できる。

18.3 パイプライン文と |

stdin
    | filter(utf8)
    | filter(json<DateTime>)
    | dates;
  • A | B | C; で A の出力が B へ、B の出力が C へ接続される。
  • 型チェックにより各ステージの入出力型整合性を確認。
  • パイプライン文そのものは値を返さない（副作用として接続を張るだけ）。

18.4 foreach と pipe

foreach (line in lines) {
    print("Input: " + line);
}
  • foreach (x in p) は内部的に p.next() を EOF まで呼ぶ。
  • 書き込み側が close() を呼ぶと読み側は EOF を検知してループを終了する。

18.5 close と EOF 伝播

  • p.close() 後の write() は無効（エラーになり得る）。
  • フィルタは入力 EOF を検知したら出力側も close() すべき。

18.6 pipe を使った並行モデル

  • タスク同士の主な通信手段は pipe<T>。
  • shared-nothing モデルを前提とし、共有可変データ構造は持たない設計を推奨。

18.7 例：簡単な REST 風パイプライン

class User {
    field id: int;
    field name: string;
}

function main(args: string[]): int {
    let req: pipe<byte> = pipe<byte>.create();
    let users: pipe<User> = pipe<User>.create();

    stdin
        | filter(utf8)
        | filter(json<User>)
        | users;

    let u: User? = users.next();
    if (u is User) {
        print("User: " + u.name);
    }

    return 0;
}

utf8 や json<T> などの標準フィルタの詳細は、 コア言語仕様ではなく別途「標準フィルタライブラリ」仕様で定義される。

以上が、元の英語仕様に基づき、 with ブロックの型対応拡張と ROP / Result<T> 型 / pipe<T> ベースのパイプ宣言 / filter.bound の Result<void> 戻り値 / フィルタオプションの省略記法 を加えた Aloe 言語仕様（日本語・HTML 版）です。
