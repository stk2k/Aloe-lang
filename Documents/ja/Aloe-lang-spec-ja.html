この文書は、英語版仕様 https://stk2k.github.io/Aloe-lang/Documents/en/Aloe-lang-spec-en.html をベースにした日本語ドラフトです。
その上で 4.2 with-blocks に class / struct / enum を対象にした型ベース with（static メンバー／列挙子への短縮アクセス） の拡張、および 18 章の pipe / filter / Result / ROP（Railway Oriented Programming） に関する拡張を追加しています。
それ以外の仕様は原文にできるだけ忠実に訳しています。

0. Overview（概要）

Aloe は静的型付けのスクリプト言語であり、シンプルで直感的な構文を目指している。
スタックベースの仮想マシン（AloeVM）上で動作し、主に次のような目標を持つ。

  • 教育用途や小規模アプリケーションで読み書きしやすいこと
  • C# ランタイムおよび WebAssembly (WASM) 環境で動作すること
  • pipe と filter によって、I/O と並行処理を静的型付けのまま制御できること

特徴：

  • 静的型付け
    • var による型推論
    • let による明示的な型指定
  • 値型 / 参照型
    • struct … 値型
    • class … 参照型
  • GC ベースのメモリ管理（具体的なアルゴリズムは VM 実装依存）
  • = null をソースコード中に直接書かない
    • 参照型は内部的には null を取り得るが、delete による糖衣構文で扱う。
  • pipe / filter によるデータフロースタイルの設計
  • Result 型と ROP（Railway Oriented Programming）によるエラー処理の明文化

1. Basic Syntax（基本文法）

1.1 Comments（コメント）

// single-line comment

/* multi-line
   comment */
  • // から行末までが行コメント。
  • /* ～ */ が複数行コメント。
  • ネストした /* */ は未定義（サポートしない想定）。

1.2 Identifiers（識別子）

  • 使用可能文字：英字 (A–Z, a–z)、数字 (0–9)、アンダースコア _
  • 先頭文字は英字または _ でなければならない。
  • 大文字小文字は区別される。
  • キーワードと同じ綴りは使用できない。
  • _ で始まる識別子は「プライベート」扱い（外部からは参照しない慣習）。
  • ユニコード文字も使用できる（空白や区切り記号などは除外）。

1.4 Literals（リテラル）

  • 整数リテラル：10, 0, -5, 0x1A, 0b1010 など
  • 浮動小数点リテラル：10.5, 0.0, -3.14
  • 10進リテラル：10.12345678901234567890:d など
  • 真偽値：true, false
  • 文字列："Hello, World!"
  • 文字：'A', 'z' など
  • null リテラル：null
  • 配列リテラル：[1, 2, 3]
  • リストリテラル：(1, 2, 3)
  • セットリテラル：{1, 2, 3}
  • マップリテラル：["key1": 10, "key2": 20]
  • enum リテラル：EnumType.Value1
  • struct リテラル：Point { x: 0, y: 1 }
  • オブジェクト生成：new ClassName()

1.5 Statement terminator（文の終端）

var x = 10;
x = x + 1;
print(x);
  • すべての文はセミコロン ; で終わる。
  • 改行による自動セミコロン挿入は行わない。

1.6 Whitespace（空白）

  • スペース、タブ、改行はトークン区切りとして扱われる。
  • 連続する空白は 1 つの区切りとみなされる。
  • インデントに意味はない（Python とは異なる）。

1.7 Case sensitivity（大文字小文字）

  • 識別子とキーワードは大文字小文字を区別する。
  • キーワードはすべて小文字。

2. Type System（型システム）

2.1 Primitive types（プリミティブ型）

Aloe における「プリミティブ型」は、次の 7 種類とする。

  • bool / boolean – 真偽値。
  • int / integer – 符号付き整数。
  • float – 浮動小数点数。
  • decimal – 高精度 10 進数。
  • string – 文字列（内部的には Unicode）。
  • null – 「値が存在しない」ことを表す特別な型兼値。
  • void – 戻り値なし（メソッド戻り値専用）。

any 型は存在せず、動的型付けはサポートしない。
char や byte は言語としてはサポートされるが、プリミティブ型のカテゴリからは外し、 数値と変換の章（3 章）で別途扱う値型とする。

2.2 Composite / user-defined types（合成型 / ユーザー定義型）

  • 配列：T[]
  • リスト：T()
  • セット：T{}
  • マップ：map<T> または map<K/V>
  • struct
  • class
  • enum, bitfield enum
  • 非同期タスク：task
  • パイプ：pipe<T>（18章参照）

any 型は存在せず、動的型付けはサポートしない。

2.3 Value types / reference types（値型 / 参照型）

  • 値型：
    • int, float, decimal, bool, byte, char
    • struct
    • enum, bitfield enum（内部表現は int32）
  • 参照型：
    • class
    • string（実装上は参照型）
    • task, pipe<T> など

Struct restrictions and copying（struct の制約とコピー）

  • struct のフィールドにはプリミティブ型と他の struct のみを持てる。
  • class などの参照型フィールドは持てない。
  • struct の代入は常に値コピー。
  • 最大サイズなどは仕様で制限しない。

Class vs struct（class と struct）

  • class は struct をフィールドとして持てる。
  • class インスタンスを別変数に代入すると参照がコピーされる（オブジェクト本体は共有）。

3. Numbers & Conversions（数値と暗黙変換）

3.1 int and float

  • int と float を混在させた式の結果は float になる。
    • int + float → float
    • int - float → float
    • int * float → float
    • int / float → float

3.2 decimal

  • decimal は高精度な 10 進数。
  • int / float と混在する式は、必要に応じて decimal に昇格しうる。
  • オーバーフロー / アンダーフロー / 精度問題は OverflowException として扱う。

3.3 enum and int

  • Aloe にはキャスト構文がない。
  • enum と int の暗黙 / 明示変換はいずれも禁止。
  • enum と数値型の混在式はコンパイルエラー（bitfield enum のビット演算を除く）。
  • 内部表現としては enum / bitfield enum はすべて int32。

3.4 byte and char

  • byte：
    • 範囲は 0–255。
    • byte 同士の演算は許されるが、範囲外はエラーになり得る。
    • byte と int の混在式は int に昇格。
  • char：
    • char の算術演算は不可。
    • 比較演算（==, <, > 等）は char 同士のみ。
    • 数値型との暗黙変換はない。

3.5 NaN / Infinity

  • 数値オーバーフロー/アンダーフローは NaN/Infinity ではなく例外で扱う。
  • ZeroDivisionException は 0 / 0.0 で常に投げる。

4. Blocks & Scope（ブロックとスコープ）

ブロックは { と } で区切られ、各ブロックが新しいスコープを導入する。

  • main ブロック
  • クラス定義ブロック
  • メソッド定義ブロック
  • 任意の { ... } ブロック

4.1 Static (lexical) scope and shadowing（静的スコープとシャドウイング）

class Sample {
    field value: int = 1;
    method foo(): void {
        var value = 2;        // shadows the field
        print(value);         // 2
        print(this.value);    // 1
    }
}
  • Aloe は静的（レキシカル）スコープ。
  • 内側ブロックで同名変数を宣言すると、外側の変数をシャドウする。
  • フィールド名とローカル変数名がぶつかった場合、ローカル変数が優先される。 フィールドは this. で参照する。

4.2 with-blocks（メンバー省略記法）

Aloe には、同一オブジェクトのメンバーアクセスを簡潔に書くための with ブロックがある。 

with (user) {
    .name = "Alice";
    .age = .age + 1;
    print(.name);
}
  • with (expr) { ... } ブロック内では：
    • .Member は (expr).Member の省略形。
    • expr はブロックの先頭で 1 度だけ評価され、内部一時変数に保持される。
    • 特別な一時変数 _ は「with 対象オブジェクト」を指す。
      • log(_); のように渡すことができる。
  • ネストした with ブロック：
    • 内側の with が . の解決および _ をシャドウする。
    • 外側オブジェクトへは通常の変数名でアクセスできる。
  • C# の using スタイルのように、同じブロックに複数の with を並べられる：
with (user), (profile) {
    .name = "Alice";  // resolved against both user and profile
}
  • どのオブジェクトのメンバーに解決されるかの順序は実装依存。
  • ただし曖昧な場合（複数オブジェクトが同じメンバーを持つ）はコンパイルエラー。

with はあくまで構文糖衣であり、意味的にはすべて 明示的な obj.member 呼び出しに書き換えた上で 型チェック・コード生成を行うことができる。

4.2.1 Type-based with（class / struct / enum の static / 列挙子ショートカット）

通常の with (expr) に加えて、型名 を対象にした with を利用できる。 

with (TypeName) {
    .SomeStaticMethod();
    .SomeStaticField = 1;
}

ここで TypeName は次のいずれか：

  • class 型
  • struct 型
  • enum / bitfield enum

この場合、with は「型コンテキスト with」として解釈される：

  • () 内の式は実行時に値として評価されない。
    • コンパイラは TypeName を型として解決し、インスタンス生成などは行わない。
  • ブロック内では：
    • .Member は TypeName.Member の省略形。
      • 対象となる Member は
        • static フィールド
        • static メソッド
        • enum / bitfield enum の列挙子
    • 一時変数 _ は、TypeName の static コンテキストを表す擬似オブジェクトとして振る舞う。
      • _.Member も TypeName.Member と等価。
    • 型ベース with 内でインスタンスメンバーにアクセスすることはコンパイルエラー。

なお、型ベース with でもローカル変数や外側スコープの変数が同名で存在すればそちらが優先される。 列挙子や static メンバーを明示したいときは TypeName.Member または .Member / _.Member を用いる。

例：class / struct の static アクセス
struct AngleUtil {
    static field Pi: float = 3.14159;

    static method degToRad(deg: float): float {
        return deg * Pi / 180.0;
    }
}

class MathUtil {
    static method max(a: int, b: int): int {
        if (a > b) return a;
        return b;
    }
}

function main(args: string[]): int {
    with (AngleUtil) {
        var rad = degToRad(90.0);  // AngleUtil.degToRad(90.0)
        print(.Pi);                // AngleUtil.Pi
    }

    with (MathUtil) {
        var m = max(10, 20);       // MathUtil.max(10, 20)
        print(m);
    }

    return 0;
}
例：enum の列挙子アクセス
enum Color {
    Red,
    Green,
    Blue
}

function main(args: string[]): int {
    with (Color) {
        let c1: Color = Red;   // Color.Red
        let c2: Color = Blue;  // Color.Blue
        print(c1);
        print(c2);
    }
    return 0;
}

型ベース with も複数ターゲット構文に混在させられる： 

with (user), (Color) {
    .name = "Alice"; // user.name
    let c: Color = Red; // Color.Red
}

この場合も解決順は実装依存だが、曖昧であればコンパイルエラーとする。

5. Variables & Constants（変数と定数）

5.1 var（型推論）

var i = 10;
var f = 10.5;
var s = "Hello";
var c = 'A';
var b = 255;
var arr = [1, 2, 3];
var lst = (1, 2, 3);
var st  = {1, 2, 3};
var mp  = ["key": 10];
var fb  = new FizzBuzz();
  • 変数の型は右辺の式から推論される。
  • var x = 1; は常に int。
  • var x = 1.0; は float として扱われる。
  • var x = null; はコンパイルエラー（型が推論できない）。

5.2 let（明示的な型）

let i: int = 10;
let f: float = 10.5;
let d: decimal = 10.12345678901234567890:d;
let s: string = "Hello, Aloe!";
let c: char = 'A';
let b: byte = 255;

let arr: int[] = [1, 2, 3];
let lst: int() = (1, 2, 3);
let st : int{} = {1, 2, 3};
let mp : map<string/int> = ["key": 10];

5.3 Constants（const）

const MaxCount: int = 100;
const Pi: float = 3.14159;
const AppName: string = "Aloe VM";
  • const 変数は再代入不可。
  • const には型注釈が必須。

6. Fields（フィールド宣言）

class / struct のフィールドは field キーワードで宣言する。

class Sample {
    field count: int = 0;
    field name: string = "default";

    readonly field id: int;

    construct(id: int) {
        this.id = id; // OK: inside the constructor
    }
    method change(): void {
        this.count = this.count + 1; // OK
        // this.id = 2; // NG: readonly field cannot be changed outside constructor
    }
}
  • field … 通常のフィールド。
  • readonly field … コンストラクタ内のみ代入可、それ以降は読み取り専用。
  • struct / class の両方で同じ意味。

7. Class / Struct / Interface / Trait

7.1 class

class FizzBuzz {
    field _count: int = 0;
    field name: string = "FizzBuzz";

    construct() {
    }

    method run(): void {
        var i = 1;
        while (i <= 100) {
            // ...
            i = i + 1;
        }
    }
}
  • class は参照型。
  • sealed class は extends による継承不可。
  • 単一継承のみサポート。

7.2 struct

struct Point {
    field x: int;
    field y: int;

    construct(x: int, y: int) {
        this.x = x;
        this.y = y;
    }
}
  • struct は値型。
  • フィールドはプリミティブ型か他の struct に限定（参照型フィールドは禁止）。
  • struct のコピーは常に値コピー。
  • struct の継承はサポートしない（sealed struct は「拡張禁止」マーカー）。

7.3 Inheritance and sealed

class Base {
    method foo(): void { }
}

class Derived extends Base {
    method bar(): void { }
}

sealed class FinalDerived extends Base {
    // cannot be extended further
}

7.4 interface

interface ILogger {
    method log(msg: string): void;
}

class ConsoleLogger implements ILogger {
    method log(msg: string): void {
        print(msg);
    }
}
  • implements キーワードでインタフェースを実装。
  • is 演算子でインタフェース実装有無を判定。

7.5 trait

trait Trait_A {
    method hello(): void {
        print("Hello from Trait_A");
    }
}

with による trait 適用

var obj = new SomeClass() with Trait_A;
obj.hello(); // Trait_A から取り込まれたメソッド（競合がなければ）

Non-destructive add/remove（+ / -）

var obj2 = obj + Trait_A;      // obj は変更されない。Trait_A を適用した新しいオブジェクトを返す
var obj3 = obj2 - Trait_A;     // obj2 から Trait_A を外した新しいオブジェクトを返す

エイリアス指定（as）

var obj2 = obj + Trait_A as a;
obj2.a.hello(); // OK

// obj2.hello(); // NG: alias 経由でのみアクセス可能

Trait conflicts（競合）

  • 複数 trait が同名フィールド / メソッドを定義するとコンパイルエラー。
  • as a などのエイリアス付き適用は衝突判定の対象外。
  • クラス本体と trait メンバとの衝突の扱いは将来バージョンで定義予定。

8. delete & null

var obj = new FizzBuzz();
delete obj;  // equivalent to: obj = null (at VM level)
  • delete obj; は VM レベルでは obj = null と等価。
  • ソースコード中で obj = null; と書くことはできない。
  • 参照型は内部的には null を取り得るが、型システムは null を追跡しない。
  • null 状態でのアクセスは NullReferenceException を引き起こす可能性がある。

9. is Operator（is 演算子）

if (obj is FizzBuzz) {
    // true if runtime type is FizzBuzz or a subclass
}

if (obj is ILogger) {
    // true if obj implements ILogger
}

if (obj is Trait_A) {
    // true if obj has Trait_A applied (with / +)
}
  • is は実行時の型チェックを行う。
  • 継承関係（Derived は Base としても扱う）を考慮する。
  • インタフェースや trait に対しても使用できる。

10. Collections（配列 / リスト / セット / マップ）

10.1 Arrays (T[])

var arr: int[] = [1, 2, 3];
var matrix: float[][] = [[1.0, 2.0], [3.0, 4.0]];

10.2 Lists (T())

var lst: int() = (1, 2, 3);
var objLst: FizzBuzz() = (new FizzBuzz(), new FizzBuzz());

10.3 Sets (T{})

var st: int{} = {1, 2, 3};
var structSt: Point{} = {Point { x:0, y:0 }, Point { x:1, y:1 }};

10.4 Maps (map<K/V>)

var mp: map<int> = ["key1": 10, "key2": 20];          // default key type is string
var mp2: map<string/float> = ["key1": 10.5, "key2": 20.5];
var objMp: map<FizzBuzz> = ["obj1": new FizzBuzz(), "obj2": new FizzBuzz()];
  • map<T> は map<string/T> のシンタックスシュガー（エイリアス）とする。
  • キー型を明示したい場合は map<K/V> を用いる。

11. enum and bitfield enum

11.1 Normal enum

enum Color {
    Red,
    Green,
    Blue
}
  • 内部表現は常に int32。
  • enum<byte>, enum<string> のような基礎型の変更は許可されない。
  • enum 型も型ベース with の対象となり、with (Color) { Red; } などで列挙子に短縮アクセスできる。

11.2 bitfield enum

bitfield enum LogFlags {
    None = 1 << 0,
    Info = 1 << 3,
    Warn,
}

b(n) 記法

bitfield enum LogFlags {
    None : b(0),
    Info : b(3),
    Warn,            // highest b(n) so far is 3; Warn becomes b(4) = 1 << 4
}
  • b(n) はビット n（1 << n）を表す。
  • Name : b(n) でその列挙子にビット n を割り当てる。
  • bitfield enum で使用可能なビットは最大 32（n は 0–31）。
  • 同一ビットを共有する列挙子があっても、コンパイラは警告にとどめ、エラーにはしない。
  • 基礎型は常に int32 であり、bitfield enum<byte> などはコンパイルエラー。

12. main Block（エントリポイント）

function main(args: string[]): int {
    var fb = new FizzBuzz();
    fb.run();
    return 0;
}
  • プログラムのエントリポイント。
  • args はコマンドライン引数。
  • main ブロックは function キーワードで宣言され、戻り値の型は常に int。
  • 戻り値は return 文で明示的に指定する。
    （従来バージョンの「main 内の _ の値がプロセス終了コードになる」仕様は削除された。）

13. Temp Variable _（一時変数）

function main(args: string[]): int {
    var x = 10;
    _ = x + 5;   // _ の型はこのブロック内で最初に代入された式から推論される（戻り値とは無関係）
    return 0;
}

method foo(): void {
    var x = 10;
    _ = x + 5;   // type of _ is inferred in this block
}
  • _ は任意のブロックで宣言なしに使用できる特別な一時変数。
  • main も含め、_ の型は「そのブロックで最初に代入された式の型」で決まる。
  • main の戻り値は return 文で決まり、_ の値とは無関係。
  • with ブロック内では、追加で「現在の with 対象」を指す意味を持つ（4.2 参照）。

14. swap Keyword（swap キーワード）

var a = 10;
var b = 20;
swap (a, b);
print(a); // 20
print(b); // 10
  • swap (x, y); は x と y の値（または参照）を入れ替える。
  • swap が妥当かどうかはコンパイル時にチェックされる。

18. Pipes & Filters（パイプとフィルタ）

Aloe は pipe<T> と filter(...) を用いて、 サーバ / CLI / Web アプリなどのストリーミング処理を構成する。 あわせて Result 型と ROP（Railway Oriented Programming）により、 成功・失敗を 2 本の「線路」として扱うスタイルを取る。

18.0 Result<T> 型と Railway Oriented Programming（概略）

18.0.1 Result<T> の定義（概念レベル）

  • Result<T> は「成功か失敗か」を表すラッパー型。
  • 成功:
    • Ok(value: T) のような形で値を保持する。
  • 失敗:
    • Error(error: Exception) のような形でエラー情報を保持する。
  • どちらのケースかはパターンマッチ（将来拡張）や match による分岐で扱う。

18.0.2 Railway Oriented Programming（ROP）のイメージ

ROP は、処理の流れを「成功の線路」と「失敗の線路」の 2 本で表現する考え方である。 それぞれのフィルタは Result<T> を受け取り、 成功なら次のステージに、失敗なら「失敗の線路」に乗せて伝播させる。 

入力
  |
  v
[ Filter1 ] ----(Error)---> [ Error Track ]
  |
  v
[ Filter2 ] ----(Error)---> [ Error Track ]
  |
  v
 出力
  • 左から右へ、「成功の線路」をたどりながらフィルタが順次適用される。
  • 途中でエラーが起きた場合は、その時点で失敗として「エラー線路」に分岐する。
  • Result 型を使うことで、この 2 本の線路を型レベルで区別できる。

18.1 pipe<T> の基本

  • pipe<T> は T 型要素のストリーム。
  • pipe<byte> … 標準入力やソケットなどのバイト列。
  • pipe<string> … 文字列ストリーム。
  • pipe は共有メモリではなく値コピーを前提とする。
  • ユーザー定義 pipe 型は pipe<T> の形で宣言する（専用の「pipe 定義文」は存在しない）。

18.1.1 組み込みパイプ

  • stdin : pipe<byte>
  • stdout : pipe<byte>
  • stderr : pipe<byte>（任意）

18.2 filter(...) の基本

filter(F) はパイプライン中の中間変換ステージ。

18.2.1 フィルタ定義

filter lineSplit {
    in:  pipe<string>;
    out: pipe<string>;

    // オプション型を指定（省略可）
    options: LineSplitOptions;

    // フィールド定義も可能だが、ROP 的には副作用のない設計が推奨される
    field delimiter: string;

    // コンストラクタ（省略可）
    constructor(delim: string) {
        this.delimiter = delim;
    }

    // ROP スタイル：Result<T> を返す
    bound(input: pipe<string>, output: pipe<string>): Result<void> {
        foreach (chunk in input) {
            let lines = chunk.split(this.delimiter);
            foreach (line in lines) {
                output.write(line);
            }
        }
        output.close();
        return Ok();
    }
}
  • in / out で入力パイプ型と出力パイプ型を指定する。
  • options で「このフィルタに対応するオプション型」を指定できる。
    • options: SomeFilterOptions; のように記述する。
  • field でフィルタ内部の状態を保持することもできるが、 ROP（Railway Oriented Programming）的には「フィルタは副作用を持たず、 入力とオプションだけに依存した純粋な変換」であることが望ましい。 そのため、フィールド使用は必要最小限にとどめることが推奨される。
  • constructor(...) でフィルタ用のコンストラクタを定義できる。
  • bound は ROP スタイルで Result<T> を返す。
    • 正常終了時は Ok() あるいは Ok(value) を返す。
    • エラー時には Error(e) のように失敗を返す実装（詳細はランタイム仕様で定義）。

18.2.2 フィルタオプション

bitfield enum JsonOptions {
    None      : b(0),
    Pretty    : b(1),
    IncludeNulls : b(2),
}

// フィルタ定義側で options を指定
filter json<T> {
    in:  pipe<string>;
    out: pipe<T>;

    options: JsonOptions;

    bound(input: pipe<string>, output: pipe<T>): Result<void> {
        // ...
        return Ok();
    }
}

// パイプライン側での指定

stdin
    | filter(utf8)
    | filter(json<User>: JsonOptions.Pretty | JsonOptions.IncludeNulls)
    | users;
  • オプションは : の後に | でつなげて指定する。
    • filter(json<User>: JsonOptions.Pretty | JsonOptions.IncludeNulls)
  • フィルタ定義側で options: JsonOptions; と指定されている場合、 パイプライン側での記述では型名を省略し、 列挙子だけで書くことも可能とする： 
    stdin
    | filter(utf8)
    | filter(json<User>: Pretty | IncludeNulls)
    | users;
  • options に指定していない型の列挙子や値を : の後に書いた場合はコンパイルエラー。

18.3 パイプライン文と | / パイプ with 演算子 |>

stdin
    | filter(utf8)
    | filter(json<DateTime>)
    | dates;
  • A | B | C; で A の出力が B へ、B の出力が C へ接続される。
  • 型チェックにより各ステージの入出力型整合性を確認。
  • パイプライン文そのものは値を返さない（副作用として接続を張るだけ）。

18.3.1 パイプ with 演算子 |>

パイプ with 演算子 |> は、 「同じ入力ストリーム（あるいはコレクション）に対して条件付き filter(if) を適用する」 ためのショートカット構文として定義する。 

// 元になる書き方
users
    | filter(if: .Age >= 20 and .Gender == Gender.Male)
    | adults;

// ショートカット
users |> .Age >= 20 and .Gender == Gender.Male
    | adults;
  • X |> 条件式 は X | filter(if: 条件式) の糖衣構文である。
  • 条件式の中では with と同様に . で現在要素のメンバーにアクセスできる。
  • and / or は && / || のエイリアスとして利用可能。

18.4 foreach と pipe

foreach (line in lines) {
    print("Input: " + line);
}
  • foreach (x in p) は内部的に p.next() を EOF まで呼ぶ。
  • 書き込み側が close() を呼ぶと読み側は EOF を検知してループを終了する。

18.5 組み込み map 関数（pipe 用）

パイプに対して要素変換を行う組み込み map 関数を用意する。 

// 各要素 x に関数 f を適用して新しい pipe を返す
function map<TIn, TOut>(source: pipe<TIn>, f: (TIn) -> TOut): pipe<TOut>;

numbers
    | map (x => x * 2)
    | doubled;
  • map は pipe<TIn> を受け取り、 各要素に関数を適用した結果を pipe<TOut> として流す。
  • 内部的には filter と類似の仕組みで実装されうるが、 「要素ごとの変換」であることを意図的に表すために別シンボルを用意する。

18.6 コレクション向け map（テンプレートを使った変換）

コレクション（リスト等）に対しても、テンプレート指定による map を利用できる。 

users as u
    | map Customer {
        Name    = u.Name,
        Address = u.Address
      }
    | customers;
  • users as u | map Customer { ... } は、 users コレクションの各要素に対してブロック内のテンプレートを適用し、 Customer 型の新しいコレクションを生成することを表す。
  • テンプレート内では u が元要素を指す。
  • 戻り値のコレクション型（例：Customer(), Customer[]）は 文脈や map のオーバーロード仕様で決定される（詳細は将来拡張）。

18.7 close と EOF 伝播

  • p.close() 後の write() は無効（エラーになり得る）。
  • フィルタは入力 EOF を検知したら出力側も close() すべき。

18.8 pipe を使った並行モデル

  • タスク同士の主な通信手段は pipe<T>。
  • shared-nothing モデルを前提とし、共有可変データ構造は持たない設計を推奨。

18.9 match によるパイプ分岐（Result<T> の分岐）

Result<T> を扱うパイプラインでは、 match を使って成功・失敗などの条件に応じて分岐できる。 

results
    | match {
        case is Ok:
            // 成功時の後続パイプ
            | filter(handleSuccess)
            | successSink;
        case is Error:
            // 失敗時の後続パイプ
            | filter(handleError)
            | errorSink;
        default:
            // どの条件にも当てはまらない場合（将来拡張用）
            break;
      };
  • | match { ... } は、直前までのパイプライン出力を条件に応じて分岐させる。
  • 一般的な書式は次の通り： 
    | match {
    case 条件式1:
        // この case にマッチしたときのパイプ列
        | filter(A)
        | filter(B);
    case 条件式2:
        | filter(C);
    default:
        break;
}
  • 各 case の後ろには、その条件にマッチしたときの「後続パイプ」を記述できる。
  • default 節は任意。いずれの case にもマッチしない場合に実行される。

18.10 例：簡単な REST 風パイプライン

class User {
    field id: int;
    field name: string;
}

function main(args: string[]): int {
    let users: pipe<User> = pipe<User>.create();

    stdin
        | filter(utf8)
        | filter(json<User>)
        | users;

    let u: User? = users.next();
    if (u is User) {
        print("User: " + u.name);
    }

    return 0;
}

utf8 や json<T> などの標準フィルタの詳細は、 コア言語仕様ではなく別途「標準フィルタライブラリ」仕様で定義される。

以上が、元の英語仕様に基づき、 with ブロックの型対応拡張・pipe/filter/Result/ROP・map/and/or などの拡張を加えた Aloe 言語仕様（日本語・HTML 版） です。
