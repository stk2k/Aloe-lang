0. Overview（概要）

Aloe は静的型付けのスクリプト言語であり、シンプルで直感的な構文を目指している。
スタックベースの仮想マシン（AloeVM）上で動作し、主に次のような目標を持つ。

  • 教育用途や小規模アプリケーションで読み書きしやすいこと
  • C# ランタイムおよび WebAssembly (WASM) 環境で動作すること
  • pipe と filter によって、I/O と並行処理を静的型付けのまま制御できること

特徴：

  • 静的型付け
    • var による型推論
    • let による明示的な型指定
  • 値型 / 参照型
    • struct … 値型
    • class … 参照型
  • GC ベースのメモリ管理（具体的なアルゴリズムは VM 実装依存）
  • = null をソースコード中に直接書かない
    • 参照型は内部的には null を取り得るが、delete による糖衣構文で扱う。
  • pipe / filter によるデータフロースタイルの設計

1. Basic Syntax（基本文法）

1.1 Comments（コメント）

// single-line comment

/* multi-line
   comment */
  • // から行末までが行コメント。
  • /* ～ */ が複数行コメント。
  • ネストした /* */ は未定義（サポートしない想定）。

1.2 Identifiers（識別子）

  • 使用可能文字：英字 (A–Z, a–z)、数字 (0–9)、アンダースコア _
  • 先頭文字は英字または _ でなければならない。
  • 大文字小文字は区別される。
  • キーワードと同じ綴りは使用できない。
  • _ で始まる識別子は「プライベート」扱い（外部からは参照しない慣習）。
  • ユニコード文字も使用できる（空白や区切り記号などは除外）。

  • package は使わず、namespace を使用する。

  • instanceof は廃止され、型チェックには is を用いる。

1.4 Literals（リテラル）

  • 整数リテラル：10, 0, -5, 0x1A, 0b1010 など
  • 浮動小数点リテラル：10.5, 0.0, -3.14
  • 10進リテラル：10.12345678901234567890:d など
  • 真偽値：true, false
  • 文字列："Hello, World!"
  • 文字：'A', 'z' など
  • null リテラル：null
  • 配列リテラル：[1, 2, 3]
  • リストリテラル：(1, 2, 3)
  • セットリテラル：{1, 2, 3}
  • マップリテラル：["key1": 10, "key2": 20]
  • enum リテラル：EnumType.Value1
  • struct リテラル：Point { x: 0, y: 1 }
  • オブジェクト生成：new ClassName()

1.5 Statement terminator（文の終端）

var x = 10;
x = x + 1;
print(x);
  • すべての文はセミコロン ; で終わる。
  • 改行による自動セミコロン挿入は行わない。

1.6 Whitespace（空白）

  • スペース、タブ、改行はトークン区切りとして扱われる。
  • 連続する空白は 1 つの区切りとみなされる。
  • インデントに意味はない（Python とは異なる）。

1.7 Case sensitivity（大文字小文字）

  • 識別子とキーワードは大文字小文字を区別する。
  • キーワードはすべて小文字。

2. Type System（型システム）

2.1 Primitive types（プリミティブ型）

Aloe における「プリミティブ型」は、次の 7 種類とする。

  • bool / boolean – 真偽値。
  • int / integer – 符号付き整数。
  • float – 浮動小数点数。
  • decimal – 高精度 10 進数。
  • string – 文字列（内部的には Unicode）。
  • null – 「値が存在しない」ことを表す特別な型兼値。
  • void – 戻り値なし（メソッド戻り値専用）。

any 型は存在せず、動的型付けはサポートしない。
char や byte は言語としてはサポートされるが、プリミティブ型のカテゴリからは外し、 数値と変換の章（3 章）で別途扱う値型とする。

2.2 Composite / user-defined types（合成型 / ユーザー定義型）

  • 配列：T[]
  • リスト：T()
  • セット：T{}
  • マップ：map<T> または map<K/V>
  • struct
  • class
  • enum, bitfield enum
  • 非同期タスク：task
  • パイプ：pipe<T>（18章参照）

any 型は存在せず、動的型付けはサポートしない。

2.3 Value types / reference types（値型 / 参照型）

  • 値型：
    • int, float, decimal, bool, byte, char
    • struct
    • enum, bitfield enum（内部表現は int32）
  • 参照型：
    • class
    • string（実装上は参照型）
    • task, pipe<T> など

Struct restrictions and copying（struct の制約とコピー）

  • struct のフィールドにはプリミティブ型と他の struct のみを持てる。
  • class などの参照型フィールドは持てない。
  • struct の代入は常に値コピー。
  • 最大サイズなどは仕様で制限しない。

Class vs struct（class と struct）

  • class は struct をフィールドとして持てる。
  • class インスタンスを別変数に代入すると参照がコピーされる（オブジェクト本体は共有）。

3. Numbers & Conversions（数値と暗黙変換）

3.1 int and float

  • int と float を混在させた式の結果は float になる。
    • int + float → float
    • int - float → float
    • int * float → float
    • int / float → float

3.2 decimal

  • decimal は高精度な 10 進数。
  • int / float と混在する式は、必要に応じて decimal に昇格しうる。
  • オーバーフロー / アンダーフロー / 精度問題は OverflowException として扱う。

3.3 enum and int

  • Aloe にはキャスト構文がない。
  • enum と int の暗黙 / 明示変換はいずれも禁止。
  • enum と数値型の混在式はコンパイルエラー（bitfield enum のビット演算を除く）。
  • 内部表現としては enum / bitfield enum はすべて int32。

3.4 byte and char

  • byte：
    • 範囲は 0–255。
    • byte 同士の演算は許されるが、範囲外はエラーになり得る。
    • byte と int の混在式は int に昇格。
  • char：
    • char の算術演算は不可。
    • 比較演算（==, <, > 等）は char 同士のみ。
    • 数値型との暗黙変換はない。

3.5 NaN / Infinity

  • 数値オーバーフロー/アンダーフローは NaN/Infinity ではなく例外で扱う。
  • ZeroDivisionException は 0 / 0.0 で常に投げる。

4. Blocks & Scope（ブロックとスコープ）

ブロックは { と } で区切られ、各ブロックが新しいスコープを導入する。

  • main ブロック（function main(...)）
  • クラス定義ブロック
  • メソッド定義ブロック
  • 任意の { ... } ブロック

4.1 Static (lexical) scope and shadowing（静的スコープとシャドウイング）

class Sample {
    field value: int = 1;
    method foo(): void {
        var value = 2;        // shadows the field
        print(value);         // 2
        print(this.value);    // 1
    }
}
  • Aloe は静的（レキシカル）スコープ。
  • 内側ブロックで同名変数を宣言すると、外側の変数をシャドウする。
  • フィールド名とローカル変数名がぶつかった場合、ローカル変数が優先される。 フィールドは this. で参照する。

4.2 with-blocks（メンバー省略記法）

Aloe には、同一オブジェクトのメンバーアクセスを簡潔に書くための with ブロックがある。 

with (user) {
    .Name = "Alice";
    .Age = .Age + 1;
    print(.Name);
}
  • with (expr) { ... } ブロック内では：
    • .Member は (expr).Member の省略形。
    • expr はブロックの先頭で 1 度だけ評価され、内部一時変数に保持される。
    • 特別な一時変数 _ は「with 対象オブジェクト」を指す。
      • log(_); のように渡すことができる。
  • ネストした with ブロック：
    • 内側の with が . の解決および _ をシャドウする。
    • 外側オブジェクトへは通常の変数名でアクセスできる。
  • C# の using スタイルのように、同じブロックに複数の with を並べられる：
with (user), (profile) {
    .Name = "Alice";  // resolved against both user and profile
}
  • どのオブジェクトのメンバーに解決されるかの順序は実装依存。
  • ただし曖昧な場合（複数オブジェクトが同じメンバーを持つ）はコンパイルエラー。

with はあくまで構文糖衣であり、意味的にはすべて 明示的な obj.Member 呼び出しに書き換えた上で 型チェック・コード生成を行うことができる。

4.2.1 Type-based with（class / struct / enum の static / 列挙子ショートカット）

通常の with (expr) に加えて、型名 を対象にした with を利用できる。 

with (AngleUtil) {
    .DegToRad(90.0);
    .Pi = 3.14159;
}

ここで TypeName は次のいずれか：

  • class 型
  • struct 型
  • enum / bitfield enum

この場合、with は「型コンテキスト with」として解釈される：

  • () 内の式は実行時に値として評価されない。
    • コンパイラは TypeName を型として解決し、インスタンス生成などは行わない。
  • ブロック内では：
    • .Member は TypeName.Member の省略形。
      • 対象となる Member は
        • static フィールド
        • static メソッド
        • enum / bitfield enum の列挙子
    • 一時変数 _ は、TypeName の static コンテキストを表す擬似オブジェクトとして振る舞う。
      • _.Member も TypeName.Member と等価。
    • 型ベース with 内でインスタンスメンバーにアクセスすることはコンパイルエラー。

なお、型ベース with でもローカル変数や外側スコープの変数が同名で存在すればそちらが優先される。 列挙子や static メンバーを明示したいときは TypeName.Member または .Member / _.Member を用いる。

例：class / struct の static アクセス
struct AngleUtil {
    static field Pi: float = 3.14159;

    static method DegToRad(deg: float): float {
        return deg * Pi / 180.0;
    }
}

class MathUtil {
    static method Max(a: int, b: int): int {
        if (a > b) return a;
        return b;
    }
}

function main(args: string[]): int {
    with (AngleUtil) {
        var rad = DegToRad(90.0);  // AngleUtil.DegToRad(90.0)
        print(.Pi);                // AngleUtil.Pi
    }

    with (MathUtil) {
        var m = Max(10, 20);       // MathUtil.Max(10, 20)
        print(m);
    }

    return 0;
}
例：enum の列挙子アクセス
enum Color {
    Red,
    Green,
    Blue
}

function main(args: string[]): int {
    with (Color) {
        let c1: Color = Red;   // Color.Red
        let c2: Color = Blue;  // Color.Blue
        print(c1);
        print(c2);
    }
    return 0;
}

型ベース with も複数ターゲット構文に混在させられる： 

with (user), (Color) {
    .Name = "Alice"; // user.Name
    let c: Color = Red; // Color.Red
}

この場合も解決順は実装依存だが、曖昧であればコンパイルエラーとする。

5. Variables & Constants（変数と定数）

5.1 var（型推論）

var i = 10;
var f = 10.5;
var s = "Hello";
var c = 'A';
var b = 255;
var arr = [1, 2, 3];
var lst = (1, 2, 3);
var st  = {1, 2, 3};
var mp  = ["key": 10];
var fb  = new FizzBuzz();
  • 変数の型は右辺の式から推論される。
  • var x = 1; は常に int。
  • var x = 1.0; は float として扱われる。
  • var x = null; はコンパイルエラー（型が推論できない）。

5.2 let（明示的な型）

let i: int = 10;
let f: float = 10.5;
let d: decimal = 10.12345678901234567890:d;
let s: string = "Hello, Aloe!";
let c: char = 'A';
let b: byte = 255;

let arr: int[] = [1, 2, 3];
let lst: int() = (1, 2, 3);
let st : int{} = {1, 2, 3};
let mp : map<string/int> = ["key": 10];

5.3 Constants（const）

const MaxCount: int = 100;
const Pi: float = 3.14159;
const AppName: string = "Aloe VM";
  • const 変数は再代入不可。
  • const には型注釈が必須。

6. Fields（フィールド宣言）

class / struct のフィールドは field キーワードで宣言する。

class Sample {
    field Count: int = 0;
    field Name: string = "default";

    readonly field Id: int;

    construct(Id: int) {
        this.Id = Id; // OK: inside the constructor
    }
    method Change(): void {
        this.Count = this.Count + 1; // OK
        // this.Id = 2; // NG: readonly field cannot be changed outside constructor
    }
}
  • field … 通常のフィールド。
  • readonly field … コンストラクタ内のみ代入可、それ以降は読み取り専用。
  • struct / class の両方で同じ意味。

7. Class / Struct / Interface / Trait

7.1 class

class FizzBuzz {
    field _Count: int = 0;
    field Name: string = "FizzBuzz";

    construct() {
    }

    method Run(): void {
        var i = 1;
        while (i <= 100) {
            // ...
            i = i + 1;
        }
    }
}
  • class は参照型。
  • sealed class は extends による継承不可。
  • 単一継承のみサポート。

7.2 struct

struct Point {
    field X: int;
    field Y: int;

    construct(X: int, Y: int) {
        this.X = X;
        this.Y = Y;
    }
}
  • struct は値型。
  • フィールドはプリミティブ型か他の struct に限定（参照型フィールドは禁止）。
  • struct のコピーは常に値コピー。
  • struct の継承はサポートしない（sealed struct は「拡張禁止」マーカー）。

7.3 Inheritance and sealed

class Base {
    method Foo(): void { }
}

class Derived extends Base {
    method Bar(): void { }
}

sealed class FinalDerived extends Base {
    // cannot be extended further
}

7.4 interface

interface ILogger {
    method Log(msg: string): void;
}

class ConsoleLogger implements ILogger {
    method Log(msg: string): void {
        print(msg);
    }
}
  • implements キーワードでインタフェースを実装。
  • is 演算子でインタフェース実装有無を判定。

7.5 trait

trait Trait_A {
    method Hello(): void {
        print("Hello from Trait_A");
    }
}

7.5.1 with による trait の適用

var obj = new SomeClass() with Trait_A;
obj.Hello(); // Trait_A からのメソッド（衝突がなければ）

7.5.2 非破壊な追加・削除（+ / -）

var obj2 = obj + Trait_A;      // obj は変更されず、obj2 に Trait_A が付与される
var obj3 = obj2 - Trait_A;     // obj3 は obj2 から Trait_A を取り除いたもの

7.5.3 as によるエイリアス適用

var obj2 = obj + Trait_A as a;
obj2.a.Hello(); // OK: a 経由で Trait_A のメンバーにアクセス

// obj2.Hello(); // NG: alias のみなのでメンバーはフラットにはならない

7.5.4 Trait の衝突

  • 複数 trait が同名フィールド / メソッドを定義するとコンパイルエラー。
  • as a などのエイリアス付き適用は衝突判定の対象外。
  • クラス本体と trait メンバとの衝突の扱いは将来バージョンで定義予定。

8. delete & null

var obj = new FizzBuzz();
delete obj;  // equivalent to: obj = null (at VM level)
  • delete obj; は VM レベルでは obj = null と等価。
  • ソースコード中で obj = null; と書くことはできない。
  • 参照型は内部的には null を取り得るが、型システムは null を追跡しない。
  • null 状態でのアクセスは NullReferenceException を引き起こす可能性がある。

9. is Operator（is 演算子）

if (obj is FizzBuzz) {
    // true if runtime type is FizzBuzz or a subclass
}

if (obj is ILogger) {
    // true if obj implements ILogger
}

if (obj is Trait_A) {
    // true if obj has Trait_A applied (with / +)
}
  • is は実行時の型チェックを行う。
  • 継承関係（Derived は Base としても扱う）を考慮する。
  • インタフェースや trait に対しても使用できる。

10. Collections（配列 / リスト / セット / マップ）

10.1 Arrays (T[])

var arr: int[] = [1, 2, 3];
var matrix: float[][] = [[1.0, 2.0], [3.0, 4.0]];

10.2 Lists (T())

var lst: int() = (1, 2, 3);
var objLst: FizzBuzz() = (new FizzBuzz(), new FizzBuzz());

10.3 Sets (T{})

var st: int{} = {1, 2, 3};
var structSt: Point{} = {Point { X:0, Y:0 }, Point { X:1, Y:1 }};

10.4 Maps (map<K/V>)

var mp: map<int> = ["key1": 10, "key2": 20];          // default key type is string
var mp2: map<string/float> = ["key1": 10.5, "key2": 20.5];
var objMp: map<FizzBuzz> = ["obj1": new FizzBuzz(), "obj2": new FizzBuzz()];

11. enum and bitfield enum

11.1 Normal enum

enum Color {
    Red,
    Green,
    Blue
}
  • 内部表現は常に int32。
  • enum<byte>, enum<string> のような基礎型の変更は許可されない。
  • enum 型も型ベース with の対象となり、with (Color) { Red; } などで列挙子に短縮アクセスできる。

11.2 bitfield enum

bitfield enum LogFlags {
    None = 1 << 0,
    Info = 1 << 3,
    Warn,
}

b(n) 記法

bitfield enum LogFlags {
    None : b(0),
    Info : b(3),
    Warn,            // highest b(n) so far is 3; Warn becomes b(4) = 1 << 4
}
  • b(n) はビット n（1 << n）を表す。
  • Name : b(n) でその列挙子にビット n を割り当てる。
  • bitfield enum で使用可能なビットは最大 32（n は 0–31）。
  • 同一ビットを共有する列挙子があっても、コンパイラは警告にとどめ、エラーにはしない。
  • 基礎型は常に int32 であり、bitfield enum<byte> などはコンパイルエラー。

12. main Block（エントリポイント）

function main(args: string[]): int {
    var fb = new FizzBuzz();
    fb.Run();
    return 0; // process exit code (int)
}
  • プログラムのエントリポイント。
  • args はコマンドライン引数。
  • main の戻り値は int 型であり、 return 文でプロセスの終了コードを指定する。

13. Temp Variable _（一時変数）

method Foo(): void {
    var x = 10;
    _ = x + 5;   // このブロック内だけで使う一時変数
}
  • _ は任意のブロックで宣言なしに使用できる特別な一時変数。
  • 各ブロックごとに独立した _ を持ち、 そのブロックで最初に代入された式の型が _ の型になる。
  • main ブロックでも通常の一時変数として扱われる（終了コードは return で決定）。
  • with ブロック内では、追加で「現在の with 対象」を指す意味を持つ（4.2 参照）。

14. swap Keyword（swap キーワード）

var a = 10;
var b = 20;
swap (a, b);
print(a); // 20
print(b); // 10
  • swap (x, y); は x と y の値（または参照）を入れ替える。
  • swap が妥当かどうかはコンパイル時にチェックされる。

15. Operators（演算子）

15.1 優先順位（高い順）

  1. メンバー / 呼び出し：x.y, x[y], x(args...)
  2. 単項：+x, -x, !x, delete x
  3. 乗除：*, /, %
  4. 加減：+, -
  5. シフト：<<, >>
  6. 関係：<, <=, >, >=
  7. 等価：==, !=
  8. ビット AND：&
  9. ビット XOR：^
  10. ビット OR：|（パイプライン文での | とは別構文）
  11. 論理 AND：&&
  12. 論理 OR：||
  13. パイプライン演算子：|, |>
    • パイプライン文専用の構文であり、通常の式の優先順位とは別に構文レベルで解釈される。
  14. 代入：=

複合代入（+= 等）はまだ仕様化されていない。

15.2 結合規則

  • 単項演算子：右結合。
  • 代入：右結合。
  • その他の 2 項演算子：左結合。
  • パイプライン文は A | B | C のように左結合で解釈される（18章参照）。

18. Pipes & Filters（パイプとフィルタ）

Aloe は pipe<T> と filter(...) を用いて、 サーバ / CLI / Web アプリなどのストリーミング処理を構成する。 本章では Railway Oriented Programming (ROP) に影響を受けた Result<T> ベースのパイプラインモデルも説明する。

18.1 pipe<T> の基本

  • pipe<T> は T 型要素のストリーム。
  • pipe<byte> … 標準入力やソケットなどのバイト列。
  • pipe<string> … 文字列ストリーム。
  • pipe は共有メモリではなく値コピーを前提とする。
  • パイプ自体は pipe<T> の静的メンバーなどで生成する： 
    let p: pipe<int> = pipe<int>.create();

18.1.1 組み込みパイプ

  • stdin : pipe<byte>
  • stdout : pipe<byte>
  • stderr : pipe<byte>（任意）

18.3 Result<T> と Railway Oriented Programming（ROP）

Aloe のパイプラインは、失敗を例外ではなく値で表現するために Result<T> 型を用いることを推奨する。

18.3.1 Result<T> の意味論

  • Result<T> は「成功」または「失敗」のいずれか 1 つの状態を持つコンテナ。
  • 概念的には： 
    // 疑似的な表現（実際の構造は標準ライブラリ仕様で定義）
struct Result<T> {
    field IsOk: bool;
    field Value: T;              // IsOk == true のとき有効
    field Error: Exception;      // IsOk == false のとき有効
}
  • 成功値を生成する Result.Ok(value)、 失敗値を生成する Result.Error(error) のようなヘルパーが 標準ライブラリ側で用意される想定。

18.3.2 Railway Oriented Programming（ROP）のイメージ

ROP では、「成功の線路」と「失敗の線路」を分けて処理をつなげる。 フィルタは Result<T> を受け取り、成功なら次へ、失敗ならそのまま失敗の線路を流す。 

入力
  ──> [ Validate ] ──> [ Transform ] ──> [ Save ] ──> Result<Ok>
           │                │                │
           └─× Error        └─× Error        └─× Error
                         （途中で失敗した場合は Result<Error> が右方向に流れ続ける）

ROP スタイルのフィルタは、できるだけ副作用を減らし「同じ入力 Result に対して 常に同じ出力 Result を返す（参照透過）」ように設計することが推奨される。

18.2 filter(...) の基本

filter(F) はパイプライン中の中間変換ステージ。

18.2.1 フィルタ定義

filter LineSplit {
    in:  pipe<string>;
    out: pipe<string>;

    // オプションクラス（後述 18.2.2）を指定できる
    options: LineSplitOptions;

    // フィールド定義も可能（ただし ROP 的には非推奨）
    field Separator: string = "\n";

    construct(sep: string) {
        this.Separator = sep;
    }

    bound(input: pipe<string>, output: pipe<string>): Result<void> {
        foreach (chunk in input) {
            let lines = chunk.split(this.Separator);
            foreach (line in lines) {
                output.write(line);
            }
        }
        output.close();
        return Result.Ok();   // 成功
    }
}
  • in / out で入力パイプ型と出力パイプ型を指定する。
  • options で、このフィルタが受け付けるオプションクラスを指定できる。
  • field でフィルタ内部の状態を保持することもできるが、 ROP（Railway Oriented Programming）的な観点からは冪等性を損なう可能性があるため推奨しない。 可能であれば、状態は引数や Result / オプションに載せて渡す。
  • construct(...) でコンストラクタを定義できる。 フィルタがインスタンス化される際に呼び出される。
  • bound(input, output) はフィルタの本体であり、 Result<T>（ここでは Result<void>）を返す。 例外ではなく Result を返すことで、パイプライン全体を ROP スタイルで構成できる。

18.2.2 フィルタオプション

bitfield enum JsonOptions {
    None        : b(0),
    PrettyPrint : b(1),
    IncludeNulls: b(2),
}

オプションは filter(json<User>: ...) のように : の後ろに | でつなげて指定する。 

filter(json<User>: JsonOptions.PrettyPrint | JsonOptions.IncludeNulls)
  • 各フィルタは自前のオプション型（通常は bitfield enum 等）を持つ。
  • フィルタ定義内で 
    filter JsonSerializer {
    in:  pipe<User>;
    out: pipe<string>;

    options: JsonOptions; // ← オプションクラスを指定
    ...
}
    のように options: JsonOptions; を指定した場合、 使用側でオプションを指定するときはクラス名を省略できる： 
    // フィルタ定義に options: JsonOptions; があるため
// JsonOptions. プレフィックスを省略して書ける
filter(json<User>: PrettyPrint | IncludeNulls)
  • 別のオプションクラスを使いたい場合は options で明示的に切り替える。

18.4 パイプライン文と | / パイプ with 演算子 |>

stdin
    | filter(utf8)
    | filter(json<DateTime>)
    | dates;
  • A | B | C; で A の出力が B へ、B の出力が C へ接続される。
  • 型チェックにより各ステージの入出力型整合性を確認。
  • パイプライン文そのものは値を返さない（副作用として接続を張るだけ）。

パイプ with 演算子 |> は「同じ入力を複数の枝に流す」用途など、 将来的な拡張に予約されている。現行ドラフトでは構文のみ予約し、 振る舞いは別途 ROP 拡張仕様で定義される。

18.5 foreach と pipe

foreach (line in lines) {
    print("Input: " + line);
}
  • foreach (x in p) は内部的に p.next() を EOF まで呼ぶ。
  • 書き込み側が close() を呼ぶと読み側は EOF を検知してループを終了する。

18.6 map によるパイプ要素変換

Aloe のパイプには「要素ごとに関数を適用する」ための 組み込み map 操作が存在する（概念的には以下のようなもの）： 

// 疑似シグネチャ
method pipe<T>.map<U>(f: T -> U): pipe<U>

使用例：

stdin
    | filter(utf8)
    | map (s: string -> s.trim())      // 前後の空白を削除
    | map (s: string -> s.toUpper())   // 大文字化
    | filter(lineSink);
  • map 自体は filter の一種として実装してもよいし、 pipe<T> のメソッドとして実装してもよい（実装依存）。
  • ROP スタイルと組み合わせる場合は Result<T> -> Result<U> を返す関数を渡すことで 失敗を線路から落とさずに伝播できる。

18.6.1 コレクションパイプ向けテンプレート map

要素がオブジェクトであるパイプに対して、テンプレート構文を用いて 別の型のコレクションを生成する map も表現できる。 

// users: pipe<User> を Customer の列に変換するイメージ
users as u | map Customer {
    Name    = u.Name,
    Address = u.Address
}
  • users as u で要素を表す別名 u を導入する。
  • map Customer { ... } で Customer 型の新しいオブジェクトを構築し、 それらを要素とする新しいパイプ / コレクションを生成する。
  • 上記は構文糖衣であり、実装上は通常の map (u -> new Customer { ... }) に 展開してもよい。

18.8 match によるパイプ分岐（Result<T> の分岐）

Result<T> を使った ROP スタイルでは、 成功・失敗・その他の条件に応じてパイプラインを分岐させるための match 構文を用意する。

18.8.1 match の基本書式

| match {
    case 条件式1:
        break;
    case 条件式2:
        break;
    default:
        break;
}
  • | match { ... } は、直前までのパイプライン結果に対して条件分岐を行う。
  • case の条件式には Result<T> を前提とした パターンや述語（例：result.IsOk）を記述できる。
  • break; は「この枝のパイプラインをここで終了する」ことを示す。
  • default: はどの case にもマッチしなかった場合に実行される。

18.8.2 case の後に後続パイプを指定する

case 節の中では、さらに後続のパイプを指定することもできる。 

stdin
    | filter(utf8)
    | filter(ParseUser)            // Result<User> を返すフィルタ
    | match {
        case result.IsOk:
            | filter(HandleUser)   // 成功時のみ HandleUser へ
        case result.IsOk == false:
            | filter(HandleError)  // 失敗時はこちらのパイプへ
        default:
            break;
    };
  • それぞれの case 節内で始まる | filter(...) は、 その条件にマッチしたデータだけを流す枝として扱われる。
  • 文法上は 
    case 条件式:
    | ...;
    のように「パイプラインの続き」を書けるが、 詳細な型チェック・評価順序は VM / コンパイラ仕様で定義される。

18.6 pipe を使った並行モデル

  • タスク同士の主な通信手段は pipe<T>。
  • shared-nothing モデルを前提とし、共有可変データ構造は持たない設計を推奨。

18.7 例：ROP スタイルの単純な検証パイプライン

filter ValidateUser {
    in:  pipe<User>;
    out: pipe<Result<User>>;

    bound(input: pipe<User>, output: pipe<Result<User>>): Result<void> {
        foreach (u in input) {
            if (u.Name == "") {
                output.write(Result.Error(new ArgumentException("Name is empty")));
            }
            else {
                output.write(Result.Ok(u));
            }
        }
        output.close();
        return Result.Ok();
    }
}

filter SaveUser {
    in:  pipe<Result<User>>;
    out: pipe<Result<User>>;

    bound(input: pipe<Result<User>>, output: pipe<Result<User>>): Result<void> {
        foreach (r in input) {
            if (r.IsOk) {
                // DB に保存するなど
                output.write(r);
            }
            else {
                // 失敗はそのまま流す
                output.write(r);
            }
        }
        output.close();
        return Result.Ok();
    }
}

18.10 例：簡単な REST 風パイプライン

class User {
    field Id: int;
    field Name: string;
}

function main(args: string[]): int {
    let users: pipe<User> = pipe<User>.create();

    stdin
        | filter(utf8)
        | filter(json<User>)
        | users;

    let u: User? = users.next();
    if (u is User) {
        print("User: " + u.Name);
    }

    return 0;
}

utf8 や json<T> などの標準フィルタの詳細は、 コア言語仕様ではなく別途「標準フィルタライブラリ」仕様で定義される。

以上が、元の英語仕様に基づき、 with ブロックの型対応拡張・Result<T> / ROP / match / map 等の拡張を加えた Aloe 言語仕様（日本語・HTML 版） です。
