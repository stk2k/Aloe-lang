0. Overview（概要）

Aloe は静的型付けのスクリプト言語であり、シンプルで直感的な構文を目指している。
スタックベースの仮想マシン（AloeVM）上で動作し、主に次のような目標を持つ。

  • 教育用途や小規模アプリケーションで読み書きしやすいこと
  • C# ランタイムおよび WebAssembly (WASM) 環境で動作すること
  • pipe と filter によって、I/O と並行処理を静的型付けのまま制御できること

特徴：

  • 静的型付け
    • var による型推論
    • let による明示的な型指定
  • 値型 / 参照型
    • struct … 値型
    • class … 参照型
  • GC ベースのメモリ管理（具体的なアルゴリズムは VM 実装依存）
  • = null をソースコード中に直接書かない
    • 参照型は内部的には null を取り得るが、delete による糖衣構文で扱う。
  • pipe / filter によるデータフロースタイルの設計

1. Basic Syntax（基本文法）

1.1 Comments（コメント）

// single-line comment

/* multi-line
   comment */
  • // から行末までが行コメント。
  • /* ～ */ が複数行コメント。
  • ネストした /* */ は未定義（サポートしない想定）。

1.2 Identifiers（識別子）

  • 使用可能文字：英字 (A–Z, a–z)、数字 (0–9)、アンダースコア _
  • 先頭文字は英字または _ でなければならない。
  • 大文字小文字は区別される。
  • キーワードと同じ綴りは使用できない。
  • _ で始まる識別子は「プライベート」扱い（外部からは参照しない慣習）。
  • ユニコード文字も使用できる（空白や区切り記号などは除外）。

  • package は使わず、namespace を使用する。

  • instanceof は廃止され、型チェックには is を用いる。

1.4 Literals（リテラル）

  • 整数リテラル：10, 0, -5, 0x1A, 0b1010 など
  • 浮動小数点リテラル：10.5, 0.0, -3.14
  • 10進リテラル：10.12345678901234567890:d など
  • 真偽値：true, false
  • 文字列："Hello, World!"
  • 文字：'A', 'z' など
  • null リテラル：null
  • 配列リテラル：[1, 2, 3]
  • リストリテラル：(1, 2, 3)
  • セットリテラル：{1, 2, 3}
  • マップリテラル：["key1": 10, "key2": 20]
  • enum リテラル：EnumType.Value1
  • struct リテラル：Point { x: 0, y: 1 }
  • オブジェクト生成：new ClassName()

1.5 Statement terminator（文の終端）

var x = 10;
x = x + 1;
print(x);
  • すべての文はセミコロン ; で終わる。
  • 改行による自動セミコロン挿入は行わない。

1.6 Whitespace（空白）

  • スペース、タブ、改行はトークン区切りとして扱われる。
  • 連続する空白は 1 つの区切りとみなされる。
  • インデントに意味はない（Python とは異なる）。

1.7 Case sensitivity（大文字小文字）

  • 識別子とキーワードは大文字小文字を区別する。
  • キーワードはすべて小文字。

2. Type System（型システム）

2.1 Primitive types（プリミティブ型）

Aloe における「プリミティブ型」は、次の 7 種類とする。

  • bool / boolean – 真偽値。
  • int / integer – 符号付き整数。
  • float – 浮動小数点数。
  • decimal – 高精度 10 進数。
  • string – 文字列（内部的には Unicode）。
  • null – 「値が存在しない」ことを表す特別な型兼値。
  • void – 戻り値なし（メソッド戻り値専用）。

any 型は存在せず、動的型付けはサポートしない。
char や byte は言語としてはサポートされるが、プリミティブ型のカテゴリからは外し、 数値と変換の章（3 章）で別途扱う値型とする。

2.2 Composite / user-defined types（合成型 / ユーザー定義型）

  • 配列：T[]
  • リスト：T()
  • セット：T{}
  • マップ：map<T> または map<K/V>
  • struct
  • class
  • enum, bitfield enum
  • 非同期タスク：task
  • パイプ：pipe<T>（18章参照）

any 型は存在せず、動的型付けはサポートしない。

2.3 Value types / reference types（値型 / 参照型）

  • 値型：
    • int, float, decimal, bool, byte, char
    • struct
    • enum, bitfield enum（内部表現は int32）
  • 参照型：
    • class
    • string（実装上は参照型）
    • task, pipe<T> など

Struct restrictions and copying（struct の制約とコピー）

  • struct のフィールドにはプリミティブ型と他の struct のみを持てる。
  • class などの参照型フィールドは持てない。
  • struct の代入は常に値コピー。
  • 最大サイズなどは仕様で制限しない。

Class vs struct（class と struct）

  • class は struct をフィールドとして持てる。
  • class インスタンスを別変数に代入すると参照がコピーされる（オブジェクト本体は共有）。

3. Numbers & Conversions（数値と暗黙変換）

3.1 int and float

  • int と float を混在させた式の結果は float になる。
    • int + float → float
    • int - float → float
    • int * float → float
    • int / float → float

3.2 decimal

  • decimal は高精度な 10 進数。
  • int / float と混在する式は、必要に応じて decimal に昇格しうる。
  • オーバーフロー / アンダーフロー / 精度問題は OverflowException として扱う。

3.3 enum and int

  • Aloe にはキャスト構文がない。
  • enum と int の暗黙 / 明示変換はいずれも禁止。
  • enum と数値型の混在式はコンパイルエラー（bitfield enum のビット演算を除く）。
  • 内部表現としては enum / bitfield enum はすべて int32。

3.4 byte and char

  • byte：
    • 範囲は 0–255。
    • byte 同士の演算は許されるが、範囲外はエラーになり得る。
    • byte と int の混在式は int に昇格。
  • char：
    • char の算術演算は不可。
    • 比較演算（==, <, > 等）は char 同士のみ。
    • 数値型との暗黙変換はない。

3.5 NaN / Infinity

  • 数値オーバーフロー/アンダーフローは NaN/Infinity ではなく例外で扱う。
  • ZeroDivisionException は 0 / 0.0 で常に投げる。

4. Blocks & Scope（ブロックとスコープ）

ブロックは { と } で区切られ、各ブロックが新しいスコープを導入する。

  • main 関数ブロック
  • クラス定義ブロック
  • メソッド定義ブロック
  • 任意の { ... } ブロック

4.1 Static (lexical) scope and shadowing（静的スコープとシャドウイング）

class Sample {
    field value: int = 1;
    method foo(): void {
        var value = 2;        // フィールドをシャドウ
        print(value);         // 2
        print(this.value);    // 1
    }
}
  • Aloe は静的（レキシカル）スコープ。
  • 内側ブロックで同名変数を宣言すると、外側の変数をシャドウする。
  • フィールド名とローカル変数名がぶつかった場合、ローカル変数が優先される。 フィールドは this. で参照する。

4.2 with-blocks（メンバー省略記法）

Aloe には、同一オブジェクトのメンバーアクセスを簡潔に書くための with ブロックがある。 

with (user) {
    .Name = "Alice";
    .Age = .Age + 1;
    print(.Name);
}
  • with (expr) { ... } ブロック内では：
    • .Member は (expr).Member の省略形。
    • expr はブロックの先頭で 1 度だけ評価され、内部一時変数に保持される。
    • 特別な一時変数 _ は「with 対象オブジェクト」を指す。
      • log(_); のように渡すことができる。
  • ネストした with ブロック：
    • 内側の with が . の解決および _ をシャドウする。
    • 外側オブジェクトへは通常の変数名でアクセスできる。
  • C# の using スタイルのように、同じブロックに複数の with を並べられる：
with (user), (profile) {
    .Name = "Alice";  // user.Name か profile.Name に解決される
}
  • どのオブジェクトのメンバーに解決されるかの順序は実装依存。
  • ただし曖昧な場合（複数オブジェクトが同じメンバーを持つ）はコンパイルエラー。

with はあくまで構文糖衣であり、意味的にはすべて 明示的な obj.Member 呼び出しに書き換えた上で 型チェック・コード生成を行うことができる。

4.2.1 Type-based with（class / struct / enum の static / 列挙子ショートカット）

通常の with (expr) に加えて、型名 を対象にした with を利用できる。 

with (TypeName) {
    .SomeStaticMethod();
    .SomeStaticField = 1;
}

ここで TypeName は次のいずれか：

  • class 型
  • struct 型
  • enum / bitfield enum

この場合、with は「型コンテキスト with」として解釈される：

  • () 内の式は実行時に値として評価されない。
    • コンパイラは TypeName を型として解決し、インスタンス生成などは行わない。
  • ブロック内では：
    • .Member は TypeName.Member の省略形。
      • 対象となる Member は
        • static フィールド
        • static メソッド
        • enum / bitfield enum の列挙子
    • 一時変数 _ は、TypeName の static コンテキストを表す擬似オブジェクトとして振る舞う。
      • _.Member も TypeName.Member と等価。
    • 型ベース with 内でインスタンスメンバーにアクセスすることはコンパイルエラー。

なお、型ベース with でもローカル変数や外側スコープの変数が同名で存在すればそちらが優先される。 列挙子や static メンバーを明示したいときは TypeName.Member または .Member / _.Member を用いる。

例：class / struct の static アクセス
struct AngleUtil {
    static field Pi: float = 3.14159;

    static method DegToRad(deg: float): float {
        return deg * Pi / 180.0;
    }
}

class MathUtil {
    static method Max(a: int, b: int): int {
        if (a > b) return a;
        return b;
    }
}

function main(args: string[]): int {
    with (AngleUtil) {
        var rad = DegToRad(90.0);  // AngleUtil.DegToRad(90.0)
        print(.Pi);                // AngleUtil.Pi
    }

    with (MathUtil) {
        var m = Max(10, 20);       // MathUtil.Max(10, 20)
        print(m);
    }

    return 0;
}
例：enum の列挙子アクセス
enum Color {
    Red,
    Green,
    Blue
}

function main(args: string[]): int {
    with (Color) {
        let c1: Color = Red;   // Color.Red
        let c2: Color = Blue;  // Color.Blue
        print(c1);
        print(c2);
    }
    return 0;
}

型ベース with も複数ターゲット構文に混在させられる： 

with (user), (Color) {
    .Name = "Alice"; // user.Name
    let c: Color = Red; // Color.Red
}

この場合も解決順は実装依存だが、曖昧であればコンパイルエラーとする。

5. Variables & Constants（変数と定数）

5.1 var（型推論）

var i = 10;
var f = 10.5;
var s = "Hello";
var c = 'A';
var b = 255;
var arr = [1, 2, 3];
var lst = (1, 2, 3);
var st  = {1, 2, 3};
var mp  = ["key": 10];
var fb  = new FizzBuzz();
  • 変数の型は右辺の式から推論される。
  • var x = 1; は常に int。
  • var x = 1.0; は float として扱われる。
  • var x = null; はコンパイルエラー（型が推論できない）。

5.2 let（明示的な型）

let i: int = 10;
let f: float = 10.5;
let d: decimal = 10.12345678901234567890:d;
let s: string = "Hello, Aloe!";
let c: char = 'A';
let b: byte = 255;

let arr: int[] = [1, 2, 3];
let lst: int() = (1, 2, 3);
let st : int{} = {1, 2, 3};
let mp : map<string/int> = ["key": 10];

5.3 Constants（const）

const MaxCount: int = 100;
const Pi: float = 3.14159;
const AppName: string = "Aloe VM";
  • const 変数は再代入不可。
  • const には型注釈が必須。

6. Fields（フィールド宣言）

class / struct のフィールドは field キーワードで宣言する。

class Sample {
    field Count: int = 0;
    field Name: string = "default";

    readonly field Id: int;

    construct(id: int) {
        this.Id = id; // OK: コンストラクタ内
    }
    method Change(): void {
        this.Count = this.Count + 1; // OK
        // this.Id = 2; // NG: readonly field はコンストラクタ外で変更不可
    }
}
  • field … 通常のフィールド。
  • readonly field … コンストラクタ内のみ代入可、それ以降は読み取り専用。
  • struct / class の両方で同じ意味。

7. Class / Struct / Interface / Trait

7.1 class

class FizzBuzz {
    field _Count: int = 0;
    field Name: string = "FizzBuzz";

    construct() {
    }

    method Run(): void {
        var i = 1;
        while (i <= 100) {
            // ...
            i = i + 1;
        }
    }
}
  • class は参照型。
  • sealed class は extends による継承不可。
  • 単一継承のみサポート。

7.2 struct

struct Point {
    field X: int;
    field Y: int;

    construct(x: int, y: int) {
        this.X = x;
        this.Y = y;
    }
}
  • struct は値型。
  • フィールドはプリミティブ型か他の struct に限定（参照型フィールドは禁止）。
  • struct のコピーは常に値コピー。
  • struct の継承はサポートしない（sealed struct は「拡張禁止」マーカー）。

7.3 Inheritance and sealed

class Base {
    method Foo(): void { }
}

class Derived extends Base {
    method Bar(): void { }
}

sealed class FinalDerived extends Base {
    // これ以上継承できない
}

7.4 interface

interface ILogger {
    method Log(msg: string): void;
}

class ConsoleLogger implements ILogger {
    method Log(msg: string): void {
        print(msg);
    }
}
  • implements キーワードでインタフェースを実装。
  • is 演算子でインタフェース実装有無を判定。

7.5 trait

trait TraitA {
    method Hello(): void {
        print("Hello from TraitA");
    }
}

with による trait の適用

var obj = new SomeClass() with TraitA;
obj.Hello(); // TraitA で定義されたメソッドが呼ばれる（衝突がなければ）

非破壊な追加 / 削除（+ / -）

var obj2 = obj + TraitA;      // obj は変更されず、obj2 に TraitA が適用される
var obj3 = obj2 - TraitA;     // obj3 は TraitA を取り除いた obj2

as によるエイリアス

var obj2 = obj + TraitA as a;
obj2.a.Hello(); // OK

// obj2.Hello(); // NG: エイリアス経由でのみ呼び出せる

Trait の衝突

  • 複数 trait が同名フィールド / メソッドを定義するとコンパイルエラー。
  • as a などのエイリアス付き適用は衝突判定の対象外。
  • クラス本体と trait メンバとの衝突の扱いは将来バージョンで定義予定。

8. delete & null

var obj = new FizzBuzz();
delete obj;  // VM レベルでは obj = null と等価
  • delete obj; は VM レベルでは obj = null と等価。
  • ソースコード中で obj = null; と書くことはできない。
  • 参照型は内部的には null を取り得るが、型システムは null を追跡しない。
  • null 状態でのアクセスは NullReferenceException を引き起こす可能性がある。

9. is Operator（is 演算子）

if (obj is FizzBuzz) {
    // 実行時型が FizzBuzz またはそのサブクラスであれば true
}

if (obj is ILogger) {
    // ILogger を実装していれば true
}

if (obj is TraitA) {
    // obj に TraitA が適用されていれば true
}
  • is は実行時の型チェックを行う。
  • 継承関係（Derived は Base としても扱う）を考慮する。
  • インタフェースや trait に対しても使用できる。

10. Collections（配列 / リスト / セット / マップ）

10.1 Arrays (T[])

var arr: int[] = [1, 2, 3];
var matrix: float[][] = [[1.0, 2.0], [3.0, 4.0]];

10.2 Lists (T())

var lst: int() = (1, 2, 3);
var objLst: FizzBuzz() = (new FizzBuzz(), new FizzBuzz());

10.3 Sets (T{})

var st: int{} = {1, 2, 3};
var structSt: Point{} = {Point { X:0, Y:0 }, Point { X:1, Y:1 }};

10.4 Maps (map<K/V>)

var mp: map<int> = ["key1": 10, "key2": 20];          // デフォルトのキー型は string
var mp2: map<string/float> = ["key1": 10.5, "key2": 20.5];
var objMp: map<FizzBuzz> = ["obj1": new FizzBuzz(), "obj2": new FizzBuzz()];

11. enum and bitfield enum

11.1 Normal enum

enum Color {
    Red,
    Green,
    Blue
}
  • 内部表現は常に int32。
  • enum<byte>, enum<string> のような基礎型の変更は許可されない。
  • enum 型も型ベース with の対象となり、with (Color) { Red; } などで列挙子に短縮アクセスできる。

11.2 bitfield enum

bitfield enum LogFlags {
    None = 1 << 0,
    Info = 1 << 3,
    Warn,
}

b(n) 記法

bitfield enum LogFlags {
    None : b(0),
    Info : b(3),
    Warn,            // ここまでの最大 b(n) が 3 のため Warn は b(4) = 1 << 4
}
  • b(n) はビット n（1 << n）を表す。
  • Name : b(n) でその列挙子にビット n を割り当てる。
  • bitfield enum で使用可能なビットは最大 32（n は 0–31）。
  • 同一ビットを共有する列挙子があっても、コンパイラは警告にとどめ、エラーにはしない。
  • 基礎型は常に int32 であり、bitfield enum<byte> などはコンパイルエラー。

12. main 関数（エントリポイント）

function main(args: string[]): int {
    var fb = new FizzBuzz();
    fb.Run();
    return 0; // プロセス終了コード
}
  • プログラムのエントリポイント。
  • main は通常の関数と同様に function で宣言し、戻り値は int とする。
  • args はコマンドライン引数。
  • プロセスの終了コードは main の return 値で指定する。

13. Temp Variable _（一時変数）

function main(args: string[]): int {
    var x = 10;
    _ = x + 5;   // このブロック内では _ の型は int になる
    // ...
    return 0;
}

method Foo(): void {
    var x = 10;
    _ = x + 5;   // ここでは _ の型は int
}
  • _ は任意のブロックで宣言なしに使用できる特別な一時変数。
  • 各ブロックで最初に代入された式の型が、そのブロックにおける _ の型になる。
  • main でも他のブロックと同様に扱われる（終了コードは return で指定）。
  • with ブロック内では、追加で「現在の with 対象」を指す意味を持つ（4.2 参照）。

14. swap Keyword（swap キーワード）

var a = 10;
var b = 20;
swap (a, b);
print(a); // 20
print(b); // 10
  • swap (x, y); は x と y の値（または参照）を入れ替える。
  • swap が妥当かどうかはコンパイル時にチェックされる。

15. Operators（演算子）

15.1 優先順位（高い順）

  1. メンバー / 呼び出し：x.y, x[y], x(args...)
  2. 単項：+x, -x, !x, delete x
  3. 乗除：*, /, %
  4. 加減：+, -
  5. シフト：<<, >>
  6. 関係：<, <=, >, >=
  7. 等価：==, !=
  8. ビット AND：&
  9. ビット XOR：^
  10. ビット OR：|（数値 / bitfield enum のとき）
  11. パイプ / パイプ with：|, |>（パイプライン文のとき）
  12. 論理 AND：&&
  13. 論理 OR：||
  14. 代入：=

複合代入（+= 等）はまだ仕様化されていない。

15.2 結合規則

  • 単項演算子：右結合。
  • 代入：右結合。
  • その他の 2 項演算子：左結合。

18. Pipes & Filters（パイプとフィルタ）

Aloe は pipe<T> と filter(...) を用いて、 サーバ / CLI / Web アプリなどのストリーミング処理を構成する。 パイプラインは Railway Oriented Programming (ROP) の考え方を取り入れ、 正常経路とエラー経路を明示的に分岐させる。

18.1 pipe<T> の基本

  • pipe<T> は T 型要素のストリーム。
  • pipe<byte> … 標準入力やソケットなどのバイト列。
  • pipe<string> … 文字列ストリーム。
  • pipe は共有メモリではなく値コピーを前提とする。

18.1.1 組み込みパイプ

  • stdin : pipe<byte>
  • stdout : pipe<byte>
  • stderr : pipe<byte>（任意）

18.2 Result<T> と ROP の概要

パイプラインとフィルタは、成功 / 失敗を Result<T> 型で表現する。 これにより、エラーを例外だけでなく「もう一つの線路」として扱う Railway Oriented Programming (ROP) を実現する。 

struct Result<T> {
    field IsOk: bool;
    field Value: T;      // IsOk == true のときだけ有効
    field Error: string; // IsOk == false のときエラーメッセージ
}

概念図（左から右に流れる二本の線路）：

[入力] ──> [ステージ1] ──> [ステージ2] ──> [ステージ3] ──> (正常完了)
   │
   └───────────────> [エラーハンドラ] ──> (終了)
  • 各フィルタは Result<T> を返し、成功 / 失敗を明示する。
  • 成功時は次のステージへ、失敗時はエラー線路へ分岐する。
  • パイプライン文の match により、成功 / 失敗ごとの処理を分けられる（18.8 参照）。

18.3 filter(...) の基本

filter(F) はパイプライン中の中間変換ステージ。

18.3.1 フィルタ定義

filter LineSplit {
    in:  pipe<string>;
    out: pipe<string>;

    // フィルタごとのオプション型（任意）
    options: LineSplitOptions;

    // 状態を持つフィルタ用のフィールド（任意だが、ROP 的には最小限を推奨）
    field _Buffer: string;

    construct() {
        _Buffer = "";
    }

    bound(input: pipe<string>, output: pipe<string>): Result<void> {
        foreach (chunk in input) {
            let lines = chunk.Split("\n");
            foreach (line in lines) {
                output.Write(line);
            }
        }
        output.Close();
        return Result<void> { IsOk: true };
    }
}
  • in / out で入出力パイプ型を指定する。
  • options で、このフィルタに対応するオプションクラス（bitfield enum 等を持つ型）を指定できる。
  • field でフィルタ内部の状態を定義できるが、 ROP（Railway Oriented Programming）的にはフィルタはできる限り冪等であることが望ましいため、 mutable なフィールドを持ちすぎないことが推奨される。
  • construct でフィルタインスタンスの初期化を行う。
  • bound(...): Result<void> は、入力 / 出力パイプが束縛された状態で呼ばれる本体。 正常終了なら IsOk = true の Result を返す。

18.3.2 フィルタオプション

filter Json<T> {
    in:  pipe<string>;
    out: pipe<T>;

    options: JsonOptions;  // オプションクラス

    bound(input: pipe<string>, output: pipe<T>): Result<void> {
        // ...
        return Result<void> { IsOk: true };
    }
}

bitfield enum JsonOptions {
    None        : b(0),
    PrettyPrint : b(1),
    IncludeNull : b(2),
}

パイプライン文での指定：

stdin
    | filter(utf8)
    | filter(Json<User> : JsonOptions.PrettyPrint | JsonOptions.IncludeNull)
    | users;
  • filter(Name : OptionA | OptionB) のように、: の後ろにオプションを列挙する。
  • オプションクラスを options で指定している場合、 パイプライン側ではクラス名を省略して filter(Json<User> : PrettyPrint | IncludeNull) と書くこともできる。

18.4 パイプライン文と | / パイプ with 演算子 |>

stdin
    | filter(utf8)
    | filter(Json<DateTime>)
    | dates;
  • A | B | C; で A の出力が B へ、B の出力が C へ接続される。
  • 型チェックにより各ステージの入出力型整合性を確認する。
  • パイプライン文そのものは値を返さない（副作用として接続を張るだけ）。

18.4.1 パイプ with 演算子 |>

|> は「条件付きフィルタ」のショートカットとして定義される。 

users
    |> .Age >= 20 and .Gender == Gender.Male
    | filter(Json<User>);

これは次の書き方の糖衣構文である： 

users
    | filter(if : .Age >= 20 and .Gender == Gender.Male)
    | filter(Json<User>);
  • filter(if : <条件式>) は、入力要素ごとに条件式を評価し、 true の要素だけを次のステージに流す組み込みフィルタを表す。
  • X |> 条件 は常に X | filter(if : 条件) に展開可能である。
  • 条件式内の . は「現在の要素」を指す擬似オブジェクトとして解釈される。

18.5 foreach と pipe

foreach (line in lines) {
    print("Input: " + line);
}
  • foreach (x in p) は内部的に p.Next() を EOF まで呼ぶ。
  • 書き込み側が Close() を呼ぶと読み側は EOF を検知してループを終了する。

18.6 関数適用 map（pipe 用）

パイプ上の各要素に関数を適用して別の型のパイプを生成するために、 組み込みの map フィルタを用意する。 

// Users: pipe<User> から Names: pipe<string> を作る例
Users
    | map(u => u.Name)
    | Names;
  • map(f) は pipe<T> -> pipe<U> 型のフィルタとして扱われる。
  • ラムダ式や with と組み合わせて、簡潔に変換処理を書ける。

18.7 組み込み関数（ドラフト）

パイプラインで頻出する関数 / フィルタの例（暫定）：

  • map(f) – 各要素を変換する。
  • filter(if : 条件) – 条件に合致する要素だけを通す。
  • tap(f) – 各要素に対して副作用だけを実行し、そのまま要素を流す（ログなど）。
  • take(n) – 先頭 n 要素だけを取り出す。
  • skip(n) – 先頭 n 要素をスキップする。

これらは標準フィルタライブラリ側で正式に定義される予定であり、本仕様ではシグネチャのみを示す。

18.8 match によるパイプ分岐（Result<T> の分岐）

Result<T> を返すパイプラインの結果に応じて処理を分岐するために、 パイプライン内で match 文を使用できる。 

Results
    | match {
        case .IsOk:
            // 成功経路用のパイプライン
            | map(r => r.Value)
            | stdout;
        case not .IsOk:
            // 失敗経路用のパイプライン
            | map(r => r.Error)
            | stderr;
        default:
            // その他（必要なら）
            | stdout;
    }
  • | match { ... } は直前のパイプを入力として受け取り、case ごとに別のパイプラインを定義する。
  • case 条件式: では、Result<T> の各要素に対して条件式を評価し、 true になった要素をその後続パイプに流す。
  • default: はどの case にもマッチしなかった要素を受け取る。

18.9 コレクション用 map 構文（テンプレート）

コレクションパイプ（pipe<T>）から、テンプレート的に別のコレクションを作成する構文を用意する。 

Users as u
    | map Customer {
        Name    = u.Name,
        Address = u.Address
    }
    | Customers;
  • Users as u で、要素を u という名前で参照する。
  • map 型名 { ... } で、各要素ごとに新しいオブジェクトを構築する。
  • 右辺では u のプロパティを参照して、新しいコレクション要素を定義できる。

18.10 例：簡単な REST 風パイプライン

class User {
    field Id: int;
    field Name: string;
}

function main(args: string[]): int {
    let UsersPipe: pipe<User> = pipe<User>.Create();

    stdin
        | filter(utf8)
        | filter(Json<User>)
        | UsersPipe;

    let u: User? = UsersPipe.Next();
    if (u is User) {
        print("User: " + u.Name);
    }

    return 0;
}

utf8 や Json<T> などの標準フィルタの詳細は、 コア言語仕様ではなく別途「標準フィルタライブラリ」仕様で定義される。

以上が、元の英語仕様に基づき、 with ブロックの型対応拡張や Result 型 / パイプライン拡張を加えた Aloe 言語仕様（日本語・HTML 版） です。
