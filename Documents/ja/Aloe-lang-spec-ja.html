この文書は、英語版仕様 https://stk2k.github.io/Aloe-lang/Documents/en/Aloe-lang-spec-en.html をベースにした日本語ドラフトです。
その上で 4.2 with-blocks に class / struct / enum を対象にした型ベース with（static メンバー／列挙子への短縮アクセス） の拡張を追加しています。
それ以外の仕様は原文にできるだけ忠実に訳しています。

0. Overview（概要）

Aloe は静的型付けのスクリプト言語であり、シンプルで直感的な構文を目指している。
スタックベースの仮想マシン（AloeVM）上で動作し、主に次のような目標を持つ。

  • 教育用途や小規模アプリケーションで読み書きしやすいこと
  • C# ランタイムおよび WebAssembly (WASM) 環境で動作すること
  • pipe と filter によって、I/O と並行処理を静的型付けのまま制御できること

特徴：

  • 静的型付け
    • var による型推論
    • let による明示的な型指定
  • 値型 / 参照型
    • struct … 値型
    • class … 参照型
  • GC ベースのメモリ管理（具体的なアルゴリズムは VM 実装依存）
  • = null をソースコード中に直接書かない
    • 参照型は内部的には null を取り得るが、delete による糖衣構文で扱う。
  • pipe / filter によるデータフロースタイルの設計

1. Basic Syntax（基本文法）

1.1 Comments（コメント）

// single-line comment

/* multi-line
   comment */
  • // から行末までが行コメント。
  • /* ～ */ が複数行コメント。
  • ネストした /* */ は未定義（サポートしない想定）。

1.2 Identifiers（識別子）

  • 使用可能文字：英字 (A–Z, a–z)、数字 (0–9)、アンダースコア _
  • 先頭文字は英字または _ でなければならない。
  • 大文字小文字は区別される。
  • キーワードと同じ綴りは使用できない。
  • _ で始まる識別子は「プライベート」扱い（外部からは参照しない慣習）。
  • ユニコード文字も使用できる（空白や区切り記号などは除外）。

  • package は使わず、namespace を使用する。

1.4 Literals（リテラル）

  • 整数リテラル：10, 0, -5, 0x1A, 0b1010 など
  • 浮動小数点リテラル：10.5, 0.0, -3.14
  • 10進リテラル：10.12345678901234567890:d など
  • 真偽値：true, false
  • 文字列："Hello, World!"
  • 文字：'A', 'z' など
  • null リテラル：null（リテラルとして存在するが、直接代入には使わない）
  • 配列リテラル：[1, 2, 3]
  • リストリテラル：(1, 2, 3)
  • セットリテラル：{1, 2, 3}
  • マップリテラル：["key1": 10, "key2": 20]
  • enum リテラル：EnumType.Value1
  • struct リテラル：Point { x: 0, y: 1 }
  • オブジェクト生成：new ClassName()

1.5 Statement terminator（文の終端）

var x = 10;
x = x + 1;
print(x);
  • すべての文はセミコロン ; で終わる。
  • 改行による自動セミコロン挿入は行わない。

1.6 Whitespace（空白）

  • スペース、タブ、改行はトークン区切りとして扱われる。
  • 連続する空白は 1 つの区切りとみなされる。
  • インデントに意味はない（Python とは異なる）。

1.7 Case sensitivity（大文字小文字）

  • 識別子とキーワードは大文字小文字を区別する。
  • キーワードはすべて小文字。

2. Type System（型システム）

2.1 Primitive types（プリミティブ型）

  • int / integer – 整数
  • float – 浮動小数点数
  • bool / boolean – 真偽値
  • decimal – 高精度 10 進数
  • string – 文字列（内部的には Unicode）
  • char – 1 文字
  • byte – 0–255 の整数
  • void – 戻り値なし（メソッド戻り値専用）
  • null – 特別なリテラル（型ではない）

any 型は存在せず、動的型付けはサポートしない。

2.2 Composite / user-defined types（合成型 / ユーザー定義型）

  • 配列：T[]
  • リスト：T()
  • セット：T{}
  • マップ：map<T> または map<K/V>
  • struct
  • class
  • enum, bitfield enum
  • 非同期タスク：task
  • パイプ：pipe<T>（18章参照）

2.3 Value types / reference types（値型 / 参照型）

  • 値型：
    • int, float, decimal, bool, byte, char
    • struct
    • enum, bitfield enum（内部表現は int32）
  • 参照型：
    • class
    • string（実装上は参照型）
    • task, pipe<T> など

Struct restrictions and copying（struct の制約とコピー）

  • struct のフィールドにはプリミティブ型と他の struct のみを持てる。
  • class などの参照型フィールドは持てない。
  • struct の代入は常に値コピー。
  • 最大サイズなどは仕様で制限しない。

Class vs struct（class と struct）

  • class は struct をフィールドとして持てる。
  • class インスタンスを別変数に代入すると参照がコピーされる（オブジェクト本体は共有）。

3. Numbers & Conversions（数値と暗黙変換）

3.1 int and float

  • int と float を混在させた式の結果は float になる。
    • int + float → float
    • int - float → float
    • int * float → float
    • int / float → float

3.2 decimal

  • decimal は高精度な 10 進数。
  • int / float と混在する式は、必要に応じて decimal に昇格しうる。
  • オーバーフロー / アンダーフロー / 精度問題は OverflowException として扱う。

3.3 enum and int

  • Aloe にはキャスト構文がない。
  • enum と int の暗黙 / 明示変換はいずれも禁止。
  • enum と数値型の混在式はコンパイルエラー（bitfield enum のビット演算を除く）。
  • 内部表現としては enum / bitfield enum はすべて int32。

3.4 byte and char

  • byte：
    • 範囲は 0–255。
    • byte 同士の演算は許されるが、範囲外はエラーになり得る。
    • byte と int の混在式は int に昇格。
  • char：
    • char の算術演算は不可。
    • 比較演算（==, <, > 等）は char 同士のみ。
    • 数値型との暗黙変換はない。

3.5 NaN / Infinity

  • 数値オーバーフロー/アンダーフローは NaN/Infinity ではなく例外で扱う。
  • ZeroDivisionException は 0 / 0.0 で常に投げる。

4. Blocks & Scope（ブロックとスコープ）

ブロックは { と } で区切られ、各ブロックが新しいスコープを導入する。

  • main ブロック
  • クラス定義ブロック
  • メソッド定義ブロック
  • 任意の { ... } ブロック

4.1 Static (lexical) scope and shadowing（静的スコープとシャドウイング）

class Sample {
    field value: int = 1;
    method foo(): void {
        var value = 2;        // shadows the field
        print(value);         // 2
        print(this.value);    // 1
    }
}
  • Aloe は静的（レキシカル）スコープ。
  • 内側ブロックで同名変数を宣言すると、外側の変数をシャドウする。
  • フィールド名とローカル変数名がぶつかった場合、ローカル変数が優先される。 フィールドは this. で参照する。

4.2 with-blocks（メンバー省略記法）

Aloe には、同一オブジェクトのメンバーアクセスを簡潔に書くための with ブロックがある。 

with (user) {
    name = "Alice";
    age = age + 1;
    print(name);
}
  • with (expr1) または with (expr1), (expr2), ... という形で指定する。
  • 各 exprN はブロックの先頭で 1 度だけ評価され、内部一時変数として保持される。
    • 以降のメンバーアクセスでは、その内部一時変数に対してアクセスが行われる。
  • with ブロック内の「単純な識別子」name の解決順は次の通り：
    1. そのブロック内で宣言されたローカル変数 / 引数
    2. with 対象 のメンバー
      • 左から右の順で expr1, expr2, ... それぞれの型の フィールド / プロパティ / メソッドを探索する。
      • 複数の with 対象が同じ名前のメンバーを持つ場合はコンパイルエラー（曖昧）。
    3. 外側のスコープ（外側のブロック、クラスのフィールド、import されたシンボルなど）
  • 特別な一時変数 _ は「最初の with 対象（expr1）」を指す。
    • log(_); のように渡すことができる。
    • 複数ターゲットのときも、_ は常に expr1 に対応する。

4.2.1 複数ターゲット with

with (user), (profile) {
    name = "Alice";        // user.name または profile.name に解決を試みる
    updateProfile();       // user.updateProfile() または profile.updateProfile()
}
  • 複数ターゲット with では、2 番目以降のターゲットもメンバー解決対象になる。
  • 同名メンバーを複数ターゲットが持っている場合はコンパイルエラー。
  • 一方、ローカル変数や外側スコープに同名シンボルが存在する場合は、それらが優先される。

4.2.2 ネストした with

with (user) {
    name = "Alice";

    with (address) {
        city = "Tokyo";     // address.city
        print(name);        // ここでは user.name かローカル name か、通常の解決順に従う
    }
}
  • 内側の with は独立した解決規則を持つ。
  • 内側で解決できない識別子は、外側 with を含む外側スコープへとフォールバックする。
  • 内側の with で同名メンバーが現れると、外側の with メンバーをシャドウする形になる。

with はあくまで構文糖衣であり、意味的には 「ローカル変数／パラメータ → with 対象 → 外側スコープ」 という名前解決ルールを追加した上で、 明示的な obj.member 呼び出しに近い形へと解釈できる。

4.2.3 Type-based with（class / struct / enum の static / 列挙子ショートカット）

通常の with (expr) に加えて、型名 を対象にした with を利用できる。 

with (TypeName) {
    SomeStaticMethod();
    SomeStaticField = 1;
}

ここで TypeName は次のいずれか：

  • class 型
  • struct 型
  • enum / bitfield enum

この場合、with は「型コンテキスト with」として解釈される：

  • () 内の識別子は式としてではなく「型」として解決される。
    • 実行時評価やインスタンス生成は行わない。
  • with ブロック内での単純識別子の解決順は、通常の with と同様に
    1. ローカル変数 / 引数
    2. with 対象の「型コンテキスト」のメンバー
      • static フィールド
      • static メソッド
      • enum / bitfield enum の列挙子
    3. 外側のスコープ
  • 特別な一時変数 _ は、「最初の型ターゲットの static コンテキスト」を表す擬似オブジェクトとして扱われる。
    • _.Member と書いてもよいが、Member と等価。
  • 型ベース with 内でインスタンスメンバーにアクセスすることはコンパイルエラー。

型ベース with は、通常の値ベース with と混在させることもできる。 その場合も、すべてのターゲットを合わせて「メンバー解決候補」として扱い、 複数のターゲットに同名メンバーが存在する場合は曖昧としてコンパイルエラーになる。

例：class / struct の static アクセス
struct AngleUtil {
    static field Pi: float = 3.14159;

    static method degToRad(deg: float): float {
        return deg * Pi / 180.0;
    }
}

class MathUtil {
    static method max(a: int, b: int): int {
        if (a > b) return a;
        return b;
    }
}

main(args: string[]) {
    with (AngleUtil) {
        var rad  = degToRad(90.0);  // AngleUtil.degToRad(90.0)
        var rad2 = degToRad(45.0);
        print(Pi);                  // AngleUtil.Pi
    }

    with (MathUtil) {
        var m = max(10, 20);        // MathUtil.max(10, 20)
        print(m);
    }
}
例：enum の列挙子アクセス
enum Color {
    Red,
    Green,
    Blue
}

main(args: string[]) {
    with (Color) {
        var c1: Color = Red;   // Color.Red
        var c2: Color = Blue;  // Color.Blue
        print(c1);
        print(c2);
    }
}
例：値ターゲットと型ターゲットの混在
with (user), (Color) {
    name = "Alice";      // user.name （他ターゲットに name があると曖昧でエラー）
    var c: Color = Red;  // Color.Red
}

この場合も、値ターゲットと型ターゲットをあわせてメンバー解決を行い、 同名候補が複数ある場合はコンパイルエラーとする。

5. Variables & Constants（変数と定数）

5.1 var（型推論）

var i = 10;
var f = 10.5;
var s = "Hello";
var c = 'A';
var b = 255;
var arr = [1, 2, 3];
var lst = (1, 2, 3);
var st  = {1, 2, 3};
var mp  = ["key": 10];
var fb  = new FizzBuzz();
  • 変数の型は右辺の式から推論される。
  • var x = 1; は常に int。
  • var x = 1.0; は float として扱われる。
  • var x = null; はコンパイルエラー（型が推論できない）。

5.2 let（明示的な型）

let i: int = 10;
let f: float = 10.5;
let d: decimal = 10.12345678901234567890:d;
let s: string = "Hello, Aloe!";
let c: char = 'A';
let b: byte = 255;

let arr: int[] = [1, 2, 3];
let lst: int() = (1, 2, 3);
let st : int{} = {1, 2, 3};
let mp : map<string/int> = ["key": 10];

5.3 Constants（const）

const MaxCount: int = 100;
const Pi: float = 3.14159;
const AppName: string = "Aloe VM";
  • const 変数は再代入不可。
  • const には型注釈が必須。

6. Fields（フィールド宣言）

class / struct のフィールドは field キーワードで宣言する。

class Sample {
    field count: int = 0;
    field name: string = "default";

    readonly field id: int;

    construct(id: int) {
        this.id = id; // OK: inside the constructor
    }
    method change(): void {
        this.count = this.count + 1; // OK
        // this.id = 2; // NG: readonly field cannot be changed outside constructor
    }
}
  • field … 通常のフィールド。
  • readonly field … コンストラクタ内のみ代入可、それ以降は読み取り専用。
  • struct / class の両方で同じ意味。

7. Class / Struct / Interface / Trait

7.1 class

class FizzBuzz {
    field _count: int = 0;
    field name: string = "FizzBuzz";

    construct() {
    }

    method run(): void {
        var i = 1;
        while (i <= 100) {
            // ...
            i = i + 1;
        }
    }
}
  • class は参照型。
  • sealed class は extends による継承不可。
  • 単一継承のみサポート。

7.2 struct

struct Point {
    field x: int;
    field y: int;

    construct(x: int, y: int) {
        this.x = x;
        this.y = y;
    }
}
  • struct は値型。
  • フィールドはプリミティブ型か他の struct に限定（参照型フィールドは禁止）。
  • struct のコピーは常に値コピー。
  • struct の継承はサポートしない（sealed struct は「拡張禁止」マーカー）。

7.3 Inheritance and sealed

class Base {
    method foo(): void { }
}

class Derived extends Base {
    method bar(): void { }
}

sealed class FinalDerived extends Base {
    // cannot be extended further
}

7.4 interface

interface ILogger {
    method log(msg: string): void;
}

class ConsoleLogger implements ILogger {
    method log(msg: string): void {
        print(msg);
    }
}
  • implements キーワードでインタフェースを実装。
  • is 演算子でインタフェース実装有無を判定。

7.5 trait

trait Trait_A {
    method hello(): void {
        print("Hello from Trait_A");
    }
}

Applying traits with with

var obj = new SomeClass() with Trait_A;
obj.hello(); // method from Trait_A (if no conflicts)

Non-destructive add/remove（+ / -）

var obj2 = obj + Trait_A;      // obj unchanged; obj2 has Trait_A
var obj3 = obj2 - Trait_A;     // obj3 is obj2 without Trait_A

Alias with as

var obj2 = obj + Trait_A as a;
obj2.a.hello(); // OK

// obj2.hello(); // NG: alias only; no direct member flattening

Trait conflicts

  • 複数 trait が同名フィールド / メソッドを定義するとコンパイルエラー。
  • as a などのエイリアス付き適用は衝突判定の対象外。
  • クラス本体と trait メンバとの衝突の扱いは将来バージョンで定義予定。

8. delete & null

var obj = new FizzBuzz();
delete obj;  // equivalent to: obj = null (at VM level)
  • delete obj; は VM レベルでは obj = null と等価。
  • ソースコード中で obj = null; と書くことはできない。
  • 参照型は内部的には null を取り得るが、型システムは null を追跡しない。
  • null 状態でのアクセスは NullReferenceException を引き起こす可能性がある。

9. is Operator（is 演算子）

if (obj is FizzBuzz) {
    // true if runtime type is FizzBuzz or a subclass
}

if (obj is ILogger) {
    // true if obj implements ILogger
}

if (obj is Trait_A) {
    // true if obj has Trait_A applied (with / +)
}
  • is は実行時の型チェックを行う。
  • 継承関係（Derived は Base としても扱う）を考慮する。
  • インタフェースや trait に対しても使用できる。

10. Collections（配列 / リスト / セット / マップ）

10.1 Arrays (T[])

var arr: int[] = [1, 2, 3];
var matrix: float[][] = [[1.0, 2.0], [3.0, 4.0]];

10.2 Lists (T())

var lst: int() = (1, 2, 3);
var objLst: FizzBuzz() = (new FizzBuzz(), new FizzBuzz());

10.3 Sets (T{})

var st: int{} = {1, 2, 3};
var structSt: Point{} = {Point { x:0, y:0 }, Point { x:1, y:1 }};

10.4 Maps (map<K/V>)

var mp: map<int> = ["key1": 10, "key2": 20];          // default key type is string
var mp2: map<string/float> = ["key1": 10.5, "key2": 20.5];
var objMp: map<FizzBuzz> = ["obj1": new FizzBuzz(), "obj2": new FizzBuzz()];

11. enum and bitfield enum

11.1 Normal enum

enum Color {
    Red,
    Green,
    Blue
}
  • 内部表現は常に int32。
  • enum<byte>, enum<string> のような基礎型の変更は許可されない。
  • enum 型も型ベース with の対象となり、with (Color) { Red; } などで列挙子に短縮アクセスできる。

11.2 bitfield enum

bitfield enum LogFlags {
    None = 1 << 0,
    Info = 1 << 3,
    Warn,
}

b(n) 記法

bitfield enum LogFlags {
    None : b(0),
    Info : b(3),
    Warn,            // highest b(n) so far is 3; Warn becomes b(4) = 1 << 4
}
  • b(n) はビット n（1 << n）を表す。
  • Name : b(n) でその列挙子にビット n を割り当てる。
  • bitfield enum で使用可能なビットは最大 32（n は 0–31）。
  • 同一ビットを共有する列挙子があっても、コンパイラは警告にとどめ、エラーにはしない。
  • 基礎型は常に int32 であり、bitfield enum<byte> などはコンパイルエラー。

12. main Block（エントリポイント）

main(args: string[]) {
    var fb = new FizzBuzz();
    fb.run();
    _ = 0; // process exit code (int)
}
  • プログラムのエントリポイント。
  • args はコマンドライン引数。
  • main ブロックでは、一時変数 _ は常に int 型であり、その値がプロセスの終了コードとなる。

13. Temp Variable _（一時変数）

main(args: string[]) {
    var x = 10;
    _ = x + 5;   // _ is int (exit code)
}

method foo(): void {
    var x = 10;
    _ = x + 5;   // type of _ is inferred in this block
}
  • _ は任意のブロックで宣言なしに使用できる特別な一時変数。
  • main では常に int。
  • その他のブロックでは、そのブロックで最初に代入された式の型が _ の型になる。
  • 4.2 で述べたように、単一ターゲット with ブロック内では、 補助的に「with 対象オブジェクト」を指す概念としても利用できるが、 実装上は単に構文糖衣として扱ってよい。

14. swap Keyword（swap キーワード）

var a = 10;
var b = 20;
swap (a, b);
print(a); // 20
print(b); // 10
  • swap (x, y); は x と y の値（または参照）を入れ替える。
  • swap が妥当かどうかはコンパイル時にチェックされる。

15. Operators（演算子）

15.1 優先順位（高い順）

  1. メンバー / 呼び出し：x.y, x[y], x(args...)
  2. 単項：+x, -x, !x, delete x
  3. 乗除：*, /, %
  4. 加減：+, -
  5. シフト：<<, >>
  6. 関係：<, <=, >, >=
  7. 等価：==, !=
  8. ビット AND：&
  9. ビット XOR：^
  10. ビット OR：|（pipeline 文では別語義）
  11. 論理 AND：&&
  12. 論理 OR：||
  13. 代入：=

複合代入（+= 等）はまだ仕様化されていない。

15.2 結合規則

  • 単項演算子：右結合。
  • 代入：右結合。
  • その他の 2 項演算子：左結合。

16. Exceptions（例外）

try {
    // code that might throw
}
catch (e) {
    // handle exception
}
finally {
    // cleanup
}

16.1 標準例外の概要

Exception
├── SystemException
│   ├── NullReferenceException
│   ├── IndexOutOfBoundsException
│   ├── OverflowException
│   ├── ZeroDivisionException
│   ├── InvalidOperationException
│   ├── NotImplementedException
│   ├── TimeoutException
│   ├── ArgumentException
│   │   ├── ArgumentNullException
│   │   └── ArgumentOutOfRangeException
│   └── FormatException

    ├── FileNotFoundException

  • OverflowException はオーバーフロー / アンダーフロー両方に用いる。
  • ZeroDivisionException は 0 / 0.0 による除算で投げる。
  • パースやフォーマットエラーは FormatException。
  • その他の VM レベルの致命的エラーは実装依存。

17. Modules / namespace / import

namespace Aloe.Sample {
    import Aloe.Core.Logging;

    class App {
        // ...
    }
}
  • namespace は論理的な名前空間を定義する。
  • import は他の namespace / シンボルへの参照を導入する。
  • ファイル構成や循環 import の扱いなど詳細はモジュール仕様で定義される。

18. Pipes & Filters（パイプとフィルタ）

Aloe は pipe<T> と filter(...) を用いて、 サーバ / CLI / Web アプリなどのストリーミング処理を構成する。

18.1 pipe<T> の基本

  • pipe<T> は T 型要素のストリーム。
  • pipe<byte> … 標準入力やソケットなどのバイト列。
  • pipe<string> … 文字列ストリーム。
  • pipe は共有メモリではなく値コピーを前提とする。

18.1.1 組み込みパイプ

  • stdin : pipe<byte>
  • stdout : pipe<byte>
  • stderr : pipe<byte>（任意）

18.2 filter(...) の基本

filter(F) はパイプライン中の中間変換ステージ。

18.2.1 フィルタ定義

filter lineSplit {
    in:  pipe<string>;
    out: pipe<string>;

    bound(input, output) {
        foreach (chunk in input) {
            let lines = chunk.split("\n");
            foreach (line in lines) {
                output.write(line);
            }
        }
        output.close();
    }
}

18.2.2 フィルタオプション

filter(json<User>: JsonOptions.PrettyPrint | JsonOptions.IncludeNulls)
  • オプションは : の後に | でつなげて指定。
  • 各フィルタは自前のオプション型（bitfield enum 等）を持つ。

18.3 パイプライン文と |

stdin
    | filter(utf8)
    | filter(json<DateTime>)
    | dates;
  • A | B | C; で A の出力が B へ、B の出力が C へ接続される。
  • 型チェックにより各ステージの入出力型整合性を確認。
  • パイプライン文そのものは値を返さない（副作用として接続を張るだけ）。

18.4 foreach と pipe

foreach (line in lines) {
    print("Input: " + line);
}
  • foreach (x in p) は内部的に p.next() を EOF まで呼ぶ。
  • 書き込み側が close() を呼ぶと読み側は EOF を検知してループを終了する。

18.5 close と EOF 伝播

  • p.close() 後の write() は無効（エラーになり得る）。
  • フィルタは入力 EOF を検知したら出力側も close() すべき。

18.6 pipe を使った並行モデル

  • タスク同士の主な通信手段は pipe<T>。
  • shared-nothing モデルを前提とし、共有可変データ構造は持たない設計を推奨。

18.7 例：簡単な REST 風パイプライン

class User {
    field id: int;
    field name: string;
}

main(args: string[]) {
    let req: pipe<byte> = pipe<byte>.create();
    let users: pipe<User> = pipe<User>.create();

    stdin
        | filter(utf8)
        | filter(json<User>)
        | users;

    let u: User? = users.next();
    if (u is User) {
        print("User: " + u.name);
    }

    _ = 0;
}

utf8 や json<T> などの標準フィルタの詳細は、 コア言語仕様ではなく別途「標準フィルタライブラリ」仕様で定義される。

以上が、元の英語仕様に基づき、 with ブロックの「ドットなし」拡張（値 / 型ターゲット両対応）を加えた Aloe 言語仕様（日本語・HTML 版） です。
